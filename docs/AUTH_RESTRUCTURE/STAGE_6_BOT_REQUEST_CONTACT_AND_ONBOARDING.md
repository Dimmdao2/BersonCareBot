# Stage 6 — Привязка номера через бот: `request_contact` и приветствие

Сводка в общем плане: `MASTER_PLAN.md` -> Stage 6.

## Актуализация (после единого гейта, 2026-04)

Исходный scope Stage 6 (приветствие + `/start`) **дополнен** в коде: **основной** контроль — в integrator (**прод:** `buildPlan` + `scripts.json` при `linkedPhone: false` + центральный гейт колбэков; см. `resolver.ts` и §8 `SCENARIOS_AND_CODE_MAP.md`); `handleUpdate` / `handleMessage` — вне webhook-пути. **Страховка** — `MiniAppShareContactGate`, `POST /api/patient/messenger/request-contact`, tier **patient** в `/api/me`. Журнал и ссылки: [`BOT_CONTACT_MINI_APP_GATE.md`](./BOT_CONTACT_MINI_APP_GATE.md), [`../PLATFORM_IDENTITY_ACCESS/AGENT_EXECUTION_LOG.md`](../PLATFORM_IDENTITY_ACCESS/AGENT_EXECUTION_LOG.md) (блок «Единый гейт контакта»).

## Цель этапа

При первом `/start`, если номер телефона еще не привязан, бот всегда отправляет приветственное сообщение (канонический текст ниже) и сразу показывает клавиатуру `request_contact`. Номер сохраняется в integrator и проецируется в webapp: обновляется `platform_users.phone_normalized`, а для соответствующего мессенджера поддерживается `user_channel_bindings`.

## Scope (только Stage 6)

- Поведение Telegram- и Max-бота на `/start` при отсутствии номера.
- Канонический onboarding-текст и эмодзи.
- Немедленный запрос контакта (`request_contact`) в том же ходе.
- Идемпотентность: после успешной привязки onboarding не повторяется.

Не включать:

- Публичный web login (`AuthFlowV2`), SMS/PIN и OAuth.

## Предусловия

- Рабочий `request_contact` path в integrator.
- Понимание текущего `handleStart` поведения.
- Для Max: подтверждение, что аналог `request_contact` поддержан, либо документированный workaround.

## Подробный план реализации

### S6.T01 — Аудит текущего flow

1. Проверить все ветки `/start` в Telegram/Max.
2. Зафиксировать, где `request_contact` уже вызывается и где нет.
3. Определить минимальную точку входа для универсального onboarding.
4. Обновить `AGENT_EXECUTION_LOG.md`.

### S6.T02 — Обновить `handleStart`

Если `!hasLinkedPhone`:

1. Отправить приветствие по канону (см. S6.T06).
2. Сразу отправить кнопку/клавиатуру `request_contact`.
3. После успешной привязки показать главное меню.
4. Обновить `AGENT_EXECUTION_LOG.md`.

### S6.T03 — Проекция в webapp

1. После сохранения номера в `channel_users` отправлять событие про связку (`phone.linked` или текущий channel-link контракт).
2. Проверить, что номер появляется в `platform_users`, а привязка Telegram/Max отражается в `user_channel_bindings`.
3. Добавить/обновить тест или проверку интеграции.
4. Обновить `AGENT_EXECUTION_LOG.md`.

### S6.T04 — Max-ветка

1. Применить аналогичное поведение для Max.
2. Если API не дает request_contact:
   - зафиксировать ограничение,
   - внедрить безопасный fallback,
   - отразить это в логе и stage-доке.
3. Обновить `AGENT_EXECUTION_LOG.md`.

### S6.T05 — Тесты

Минимум:

1. `/start` без номера -> приветствие -> сразу запрос контакта -> пользователь делится контактом -> номер сохранен -> главное меню.
2. Повторный `/start` после привязки -> onboarding не повторяется, контакт не запрашивается повторно.
3. Для Max: аналогичный тест или зафиксированный fallback behavior.
4. Прогнать релевантные тесты + `pnpm run ci`.
5. Зафиксировать checks/evidence в `AGENT_EXECUTION_LOG.md`.

### S6.T06 — Канонический текст и эмодзи

Правила:

- После "Привет!" ставить 👋.
- Список возможностей маркировать ✅.
- Абзац про обязательность номера начинать с ❗.
- Не использовать литералы `(!)` вместо эмодзи.

## Два слоя контроля (бот + Mini App)

- **Бот (основной):** без привязанного номера не выдаётся полноценное меню и действия с WebApp — везде, где это обрабатывается в домене, уходит `requestPhoneLink` / сценарные правила `linkedPhone: false` (см. `handleUpdate`, `handleMessage`, `requestContactFlow`, `scripts.json`).
- **Mini App (страховка):** если WebApp уже открыт, а tier пациента ещё не `patient`, гейт в layout и `POST …/messenger/request-contact` дотаскивают пользователя до чата с кнопкой контакта. Подробнее: [`BOT_CONTACT_MINI_APP_GATE.md`](./BOT_CONTACT_MINI_APP_GATE.md).

## Webapp: Mini App до контакта

Реализован гейт `MiniAppShareContactGate` в layout пациента (`app/app/patient/layout.tsx`): при открытии приложения из мессенджера, если в сессии есть привязка Telegram/Max, но ещё нет tier **patient** в webapp, показывается экран с опросом `/api/me`, кнопкой запроса контакта в чат и при необходимости ссылкой на бота. Подробнее: `BOT_CONTACT_MINI_APP_GATE.md`.

## Gate (критерий готовности)

Первый `/start` без номера -> приветствие по S6.T06 -> сразу `request_contact` -> после привязки номер и messenger binding доступны в webapp; повторный `/start` не запускает onboarding заново. `pnpm run ci` зеленый.
