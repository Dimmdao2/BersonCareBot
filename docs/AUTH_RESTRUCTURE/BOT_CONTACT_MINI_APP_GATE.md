# Mini App: гейт «сначала контакт в боте»

## Два слоя (модель продукта)

1. **Основной контроль — в чате с ботом (integrator), прод:** входящие Telegram/Max webhook обрабатываются **оркестратором**: `processAcceptedIncomingEvent` → `handleIncomingEvent` → **`buildPlan`** по сценариям из [`apps/integrator/src/content/telegram/user/scripts.json`](../../apps/integrator/src/content/telegram/user/scripts.json) и [`apps/integrator/src/content/max/user/scripts.json`](../../apps/integrator/src/content/max/user/scripts.json). Пока в контексте **`linkedPhone === false`**, срабатывают правила с `match.context.linkedPhone: false` (обычно с `priority: 30`) и **центральный гейт** в `buildPlan` для **`callback.received`** (в [`resolver.ts`](../../apps/integrator/src/kernel/orchestrator/resolver.ts)) — план запроса контакта строится **до** матчинга сценариев из JSON. В т.ч. `/start <внешний_id_записи>` **без** телефона в записи ведёт в запрос контакта, а не в полное меню с WebApp.  
   **Дополнительно (executor):** при флаге `sendMenuOnButtonPress` главная reply-клавиатура с WebApp подмешивается к `message.send` / `message.compose` **только если** `ctx.base.linkedPhone === true` ([`delivery.ts`](../../apps/integrator/src/kernel/domain/executor/handlers/delivery.ts)), иначе возможен обход гейта — см. [`INTEGRATOR_CONTRACT.md`](../../apps/webapp/INTEGRATOR_CONTRACT.md) Flow 6b.  
   **Регрессия контента:** vitest [`userScriptsLinkedPhoneGate.test.ts`](../../apps/integrator/src/content/userScriptsLinkedPhoneGate.test.ts) — ни одно правило с `match.context.linkedPhone: false` не содержит в `steps` ключей `webAppUrlFact` или `web_app`.  
   **Уточнение по коду:** [`handleUpdate`](../../apps/integrator/src/kernel/domain/usecases/handleUpdate.ts) / [`handleMessage`](../../apps/integrator/src/kernel/domain/usecases/handleMessage.ts) — **не** подключены к webhook в проде (остались для тестов и исторической совместимости); [`requestPhoneLink`](../../apps/integrator/src/kernel/domain/usecases/requestContactFlow.ts) описывает тот же продуктовый UX, что повторяют сценарии и M2M `request-contact`. Сценарии `*.need_phone` с событием `callback.received` в JSON дублируют центральный гейт и в проде обычно не выбираются — при правках не путать с единственным источником истины для колбэков.

2. **Страховка — Mini App (webapp):** если пользователь **уже открыл** WebApp из кнопки/ссылки при сессии с привязкой к мессенджеру, но в ответе `/api/me` **ещё нет** tier **patient** (нет доверенной активации телефона на каноне — см. `patientMessengerContactGate` / `resolvePlatformAccessContext`), показывается полноэкранный гейт: кнопка «Предоставить контакт» → `POST /api/patient/messenger/request-contact` → integrator шлёт в **чат** клавиатуру/кнопку запроса контакта → `WebApp.close()`. Подстраховка на гонку проекции (`contact.linked`), а не замена п.1.

## Зачем (страховочный слой)

Пользователь может открыть Telegram Mini App **до** того, как в чате с ботом нажал «Поделиться контактом». Сессия webapp создаётся по `initData` ([`exchangeTelegramInitData`](../../apps/webapp/src/modules/auth/service.ts)), а номер в `platform_users` приходит асинхронно после события `contact.linked` из integrator.

## Поведение

- Клиент: мессенджерный Mini App — [`isMessengerMiniAppHost`](../../apps/webapp/src/shared/lib/messengerMiniApp.ts): **Telegram** — непустой `initData` **или** cookie `bersoncare_platform=bot` при наличии `Telegram.WebApp` (initData в WebView может появиться позже); **MAX** — глобальный `window.WebApp` с `ready` (см. dev.max.ru). Для M2M `request-contact` при **двух** привязках (Telegram+Max) заголовок **`X-Bersoncare-Contact-Channel: telegram | max` обязателен**; без него или при неверном значении — **`400`** ([`inferMessengerChannelForRequestContact`](../../apps/webapp/src/shared/lib/messengerMiniApp.ts) на клиенте, валидация в route).
- **Лимиты `POST /api/patient/messenger/request-contact`:** счётчик **60 с** обновляется только после **успешного** ответа интегратора (`ok: true`, в т.ч. **`status: "duplicate"`** — считаем успехом, чат уже получил/дедупнул запрос); идемпотентность на стороне integrator — **в памяти процесса** (`Map` по `idempotencyKey`, см. [`INTEGRATOR_CONTRACT.md`](../../apps/webapp/INTEGRATOR_CONTRACT.md) Flow 6b); webapp — **окно 5 минут** на пару `(channel, recipientId)` в ключе идемпотентности.
- **401 на `/api/me`:** перед проверкой гейта вызывается восстановление сессии — [`ensureMessengerMiniAppWebappSession`](../../apps/webapp/src/shared/lib/miniAppSessionRecovery.ts): сначала `POST /api/auth/telegram-init` при наличии Telegram `initData`, иначе при наличии в URL `?t=` / `?token=` — `POST /api/auth/exchange` (типичный вход из Max/TG по ссылке бота без предварительного захода на `/app`). Если после этого всё ещё 401 — экран «Нет сессии», не разблокирование контента.
- **Ошибка `/api/me` (5xx и пр., не 401):** контент **не** открывается — состояние `me_unavailable` в [`getPatientMessengerContactGateDetail`](../../apps/webapp/src/shared/lib/patientMessengerContactGate.ts): остаётся блокирующий экран с «Проверить снова» (fail-safe для tier **patient**).
- **Когда показывается оверлей:** по [`getPatientMessengerContactGateDetail`](../../apps/webapp/src/shared/lib/patientMessengerContactGate.ts) — пользователь с **привязкой Telegram или Max**, но **ещё не** в tier **patient** в webapp: при наличии **`platformAccess`** решает **`platformAccess.tier === "patient"`**; при **`platformAccess === null`** без флага ошибки — прежняя эвристика непустого `user.phone` (legacy / без `DATABASE_URL`). Если в ответе есть **`platformAccessUnresolved: true`** (ошибка `resolvePlatformAccessContext` при настроенной БД) и есть привязка мессенджера — **блокирующее состояние** как при недоступности `/api/me` (**`me_unavailable`**), без снятия гейта по snapshot-телефону (аналог fail-safe [`patientClientBusinessGate`](../../apps/webapp/src/modules/platform-access/patientClientBusinessGate.ts)). Дальше — полноэкранный экран: ссылка на бота, опрос `/api/me`, кнопка «Предоставить контакт» (M2M + `close`).
- **`/app/patient/bind-phone`:** оверлей-гейт не показывается; в Mini App с привязкой к боту страница показывает встроенную подсказку «через бота» ([`PatientBindPhoneClient`](../../apps/webapp/src/app/app/patient/bind-phone/PatientBindPhoneClient.tsx)), без формы SMS. В обычном браузере — прежний поток (Telegram channel / web + OTP по политике `AuthFlowV2`).
- **Таймаут опроса** (~90 с): затем текст про возможную задержку синхронизации и кнопка «Проверить снова».
- **Контракт `contact.linked`**: событие должно нести не только `phoneNormalized`, но и `channelCode` / `externalId`, чтобы webapp синхронно восстановил и `platform_users.phone_normalized`, и `user_channel_bindings`.
- **Риск `contact.linked`**: если в integrator при `user.phone.link` не найден `userId` по identity, событие в webapp не ставится — номер может задержаться; см. [`writePort`](../../apps/integrator/src/infra/db/writePort.ts).
- **Риск частичной проекции**: UI блока «Привязанные каналы» читает `user_channel_bindings`, а не только `platform_users.phone_normalized`. Поэтому корректная доставка `contact.linked` важна не только для номера, но и для согласованного статуса Telegram в ЛК.
- **Другие разделы**: гейт только в layout пациента; логика «достаточно мессенджера без телефона» (`hasMessengerBinding` и т.д.) на остальных экранах не менялась.
