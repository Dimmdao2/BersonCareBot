# Отчёт по глубокой оценке кода

**Дата:** 2025-03-14  
**Объект:** BersonCareBot — оркестратор (интегратор tgcarebot) и webapp, их границы и взаимодействие.  
**Без изменений в коде:** отчёт только фиксирует находки и рекомендации.

---

## 1. Архитектурные границы оркестратора (интегратора)

### 1.1 Соответствие заявленной архитектуре

Документ `ARCHITECTURE.md` задаёт жёсткий контракт: интеграции только маппят payload в `IncomingEvent` и прикладывают нормализованные facts; ядро не должно знать transport/business-смысл ключей интеграции.

**Найденные нарушения и отклонения:**

| Место | Проблема | Детали |
|-------|----------|--------|
| **`kernel/domain/executor/executeAction.ts`** | Знание transport и бизнес-смысла | Executor явно использует: `ctx.base.facts.adminChatId`, `resource === 'telegram'`, шаблоны `telegram:adminForward`, `telegram:admin.reply.button`, `telegram:admin.dialog.*`, `delivery.channels: ['telegram']`, действия `admin.forward`, `conversation.admin.reply`, `dialogs.view`, `unanswered.list`. По контракту Domain/Executor не должен знать схему webhook и смысл интеграционных ключей. |
| **`kernel/domain/executor/executeAction.ts`** | Знание бизнес-сценариев | Ветвления по `notifications.toggle.spb/msk/online/bookings`, `booking.upsert`, `reminders.occurrence.deliver` с жёстко заданными категориями и ключами шаблонов (`telegram:reminder.exercise` и т.д.). Orchestrator по правилам не должен содержать hardcoded knowledge про booking, calendar, telegram-menu. |
| **`src/content/*/scripts.json`** | Transport в content | В content явно заданы: `source: "telegram"`, `delivery.channels: ["telegram"]`, `templateKey: "telegram:..."`, `chatId: "{{facts.adminChatId}}"`. В ARCHITECTURE это уже отмечено как отклонение: «Transport/delivery knowledge частично живет в content». |
| **`infra/db/readPort.ts`** | Жёсткая привязка к каналу | Для `notifications.settings` выполняется `if (resource !== 'telegram') return null` — порт знает единственный канал по имени. Допустимо для адаптера, но сужает расширяемость. |

**Итог:** Ядро (executor) и content содержат значительный объём знаний о Telegram, админ-диалогах и уведомлениях. Граница «универсальный контейнер данных / generic actions» нарушена: executor интерпретирует факты и параметры в transport- и бизнес-смысле.

### 1.2 Зависимости слоёв

- **Интеграции → kernel:** Импорт только контрактов и типов (`EventGateway`, `IncomingEvent`, `IncomingUpdate`) — корректно.
- **Интеграции → infra:** Только `infra/observability/logger` — допустимо для логирования.
- **`integrations/webappEntryToken.ts`:** Зависит от `telegram/config.js` (adminTelegramId для роли). Логика «кто админ» остаётся в интеграции; контракт с webapp (payload токена) не дублирует типы из webapp — возможен дрейф формата при изменении только одной стороны.

### 1.3 Избыточность и дублирование

- Подпись токена (HMAC-SHA256, base64url) реализована в оркестраторе (`webappEntryToken.ts`) и в webapp (`auth/service.ts`) независимо. Формат совпадает по контракту, но при смене алгоритма или полей нужны синхронные правки в двух местах.
- В `buildTelegramFacts` смешаны: данные для сценариев (actor, links), проверка админа (isAdmin, adminChatId) и построение ссылки на webapp. Функция разрослась и отвечает за несколько разных аспектов.

## 2. Архитектурные границы webapp

### 2.1 Слои и зависимости

- **`app` (страницы, API route handlers):** Используют только `buildAppDeps()` и guards; бизнес-решений в страницах нет — норма.
- **`app-layer`:** Единственное место сборки зависимостей; импортирует модули и инфраструктуру — соответствует заявленной схеме.
- **`modules`:** Импорт из `@/config/env` только в `auth/service` — по правилам модули могут зависеть от config. Остальные модули не тянут infra напрямую.
- **`infra`:** Реализация ограничена `db/client`, `webhooks/verifyIntegratorSignature`. Репозиториев под домены webapp (диарии, напоминания, пользователи) нет: диарии — in-memory в модулях, что для MVP допустимо, но выходит за рамки заявленной «инфраструктуры» (persistence, порты).

### 2.2 Расхождение с описанной структурой

В `ARCHITECTURE.md` webapp указаны папки модулей: `contacts`, `billing`, `diaries`, `reminders`, `appointments`, `purchases`. Фактически:

- `contacts`, `billing` — в коде не представлены (ни модулей, ни страниц).
- `diaries` — реализованы как два сервиса (symptom, lfk) с in-memory хранилищем внутри модуля; отдельного слоя persistence/ports нет.
- Напоминания: только валидация payload и заглушка `listReminderRules`; нет планировщика, правил в БД, связи с интегратором по контракту.

То есть заявленная «эволюция» (billing, diaries, reminders) частично только на уровне имён и заглушек, без единого подхода к портам и хранилищу.

### 2.3 Риски in-memory хранилищ (диарии)

- `symptom-service.ts` и `lfk-service.ts`: глобальные массивы и счётчики в процессе. Данные теряются при рестарте; при нескольких инстансах Next.js (горизонтальное масштабирование) хранилища не разделяются. Для MVP приемлемо, но как только понадобится персистентность или несколько воркеров — потребуется вынос в порт (БД/хранилище) и перенос логики хранения из модуля в infra.

## 3. Взаимодействие оркестратор ↔ webapp

### 3.1 Контракт (INTEGRATOR_CONTRACT.md)

- **Вход в webapp:** Подписанный токен в `?t=...`, обмен через `POST /api/auth/exchange`. Реализация соответствует: оркестратор собирает токен с `purpose: "webapp-entry"`, webapp проверяет подпись, срок, whitelist, выставляет cookie.
- **Риск рассинхрона:** Формат payload токена (поля, имена) задан текстом в контракте и реализован в двух кодовых базах. Нет общего типа (например, общего пакета или OpenAPI): при добавлении поля в одну сторону вторая может не обновиться.

### 3.2 Webhook-и webapp

| Endpoint | Реализация | Замечания |
|---------|------------|-----------|
| `POST /api/integrator/events` | Есть | Проверяет заголовки и подпись, требует `X-Bersoncare-Idempotency-Key`, возвращает `accepted: true`. Тела запроса не парсит, доменной обработки нет, idempotency по ключу не сохраняется. Контракт предполагает «link messenger action to webapp-side domain reaction» — сейчас это не выполняется. |
| `POST /api/integrator/reminders/dispatch` | Есть | Валидирует payload, проверяет подпись, возвращает `dispatchMode: "bridge-to-integrator"`. Фактической отправки в оркестратор или в очередь нет — заглушка. |

Итого: контракт описан, подпись проверяется, но доменная часть (события, диспетчеризация напоминаний) не реализована; idempotency для webhook-ов не используется.

### 3.3 Секреты и конфигурация

- Общий секрет для токенов и webhook-подписи: `INTEGRATOR_SHARED_SECRET` — один и тот же для подписи entry-токена и для `verifyIntegratorSignature`. Разделение ключей (например, отдельно для webapp-entry и для webhook) могло бы ограничить последствия утечки одного из них.
- Webapp: whitelist `ALLOWED_TELEGRAM_IDS` и `ADMIN_TELEGRAM_ID` задают доступ по Telegram ID. Роль `admin` в токене доверяется оркестратору (он выставляет её по `adminTelegramId`). Двойная проверка (роль + whitelist) уменьшает риск подделки роли при компрометации только webapp.

## 4. Безопасность

### 4.1 Критичные и высокие риски

| Риск | Место | Описание |
|------|--------|----------|
| **Секреты в коде** | `src/integrations/telegram/config.ts` | В репозитории захардкожены `botToken`, `adminTelegramId`, `webhookSecret`. В `integrations/config.ts` указано «Secrets live in source by explicit project decision». Это осознанное решение, но при любом доступе к репозиторию секреты считаются скомпрометированными; ротация требует правки кода и деплоя. |
| **Replay webhook-ов** | `webapp/src/infra/webhooks/verifyIntegratorSignature.ts` | Проверяется только совпадение подписи `timestamp + "." + body`. Проверки свежести `timestamp` (окно времени) нет — старый валидный запрос можно повторно отправить. |
| **Dev-обход auth** | `webapp/src/modules/auth/service.ts` | При `ALLOW_DEV_AUTH_BYPASS=true` токены `dev:client`, `dev:doctor`, `dev:admin` принимаются без whitelist и без проверки подписи интегратора. Если переменная случайно включена в prod, возможен полный обход контроля доступа. |

### 4.2 Средние риски

| Риск | Место | Описание |
|------|--------|----------|
| **Нет idempotency для webhook** | `webapp/src/app/api/integrator/reminders/dispatch/route.ts`, `events/route.ts` | Заголовок `X-Bersoncare-Idempotency-Key` принимается, но не используется для дедупликации. Повторные запросы с тем же ключом будут обработаны многократно. Контракт требует idempotent writes. |
| **Сессия в cookie** | `webapp/src/modules/auth/service.ts` | Cookie подписывается HMAC, httpOnly, sameSite: lax, secure в production — хорошо. TTL 12 часов; отсутствует механизм продления или отзыва по списку — при утечке cookie сессия живёт до истечения срока. |
| **Роль из токена без проверки в webapp** | Webapp auth | Роль приходит в токене от оркестратора; webapp доверяет ей для редиректа и доступа к /app/doctor и /app/patient. При компрометации оркестратора или поддельном токене (если секрет утёк) злоумышленник может получить роль admin. Whitelist по telegramId частично смягчает, но не заменяет принцип наименьших привилегий на стороне webapp. |

### 4.3 Низкие риски и положительное

- Сравнение подписей: везде используется `timingSafeEqual` (auth, verifyIntegratorSignature) — защита от timing-атак.
- В разметке webapp не обнаружено `dangerouslySetInnerHTML`, `eval` — снижен риск XSS со стороны рендера.
- Прямых конкатенаций SQL в запросах в обзоре не видно; webapp использует только `client.query("select 1")` для health. Оркестратор использует параметризованные запросы через порты и репозитории.

## 5. Некорректный и избыточный код

### 5.1 Оркестратор

- **`executeAction.ts`:** Очень большой файл (~1700+ строк), множество `case` по типам действий. Трудно сопровождать и тестировать; часть кейсов (admin, notifications, reminders, booking) можно вынести в отдельные обработчики или под-модули с явными контрактами.
- **Дублирование логики рендера для админа:** Повторяющийся паттерн: взять `adminChatId`, сформировать текст через `templatePort`, отправить intent с кнопками. Можно обобщить в одну функцию/хелпер.
- **Жёстко заданные ключи шаблонов:** Строки вида `'telegram:admin.reply.button'`, `'telegram:reminder.exercise'` разбросаны по executor. Имеет смысл вынести в константы или конфиг content, чтобы связь «действие → шаблон» была в одном месте.

### 5.2 Webapp

- **Страница `/app` при наличии сессии:** Показывает «Рабочее пространство» и меню карточек; по сути дублирует навигацию, которую пользователь всё равно видит в разделах patient/doctor. Можно сразу редиректить на `/app/patient` или `/app/doctor` в зависимости от роли и не рендерить промежуточную страницу.
- **`getPatientCabinetState(appointmentCount)`:** Принимает число записей; вызывающий код (страница cabinet) сначала получает список записей, затем передаёт его длину. Логичнее передавать в сервис `userId` (или сессию) и инкапсулировать получение записей внутри сервиса кабинета, чтобы страница не знала о двух источниках данных.
- **Модуль `purchases`:** Только заглушка с текстом. Модуль `reminders` — валидация и `listReminderRules` (пустой массив); доменная логика напоминаний (правила, очередь) в webapp не реализована. Код минимален, но в архитектуре они заявлены как домены — стоит явно помечать заглушки и планировать границы портов.

### 5.3 Общее

- **Контракт токена:** В оркестраторе и webapp разная типовая база (TypeScript-типы в каждом проекте). При изменении формата (новые поля, смена имени) легко забыть обновить вторую сторону. Вариант улучшения: общий пакет типов или явная схема (JSON Schema/OpenAPI) и кодогенерация или ручная сверка по чек-листу при изменениях.

## 6. Рекомендации по структуре и взаимодействию

### 6.1 Оркестратор

1. **Разделение знаний в executor:** Вынести обработку «админ» (диалоги, пересылки, списки) в отдельный модуль/набор действий с явным контрактом по `facts` и шагам; executor оставить тонкой диспетчеризацией по типу действия. Альтернатива — ввести слой «action handlers» с регистрацией по типу.
2. **Content:** Постепенно переходить к более generic-именам в шагах и шаблонах (например, «channel_admin» вместо «telegram:admin»), а маппинг «generic → telegram» держать в интеграционном слое или в конфиге контента по source.
3. **Секреты:** Вынести конфиг Telegram (и при необходимости другие интеграции) в env без дефолтов в коде; убрать захардкоженные токены и webhookSecret из репозитория.
4. **Webapp entry token:** Рассмотреть вынос формата payload и алгоритма подписи в общий описанный контракт (документ + при необходимости общий пакет), чтобы изменения формата делались согласованно.

### 6.2 Webapp

1. **Репозитории и порты:** Для диариев и напоминаний ввести порты (например, `SymptomDiaryPort`, `LfkDiaryPort`, `ReminderRulesPort`) и реализации в `infra` (БД или внешний сервис). Модули вызывают только порты; данные не хранить в памяти модуля при выходе из MVP.
2. **Webhook-и:** Для `POST /api/integrator/events` и `POST /api/integrator/reminders/dispatch`: парсить тело, проверять idempotency по ключу (хранение обработанных ключей в БД или кэше), вызывать доменную логику (события → обработчики, напоминания → очередь или вызов оркестратора). Иначе контракт остаётся нереализованным.
3. **Replay-защита:** В `verifyIntegratorSignature` (или в вызывающем коде) добавить проверку: `timestamp` в допустимом окне (например, ±5 минут от текущего времени).
4. **Редирект с /app:** При валидной сессии сразу делать redirect на `/app/patient` или `/app/doctor`, чтобы не показывать промежуточную страницу «Рабочее пространство» без необходимости.
5. **ALLOW_DEV_AUTH_BYPASS:** В production-сборке или при `NODE_ENV=production` принудительно игнорировать эту переменную (не включать dev-токены), чтобы исключить случайное включение на проде.

### 6.3 Взаимодействие

1. **Единый контракт:** Оформить формат токена webapp-entry и формат webhook (events, reminders) в виде схемы (JSON Schema или OpenAPI fragment) в отдельном репозитории или в монорепе; при изменении — обновлять обе стороны и тесты.
2. **Секреты:** Разделить ключи: один для подписи webapp-entry токенов, другой для webhook-запросов webapp ↔ оркестратор, чтобы компрометация одного не давала полного контроля.
3. **Направление вызовов:** Сейчас оркестратор вызывает webapp только косвенно (ссылка с токеном). Webapp по контракту может вызывать оркестратор (reminders/dispatch). Реализовать фактическую отправку напоминаний в оркестратор (HTTP-вызов с подписью) и при необходимости idempotency на стороне оркестратора.
