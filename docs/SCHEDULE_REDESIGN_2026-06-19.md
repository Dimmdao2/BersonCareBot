## 1. Current State

Two disconnected schedule editing systems exist:

### System A — Per-date editor (`ScheduleWorkTab.tsx`, "По датам" mode)
- File: `apps/webapp/src/app/app/doctor/schedule/tabs/ScheduleWorkTab.tsx` (1113 lines)
- Data: `be_working_days` table — per-date rows with `{workDate, startMinute, endMinute, breaks, isClosed, branchId}`
- API: `WD_BASE = /api/admin/booking-engine/working-days` (GET/PUT with action=upsert/clear/close)
- UI: month grid → select dates → right panel → set hours → Save / Очистить расписание
- Also has: named template panel at bottom (`be_schedule_templates` via `TPL_BASE`; templates are name+time patterns applied to arbitrary date sets — they do NOT have a weekday column)
- "Недельный шаблон" tab in the mode switcher → shows `BookingSoloScheduleSection` (System B)

### System B — Weekday editor (`BookingSoloScheduleSection.tsx`)
- File: `apps/webapp/src/app/app/settings/BookingSoloScheduleSection.tsx` (492 lines)
- Data: `be_working_hours` table — rows with `{weekday: 0–6, startMinute, endMinute, isActive, specialistId, branchId}`
- API: `WH_BASE = /api/admin/booking-engine/working-hours` (GET/POST/PATCH/DELETE)
- UI: 7-day list (Пн–Вс), per-day interval rows, inline edit/delete, copy-from/copy-to, buffer/notice settings
- This is the TRUE weekday template model the booking slot engine reads

### How the Calendar Uses Both
- `modules/booking-calendar/service.ts` (lines 136–164): loads `listWorkingHours` (be_working_hours weekday rows) + `listWorkingDays` (per-date overrides) in parallel
- Per-date override takes priority over weekday hours for that date
- Calendar renders gray non-working fill based on effective working windows per date

### Key Architecture Clarity
- **"Permanent schedule" (постоянное расписание)** = `be_working_hours` (weekday model), API `/working-hours`
- **"Named templates"** = `be_schedule_templates`, API `/working-schedule-templates` — does NOT have a weekday column, used only for apply-to-date-set
- The QUEUE.md CR-4 spec uses "working-schedule-templates" loosely to mean the weekday model; implementors must save weekday selections to **`be_working_hours`** via `/working-hours` POST

## 3. Data Model Notes

### be_working_hours (permanent weekday template)
```
id, organizationId, specialistId, branchId, roomId
weekday: int (0=Sun, 1=Mon, …, 6=Sat)
startMinute: int, endMinute: int
isActive: bool
```
Multiple rows allowed per weekday (intervals). Saved via POST `/working-hours`, updated via PATCH, deleted via DELETE.

### be_working_days (per-date overrides)
```
id, organizationId, specialistId, branchId, roomId
workDate: date (YYYY-MM-DD)
startMinute: int|null, endMinute: int|null
breaks: jsonb [{startMinute, endMinute}]
isClosed: bool
```
Upsert action: `PUT /working-days` `{action:"upsert", dates, startMinute, endMinute, breaks, specialistId, branchId}`
Clear action: `PUT /working-days` `{action:"clear", dates, specialistId}` — deletes rows, day falls back to weekday template

### Weekday Number Mapping
- `BookingSoloScheduleSection` uses: 0=Sunday, 1=Mon, …, 6=Sat (JS `Date.getDay()`)
- Luxon's `DateTime.weekday` is 1=Mon…7=Sun
- `buildMonthGrid` in `ScheduleWorkTab` uses Mon-first grid: column 0=Mon (weekday 1), …, col 6=Sun (weekday 0)
- When converting grid column index to be_working_hours weekday: `[1,2,3,4,5,6,0][colIndex]`

### What To Render in Cells
To show effective hours per date in the calendar, the component needs:
1. Per-date `be_working_days` rows (already loaded via `loadMonth` → `WD_BASE`)
2. Weekday hours from `be_working_hours` (needs new load: `GET /working-hours?specialistId=X&branchId=Y`)
3. Effective = perDateRow if exists; else weekday row matching `date.weekday`; else no schedule

## 7. Open Questions / Risks

### CR-4 Risks

1. **Weekday number encoding.** `be_working_hours` uses JS `Date.getDay()` (0=Sun, 1=Mon…6=Sat). Luxon's `weekday` is 1=Mon…7=Sun. The grid column index 0=Mon…6=Sun maps to weekday `[1,2,3,4,5,6,0][col]`. Executor MUST use this mapping; off-by-one on weekday = bugs visible only after a week.

2. **`replace=true` atomicity.** SCH-R-07 deactivates then inserts. If the insert fails, the deactivation has already happened. Consider a transaction wrapper in `pgBookingScheduling.ts` or accept the rare inconsistency (a retry will re-insert).

3. **Multiple intervals per weekday.** `be_working_hours` allows multiple rows per weekday (e.g. morning + afternoon shift). The right panel currently only has one start/end pair. For CR-4, treat as single-interval (deactivate all existing, insert one). Edge case: if doctor had split shifts, the unified save will flatten them. Not a regression from current behavior but worth noting.

4. **Calendar rendering needs both datasets.** `ScheduleWorkTab` currently only loads `be_working_days` (not `be_working_hours`). SCH-R-01 adds the second load. If `specialistId` is not yet set when the component mounts, both loads must wait (already gated by `if (!specialistId) return`).

5. **Branch filter and weekday template.** Weekday hours in `be_working_hours` are scoped to `(org, specialistId, branchId)`. When saving «постоянное расписание», use `panelBranchId` as the branch. The effective hours for cells need to be filtered by `gridBranchFilter` if active (or show all if "all"). SCH-R-02 must handle branch matching.

6. **CR-5 already fixed.** Per QUEUE.md CR-5 note: `computeSlots.ts:90` no longer returns `DEFAULT_WORKING` phantom (confirmed: `pickWorkingHours` at line 90 just returns rows as-is). CR-4 implementation should not reintroduce any DEFAULT_WORKING hardcoding.

### §A Risks

7. **§A-1 sets field (подходы).** Current `ProgramItemCompleteDialog` has `reps` and `weightKg` but NO `sets` field. The spec says "3 числа в ряд — повторения · подходы · вес". Adding `sets` requires checking if the API `/progress/complete` accepts it — verify in `PatientProgramStageItemPageClient.tsx` around lines 815–838 where the POST is made.

8. **§A-3 API gap.** The tile card currently does NOT receive unread comment counts or contraindications data. This requires the parent data-fetching to include these per-item. May involve a new API field or a secondary fetch — scoping can be larger than 30 min.

9. **§A-4 dialog to inline.** The discussion dialog loads its own data (`fetch` with abort controller). Moving this inline to the page will change the load timing. Ensure loading state is handled so the page doesn't stall awaiting comments.
