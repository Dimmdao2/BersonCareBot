# Схема блоков: страницы «План» / программа лечения (пациент)

**Назначение.** Одна страница-шпаргалка: примитивные блок-схемы по уже зафиксированным документам, без новых продуктовых решений.

**Эталонный вариант детали программы** — это **`ROADMAP_2` §1.1b** («визуальный редизайн / эталон-дизайн»): порядок секций **сверху вниз** и перенос полного содержимого текущего этапа на отдельный маршрут **`stages/[stageId]`**. Текст roadmap допускает уточнение порядка по ходу реализации; ниже — дословная нумерация из §1.1b.

## 2. Деталь программы — MVP-проход (`ROADMAP_2` §1.1a)

Промежуточный экран **до** эталона §1.1b: вся логика текущего этапа **на той же странице**, без отдельного маршрута этапа.

```
┌─────────────────────────────────────────────────────────┐
│  ШАПКА                                                  │
│  · название программы                                   │
│  · номер/название текущего этапа                        │
│  · CTA «Открыть текущий этап» (якорь/скролл в блок)    │
│  · ссылка «Архив этапов»                                │
│  · «План обновлён»                                      │
│  · дата контроля: started_at + expected_duration_days   │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  БЛОК «Общие рекомендации» (этап 0, sort_order = 0)      │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  ТЕКУЩИЙ ЭТАП — единый список назначений               │
│  exercise | lfk_complex | recommendation | test_set |   │
│  lesson                                                 │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  <details> архив: завершённые / пропущенные этапы       │
└─────────────────────────────────────────────────────────┘

  (в этом MVP-проходе «Чек-лист на сегодня» с детали убран)
```

## 3. Эталон: деталь программы — `/treatment-programs/[instanceId]` (`ROADMAP_2` §1.1b)

**Порядок блоков** соответствует §1.1b по смыслу. Отдельные блоки roadmap **C4** («превью текущего этапа») и **C6** («предыдущие этапы») в текущей реализации **объединены** в одну секцию **«Активный этап»**: вертикальный список всех этапов с `sort_order > 0` (архив + активный pipeline), с модалкой состава для активной строки и основным CTA **«Начать занятие»** → `stages/[stageId]`. Верхний `AppShell`-title подавляется (`patientSuppressShellTitle`), чтобы не дублировать название программы из hero.

Якорь **`#patient-program-current-stage`** находится на секции timeline (совместимость с deep link / текстом roadmap про scroll-цель; основной CTA ведёт на маршрут этапа).

```
┌─────────────────────────────────────────────────────────┐
│ 1. HERO «Мой план»                                      │
│    бейдж «Мой план» · бейдж статуса (завершена /        │
│    назначена / старт с датой)                           │
│    название программы                                   │
│    описание шаблона (если есть: RSC `getTemplate`)      │
│    «План обновлён» · при необходимости «Ожидает старта» │
│    CTA «Начать занятие» → stages/[currentStageId]       │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ 2. КАРТОЧКА «Следующий контроль» / запись               │
│    дата · подпись про консультацию                      │
│    «Выполнить тесты» · «Записаться на приём»           │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ 3. «Рекомендации на период» (этап 0)                    │
│    Collapsible — по умолчанию ЗАКРЫТ                    │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ 4. «Активный этап» — timeline (этапы с sort_order > 0)  │
│    merge архив + pipeline; активная строка: Play,       │
│    «N из K», кнопка списка → Dialog; прошлые/будущие   │
│    стили по токенам; id=#patient-program-current-stage  │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ 5. «История тестирования» — одна точка входа            │
│    ссылка «Открыть текущий этап» → stages/[stageId]    │
└─────────────────────────────────────────────────────────┘
```

**Цветовые зоны (из §1.1b, привязка к токенам):** hero — `patientHomeCardHeroClass` / программные бейджи из `patientHomeCardStyles`; карточка контроля — `patientSurfaceWarningClass`; этап 0 — `patientSurfaceSuccessClass`; timeline и прочие карточки — `patientCardClass` / `patientListItemClass` (прошлые строки: `bg-muted/20 opacity-70`; заблокированные будущие: `opacity-50`).

## 4. Страница этапа — `/treatment-programs/[instanceId]/stages/[stageId]` (§1.1b, B5)

Полное тело этапа (`PatientInstanceStageBody` и назначения) переносится **сюда** из детали программы.

```
┌─────────────────────────────────────────────────────────┐
│  Шапка / навигация этапа (как примет реализация)        │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  ПОЛНОЕ СОДЕРЖИМОЕ ЭТАПА                                │
│  цели / задачи / срок · группы (аккордеоны) · items     │
│  exercise / lfk_complex / recommendation / test_set /   │
│  lesson · run-screen / отметки по доменным правилам    │
└─────────────────────────────────────────────────────────┘
```

Детальная разметка внутри этапа согласуется с [`PROGRAM_PATIENT_SHAPE_PLAN.md`](../APP_RESTRUCTURE_INITIATIVE/PROGRAM_PATIENT_SHAPE_PLAN.md) и [`TARGET_STRUCTURE_PATIENT.md`](../APP_RESTRUCTURE_INITIATIVE/TARGET_STRUCTURE_PATIENT.md) §4.2 (там шире продуктовая схема, включая чек-лист и группы).

## 5. Сверка: эскиз vs реальный экран vs код

Эскиз (мобильный макет) в документах **не хранится как файл** — он описан текстом в `ROADMAP_2` §1.1b.
Ниже — построчная сверка блоков эскиза, состояния кода и того, что видно на рабочем dev-экране.

> Эскиз: третий скриншот (мобильный макет).
> Dev-экран: второй скриншот (Safari, desktop dev-сервер).

| № | Блок эскиза | Описан в docs? | Есть в коде? | Виден на dev? | Зазор / примечание |
|---|-------------|----------------|--------------|---------------|--------------------|
| 1 | Hero: бейдж «Мой план» + бейдж статуса программы (даты) | §1.1b B1 ✓ | ✓ | ✓ | Номер этапа «N из K» вынесен в строку активного этапа в timeline, не в hero |
| 1 | Hero: **описание программы** под заголовком | §1.1b B1 «если есть данные» ✓ | ✓ RSC `getTemplate` + `programDescription` | при `templateId` и непустом `description` шаблона | `line-clamp-3` |
| 1 | Hero: **иллюстрация справа** (анатомия) | §1.1b B1 «опционально» ✓ | ✗ | ✗ | В hero нет декоративного `<img>` |
| 1 | Основной CTA + play-иконка | §1.1b B1 ✓ | ✓ «Начать занятие» → `stages/[stageId]` | ✓ | Текст отличается от раннего макета «Открыть план» |
| 1 | Hero: «● План обновлён», «Ожидает старта» | §1.1b B1 ✓ | ✓ | nudge по данным | «Ожидает старта» при `active` + рабочий этап `available` |
| 2 | Карточка «Следующий контроль»: дата, консультация, 2 CTA | §1.1b B3 ✓ | ✓ `PatientProgramControlCard` | ✗ на dev | Нужны `started_at` + `expected_duration_days` |
| 3 | Блок «Рекомендации на период» (зелёный, `Collapsible`) | §1.1b B4 ✓ | ✓ | ✗ на dev | Нужен этап с `sort_order = 0` |
| 4 | Этапы программы (текущий + архив) | §1.1b B5/B6 ✓ | ✓ секция «Активный этап» | ✓ | Превью-карточка C4 и отдельный блок C6 **не используются** — заменены timeline + Dialog |
| 5 | **«История тестирования»** — отдельная точка входа | §1.1b B7 ✓ | ✓ секция | ✓ | CTA «Открыть текущий этап» → текущий `stages/[stageId]` |

### Итог сверки

- **Совпадает с целями §1.1b:** hero с программными бейджами и основным CTA на этап; контроль; этап 0; единая точка входа «История тестирования»; список этапов с переходом на `stages/[stageId]`; якорь `#patient-program-current-stage` на секции timeline.
- **Закодировано, но не видно на dev** (нет тестовых данных): карточка контроля; блок этапа 0.
- **Сознательное отличие от раннего эскиза roadmap:** нет отдельной карточки-превью «Текущий этап» — контент перенесён в timeline и страницу этапа.
- **Опционально в эскизе, не в коде:** декоративная иллюстрация в hero.
- **Данные:** дата «Завершён …» у строки этапа — только если появится `completedAt` на stage (сейчас в типах нет).
- **Технический хвост:** state `testResults` обновляется при refresh, на экране не используется (зарезервировано под post-MVP).

## 7. Сводка: что считать «эталоном»

| Экран | Эталон в документах |
|-------|---------------------|
| Список | §1.1 — при **отсутствии** активной программы: empty state + `<details>` архив; при **наличии** активной — серверный `redirect` на detail инстанса |
| Деталь программы (визуальный финал) | **§1.1b** — блоки по §3 этого файла (hero → контроль → этап 0 → **Активный этап** timeline → история тестов) |
| Работа по назначениям внутри этапа | Страница **`stages/[stageId]`** — полное тело этапа |

MVP **§1.1a** — исполняемый шаг до эталона; структура блоков проще (всё на одной странице, архив этапов в `<details>`).
