# Схема блоков: страницы «План» / программа лечения (пациент)

**Назначение.** Одна страница-шпаргалка: примитивные блок-схемы по уже зафиксированным документам, без новых продуктовых решений.

**Эталонный вариант детали программы** — это **`ROADMAP_2` §1.1b** («визуальный редизайн / эталон-дизайн»): порядок секций **сверху вниз** и перенос полного содержимого текущего этапа на отдельный маршрут **`stages/[stageId]`**. Текст roadmap допускает уточнение порядка по ходу реализации; ниже — дословная нумерация из §1.1b.


## 2. Деталь программы — MVP-проход (`ROADMAP_2` §1.1a)

Промежуточный экран **до** эталона §1.1b: вся логика текущего этапа **на той же странице**, без отдельного маршрута этапа.

```
┌─────────────────────────────────────────────────────────┐
│  ШАПКА                                                  │
│  · название программы                                   │
│  · номер/название текущего этапа                        │
│  · CTA «Открыть текущий этап» (якорь/скролл в блок)    │
│  · ссылка «Архив этапов»                                │
│  · «План обновлён»                                      │
│  · дата контроля: started_at + expected_duration_days   │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  БЛОК «Общие рекомендации» (этап 0, sort_order = 0)      │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  ТЕКУЩИЙ ЭТАП — единый список назначений               │
│  exercise | lfk_complex | recommendation | test_set |   │
│  lesson                                                 │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  <details> архив: завершённые / пропущенные этапы       │
└─────────────────────────────────────────────────────────┘

  (в этом MVP-проходе «Чек-лист на сегодня» с детали убран)
```

## 3. Эталон: деталь программы — `/treatment-programs/[instanceId]` (`ROADMAP_2` §1.1b)

**Это и есть целевой порядок блоков** после закрытия §1.1a и предусловий (shadcn `Collapsible` / `Accordion` и т.д.).

```
┌─────────────────────────────────────────────────────────┐
│ 1. HERO «Мой план»                                      │
│    бейджи · название · описание (если есть)             │
│    CTA «Открыть план» · строка «План обновлён»          │
│    [опц. иллюстрация справа]                            │
└─────────────────────────────────────────────────────────┘
          │
          ▼ (CTA «Открыть план» → scroll к #patient-program-current-stage)
┌─────────────────────────────────────────────────────────┐
│ 2. КАРТОЧКА «Следующий контроль» / запись               │
│    дата · подпись про консультацию                      │
│    «Выполнить тесты» · «Записаться на приём»           │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ 3. «Рекомендации на период» (этап 0)                    │
│    Collapsible — по умолчанию ЗАКРЫТ                    │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ 4. «Текущий этап» — ПРЕВЬЮ только                       │
│    заголовок / цель·задачи (кратко)                     │
│    CTA «Открыть этап» → stages/[stageId]               │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ 5. «История тестирования» — одна точка входа             │
│    (не длинный список результатов на этой странице)    │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ 6. «Предыдущие этапы» — компактный список строк         │
│    строка → Link на stages/[stageId] для этапа          │
└─────────────────────────────────────────────────────────┘
```

**Цветовые зоны (из §1.1b, привязка к токенам):** hero — `patientSurfaceInfoClass` (или будущий `patientSurfaceProgramClass`); карточка контроля — `patientSurfaceWarningClass`; этап 0 — `patientSurfaceSuccessClass`; превью/списки — `patientCardClass` / `patientListItemClass`.

## 4. Страница этапа — `/treatment-programs/[instanceId]/stages/[stageId]` (§1.1b, B5)

Полное тело этапа (`PatientInstanceStageBody` и назначения) переносится **сюда** из детали программы.

```
┌─────────────────────────────────────────────────────────┐
│  Шапка / навигация этапа (как примет реализация)        │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  ПОЛНОЕ СОДЕРЖИМОЕ ЭТАПА                                │
│  цели / задачи / срок · группы (аккордеоны) · items     │
│  exercise / lfk_complex / recommendation / test_set /   │
│  lesson · run-screen / отметки по доменным правилам    │
└─────────────────────────────────────────────────────────┘
```

Детальная разметка внутри этапа согласуется с [`PROGRAM_PATIENT_SHAPE_PLAN.md`](../APP_RESTRUCTURE_INITIATIVE/PROGRAM_PATIENT_SHAPE_PLAN.md) и [`TARGET_STRUCTURE_PATIENT.md`](../APP_RESTRUCTURE_INITIATIVE/TARGET_STRUCTURE_PATIENT.md) §4.2 (там шире продуктовая схема, включая чек-лист и группы).

## 5. Сверка: эскиз vs реальный экран vs код

Эскиз (мобильный макет) в документах **не хранится как файл** — он описан текстом в `ROADMAP_2` §1.1b.
Ниже — построчная сверка блоков эскиза, состояния кода и того, что видно на рабочем dev-экране.

> Эскиз: третий скриншот (мобильный макет).
> Dev-экран: второй скриншот (Safari, desktop dev-сервер).

| № | Блок эскиза | Описан в docs? | Есть в коде? | Виден на dev? | Зазор / примечание |
|---|-------------|----------------|--------------|---------------|--------------------|
| 1 | Hero: бейджи «МОЙ ПЛАН» + «Этап X из Y» | §1.1b B1 ✓ | ✓ | ✓ | — |
| 1 | Hero: **описание программы** под заголовком | §1.1b B1 «если есть данные» ✓ | ✗ не рендерится | ✗ | Нет поля `description` на `TreatmentProgramInstanceSummary`/`Detail` — только на шаблоне и этапах |
| 1 | Hero: **иллюстрация справа** (анатомия) | §1.1b B1 «опционально» ✓ | ✗ | ✗ | В коде нет `<img>` в hero-блоке, только `play.svg` на кнопке |
| 1 | Hero: «Открыть план» + play-иконка | §1.1b B1 ✓ | ✓ | ✓ | — |
| 1 | Hero: «● План обновлён» | §1.1b B1 ✓ | ✓ | только если nudge активен | — |
| 2 | Карточка «Следующий контроль»: дата, консультация, 2 CTA | §1.1b B3 ✓ | ✓ `PatientProgramControlCard` | ✗ на dev | Рендерится только если есть `started_at + expected_duration_days`; у тестовой программы этих данных нет |
| 3 | Блок «Рекомендации на период» (зелёный, `Collapsible`) | §1.1b B4 ✓ | ✓ | ✗ на dev | Рендерится только если есть этап с `sort_order = 0`; у тестовой программы его нет |
| 4 | Текущий этап: превью + subtitle + «Открыть этап» | §1.1b B5 ✓ | ✓ | ✓ | Subtitle из `goals` или `objectives` (до 80 символов); у тестовых данных поля пусты |
| 5 | **«История тестирования»** — отдельная точка входа | §1.1b B7 ✓ | ✓ секция | ✓ | CTA ведёт на страницу **текущего этапа** (не на отдельную историю); отдельного маршрута нет — roadmap допускает «прорабатывается по ходу» |
| 6 | Предыдущие этапы: строки с датой «Завершён DD мм» | §1.1b B6, «если поле доступно» ✓ | ✓ список, но без даты | н/д (на dev нет завершённых) | `TreatmentProgramInstanceStageRow` не имеет `completedAt`; поле есть на item-ах, не на stage |

### Итог сверки

- **Полностью совпадает с эскизом:** hero-бейджи + CTA + план обновлён; превью текущего этапа; точка входа «История тестирования»; компактный архив этапов.
- **Закодировано, но не видно на dev** (нет тестовых данных): карточка контроля (нужен `started_at` + `expected_duration_days`), блок этапа 0 (нужен stage с `sort_order = 0`), subtitle под заголовком этапа (нужен `goals` / `objectives`).
- **Есть в эскизе, описано в docs, но не реализовано:**
  - описание программы в hero (нет поля в типе инстанса),
  - иллюстрация справа в hero (нет в коде).
- **Частичное:** дата «Завершён DD мм» в архиве этапов — поля `completedAt` на stage нет в типах (roadmap это учитывает: «не показывать дату»).
- **Ненужный остаток:** `testResults` state на детали подгружается в refresh, но нигде не рендерится.

## 7. Сводка: что считать «эталоном»

| Экран | Эталон в документах |
|-------|---------------------|
| Список | §1.1 — hero + `<details>` архив + empty state |
| Деталь программы (визуальный финал) | **§1.1b** — блоки 1–6 в порядке таблицы выше |
| Работа по назначениям внутри этапа | Отдельная страница **`stages/[stageId]`** после §1.1b (превью на детали → полный этап здесь) |

MVP **§1.1a** — исполняемый шаг до эталона; структура блоков проще (всё на одной странице, архив этапов в `<details>`).
