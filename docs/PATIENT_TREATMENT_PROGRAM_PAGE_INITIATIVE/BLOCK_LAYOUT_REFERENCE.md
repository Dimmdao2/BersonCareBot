## 3. Деталь программы — `/app/patient/treatment/[instanceId]` (вкладки)

Реализация: **hero** (название, описание шаблона при наличии, история, бейджи, CTA «Начать занятие» → модалка пункта) + **липкая панель из трёх вкладок** — **Программа** (цели/задачи + `PatientTreatmentProgramStagePageProgramSection` для рабочего этапа), **Рекомендации** (две секции плоским списком), **Прогресс** (timeline этапов, «Следующий контроль», статистика). Переключение вкладок **не размонтирует** скрытые панели (`hidden`). Счётчик «N дней» на вкладке «Прогресс» считается в RSC с рубежом **03:00** в IANA пациента (`computeProgressDaysAt0300`).

Строка «Вы занимаетесь N дней» в hero **не используется**.

```
┌─────────────────────────────────────────────────────────┐
│  HERO — программа                                       │
│  · название, описание (RSC), история, CTA → модалка      │
└─────────────────────────────────────────────────────────┘
┌───────────────┬─────────────────┬───────────────────────┐
│ Программа     │ Рекомендации    │ Прогресс              │
│ (tablist)     │                 │                       │
└───────────────┴─────────────────┴───────────────────────┘
│  Контент активной вкладки (см. выше)                    │
└─────────────────────────────────────────────────────────┘
```

**Модалка «Состав этапа»** (timeline): для пациента **ЛФК-комплекс** не показывается как один «контейнер» — в списке **отдельная строка на каждое упражнение из снимка ЛФК** (название; комментарий из снимка строки — при наличии). **Наборы тестов (`test_set`) в модалке не показываются** (как и в списке назначений на detail / во вкладке «Программа»); прохождение — на странице тестирования. В журнале **`program_action_log`** отметки **`done`** по тестам по-прежнему с ключами `:test:<uuid>`; **GET …/checklist-today** отдаёт агрегаты в **`doneTodayCountByActivityKey`** / **`lastDoneAtIsoByActivityKey`** и для тестов тоже. Зелёные точки «Сегодня» в модалке — только у строк, которые в модалке есть (упражнения ЛФК, рекомендации, уроки и т.д.); при большом числе отображаются **до 24** точек и суффикс **`+N`**, полное значение — в **`aria-label`**.

Превью в строке ЛФК — из **`exercises[].media`** в JSON-снимке (тот же формат, что у элемента **`exercise`**), поле заполняется при **`buildSnapshot`** для `lfk_complex` в [`apps/webapp/src/infra/repos/pgTreatmentProgramItemSnapshot.ts`](../../apps/webapp/src/infra/repos/pgTreatmentProgramItemSnapshot.ts). Ключ **`media`** у строки комплекса и у снимков типов **`exercise`** / **`recommendation`** в JSON **не сериализуется**, если список пустой (меньше шум в БД; клиент и так трактует отсутствие ключа как «нет превью»). Разбор строк для UI и ключей активности — [`listLfkSnapshotExerciseLines`](../../apps/webapp/src/modules/treatment-program/programActionActivityKey.ts): в объект строки попадает только непустой массив **объектов** медиа (примитивы и вложенные массивы отбрасываются). Юнит-тесты: [`programActionActivityKey.test.ts`](../../apps/webapp/src/modules/treatment-program/programActionActivityKey.test.ts). В модалке «Состав этапа»: при **есть** выбранное медиа — статичное превью (`PatientCatalogMediaStaticThumb`); при **пустом** медиа у упражнения (в т.ч. строка ЛФК и элемент **`exercise`**) — иконка гантели, у **`recommendation`** — иконка текста (`ScrollText`). **Бэклог (отдельная инициатива):** при назначении шаблона пациенту разворачивать комплекс в несколько элементов типа **`exercise`** вместо одного **`lfk_complex`** — затронет assign (`instance-service`), чек-лист, `lfk-session`, ключи журнала; нужны продуктовое решение и миграция данных.

**Цветовые зоны (из §1.1b, привязка к токенам):** hero — `patientHomeCardHeroClass` / программные бейджи из `patientHomeCardStyles`; карточка контроля — `patientSurfaceWarningClass`; этап 0 — `patientSurfaceSuccessClass`; timeline и прочие карточки — `patientCardClass` / `patientListItemClass` (прошлые строки: `bg-muted/20 opacity-70`; заблокированные будущие: `opacity-50`).

## 4. Отдельная страница этапа у пациента (§1.1b B5) — не используется

Исторически §1.1b предполагал маршрут **`/treatment-programs/[instanceId]/stages/[stageId]`**. Сейчас: **нет** отдельного `page.tsx` под этап; тело этапа — во **вкладке «Программа»** (`PatientTreatmentProgramStagePageClient` с `embedded`). Клиент по-прежнему запрашивает **`GET …/checklist-today`** для подписей «Отметок в журнале за сегодня» и модалки пункта.

## 5. Сверка: эскиз vs реальный экран vs код

Эскиз (мобильный макет) в документах **не хранится как файл** — он описан текстом в `ROADMAP_2` §1.1b.
Ниже — построчная сверка блоков эскиза, состояния кода и того, что видно на рабочем dev-экране.

> Эскиз: третий скриншот (мобильный макет).
> Dev-экран: второй скриншот (Safari, desktop dev-сервер).

| № | Блок эскиза | Описан в docs? | Есть в коде? | Виден на dev? | Зазор / примечание |
|---|-------------|----------------|--------------|---------------|--------------------|
| 1 | Hero: бейдж «Мой план» + бейдж статуса программы (даты) | §1.1b B1 ✓ | ✓ | ✓ | Номер этапа «N из K» вынесен в строку активного этапа в timeline, не в hero |
| 1 | Hero: **описание программы** под заголовком | §1.1b B1 «если есть данные» ✓ | ✓ RSC `getTemplate` + `programDescription` | при `templateId` и непустом `description` шаблона | `line-clamp-3` |
| 1 | Hero: **иллюстрация справа** (анатомия) | §1.1b B1 «опционально» ✓ | ✗ | ✗ | В hero нет декоративного `<img>` |
| 1 | Основной CTA + play-иконка | §1.1b B1 ✓ | ✓ «Начать занятие» → модалка пункта программы | ✓ | Текст отличается от раннего макета «Открыть план» |
| 1 | Hero: «● План обновлён», «Ожидает старта» | §1.1b B1 ✓ | ✓ | nudge по данным | «Ожидает старта» при `active` + рабочий этап `available` |
| 2 | Карточка «Следующий контроль»: дата, консультация, 2 CTA | §1.1b B3 ✓ | ✓ `PatientProgramControlCard` | ✗ на dev | Вкладка «Прогресс»; нужны `started_at` + `expected_duration_days` |
| 3 | Блок «Рекомендации на период» (зелёный) | §1.1b B4 ✓ | ✓ вкладка «Рекомендации» | ✗ на dev | Нужен этап с `sort_order = 0` и/или persistent на рабочем этапе |
| 4 | Этапы программы (текущий + архив) | §1.1b B5/B6 ✓ | ✓ вкладка «Прогресс» (timeline) | ✓ | Превью-карточка C4 и отдельный блок C6 **не используются** — заменены timeline + Dialog |
| 5 | **«История тестирования»** — отдельная точка входа | §1.1b B7 ✓ | ✓ секция | ✓ | CTA ведёт во вкладку «Программа» / модалку пункта (без отдельного URL этапа) |

### Итог сверки

- **Совпадает с целями §1.1b (по смыслу):** hero с программными бейджами и основным CTA; контроль и timeline на вкладке «Прогресс»; рекомендации этапа 0 / persistent — вкладка «Рекомендации»; назначения рабочего этапа — вкладка «Программа»; единая точка входа «История тестирования»; список этапов в timeline с модалкой состава; якорь **`#patient-program-current-stage`** на секции timeline.
- **Закодировано, но не видно на dev** (нет тестовых данных): карточка контроля; контент вкладок при пустых данных.
- **Сознательное отличие от раннего эскиза roadmap:** нет отдельной карточки-превью «Текущий этап» и **нет** отдельного пациентского URL этапа — контент этапа встроен во вкладку «Программа».
- **Опционально в эскизе, не в коде:** декоративная иллюстрация в hero.
- **Данные:** дата «Завершён …» у строки этапа — только если появится `completedAt` на stage (сейчас в типах нет).
- **Технический хвост:** state `testResults` обновляется при refresh, на экране не используется (зарезервировано под post-MVP).

## 6. Сводка: что считать «эталоном»

| Экран | Эталон в документах |
|-------|---------------------|
| Список | §1.1 — при **отсутствии** активной программы: empty state + `<details>` архив; при **наличии** активной — серверный `redirect` на detail инстанса (**`/app/patient/treatment/[instanceId]`**) |
| Деталь программы (визуальный финал) | **§1.1b** — по смыслу блоки §3 этого файла; в коде — **вкладки** (программа / рекомендации / прогресс) вместо одной длинной колонки |
| Работа по назначениям внутри этапа | Вкладка **«Программа»** на **`/app/patient/treatment/[instanceId]`** (`PatientTreatmentProgramStagePageClient` с `embedded`) |

MVP **§1.1a** — исполняемый шаг до эталона; структура блоков проще (всё на одной странице, архив этапов в `<details>`).
