# VIDEO_HLS_DELIVERY — Phase 10: watermark policy и производительность

## Флаг

- **`video_watermark_enabled`** (`system_settings`, scope `admin`, boolean, default **false**).
- Управление: **Настройки** → вкладка «Параметры приложения» → карточка **HLS: watermark при транскоде**.
- Воркер читает то же значение из БД (`apps/media-worker`). Без включённого **`video_hls_pipeline_enabled`** очередь транскода не используется — watermark не применяется к новым артефактам.

## Политика PII

**Выбранный вариант: без ПДн в тексте watermark.**

- На кадр наносится одна строка вида `id <uuid>` — это **`media_files.id`** (UUID медиафайла в нашей БД).
- **Не** используются: email, телефон, имя, `platform_users.id`, session id, произвольный текст из админки.
- Обоснование: при офлайн-транскоде до просмотра у воркера нет контекста пациента без передачи чувствительных данных; UUID файла достаточен для грубой привязки к записи в админке/CMS при расследовании утечек **внутри** системы. Юридическая трактовка UUID как ПДн зависит от контекста продукта; при необходимости флаг остаётся выключенным.

Техника: FFmpeg **`drawtext`** с **`textfile=`** (UTF-8 файл на диске воркера), не inline-текст из запросов.

## Шрифт (bootstrap, не секрет)

Путь к TTF задаётся **только** инфраструктурой воркера:

- env **`MEDIA_WORKER_WATERMARK_FONT`** (абсолютный путь к `.ttf`/`.ttc`), иначе поиск в типичных путях Linux (`DejaVuSans`, `Liberation Sans`).
- При включённом watermark **без** доступного шрифта job завершается **`failed`** (постоянная ошибка) с сообщением про установку шрифта или env.

## Влияние на pipeline

- Дополнительный фильтр **`drawtext`** на каждой рендер-ветке (720p, 480p) и на извлечении постера (тот же граф, что 720p при включённом watermark).
- Ориентир по CPU/времени: **+20–50%** wall-time относительно того же графа без `drawtext` (зависит от CPU, preset `veryfast`, разрешения).
- Таймаут одного вызова ffmpeg: **`min(1.45 × FFMPEG_TIMEOUT_MS, FFMPEG_TIMEOUT_MS + 45 min)`** при включённом watermark (см. `processTranscodeJob` в `apps/media-worker`).

### Benchmark wall-time (CI vs ops)

Автоматизированного замера «до/после» на эталонном ролике в **CI нет**: нужны стабильный `ffmpeg`, входной файл и воспроизводимая среда; время на shared runners не даёт жёсткого SLA. Для v1 достаточно **эвристики +20–50%** и **формулы таймаута** в коде. Численный benchmark на **staging / выделенном хосте** — опционально (finding **P10-0** в [`AUDIT_PHASE_10.md`](./AUDIT_PHASE_10.md): **DEFERRED (ops)**).

## Проверки (фаза)

- **Snapshot аргументов ffmpeg:** unit-тесты `watermarkVideoFilter.test.ts`, `workerToolkit.test.ts` (цепочка `-vf` с `scale,format,drawtext`).
- **Visual smoke (ручной, опционально):** после транскода скачать один сегмент или постер из S3 / открыть кадр в плеере и убедиться, что в правом нижнем углу читается строка `id …`. Либо локально:
  - `ffmpeg -y -ss 1 -i <source.mp4> -vf "<та же vf-цепочка что в логе воркера>" -vframes 1 /tmp/frame.jpg`
  - либо извлечь кадр из `*.ts` сегмента HLS тем же способом.

## Rollback (безопасный)

- **Выключить флаг:** **`video_watermark_enabled = false`** в админке (**Настройки → Параметры приложения → HLS: watermark**) или эквивалентный upsert в **`system_settings`** (при прямом SQL — повторить правила зеркала **`integrator.system_settings`**, см. `.cursor/rules/system-settings-integrator-mirror.mdc`).
- **Следующие транскоды:** выполняются **без** burn-in; граф **`-vf`** и таймаут совпадают с базовым pipeline (без `drawtext`, без множителя таймаута).
- **Уже выгруженные HLS и постеры** с watermark **не** удаляются и **не** перекодируются автоматически — откат **не ломает** воспроизведение существующего контента; меняется только политика для **новых** jobs.
- **Рестарт воркера** только из‑за выключения флага **не обязателен**: воркер читает флаг из БД при обработке каждого job.
