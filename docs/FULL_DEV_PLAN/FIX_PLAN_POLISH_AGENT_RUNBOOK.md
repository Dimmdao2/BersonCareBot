## 3) Явная схема чатов

### Чат 0 (этот) — Декомпозиция ✅ DONE

- Модель: opus4.6
- Результат: файлы в `EXEC/`

### Чат N (новый для каждого пакета) — Реализация

- Режим: **Auto**
- Модель: Auto default (пул подписки)
- Промпт: секция 4 ниже + ссылка на файл пакета

### Чат N (тот же) — Быстрая проверка

- После завершения пакета, **в том же чате**
- Промпт: секция 5 ниже

### Чат AUDIT (новый) — Глубокий аудит

- Модель: gpt5.4 или opus4.6
- Когда: после Pack B+C и после Pack E
- Промпт: секция 6 ниже

## 4) Промпт для Auto-агента (реализация пакета)

```text
Выполни пакет [БУКВА] из декомпозиции.

Инструкции: @docs/FULL_DEV_PLAN/EXEC/EXEC_[БУКВА]_[НАЗВАНИЕ].md
Решения владельца: @docs/FULL_DEV_PLAN/USER_TODO_STAGE.md
Архитектура: @docs/README.md, @docs/ARCHITECTURE/SERVER CONVENTIONS.md

Обязательные правила:
1) Выполняй шаги строго по порядку из файла инструкций.
2) После каждого шага: pnpm run ci
3) При FAIL:
   - исправь проблему
   - повтори pnpm run ci
   - если после 3 попыток всё ещё FAIL → СТОП
   - запиши что падает и почему в docs/FULL_DEV_PLAN/finsl_fix_report.md
4) Не переходи к следующему шагу пока текущий не зелёный.
5) После каждого шага обновляй отчёт:
   docs/FULL_DEV_PLAN/finsl_fix_report.md
6) Все решения по policy/ролям/fallback — ТОЛЬКО из USER_TODO_STAGE.md
7) Не изобретай новых требований вне контекста инструкций.
8) Не менять существующие миграции — только новые файлы.

В конце пакета:
- список всех изменённых файлов
- результаты pnpm run ci
- блокеры/риски
```

```text
Верни:

2) Какие тесты добавить/усилить

Если нашёл проблемы — исправь их все сам.

По реузльтатам обновляй отчеты, если надо

@docs/FULL_DEV_PLAN/FIX_PLAN_EXECUTION_REPORT.md @docs/FULL_DEV_PLAN/finsl_fix_report.md 
```

## 8) Чеклист перед запуском пакета

- Предыдущий пакет завершён и проверен.
- `MASTER_PLAN_EXEC.md` статус предыдущего = `done`.
- Файл инструкций пакета прочитан и понятен.
- `USER_TODO_STAGE.md` актуален.
- `pnpm run ci` зелёный перед началом.
- Ветка git чистая (нет uncommitted changes).
