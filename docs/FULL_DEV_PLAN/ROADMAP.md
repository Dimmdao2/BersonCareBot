# BersonCare — Дорожная карта

> Дата: 2026-03-22 (обновлено 2026-03-23)
> На основе: `RAW_PLAN.md` + `ANALYSIS.md` + ревью владельца продукта

---

## Общие принципы

1. **Порядок:** стабилизация → дизайн-система → доработка существующего → новые модули → интеграции → PWA → платежи.
2. **Каждый этап** — самодостаточный: после завершения можно деплоить в prod.
3. **Подэтапы** спроектированы для выполнения одним авто-агентом за одну сессию.
4. **Тесты и CI** (`pnpm run ci`) должны проходить после каждого подэтапа.

### Паттерны кодирования

| Паттерн | Где применять |
|---------|--------------|
| **Server Components by default** | Все страницы, client boundary только для интерактива |
| **Server Actions** | Мутации данных (формы, кнопки) |
| **Colocated modules** | `modules/<name>/` с `types.ts`, `actions.ts`, `components/`, `repos/` |
| **Repository pattern** | Все запросы к БД через `pg*Repo.ts` файлы |
| **Zod schemas** | Валидация на входе в каждый action/endpoint |
| **Tailwind utility classes** | Стили в компонентах, shadcn/ui для примитивов |
| **Feature flags** | `system_settings` таблица, проверка на сервере |
| **Soft-delete** | Все пользовательские данные помечаются `deleted_at`, не удаляются |
| **Idempotency** | Все мутации через integrator — idempotent |
| **Migration-per-feature** | Каждая фича = своя numbered миграция |

### Ключевые правила для агентов

1. Перед пушем — `pnpm run ci` (lint, typecheck, test, build).
2. Не менять существующие миграции — только добавлять новые.
3. Новые компоненты — в `modules/<name>/components/`, не в `shared/ui/` (кроме дизайн-системы).
4. Все тексты UI на русском языке.
5. SQL-миграции — идемпотентные (`IF NOT EXISTS`, `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`).
6. Не добавлять ORM, тяжёлые UI-фреймворки (MUI, Ant Design, Chakra).
7. **Стили: Tailwind CSS 4 + shadcn/ui.** Все новые/изменяемые компоненты — на Tailwind. Старый CSS из globals.css переводить при касании файла.
8. Медиа-файлы загружать в S3 (`S3_ENDPOINT`).
9. Новые таблицы — заранее добавлять `tenant_id` / `owner_id` (nullable) для будущей мультитенантности.
10. E2E-тесты обязательны для каждого нового функционала.
11. Для интерактивных UI-примитивов (Dialog, Popover, Tabs, Select) — использовать Radix UI primitives точечно.
12. При изменениях всегда синхронизировать документацию с реальностью.

## Тесты и проверки

### После каждого подэтапа

1. `pnpm run ci` — полный CI (lint, typecheck, test, test:webapp, webapp:typecheck, build, audit).
2. **E2E-тесты** покрывают весь новый функционал сразу.
3. Визуальная проверка на мобильном экране (375px).
4. Проверка на отсутствие регрессий в существующих страницах.
5. `@next/bundle-analyzer` — при добавлении новых библиотек проверять размер бандла.

### После каждого этапа

1. Полный прогон CI.
2. E2E smoke: `pnpm run test:webapp`.
3. Ручная проверка основных пользовательских сценариев:
   - Вход → главная → дневник → добавить запись → статистика.
   - Вход доктором → клиенты → карточка → сообщение.
4. Проверка мобильной версии (iOS Safari, Android Chrome).
5. Деплой в staging (если настроен).

### Тестирование интеграций

Для внешних интеграций (Google Calendar, Rubitime, Max, email) агент предоставляет владельцу:
- Чёткие инструкции: какое тестовое событие отправить.
- Какую команду выполнить на сервере и под каким пользователем.
- Какой вывод прислать для проверки.
- Ключи и подключения к внешним сервисам обеспечивает владелец.
