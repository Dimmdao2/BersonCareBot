## Подэтап 8.2: API сообщений

**Задача:** CRUD для диалогов и сообщений.

**Файлы:**
- Новый модуль: `apps/webapp/src/modules/messaging/`

**Действия:**
1. Server actions / API:
   - `getConversations(userId)` → список диалогов с last message preview.
   - `getConversationMessages(conversationId, cursor)` → сообщения с пагинацией.
   - `sendMessage(conversationId, text)` → сохранение + relay через integrator.
   - `markAsRead(conversationId)` → обновить `read_at`, обнулить `unread_count`.
2. Для доктора: `getUnreadCount()` → общее число непрочитанных.
3. Relay в мессенджеры: через integrator API (`POST /api/bersoncare/send-sms` или новый endpoint).

**Критерий:**
- Все API работают.
- Сообщения сохраняются в БД.
- Relay в мессенджер (через integrator).

## Подэтап 8.3: UI чата (клиент)

**Задача:** интерфейс чата для клиента.

**Файлы:**
- Новый: `apps/webapp/src/modules/messaging/components/ChatView.tsx`
- Страница: `/app/patient/messages`

**Действия:**
1. Компонент `ChatView`:
   - Пузырьки сообщений (как в Telegram):
     - Исходящие: прижаты вправо, белый фон, больший отступ слева.
     - Входящие: прижаты влево, серо-голубой полупрозрачный фон, больший отступ справа.
   - Скругленные углы (12px).
   - Дата по центру при смене дня (мелко, серым).
   - Время под сообщением (мелко, ЧЧ:ММ).
   - Скроллится вверх для просмотра истории.
   - При пустой истории: «Тут никто ничего не писал».
2. Внизу: поле ввода + кнопка отправить (кружочек со стрелкой).
3. Автопрокрутка к последнему сообщению при открытии и при отправке.

**Критерий:**
- Чат визуально соответствует описанию.
- Сообщения отправляются и отображаются.
- Автопрокрутка работает.

## Подэтап 8.7: Polling / SSE для real-time

**Задача:** автообновление сообщений.

**Файлы:**
- Новый hook: `useMessagePolling.ts`

**Действия:**
1. На странице чата: polling каждые 5 секунд (fetch новых сообщений с `since` timestamp).
2. На странице списка диалогов: polling каждые 15 секунд.
3. В шапке (unread count): polling каждые 30 секунд.
4. **Альтернатива (SSE):** если нагрузка от polling высока — перейти на Server-Sent Events через API route. Но для MVP polling достаточен.
5. Оптимистичное обновление: при отправке сообщения — показывать сразу (pending state).

**Критерий:**
- Новые сообщения появляются без перезагрузки.
- Счётчик непрочитанных обновляется.
- Нет лишних запросов при неактивной вкладке (использовать `document.visibilityState`).
