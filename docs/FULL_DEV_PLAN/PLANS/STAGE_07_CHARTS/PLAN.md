# Этап 7: Графики и статистика

> Приоритет: P2  
> Зависимости: Этап 6 (расширенные дневники), Этап 2 (дизайн-система)  
> Риск: низкий (преимущественно UI + read-only API; ошибки в агрегации влияют на отображение)

---

## Для агента в режиме «авто»: как пользоваться этим документом

1. **Выполнять один микрошаг за прогон** (например, только `7.2.2`), если явно не сказано объединить.
2. **Не трогать** этап 6 и миграции дневников без необходимости; новые миграции — только если в ревью индексов/схемы не хватает (см. 7.0).
3. **Перед кодом:** `docs/ARCHITECTURE/SERVER CONVENTIONS.md` (при БД), `README.md`, `docs/README.md`; паттерны webapp: `buildAppDeps`, `requirePatientAccess`, маршруты под `apps/webapp/src/app/api/`.
4. **Жёсткие запреты (весь этап 7):**
   - не отдавать в API статистики **чужие** `trackingId` / `complexId` (строго проверять владельца = `session.user.userId`);
   - не ослаблять авторизацию на `/api/patient/*` «для тестов»;
   - не дублировать установку **recharts**, если пакет уже в `apps/webapp/package.json` (проверить и зафиксировать в 7.1.1);
   - не писать секреты в репозиторий;
   - новые зависимости — только с обоснованием и `pnpm run ci`.
5. **Финальная проверка этапа 7:** `pnpm install --frozen-lockfile && pnpm run ci` и `pnpm --dir apps/webapp run test:e2e` (см. `.cursor/rules/pre-push-ci.mdc`).

### 7.0 Инвентаризация: recharts, БД, файлы

**Цель:** не ломать существующее и понять, что уже есть.

**Разрешено:** чтение репозитория; при необходимости — правка только этого `PLAN.md` (таблица «факт»).

**Запрещено:** менять бизнес-логику.

**Действия:**

1. Убедиться, что в `apps/webapp/package.json` есть `recharts` (уже используется в `MiniStatsChart` / `MiniStatsRecharts.tsx`).
2. Проверить индексы в миграциях для `symptom_entries(tracking_id, recorded_at)` и `lfk_sessions(complex_id, completed_at)` (или эквивалент). Если индексов нет — завести **одну** миграцию `025_*` (или следующий свободный номер после последнего в `apps/webapp/migrations/`) **только** с `CREATE INDEX IF NOT EXISTS` — без изменения семантики таблиц.
3. Зафиксировать пути: страница пациента `app/patient/page.tsx`, дневник `app/patient/diary/page.tsx`, компоненты `modules/diaries/components/*`.

**Проверки конца шага:** список файлов и решение «миграция индексов нужна / не нужна» записано в задаче или в таблице ниже.

**Критерий успеха:** нет противоречий с текущим кодом; номер новой миграции не конфликтует с существующими.

**Факт (ревью кода):** `recharts` в `apps/webapp/package.json`; индексы в `004_symptom_trackings_and_entries.sql` и `005_lfk_complexes_and_sessions.sql` уже покрывают выборки по `(tracking_id, recorded_at)` и `(complex_id, completed_at)`; отдельная миграция только с индексами **не** требовалась.

### 7.1.1 Аудит recharts и паттерн загрузки

**Цель:** единообразный lazy-import, без дублирования тяжёлого бандла на SSR.

**Разрешено:** `apps/webapp/src/modules/diaries/components/**`, небольшой рефактор без смены поведения.

**Запрещено:** удалять `MiniStats`; ломать главную пациента.

**Действия:**

1. Зафиксировать паттерн: `next/dynamic` + `{ ssr: false }` + `loading` skeleton (как в `MiniStatsChart.tsx`).
2. Для новых графиков **не** импортировать recharts в серверных компонентах.
3. Если `ChartWrapper` из старого черновика плана не нужен — не создавать файл ради файла; при дублировании 3+ раз — вынести общий wrapper.

**Тесты:** e2e in-process — импорт `MiniStats` / `MiniStatsChart` по-прежнему успешен (обновить при переносах).

**Проверки:** `pnpm run ci`.

**Критерий успеха:** главная и существующие графики работают; документирован паттерн для `SymptomChart`.

**Паттерн (факт):** `next/dynamic(() => import("…Recharts"), { ssr: false, loading: skeleton })` — `MiniStatsChart.tsx`, `SymptomChart.tsx` (линия симптомов); recharts не импортируется в серверных компонентах страниц.

### 7.2.1 Функции агрегации (чистая логика)

**Цель:** детерминированная агрегация для тестов без БД.

**Разрешено:** `apps/webapp/src/modules/diaries/**` (например `stats/` или `aggregation.ts`), без HTTP.

**Запрещено:** импортировать React.

**Действия:**

- Вход: массив записей (симптом) с `recordedAt`, `value0_10`, `entryType`; период `week | month | all`, `offset`.
- Выход: массив точек `{ date: string (ISO date), value: number, entryType?: ... }` по **дням** (или согласованное правило: max за день / последняя запись — зафиксировать в JSDoc и тестах).
- Аналогично для ЛФК: по дням `completed`, `durationMinutes`, `difficulty`, `pain` из полей сессий (nullable обработать).

**Тесты (обязательно):** `*.test.ts` рядом или в `modules/diaries/**/__tests__` — краевые случаи: пустой массив, один день, смена offset, граница периода.

**Проверки:** `pnpm --dir apps/webapp test` (затрагиваемые файлы), `pnpm run ci`.

**Критерий успеха:** unit-тесты зелёные; поведение задокументировано в комментарии к функции.

### 7.2.2 HTTP: `GET /api/patient/diary/symptom-stats`

**Цель:** JSON для графика симптомов только для владельца.

**Контракт (базовый):**

- Query: `trackingId` (обязателен), `period=week|month|all`, `offset` (целое, по умолчанию 0).
- Ответ: `{ ok: true, points: [...] }` (имена полей согласовать с `SymptomChart`).

**Разрешено:** `apps/webapp/src/app/api/patient/diary/symptom-stats/route.ts`, guards, zod/query parsing.

**Запрещено:** отдавать данные без `getCurrentSession` + роль пациента + `tracking.user_id === session.user.userId`.

**Тесты (обязательно):** `route.test.ts` — мок сессии: 401 без сессии; 403/404 для чужого `trackingId`; 200 для своего; проверка границ `period`/`offset`.

**Проверки:** `pnpm run ci`.

**Критерий успеха:** маршрут покрыт тестами; нет утечки чужих данных.
