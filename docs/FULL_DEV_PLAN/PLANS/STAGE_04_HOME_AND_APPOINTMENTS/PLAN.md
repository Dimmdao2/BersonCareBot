> Приоритет: P1

---

## Достаточность декомпозиции (для агентов в режиме «авто»)

Исходный текст этапа 4 был **крупнозернистым**: без явных якорей в репозитории, без порядка микро-шагов и без фиксации расхождений «план ↔ текущий код». Для младшего агента это приводит к догадкам и половинчатым PR.

Ниже добавлены блоки в стиле этапа 3: **ограничения**, **якоря**, **порядок**, **микро-задачи с DoD**, **когда остановиться**. Выполнять **сверху вниз** в пределах выбранного подэтапа, после каждого логического куска — **`pnpm run ci`**.

## Важно (ограничения)

- **Tailwind + shadcn + `cn()`** — не плодить одноразовые глобальные классы в `globals.css`; локальные исключения — согласовать с этапом 2.
- **Гость на `/app/patient`**: текущая страница доступна без входа — сохранить контракт: гость видит допустимые блоки; блоки «только для авторизованных» оборачивать проверкой сессии (как сейчас с `ConnectMessengersBlock`).
- **Переиспользование**: для привязки телефона в проекте уже есть **`PatientBindPhoneSection`** (`apps/webapp/src/app/app/patient/PatientBindPhoneSection.tsx`) и **`BindPhoneBlock`**. В тексте плана ниже встречается имя `BindPhoneBlock` — в коде ориентироваться на **тот же паттерн**, что в кабинете/уведомлениях (`PatientBindPhoneSection` + нужные props), если не требуется именно общий `BindPhoneBlock`.
- Перед пушем: **`pnpm run ci`** (полный пайплайн монорепо, см. `.cursor/rules/pre-push-ci.mdc`).
- Тексты UI — **на русском**.

## Подэтап 4.1: Главная — реструктуризация блоков

**Задача:** главная страница клиента по плану.

**Файлы:**
- `apps/webapp/src/app/app/patient/page.tsx`
- `apps/webapp/src/app/app/patient/cabinet/page.tsx`
- `apps/webapp/src/shared/ui/FeatureCard.tsx`

**Действия:**
1. Реструктурировать главную страницу. Блоки сверху вниз:
   - **Кабинет:** карточки (Дневник, Мои записи, Мои комплексы ЛФК — скрыт).
   - **Уроки:** карточки (Скорая помощь, Разминки, Тренировки, Полезные материалы).
   - **Новости:** InfoBlock (видим только при наличии активной новости). Красный при `importance = 'important'`.
   - **Уведомления:** список рассылок с кнопкой «Просмотрено».
   - **Мотивашка:** блок с приятным фоном, текст без заголовка, меняется ежедневно.
   - **Статистика:** мини-график (заглушка до этапа 7).
2. Использовать `FeatureCard` для карточек кабинета и уроков.
3. Блоки новостей и уведомлений: скрыты при отсутствии данных.

**Критерий:**
- Все блоки на месте в правильном порядке.
- Скрытые блоки не отображаются при пустых данных.
- FeatureCard переиспользуется.

**Микро-задачи (выполнять по порядку; одна ≈ один коммит):**

| ID | Действие | DoD (готово, когда…) |
|----|-----------|----------------------|
| **4.1a** | Вынести разметку секций в отдельные компоненты в `apps/webapp/src/app/app/patient/` или `shared/ui/patient/` (например `PatientHomeCabinetSection`, `PatientHomeLessonsSection`, …) — **без** смены бизнес-логики | Страница компилируется; визуально порядок секций соответствует списку в п. «Действия» выше |
| **4.1b** | **Новости:** найти источник «активной новости» (проекции рассылок, CMS, env — только то, что уже есть в репо) или добавить тонкий порт + заглушку `null` | Если данных нет — секция **не рендерится**; если есть — `InfoBlock` с `variant="important"` при важности |
| **4.1c** | **Уведомления на главной:** список последних рассылок пользователя + кнопка «Просмотрено» (если нет готового API — сначала заглушка с `empty`, скрытая секция) | Пустой список → секция скрыта |
| **4.1d** | **Мотивашка:** текст без заголовка, смена по дням (детерминированно: `userId + date` или календарный день UTC — зафиксировать в комментарии) | Один и тот же пользователь в течение суток видит один текст |
| **4.1e** | **Статистика:** до этапа 7 — вставить **заглушку-плейсхолдер** (серый блок) или связать с будущим `MiniStats` из 4.4 | Не ломает гостевой режим |
| **4.1f** | Стабильные `id` на секции для e2e (по конвенции проекта) | Линтер чистый |

После **4.1a–4.1f** — `pnpm run ci`.

## Подэтап 4.2: Экран «Мои записи»

**Задача:** переработать экран записей на прием.

**Файлы:**
- `apps/webapp/src/app/app/patient/cabinet/page.tsx` (или отдельная страница)
- `apps/webapp/src/modules/doctor-appointments/`
- `apps/webapp/src/app/globals.css`

**Действия:**
1. Убрать: заголовок «Кабинет клиента», текст про «подключение к системе», заголовок «Ближайшие записи».
2. Логика по состоянию:
   - **Нет телефона:** InfoBlock «Для отображения записей на прием привяжите номер телефона» + `<BindPhoneBlock />`.
   - **Есть телефон, нет записей:** обычный текст «У вас нет активных записей».
   - **Есть записи:** список с цветными статусами.
3. Статусы записей (badge справа):
   - `created` / подтверждён → зелёный.
   - Перенос → сиреневый.
   - Отменён → тёмно-красный (при нажатии — причина отмены tooltip).
4. Каждая запись — ссылка (из `payload_json` или стандартизированный URL).
5. Кнопка «Записаться» по центру → виджет Rubitime:
   - Загрузить через `<iframe src="https://dmitryberson.rubitime.ru/widget">`.
   - Обернуть в компонент `RubitimeWidget`.
   - Если iframe конфликтует с Next.js — fallback: простая ссылка на страницу записи.
6. Информационный блок после кнопки: «Адрес» (ссылка) + «Как подготовиться к приему».
7. Блок «История записей» — прошедшие записи, таблица с чередованием фона строк.

**Критерий:**
- Экран записей чистый, без мусорных заголовков.
- Статусы цветные.
- Виджет Rubitime открывается.
- История отображается.

**Микро-задачи:**

| ID | Действие | DoD |
|----|-----------|-----|
| **4.2a** | Убрать лишние заголовки/тексты из `cabinet/page.tsx` (см. п. 1 «Действия») | Нет строк «Кабинет клиента» / «подключение к системе» / «Ближайшие записи» в финальной вёрстке |
| **4.2b** | Ветка **нет телефона:** `InfoBlock` + тот же паттерн привязки, что на кабинете сейчас (`PatientBindPhoneSection` или эквивалент) | Поведение согласовано с `hasMessengerBinding` / телефоном из сессии |
| **4.2c** | Ветка **есть телефон, нет записей** / **есть записи** — тексты и список | Пустой список ≠ ошибка загрузки |
| **4.2d** | **Типы записей:** расширить `AppointmentSummary` (или отдельный DTO) полями статуса, `payload_json`, причины отмены — **откуда брать** — см. `patientCabinet` / integrator; если данных нет — расширить мок в `modules/appointments/service.ts` с пометкой `TODO: Rubitime` | Badge справа с цветами из плана; tooltip на отменённых |
| **4.2e** | Компонент **`RubitimeWidget`**: client component, `iframe` + `fallback` `<a href="…">` | В dev/prod открывается хотя бы fallback при блокировке iframe |
| **4.2f** | Блок «Адрес» + «Как подготовиться» (константы или env **без секретов** в коде) | Ссылки `https://` с `rel` |
| **4.2g** | **История:** discovery — есть ли прошлые записи в БД/API; если нет — таблица с `empty` + issue/TODO | Не падать на `undefined` |

После каждых **4.2a–4.2c** и **4.2d–4.2g** — `pnpm run ci`.

## Чеклист режима «авто» (кратко)

1. Прочитать разделы **Важно**, **Контекст**, **План ↔ код**.
2. Выбрать **один** подэтап (4.1 … 4.4); не смешивать крупные рефакторинги разных экранов без необходимости.
3. Выполнить **микро-задачи по таблице**; после каждого логического блока — **`pnpm run ci`**.
4. Зафиксировать в описании PR: **стратегия меню** (секции по `id`), **источник новостей/истории записей**, **fallback Rubitime**.
5. Если обязательное условие невыполнимо без прод-данных — **скрытый блок** или **TODO** + задача в backlog, без «вранья» в UI.
