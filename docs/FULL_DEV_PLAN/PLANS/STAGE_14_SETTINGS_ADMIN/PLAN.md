## Подэтап 14.1 — База данных: `system_settings`

### Шаг 14.1.1 — Выбор номера миграции и отсутствие коллизий

1. **Цель шага:** Файл миграции не пересекается с уже применёнными в `apps/webapp/migrations/`.
2. **Точная область изменений:** Каталог `apps/webapp/migrations/` (список файлов), новый файл `027_system_settings.sql` **только если** `027_*.sql` ещё не существует; иначе взять следующий свободный номер.
3. **Конкретные действия:** Выполнить `ls apps/webapp/migrations/*.sql` и зафиксировать следующий номер; переименовать целевой файл миграции в соответствии.
4. **Проверки после шага:** Нет двух файлов с одним числовым префиксом.
5. **Критерий успешного выполнения:** Имя файла соответствует принятой нумерации репозитория.
6. **Тесты:** E2E не нужен. Автотестов на миграции может не быть — ручная проверка применения.
7. **Обновление документации:** Упомянуть номер миграции в changelog этапа (внутренний тикет) или в этом плане в шаге 14.1.2.

### Шаг 14.1.2 — Создание таблицы и seed

1. **Цель шага:** Хранить ключи настроек с областью видимости и JSON-значением.
2. **Точная область изменений:** Новый SQL-файл в `apps/webapp/migrations/` (см. 14.1.1), ссылка на `platform_users(id)` как в черновике ниже.
3. **Конкретные действия:** Создать `CREATE TABLE IF NOT EXISTS system_settings` с колонками: `key TEXT NOT NULL`, `scope TEXT NOT NULL DEFAULT 'global'` с `CHECK (scope IN ('global', 'doctor', 'admin'))`, `value_json JSONB NOT NULL DEFAULT '{}'`, `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()`, `updated_by UUID REFERENCES platform_users(id)`, `PRIMARY KEY (key, scope)`; seed строки: `patient_label` (scope `doctor`), `sms_fallback_enabled` (scope `admin`), `debug_forward_to_admin` (scope `admin`), `dev_mode` (scope `admin`) с значениями как в предыдущей версии плана этапа; использовать `ON CONFLICT DO NOTHING` для idempotent seed.
4. **Проверки после шага:** `pnpm` скрипт миграций (как принято в проекте) применяется на чистой dev-БД без ошибки.
5. **Критерий успешного выполнения:** В `\d system_settings` или аналоге видна таблица и seed-строки.
6. **Тесты:** Integration: тест репозитория с in-memory PG или существующий паттерн миграций в webapp — если есть; иначе ручная проверка. E2E не нужен.
7. **Обновление документации:** Несекретно: описать схему в комментарии к модулю репозитория настроек (см. 14.2).

## Подэтап 14.2 — Репозиторий и API: чтение/запись настроек

### Шаг 14.2.1 — Порты и PG-реализация

1. **Цель шага:** Доменный слой читает настройки без прямого SQL в route handlers.
2. **Точная область изменений:** Новый модуль в `apps/webapp/src/modules/` (например `system-settings/` с `ports.ts`, `service.ts`) и `apps/webapp/src/infra/repos/pgSystemSettings.ts` (имя согласовать с существующими `pg*.ts`); регистрация в `apps/webapp/src/app-layer/di/buildAppDeps.ts`.
3. **Конкретные действия:** Реализовать `getSettings(scope)`, `updateSetting(key, scope, value, updatedBy)` с валидацией ключей из whitelist; использовать пул из `@/infra/db/client`.
4. **Проверки после шага:** Unit-тест сервиса с моком репозитория.
5. **Критерий успешного выполнения:** Невозможно записать произвольный ключ без whitelist.
6. **Тесты:** Unit обязательны в `service.test.ts`. Integration с PG — опционально в CI. E2E не нужен.
7. **Обновление документации:** JSDoc на порт 1–2 строки.

### Шаг 14.2.2 — HTTP API для врача и админа

1. **Цель шага:** Клиентские компоненты получают настройки через signed session + role guard.
2. **Точная область изменений:** Новые маршруты под `apps/webapp/src/app/api/` (например `apps/webapp/src/app/api/doctor/settings/route.ts` и/или `apps/webapp/src/app/api/admin/settings/route.ts`); использовать `apps/webapp/src/app-layer/guards/requireRole.ts` для `doctor` | `admin` как требует сценарий.
3. **Конкретные действия:** GET возвращает безопасный срез JSON (без секретов интеграций на этом шаге); PATCH/PUT принимает только разрешённые ключи; ответы с кодами 400/403/401 согласованы с остальным API webapp.
4. **Проверки после шага:** Запрос без сессии — 401; пациент — 403 на admin-only endpoint.
5. **Критерий успешного выполнения:** Совпадение с контрактом, используемым UI.
6. **Тесты:** Integration: тест route с in-process app deps (паттерн `buildAppDeps.test.ts`). E2E опционально.
7. **Обновление документации:** Не требуется внешний markdown, если API внутренний; при публичном контракте — добавить в `docs/` по запросу.

## Подэтап 14.4 — Режим администратора (UI + сессия)

### Шаг 14.4.1 — Флаг admin mode в сессии или cookie

1. **Цель шага:** «Режим админа» — явное состояние, не только `role === admin`.
2. **Точная область изменений:** `apps/webapp/src/modules/auth/service.ts` или аналог сессии, `cookies`, server actions для toggle; типы в `@/shared/types/session`.
3. **Конкретные действия:** Только пользователь с `role === 'admin'` может установить `adminMode: true`; хранить в защищённой cookie или в сессионной записи БД — согласовать с существующей auth-архитектурой этапа 5; при logout сбрасывать.
4. **Проверки после шага:** Пользователь с ролью doctor не видит toggle; admin видит toggle.
5. **Критерий успешного выполнения:** Невозможно включить admin mode без роли admin.
6. **Тесты:** Unit тест guard + integration тест cookie. E2E не обязателен.
7. **Обновление документации:** Комментарий в guard-файле.

### Шаг 14.4.2 — Подтверждение включения и визуальная индикация в шапке

1. **Цель шага:** Включение режима требует подтверждения; шапка визуально отличается.
2. **Точная область изменений:** `apps/webapp/src/shared/ui/DoctorHeader.tsx`, `apps/webapp/src/shared/ui/AppShell.tsx`, страница настроек (admin секция); CSS-переменная `var(--admin-header-bg)` или эквивалент в глобальных стилях проекта.
3. **Конкретные действия:** Modal confirm (shadcn `AlertDialog` если используется в проекте); при `adminMode` — класс фона шапки на красноватый токен дизайн-системы этапа 2.
4. **Проверки после шага:** Включение без подтверждения не меняет состояние; выключение возвращает обычный фон.
5. **Критерий успешного выполнения:** Визуально отличимо на скриншоте до/после.
6. **Тесты:** Unit: компонент modal; snapshot опционально. E2E не обязателен.
7. **Обновление документации:** Не требуется.

### Шаг 14.4.3 — Скрытие admin-only действий до включения режима

1. **Цель шага:** Опасные действия и блоки настроек не видны при `adminMode === false`.
2. **Точная область изменений:** Те же компоненты, что в 14.5; условный рендер по `adminMode` из сервера/клиента.
3. **Конкретные действия:** Централизовать проверку в маленьком хуке или helper `useAdminMode()` / `getAdminMode()`; не дублировать логику в 10 местах без необходимости.
4. **Проверки после шага:** С `adminMode` false — сетевых вызовов к admin API на скрытые кнопки нет (проверка в devtools).
5. **Критерий успешного выполнения:** Прямой вызов API admin без роли/режима по-прежнему отдаёт 403 (защита на сервере обязательна).
6. **Тесты:** Integration тест API 403; unit тест условного рендера. E2E не обязателен.
7. **Обновление документации:** Не требуется.

### Шаг 14.5.2 — Связь `dev_mode` и рассылок с integrator

1. **Цель шага:** Флаг в webapp влияет на исходящие рассылки согласованно с `apps/integrator` (если рассылки идут через integrator).
2. **Точная область изменений:** Точка входа рассылок в webapp (`apps/webapp/src/modules/` — поиск `mailing`, `reminder`, `integrator`); клиент `apps/webapp/src/infra/adapters/webappEventsClient.ts` не трогать без задачи; вместо этого: либо integrator опрашивает webapp, либо webapp не ставит задачи в очередь при `dev_mode` — выбрать архитектуру и реализовать.
3. **Конкретные действия:** При `dev_mode === true` блокировать отправку для всех пользователей кроме whitelist (id из `integration_test_ids`); whitelist хранить в `system_settings`.
4. **Проверки после шага:** На staging с `dev_mode` true реальное сообщение не уходит не-whitelist пользователю (проверка логов integrator).
5. **Критерий успешного выполнения:** Нет расхождения: UI говорит «off», а фоновые job всё ещё шлют.
6. **Тесты:** Unit: функция `shouldDispatch(userId)`; integration: опционально. E2E не обязателен.
7. **Обновление документации:** Комментарий в коде scheduler + ссылка на ключ `dev_mode`.
