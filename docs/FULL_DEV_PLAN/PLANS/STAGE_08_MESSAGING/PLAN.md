# Этап 8: Модуль сообщений

> Приоритет: P2  
> Зависимости: Этап 2 (дизайн-система), существующая проекция поддержки (`009_support_communication_history.sql`, `pgSupportCommunication`)  
> Риск: средний (синхронизация с integrator, polling, кросс-роли пациент/врач)

---

## Для агента в режиме «авто»: как пользоваться этим документом

1. **Выполнять один микрошаг за прогон** (например, только `8.2.2`), если явно не сказано объединить.
2. **Перед кодом:** `docs/ARCHITECTURE/SERVER CONVENTIONS.md`, `README.md`, `docs/README.md`; паттерны webapp: `buildAppDeps`, `requirePatientAccess` / `requireDoctorAccess` (или аналог из `app-layer/guards`), маршруты API под `apps/webapp/src/app/api/`.
3. **Номер миграции:** брать **следующий свободный** после максимального `NNN_*.sql` в `apps/webapp/migrations/` (на момент ревью плана последний был `024_*` → новая **`025_*`**). Не использовать имя `021_chat_messages.sql` — **`021_` уже занят в репозитории**.
4. **Жёсткие запреты (весь этап 8):**
   - не отдавать в API **чужие** `conversationId` / `platform_user_id` (проверка владельца / роли врача по доменным правилам);
   - не ослаблять авторизацию на `/api/patient/*` и `/api/doctor/*` «для тестов»;
   - не писать секреты и токены integrator в репозиторий;
   - не дублировать тяжёлый SQL в `route.ts` — выносить в `infra/repos` / сервисы;
   - новые зависимости npm — только с обоснованием и `pnpm run ci`;
   - не ломать существующую доставку событий из integrator в `pgSupportCommunication` без миграции и тестов.
5. **Финальная проверка этапа 8:** `pnpm install --frozen-lockfile && pnpm run ci` и `pnpm --dir apps/webapp run test:e2e` (см. `.cursor/rules/pre-push-ci.mdc`).

### 8.0 Инвентаризация

**Цель:** не сломать существующую модель поддержки и зафиксировать факты.

**Разрешено:** чтение репозитория; правка только этого `PLAN.md` (таблица «факт»).

**Запрещено:** менять бизнес-логику и схему БД до шага 8.1.2.

**Действия:**

1. Прочитать `apps/webapp/migrations/009_support_communication_history.sql`: таблицы `support_conversations`, `support_conversation_messages` (поля, `sender_role`, `integrator_message_id NOT NULL UNIQUE`).
2. Прочитать `apps/webapp/src/infra/repos/pgSupportCommunication.ts` (или актуальный путь) — как вставляются сообщения из integrator.
3. Зафиксировать: страница `apps/webapp/src/app/app/patient/messages/page.tsx` сейчас — **заглушка** (`PlaceholderPage`).
4. Записать **следующий номер миграции** (например `025_` если `024_` последний).
5. **Критично:** если webapp будет создавать сообщения без integrator, нужно решение для `integrator_message_id` (например префикс `webapp:<uuid>`, отдельная колонка, или смена ограничения в миграции). Зафиксировать решение в задаче или в подразделе «Решение по 8.1.1».

**Проверки конца шага:** таблица фактов заполнена в этом `PLAN.md` или в комментарии к PR.

**Критерий успеха:** нет противоречий с текущим кодом; номер миграции не конфликтует с существующими.

### 8.1.1 Проектирование расширения схемы

**Цель:** явный контракт полей до написания SQL.

**Разрешено:** документирование; обсуждение в PR.

**Запрещено:** удалять таблицы `support_*`; менять семантику `integrator_conversation_id` без миграции данных.

**Пример направления (уточнить при реализации):**

- `support_conversation_messages`: `media_url`, `media_type`, `read_at`, `delivered_at` (как в старом черновике этапа 8).
- `support_conversations`: `unread_count_user`, `unread_count_admin` (или считать агрегатом — если считать агрегатом, миграция счётчиков не обязательна; зафиксировать выбор).

**Проверки:** ревью схемы вторым глазом (или self-check по чеклисту ниже).

**Критерий успеха:** решение по `integrator_message_id` для исходящих из webapp задокументировано.

---

### 8.1.2 Миграция БД

**Цель:** одна миграция `025_*_support_messaging*.sql` (имя по смыслу).

**Разрешено:** `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, при необходимости — снятие/замена ограничения на `integrator_message_id` **согласно 8.1.1**.

**Запрещено:** `DROP TABLE support_*`; миграции без `IF NOT EXISTS` там, где возможны повторные прогоны.

**Индексы (пример):** частичный индекс на непрочитанное по `(conversation_id)` WHERE `read_at IS NULL` — если поле добавлено.

**Тесты:** нет unit на SQL; проверка — `pnpm run migrate` (или скрипт проекта) на чистой/тестовой БД.

**Проверки конца шага:** `pnpm run ci` (если CI не гоняет migrate — хотя бы локально применить миграцию).

**Критерий успеха:** миграция применяется idempotently; типы колонок согласованы с TypeScript.

### 8.2.1 Репозиторий / порт

**Цель:** доступ к новым полям и операциям list/send/markRead без дублирования SQL в route.

**Разрешено:** `apps/webapp/src/infra/repos/pgSupportCommunication.ts`, соответствующие порты в `modules/*`.

**Запрещено:** копировать 100+ строк SQL в два API-файла.

**Тесты (обязательно):** `*.test.ts` рядом с репозиторием или в `modules/**/__tests__` — моки `pg` pool / существующий паттерн проекта.

**Проверки:** `pnpm --dir apps/webapp test` (затронутые файлы), `pnpm run ci`.

**Критерий успеха:** вызовы методов покрыты тестами с моками; нет регрессии вставки из integrator (если есть тесты — обновить).

### 8.2.2 Доменный сервис сообщений

**Цель:** оркестрация: валидация текста, обновление счётчиков, вызов репо.

**Разрешено:** `apps/webapp/src/modules/messaging/**` (или расширение существующего модуля поддержки).

**Запрещено:** импортировать React в сервисе.

**Тесты (обязательно):** `service.test.ts` — мок порта; валидация длины, пустого текста, ошибки репо.

**Проверки:** `pnpm run ci`.

**Критерий успеха:** сервис покрыт тестами; контракт методов описан в JSDoc.

### 8.2.3 `buildAppDeps`

**Цель:** единая точка входа для страниц и API.

**Разрешено:** `apps/webapp/src/app-layer/di/buildAppDeps.ts`, типы.

**Запрещено:** импортировать `buildAppDeps` из `modules/*` (кроме composition root — см. правила проекта).

**Тесты:** `buildAppDeps.test.ts` — наличие новых методов.

**Проверки:** `pnpm run ci`.

**Критерий успеха:** `deps.messaging` (или имя согласовано) доступен.

### 8.3.1 API пациента

**Цель:** HTTP-контракт для списка диалогов, сообщений, отправки, mark read.

**Разрешено:** `apps/webapp/src/app/api/patient/.../route.ts`, zod, `getCurrentSession` + роль пациента.

**Запрещено:** отдавать данные без проверки, что `conversation` принадлежит пользователю (по `platform_user_id` или аналогу в вашей модели).

**Тесты (обязательно):** `route.test.ts` — 401 без сессии; 403 не пациент; 404 чужой диалог; 200 свой кейс; границы query (cursor, limit).

**Проверки:** `pnpm run ci`.

**Критерий успеха:** маршруты покрыты тестами; нет утечки чужих диалогов.

---

### 8.3.2 API врача

**Цель:** список диалогов и переписка с проверкой роли врача.

**Разрешено:** `apps/webapp/src/app/api/doctor/.../route.ts`, guards существующего кабинета врача.

**Запрещено:** открывать все диалоги без фильтра домена (если в продукте врач видит только своих клиентов — зафиксировать правило в коде и тесте).

**Тесты (обязательно):** `route.test.ts` — 401/403/200 по ролям.

**Проверки:** `pnpm run ci`.

**Критерий успеха:** тесты зелёные; JSDoc описывает контракт.

---

### 8.3.3 Relay / integrator

**Цель:** доставка исходящего в каналы (или задел под неё).

**Разрешено:** вызов существующего integrator client из `apps/integrator` или webapp adapter; feature-flag `env`.

**Запрещено:** хардкод секретов; «тихий» no-op в prod без логирования/метрики (минимум `console.warn` или структурированный лог).

**Тесты:** unit на ветку «relay отключён»; интеграционный тест с реальным integrator — опционально.

**Проверки:** `pnpm run ci`.

**Критерий успеха:** при отключённом relay сообщения всё равно сохраняются в БД (если так задумано).

### 8.4.1–8.4.3 UI пациента (`ChatView` + страница)

**Цель:** пузырьки, даты, скролл, пустое состояние — по дизайн-системе этапа 2 (классы `panel`, `auth-input`, токены темы).

**Разрешено:** `apps/webapp/src/modules/messaging/components/**`, `apps/webapp/src/app/app/patient/messages/page.tsx`.

**Запрещено:** хардкод цветов вне `hsl(var(--...))` / утилит Tailwind темы; ломать `AppShell`.

**Тесты:** минимум один unit на чистую функцию форматирования даты/группировки по дням; e2e in-process — импорт `ChatView`.

**Проверки:** `pnpm run ci`, ручной smoke опционален.

**Критерий успеха:** страница `/app/patient/messages` не заглушка; сообщения отображаются после отправки (happy path).

---

### 8.5.1–8.5.2 UI врача

**Цель:** список диалогов + переписка; непрочитанные сверху; сортировка `last_message_at DESC`.

**Разрешено:** `apps/webapp/src/app/app/doctor/messages/page.tsx`, компоненты рядом.

**Запрещено:** дублировать всю логику API в клиенте — только `fetch` к 8.3.2.

**Тесты:** unit на сортировку/merge списка на фикстурах (pure function); e2e — импорт модулей.

**Проверки:** `pnpm run ci`.

**Критерий успеха:** врач видит список и переписку; метки канала — если есть в данных.

---

### 8.6.1–8.6.2 FAB

**Цель:** спокойный FAB, переход в чат или открытие панели (один вариант в PR).

**Разрешено:** `apps/webapp/src/shared/ui/AskQuestionFAB.tsx`, CSS в `globals.css` только через токены.

**Запрещено:** ломать существующий POST `/api/patient/question` без миграции сценария «вопрос админу».

**Тесты:** e2e in-process — компонент импортируется.

**Проверки:** `pnpm run ci`.

**Критерий успеха:** клик ведёт в сообщения или открывает панель — поведение задокументировано в JSDoc компонента.

---

### 8.7.1–8.7.2 Шапки + polling

**Цель:** бейдж непрочитанных; polling с учётом `visibilityState`.

**Разрешено:** `PatientHeader.tsx`, `DoctorHeader.tsx`, хук `useMessagePolling.ts`.

**Запрещено:** polling каждые 100 мс; шторм запросов при неактивной вкладке.

**Тесты:** unit на хук с моком `document.visibilityState` и таймеров (`vi.useFakeTimers` если принято в проекте).

**Проверки:** `pnpm run ci`.

**Критерий успеха:** при скрытой вкладке интервал увеличивается или запросы останавливаются (как задокументировано в комментарии к хуку).

### 8.8 E2E in-process

**Цель:** регрессия импорта и минимальный smoke.

**Разрешено:** `apps/webapp/e2e/*` — новый файл или расширение существующего.

**Действия:** `import()` модуля `ChatView`, `ConversationList`, API routes (без реального HTTP-сервера), навигация по `routePaths` если есть паттерн.

**Проверки:** `pnpm --dir apps/webapp run test:e2e`.

**Критерий успеха:** e2e зелёные.

**Проверки:** `pnpm install --frozen-lockfile && pnpm run ci` и `pnpm --dir apps/webapp run test:e2e`.
