## 1. Зафиксированные решения до реализации

### B1 — `test_sets` тоже получают ось публикации

**Решение (обновлено):**

- В B1 **сразу** делаем разделение «архив» и «публикация» для `test_sets`:
  - фильтр архива: `active | archived`;
  - фильтр публикации: `all | draft | published`.
- UI: рядом с полем `active/archived` добавляется поле `all/draft/published` для каталогов, где есть публикация.
- Для `test_sets` это включает **миграцию схемы** (добавление статуса публикации) в рамках B1.

### B6 vs A — проверка фактического состояния кода перед стартом

**Решение (обновлено):**

- Перед EXEC B6 обязательно выполнить короткий code-state check по конструктору шаблонов **после завершения параллельной фазы A**.
- На основании этого check фиксируется актуальный scope B6 в `LOG.md` (что реально уже есть от A, что трогаем в B6).
- B6 остаётся UX-pass этапом (layout/preview/CTA), без произвольного расширения доменной логики.

### Q5 — UUID fallback удаляем

**Решение (обновлено):**

- Toggle «Сырой режим» / UUID-textarea в B3 **не оставляем**.
- Старый UUID-input удаляется из пользовательского и админского сценария; editor работает только через нормальный UI.

### B4 / Q3-Q4

- Q4: в B4 остаётся колонка/код `domain`, UI-подпись «Тип»; отдельным этапом допускается переименование `domain` → `kind` после оценки объёма.
- Q3: без «слияния» legacy значений в той же миграции; расширение значений и совместимость — аккуратно, без массовой нормализации исторических записей в B4.

### Q1 / Q2 / Q6 (уточнения 2026-05-03)

- Q1: `assessmentKind` переводим в системный справочник БД (редактируемый), текущий enum — стартовый канон для сида/миграции.
- Q2: для `qualitative` в инстансе сценарий прохождения такой же, как для остальных типов: результат отмечен -> этапный прогресс обновлён -> открывается следующий этап.
- Q6: первый шаг — UI-доступ к системному справочнику `measure_kinds`; merge/dedup и «тяжёлая» модерация откладываются.

### B7 / Q7 и patient scope

- Q7: отдельный template-comment у **каталога** рекомендации не вводим; комментарий остаётся контекстным (template/instance item), `body_md` — описание.
- Формулировка «v1 без широкого patient Plan UI» означает: в B7 обязательны doctor-редактирование + copy template→instance + корректный read API; полная отдельная пациентская переработка экранов Plan не является обязательной частью B7, если она не требуется для закрытия DoD.

### B7 — ЛФК instance

- Для `lfk_complex_exercises` добавить `local_comment` (nullable) и семантику override поверх template `comment`.
