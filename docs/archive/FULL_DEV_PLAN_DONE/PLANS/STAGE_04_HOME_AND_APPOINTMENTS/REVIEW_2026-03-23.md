# Ревью этапа 4 (качество и готовность к этапу 5)

**Дата:** 2026-03-23

## Выявленные проблемы (исправлено в коде)

| Область | Проблема | Исправление |
|---------|-----------|-------------|
| **Безопасность** | Server action `setChannelNotificationEnabled` принимал типизированный `ChannelCode` без runtime-проверки; можно было обновить `user_channel_preferences` для канала **без привязки** (например Telegram) | Zod-валидация `parseChannelPreferenceInput`; отказ, если `!card.isLinked`; общее сообщение об ошибке без утечки `Error.message` |
| **UX / честность данных** | Мок `getUpcomingAppointments` без БД всегда показывал 2 записи — «пустое состояние» не проверялось | Пустой массив по умолчанию; реальные записи — через `DATABASE_URL` + projection |
| **Rubitime** | `onError` на `<iframe>` почти не срабатывает в браузерах; лишний state | Убраны мёртвые ветки; fallback — постоянная ссылка под виджетом |
| **Производительность / a11y** | Каждый бейдж с tooltip создавал свой `TooltipProvider` | Один `TooltipProvider` в `CabinetUpcomingAppointments` |
| **Дублирование** | `isSafeHref` дублировался в `cabinet/page` | `lib/url/isSafeExternalHref.ts` + использование в клиентском списке |
| **Маршруты** | Жёстко заданные строки путей | `routePaths.patientHelp` для «Как подготовиться» |
| **Уведомления без телефона** | Только форма привязки без поясняющего блока | `InfoBlock` перед `PatientBindPhoneSection` |
| **Ошибки toggles** | Молчаливый сбой server action | Сообщение об ошибке в `ChannelNotificationToggles` |

## Оставшийся техдолг (не блокирует этап 5)

- История записей пациента — по-прежнему без API (`getPastAppointments` пустой).
- Баннер «новости» — первая строка в `mailing_topics_webapp` (эвристика important по `key`/`code`).
- CSP / полная изоляция iframe Rubitime — не настраивались (только `referrerPolicy`).

## Тесты

- Добавлены unit-тесты: `parseChannelPreferenceInput`, `isSafeExternalHref`, обновлён `appointments/service.test.ts`.

## Критерий готовности к этапу 5

- Нет критичных уязвимостей в изменённых server actions; CI зелёный; контракты гостя/авторизованного на главной сохранены.
