# FIX_PLAN — этап 8 (Messaging)

## 1. Статус этапа

**Частично выполнен:** миграция `025_support_messaging_extension.sql`, сервисы patient/doctorSupport, UI `ChatView`, инбокс врача, интеграция в `buildAppDeps`, relay с предупреждением при пустом `INTEGRATOR_API_URL` — в репозитории есть. **Не выполнено** по плану: **8.5.2** (приоритизация диалогов с непрочитанными от пользователя сверху). Тесты API и e2e — **частично** (см. приложение ниже).

## 2. Найденные проблемы

### Критичные

- Нет при принятой модели MVP «общая очередь поддержки» для врача. Если продукт требует изоляцию диалогов по специалисту — текущая модель **не** соответствует (любой doctor/admin с UUID видит диалог). См. `doctorSupportMessagingService` и раздел 2.2 приложения. Шаг: **8.3.2** (смысл контроля доступа). Тип: **логика**, **backend**.

### Средние

- **`createDoctorSupportMessagingService.getMessages`**: двойная тяжелая нагрузка (`getConversationWithMessages` + `listMessagesSince`); `sendAdminReply` тянет полный лог для проверки существования. Шаг: **8.3** (порты/сервис). Тип: **backend**, **архитектура**.
- **`useMessagePolling`**: при hidden вкладке интервал не увеличивается, таймер продолжает тикать (сеть не дергается). Шаг: **8.7.2**. Тип: **логика**, **фронт**.
- **`PatientHeader` / `DoctorHeader`**: импорт хуков из `modules/messaging` внутри `shared/ui` — граница слоев. Шаг: **8.6** (интеграция с шапкой). Тип: **архитектура**.
- **Тесты**: отсельные кейсы для `read`, `unread-count`, POST 403, poll success; врач: `unread-count`, `[id]/read`. Шаг: **8.4**–**8.5**. Тип: **тесты**.
- **Сортировка 8.5.2**: не реализована. Тип: **логика**, **backend**.

### Мелкие

- Дублирование разметки списка диалога врача; возможный вынос в `ConversationList.tsx` для соответствия именованию плана. Тип: **фронт**, **мёртвый код** (нет отдельного файла — не баг).
- Двойной polling (бейдж + чат) на одной странице — оптимизация. Тип: **фронт**.

## 3. Пропущенные шаги

- **8.5.2** — сортировка «непрочитанные сверху» (приоритет диалогов с непрочитанными от пользователя).

## 4. Лишние изменения

Не выявлены в рамках ревью этапа 8.

## 5. План исправлений

1. Реализовать **8.5.2**: либо сортировка на сервере в запросе списка диалогов, либо чистая функция на клиенте после получения флагов непрочитанных; unit-тест на фикстурах.
2. Рефакторинг **`doctorSupportMessagingService.getMessages`**: лёгкий `conversationExists` / `getConversationById` в порту PG; не загружать полную историю в горячем пути.
3. **`useMessagePolling`**: при `document.visibilityState === hidden` снимать interval или переключать на длинный период; возобновлять при `visible`.
4. Вынести хуки непрочитанных в **shared/hooks** или прокинуть счётчик с сервера — уменьшить coupling `shared/ui` -> `modules/messaging`.
5. Добавить недостающие **`route.test.ts`** для перечисленных в приложении §4 endpoints.
6. Задокументировать в JSDoc API врача модель **«глобальная очередь»** или ввести фильтр по продукту + тесты 403.
7. E2E: импорт `DoctorSupportInbox` / проверка `routePaths.patientMessages` по мере рефакторинга.
8. `pnpm run ci` после изменений.

### Уязвимости и доменная безопасность

**Пациент:** владение диалогом через `getConversationIfOwnedByUser` / `pollNew` / `sendText` — корректно. `POST .../read` проверяет владельца перед `markInboundRead` — корректно.

**Врач:** любой `doctor`/`admin` получает все открытые диалоги поддержки по projection; зная UUID — читает/пишет. MVP «общая очередь»; изоляция по специалисту не реализована. Решение: задокументировать модель или ввести фильтр + тесты.

**Rate limiting:** общая тема приложения.

### Архитектура и слои

**shared/ui → modules/messaging:** `PatientHeader` / `DoctorHeader` импортируют `useSupportUnreadPolling`. Варианты: `shared/hooks`, SSR-агрегат, обёртка у shell.

**Дублирование запросов:** `getMessages` вызывает `getConversationWithMessages` (все сообщения) затем `listMessagesSince`; `sendAdminReply` снова тяжёлый get. Нужны лёгкие методы порта.

**useMessagePolling vs 8.7.2:** при hidden не меняется interval — улучшить через снятие interval / длинный период.

### Покрытие тестами (backlog)

1. Пациент: POST 403; GET poll согласованность; POST read; GET unread-count.
2. Врач: unread-count; POST read.
3. Репозиторий: расширить моки pool для append/list/mark/count.
4. Сервис: `too_long`; relay: один warn при пустом URL.
5. Хуки: `useMessagePolling` + visibilitychange.
6. E2E: DoctorSupportInbox / `routePaths`.
7. Сортировка непрочитанных — unit при появлении функции.

### Верстка и стиль

`doctor/messages/page.tsx`: инлайн-стили у журнала заменены на Tailwind. `ChatView`: токены `hsl(var(--primary))`. Дублирование списка — опциональный вынос в `ConversationList`.

### Избыточность

Refetch после отправки у пациента — OK для MVP. Двойной polling бейджа и чата — можно объединить.

### Когда нужен крупный рефакторинг

Изоляция по врачу или отказ от полной загрузки истории на каждый запрос — отдельная задача: порты, миграции, тесты, UX.
