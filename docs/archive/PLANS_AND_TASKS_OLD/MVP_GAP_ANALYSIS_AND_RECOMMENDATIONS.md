# Анализ текущего состояния и рекомендации по доведению до MVP

Дата анализа: 2026-03-20.

---

## 1. Общее состояние проекта

Платформа BersonCareBot — монорепозиторий (pnpm) из двух основных сервисов:

| Сервис | Стек | Роль |
|--------|------|------|
| **integrator** | Fastify, grammY, PostgreSQL | Telegram/MAX/Rubitime бот, event pipeline, projection outbox |
| **webapp** | Next.js 16, React 19, PostgreSQL | UI пациента и врача, хранилище продуктовых данных |
| **admin** | Vite + React 18 | Админка (standalone, минимальный функционал) |

**Инфраструктура готова:** CI (lint, typecheck, test, build, audit), GitHub Actions deploy на host, systemd-юниты, nginx, projection outbox integrator→webapp.

**Миграция данных (DB Zones Restructure):** Этапы 0–13 выполнены. Все продуктовые домены (person, communication, reminders, appointments, subscription/mailing) перенесены в webapp. Legacy таблицы заморожены. Reconcile-скрипты, gate-проверки, backfill — на месте.

---

## 2. Определение MVP (из FULL PLATFORM MODEL)

По документам MVP включает:

| Область | Что входит в MVP |
|---------|-----------------|
| **Дневники** | Симптомы, ЛФК |
| **Web-App** | Базовое меню, скорая помощь, полезные уроки, кабинет клиента, настройки |
| **Напоминания** | Упражнения, ввод симптомов |
| **Telegram** | Задать вопрос, записаться, меню |

Что **откладывается**: AI-анализ, фото еды, платный контент, полноценные курсы, индивидуальные программы, сложная статистика.

## 3. Карта реализованности: что работает, что нет

### 3.1 Полностью работающие фичи (end-to-end)

| Фича | Комментарий |
|------|-------------|
| **Telegram бот** | Сценарии, меню, запись на приём, вопрос админу, контакт, уведомления. Работает через EventGateway → Orchestrator → Executor. |
| **Авторизация** | Три метода: signed token (из бота), Telegram initData, телефон/SMS. Полный цикл. |
| **Пациентское меню** | Главный экран с карточками (скорая помощь, уроки, кабинет, дневник симптомов, ЛФК). |
| **Дневник симптомов** | Данные из бота (projection), отображение трекингов и записей в webapp. |
| **Дневник ЛФК** | Данные из бота + форма «Отметить занятие» в webapp. |
| **Кабинет клиента** | Ближайшие записи на приём (из Rubitime через projection). |
| **Настройки каналов** | Карточки Telegram/MAX/VK, тогглы сообщений/уведомлений, персистенция в БД. |
| **Привязка телефона** | Форма привязки, SMS-подтверждение, guard на страницах с персональными данными. |
| **Кабинет врача** | Дашборд (KPI, ближайшие приёмы, список клиентов), клиенты (поиск, фильтры, master-detail), приёмы, статистика. |
| **Data pipeline** | Projection outbox, durable delivery, idempotency, reconcile — надёжный канал integrator→webapp. |

## 4. Рекомендуемый план доведения до MVP

Порядок определён по сочетанию: (а) критичность для пользовательской ценности, (б) блокирующие зависимости, (в) production safety.

### Блок A: Production Safety (перед любым продуктовым релизом)

**A1. Выполнить R1–R7 из PROD_READINESS_REMEDIATION_PLAN.md**

Прежде чем выкатывать MVP-фичи на прод, необходимо закрыть production safety gaps. R1 (backup перед webapp migrate) — критический, без него деплой миграций на прод рискован.

Затрагивает: `deploy/host/deploy-webapp-prod.sh`, `deploy/HOST_DEPLOY_README.md`, `docs/`.
Инвазивность: минимальная (скрипты деплоя + документация).

---

### Блок B: Контент — основа пользовательской ценности

Без реального контента разделы «Скорая помощь» и «Полезные уроки» — пустые оболочки. Это центральная часть продукта.

**B1. Контент-каталог: переход от статики к данным**

Текущее: 7 hardcoded записей в `STUB_ENTRIES` (файл `apps/webapp/src/modules/content-catalog/catalog.ts`) с placeholder-текстами.

Рекомендация:
- Вариант 1 (минимальный): Заменить placeholder-тексты на реальный контент прямо в статических данных. Добавить ссылки на реальные видео (YouTube/встроенные). Это позволяет запустить MVP без БД для контента.
- Вариант 2 (с БД): Создать таблицу `content_articles` в webapp, миграцию, repo, seed-скрипт. Это масштабируемее, но тяжелее.

Для MVP рекомендуется **Вариант 1** — наполнить статический каталог реальным контентом (тексты + видео URL). Это быстрее и достаточно для первых пользователей.

Затрагивает: `apps/webapp/src/modules/content-catalog/catalog.ts`, `apps/webapp/src/modules/lessons/service.ts`, `apps/webapp/src/modules/emergency/service.ts`.

**B2. Видео-плейер в контентных страницах**

Текущее: placeholder-изображение вместо видео.

Рекомендация: Добавить компонент видео-плейера (iframe для YouTube или HTML5 `<video>`) на страницу `content/[slug]/page.tsx`. В каталоге добавить поле `videoUrl`.

Затрагивает: `apps/webapp/src/app/app/patient/content/[slug]/page.tsx`, каталог.

**B3. Наполнение контента**

Требуется от владельца продукта: тексты рекомендаций, ссылки на видео упражнений, описания для скорой помощи. Это не задача разработки, но блокирует реальную ценность.

### Блок C: Напоминания — ключевая MVP-фича

По плану MVP напоминания (упражнения, ввод симптомов) — обязательная часть. Сейчас: backend projection есть, dispatch не реализован, UI нет.

**C1. Reminder dispatch (integrator → webapp → бот)**

Текущее: `handleReminderDispatch` возвращает `accepted: false, reason: "durable reminder dispatch is not implemented"`.

Рекомендация:
1. Реализовать `handleReminderDispatch` в webapp: получить правила, определить due occurrences, отправить delivery intent обратно в integrator (или напрямую в бота через webhook).
2. Альтернатива: scheduler в integrator сам обрабатывает `reminders.dispatchDue` и отправляет через DispatchPort (Telegram/MAX). Webapp хранит правила и историю, а integrator — runtime (occurrences, due, delivery).

Из кода видно, что `reminders.occurrences.*` и `reminders.due` остаются на integrator (runtime). Значит, dispatch должен происходить в integrator, а webapp только хранит конфигурацию правил.

Рекомендуемый подход: дореализовать scheduler в integrator для dispatch напоминаний, используя правила из webapp (через `remindersReadsPort`).

Затрагивает: `apps/integrator/src/infra/runtime/`, `apps/integrator/src/kernel/domain/`, scheduler, DispatchPort.

**C2. Reminder UI в webapp (настройки)**

Текущее: в Settings заглушка «В MVP настройки уведомлений готовятся как каркас под reminder bridge».

Рекомендация:
1. На странице Settings показать текущие правила напоминаний (из `reminderRules`).
2. Позволить включать/выключать категории (toggle).
3. Позволить менять пресет расписания (утро/вечер/3 раза в день).
4. Сохранение через server action → integrator webhook (или напрямую в webapp + projection обратно в integrator).

Затрагивает: `apps/webapp/src/app/app/settings/page.tsx`, `apps/webapp/src/modules/reminders/`, новый UI-компонент.

---

### Блок D: Дневник симптомов — добавление записей из webapp

**D1. Форма добавления симптома в webapp**

Текущее: «Добавить симптом или запись можно в боте.»

Рекомендация: Добавить минимальную форму на странице дневника симптомов: выбор трекинга, шкала 1–10, опциональный комментарий. Server action записывает в `symptom_entries`.

Это важно для MVP: пользователь не всегда хочет переключаться в бота для записи симптома.

Затрагивает: `apps/webapp/src/app/app/patient/diary/symptoms/page.tsx`, `apps/webapp/src/modules/diaries/`.

---

### Блок E: Doctor messaging — реальная доставка

**E1. Доставка сообщений врача через integrator**

Текущее: `sendMessage` пишет в `message_log`, но не доставляет сообщение пациенту.

Рекомендация: После записи в `message_log`, отправить запрос в integrator (например, `POST /api/bersoncare/send-message` или через projection outbox в обратном направлении webapp→integrator) для доставки через Telegram/MAX.

Затрагивает: `apps/webapp/src/modules/doctor-messaging/`, integrator (новый endpoint или webhook).
Зависимости: нужен обратный канал webapp→integrator.

### Блок F: UX-полировка

**F1. Мобильная оптимизация**

Webapp открывается из Telegram Mini App — mobile-first. Проверить:
- Все страницы корректно отображаются на мобильных экранах
- Тач-взаимодействия комфортны
- Скролл не ломается в Telegram WebView

**F2. Loading/Error states**

Проверить, что все страницы обрабатывают:
- Загрузку (скелетоны или спиннеры)
- Ошибки (fallback UI)
- Пустые состояния (когда данных нет)

**F3. Favicon, meta, PWA basics**

Добавить иконку приложения, мета-теги для Telegram Mini App, базовый manifest.

## 7. Риски и зависимости

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Контент не готов (тексты, видео) | Высокая | Блокирует ценность MVP | Подготовить шаблон, попросить владельца продукта заполнить |
| Reminder dispatch сложнее ожидаемого | Средняя | Задержка C1 | Начать с простого: scheduler tick → отправка через существующий DispatchPort |
| Telegram Mini App WebView quirks | Средняя | UX-баги | Тестирование на реальном устройстве в Telegram |
| Deploy без backup (R1 не сделано) | Высокая при деплое | Потеря данных | Сделать R1 первым делом |

## 8. Итого: минимальный путь до MVP

1. **Закрыть production safety** (R1–R7) — без этого нельзя безопасно деплоить.
2. **Наполнить контент** (скорая помощь, уроки) — без этого пустая оболочка.
3. **Запустить напоминания** (dispatch + минимальный UI) — ключевая фича платформы.
4. **Добавить запись симптома** из webapp — минимальная интерактивность дневника.
5. **Полировка** (mobile, loading states, ошибки).

После этих шагов платформа будет предоставлять реальную ценность: пациент получает контент (уроки, скорая помощь), ведёт дневники, получает напоминания, видит свои записи и приёмы. Врач видит пациентов, их данные и статистику.
