## 1. Технический стек и рекомендации по дополнению

### 1.1 Текущий стек

| Пакет | Версия | Роль |
|-------|--------|------|
| Next.js | 16.1.7 | App Router, SSR, Server Components, Server Actions |
| React | 19.2.4 | UI-фреймворк |
| TypeScript | 5.9.3 | Типизация |
| Zod | 4.3.6 | Валидация данных (используется в API и сервисах) |
| pg | 8.20.0 | PostgreSQL-клиент (прямые SQL-запросы) |
| dotenv | 17.3.1 | Переменные окружения |

**Стилизация:** один монолитный файл `globals.css` (~860 строк) с кастомными CSS-классами. Никаких внешних UI-библиотек, CSS-модулей или CSS-in-JS.

**State management:** отсутствует. Используются `useState`, `useEffect`, `useTransition`, `useActionState` на уровне отдельных компонентов.

**Иконки:** текстовые символы Unicode (`←` для «Назад», `×` для «Закрыть», `span`-элементы для гамбургера).

**Форматирование дат:** `new Date().toLocaleDateString("ru-RU")` — нет библиотеки для дат.

## 2. Что переделать в вёрстке

### 2.1 Монолитный CSS → Tailwind / CSS-модули

**Проблема:** Все стили в одном `globals.css` (860 строк). Классы не модульные, рефакторинг рискован (не ясно, кто использует какой класс). Много `style={{}}` inline-стилей в компонентах врача.

**Рекомендация:**
1. Внедрить Tailwind CSS.
2. Поэтапно мигрировать: сначала новые компоненты — на Tailwind; потом рефакторить старые.
3. Для doctor-компонентов (ClientProfileCard, NewMessageForm, и т.д.) убрать все `style={{}}` и заменить на утилитарные классы.

### 2.2 Нативные элементы без стилей

**Проблема:**
- `<select>` (выбор комплекса ЛФК, категории сообщения, получателя) — не стилизованы (нативный вид ОС).
- `<input type="checkbox">` (настройки каналов, каналы доставки сообщений) — нативные чекбоксы без кастомизации.
- `<textarea>` использует класс `.auth-input`, который по имени предназначен для авторизации — семантически неверно.

**Рекомендация:** Использовать компоненты из shadcn/ui (Select, Checkbox, Switch, Textarea) или стилизовать через Tailwind. Заменить чекбоксы для каналов (Telegram, уведомления) на Switch-тогглы — это интуитивнее для настроек вкл/выкл.

### 2.3 Кнопки

**Проблема:** Один класс `.button` (pill-shaped, `border-radius: 999px`). Используется и для основных действий («Отправить»), и для навигации («Клиенты», «Записи»), и для back-ссылок. Нет визуального различия между primary/secondary/destructive.

**Рекомендация:** Ввести варианты кнопок:
- `button--primary` (заливка, основной цвет)
- `button--secondary` (outline)
- `button--ghost` (уже есть, но используется редко)
- `button--destructive` (для отмены, удаления)

### 2.4 Пустые состояния

**Проблема:** Класс `.empty-state` — просто серый текст. Одинаковое оформление для «Нет данных», «Ошибка», «Скоро будет». Нет иллюстраций или иконок.

**Рекомендация:** Создать компонент `EmptyState` с вариантами:
- `no-data` (иконка пустого списка + текст)
- `coming-soon` (иконка часов + текст)
- `error` (иконка ошибки + текст + кнопка повторить)

### 2.5 Responsive для врача

**Проблема:** Doctor layout `max-width: 1200px` на десктопе хорошо, но на планшете (768-1024px) master-detail клиентов (1/3 + 2/3 grid) может быть тесным. На мобильном (<768px) detail-колонка скрыта (`display: none`) — пользователь не видит выбранный профиль.

**Рекомендация:** На мобильном при нажатии на клиента переходить на `/app/doctor/clients/[userId]` (уже работает через `ClientListLink`), но в master-detail на десктопе detail-панель не получает фокуса — нужна анимация/подсветка при переключении.

### 2.6 Отступы и выравнивание

**Проблема:** PatientHeader `margin: 0 -16px` — отрицательные отступы для «full-bleed» шапки. Работает, но fragile — при изменении padding контейнера сломается.

**Рекомендация:** Переверстать шапку как `position: sticky; left: 0; right: 0` вне контейнера `.app-shell--patient`, а контент сделать отдельным `<main>` с padding.

### 3.M Определение роли администратора/специалиста — КРИТИЧНО (P1)

**P-22.** После обновления БД (миграция данных, stages 0–13) сломался механизм определения администратора. При запуске webapp открывается обычный личный кабинет вместо кабинета специалиста.

**Корневая причина:**
- В `platform_users` колонка `role` по умолчанию `'client'`.
- При backfill/reconcile пользователи создаются с `role = 'client'`.
- Единственный способ стать `admin` — `ADMIN_TELEGRAM_ID` в env, но он работает только для **новых** пользователей. Если пользователь уже существует в `platform_users`, роль берётся из БД (`pgIdentityResolution.ts`), а не из env.
- `doctor` роль вообще не назначается ни через env, ни через токен. Только вручную в БД.

**Решение:**
1. При авторизации через Telegram initData или token exchange: если `telegramId === ADMIN_TELEGRAM_ID`, принудительно обновлять `platform_users.role = 'admin'` в БД (не только для новых, но и для существующих).
2. Ввести env-переменную `DOCTOR_TELEGRAM_IDS` (или `SPECIALIST_TELEGRAM_IDS`) — список Telegram ID специалистов. При входе через Telegram — обновлять роль на `doctor`.
3. Альтернатива: admin-интерфейс для назначения ролей.

**Логика привязки специалиста (описание владельца продукта):**
- Пользователь заходит через webapp (не из Telegram).
- Привязывает номер телефона.
- Привязывает аккаунт к Telegram.
- Если Telegram ID совпадает с `ADMIN_TELEGRAM_ID` или входит в `DOCTOR_TELEGRAM_IDS` — система автоматически обновляет роль на `admin`/`doctor`.

### 3.N Контент-менеджмент: БД, структура, редактирование (P2–P4)

### 4.1 Drawer-меню пациента: 3 из 5 ссылок ведут в одно место

### 4.2 Кнопка «Назад» на контентных страницах

**Проблема:** `backHref="/app/patient"` — всегда возвращает в главное меню. Если пользователь зашёл из «Полезные уроки» → «Разминка для шеи» → нажал «Назад» — попадает в главное меню, а не обратно в список уроков.

**Рекомендация:** Передавать в content page информацию об источнике (referer) и ставить `backHref` на `/app/patient/lessons` или `/app/patient/emergency` в зависимости от того, откуда пришёл пользователь. Или использовать `router.back()`.

### 4.3 FeatureCard показывает статус на английском

**Проблема:** В `FeatureCard.tsx` — `status.replace("-", " ")` рендерит «available», «coming soon», «locked» — на английском.

**Рекомендация:** Маппинг:
```
available → «Доступно»
coming-soon → «Скоро»
locked → «Заблокировано»
```

### 4.4 PatientHeader: заголовок — ссылка домой, но показывает название страницы

**Проблема:** В `PatientHeader` prop `title` отображается по центру шапки и является кликабельной ссылкой на `/app/patient`. Например, на странице «Дневник ЛФК» заголовок шапки — «Дневник ЛФК» и он кликабелен. Но пользователь может подумать, что это просто заголовок текущей страницы, а не навигация. Особенно нет визуального указателя (подчёркивания, иконки).

**Рекомендация:** Либо:
- Показывать в шапке всегда «BERSONCARE» (бренд), а заголовок страницы — ниже шапки.
- Либо добавить визуальный индикатор «домой» (иконка дома или стрелка).

### 4.5 Настройки в навигации врача разрывают контекст

**Проблема:** `DoctorNavigation` содержит пункт «Настройки» → `/app/settings`. Но Settings не обёрнут в doctor layout — навигация врача исчезает, пользователь «выпадает» из кабинета. Кнопка «Меню» на странице настроек ведёт обратно в `/app/doctor`, но переход кажется разрывом.

**Рекомендация:**
- Либо обернуть Settings в doctor layout (добавить `/app/doctor/settings`).
- Либо открывать настройки как Sheet/Dialog поверх текущей страницы.
- Либо убрать «Настройки» из doctor nav и оставить только в user-pill (уже есть кнопка «Настройки» в top-bar actions).

### 4.6 «Мои покупки» нет в главном меню пациента

**Проблема:** В `menu/service.ts` для `client` — 5 пунктов: скорая помощь, уроки, мои записи, дневник симптомов, дневник ЛФК. «Мои покупки» отсутствует. Но в `FULL PLATFORM MODEL` первый пункт меню — «Мои покупки». Страница `/app/patient/purchases` существует и доступна.

**Рекомендация:** Добавить «Мои покупки» в меню (можно со статусом `coming-soon` до реализации).

### 4.7 ConnectMessengersBlock: кнопка «Подключить» ведёт во внешний сервис

**Проблема:** Кнопка «Подключить» для Telegram/MAX открывает `target="_blank"` ссылку на бота. Пользователь может потеряться — ушёл в Telegram, не понял что делать, не вернулся.

### 4.8 Нет индикатора текущей страницы в patient drawer

**Проблема:** В drawer-меню все пункты выглядят одинаково. Нет подсветки текущего раздела.

**Рекомендация:** Добавить `active`-состояние для текущего пункта (выделение фоном или цветом), как сделано в `DoctorNavigation`.
