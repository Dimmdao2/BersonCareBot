# Stage 8 - Контрактные e2e тесты времени

## Цель этапа

Зафиксировать сквозной контракт времени: один и тот же бизнес-момент проходит через ingest -> БД -> projection -> UI/бот без потери смысла.

## Scope (только Stage 8)

- Контрактные тесты для Moscow и Samara сценариев.
- Проверки по БД, событиям и отображению.

Не включать:

- Новые runtime-фичи.
- Массовый refactor вне тестового слоя.

## План реализации (детально)

### S8.T01 - Фикстуры входных webhook

Подготовить минимум две фикстуры:

- Branch timezone `Europe/Moscow`, raw `"2026-04-07 11:00:00"`.
- Branch timezone `Europe/Samara`, raw `"2026-04-07 11:00:00"`.

### S8.T02 - Сквозные ассерты для Moscow

Проверить цепочку:

- `rubitime_records.record_at` = `2026-04-07 08:00:00+00`.
- Projection `recordAt` = `"2026-04-07T08:00:00.000Z"`.
- `appointment_records.record_at` = `2026-04-07 08:00:00+00`.
- `patient_bookings.slot_start` = `2026-04-07 08:00:00+00`.
- Отображение:
  - бот: `"7 апр. 2026 г., 11:00"` при display timezone `Europe/Moscow`.
  - кабинет пациента: `"7 апр. 2026 г., 11:00"`.
  - дашборд врача: `"11:00 07.04"`.

### S8.T03 - Сквозные ассерты для Samara

Проверить:

- Webhook `"2026-04-07 11:00:00"` -> UTC `2026-04-07 07:00:00+00`.
- При display timezone `Europe/Moscow`: текст `"7 апр. 2026 г., 10:00"`.
- При display timezone `Europe/Samara`: текст `"7 апр. 2026 г., 11:00"`.

### S8.T04 - Негативный контракт: невалидный datetime

Сценарий:

- Вебхук содержит raw-время (`record`/`datetime`), но значение невалидно.

Проверить:

- запись не потеряна (вариант A),
- `record_at` сохраняется как `NULL`,
- создается инцидент качества данных,
- отправляется Telegram-алерт админу,
- Google Calendar событие не создается/не обновляется.

### S8.T05 - Негативный контракт: невалидная IANA timezone филиала

Сценарий:

- Для филиала сохранена невалидная IANA, из-за чего используется fallback.

Проверить:

- runtime не падает,
- fallback применяется,
- создается инцидент конфигурации,
- отправляется Telegram-алерт админу.

## Проверки и тесты

- Прогнать все новые контрактные тесты.
- Прогнать `pnpm run ci`.

## Gate (обязателен для PASS)

- Оба сценария (Moscow, Samara) зеленые во всех слоях.
- Нет расхождений между БД, projection и UI-текстом.
- Негативные сценарии (`invalid datetime`, `invalid timezone`) дают предсказуемый результат: запись не теряется, инцидент + алерт обязательны.
- `pnpm run ci` зеленый.
