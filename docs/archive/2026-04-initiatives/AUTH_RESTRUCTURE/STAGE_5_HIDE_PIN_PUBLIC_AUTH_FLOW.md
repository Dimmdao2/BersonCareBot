# Stage 5 - Скрыть PIN из публичного auth-flow

Сводка в общем плане: `MASTER_PLAN.md` -> Stage 5.

## Цель этапа

Временно убрать шаг установки/ввода PIN из публичного сценария входа. PIN остается опциональной функцией вне login flow.

## Scope (только Stage 5)

- Удаление/скрытие `set_pin` шага из `AuthFlowV2`.
- Проверка, что после успешного login всегда идет редирект.
- Сохранение backend возможностей PIN без удаления доменной логики.

Не включать:

- Новые сценарии установки PIN в профиле (кроме документации/контракта).
- Изменения методов входа по номеру (Stage 4).

## Предусловия

- Stage 3 и Stage 4 готовы, чтобы основной вход не зависел от PIN.

## Подробный план реализации

### S5.T01 - Убрать автопереход на `set_pin`

1. Открыть `AuthFlowV2.tsx`.
2. Найти переходы на шаг `set_pin` после OTP/логина.
3. Заменить переход на финальный редирект в приложение.
4. Проверить все ветки успеха (SMS/Telegram).
5. Обновить `AGENT_EXECUTION_LOG.md`.

### S5.T02 - Скрыть PIN UI в публичном login

1. Отключить рендер `PinInput` в публичном auth flow.
2. Удалить недостижимые ветки состояния шага PIN в login экране.
3. Не удалять backend API PIN, если он нужен как опция профиля.
4. Обновить `AGENT_EXECUTION_LOG.md`.

### S5.T03 - Док-контракт про опциональность PIN

1. Зафиксировать в `auth.md` и/или рядом с `AuthFlowV2`, что PIN в login скрыт временно.
2. Указать, что включение PIN возможно только в отдельных сценариях (профиль/будущая версия).
3. Обновить `AGENT_EXECUTION_LOG.md`.

### S5.T04 - Тесты

1. Успешный SMS login не показывает `set_pin`.
2. Успешный Telegram login не показывает `set_pin`.
3. Проверка редиректа сразу после login.
4. Прогнать релевантные тесты + `pnpm run ci`.
5. Зафиксировать evidence в `AGENT_EXECUTION_LOG.md`.

## Gate (критерий готовности)

- В публичном login-flow отсутствует шаг PIN.
- После успешного входа пользователь сразу попадает в приложение.
- Backend PIN не ломается для непубличных сценариев.
- `pnpm run ci` зеленый.
