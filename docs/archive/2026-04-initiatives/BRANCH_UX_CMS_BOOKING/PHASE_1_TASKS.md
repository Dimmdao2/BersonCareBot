## Блок 1.A — Загрузка файлов

### Задача 1.1: Множественная загрузка файлов (`<input multiple>`)

**Цель:** врач может загрузить несколько файлов за одно действие, без повторного выбора каждого файла вручную.

**Предусловия:**
- Текущая загрузка работает для одного файла (`MediaLibraryClient.tsx` + `POST /api/media/upload`).
- Роли доступа и MIME-валидация в upload-route уже реализованы.

**Файлы для изменения:**
1. `apps/webapp/src/app/app/doctor/content/library/MediaLibraryClient.tsx` — заменить single-file сценарий на batch upload.
2. `apps/webapp/src/app/api/media/upload/route.ts` — принять несколько файлов в одном multipart-запросе (с backward compatibility для `file`).
3. `apps/webapp/src/app/api/media/upload/route.test.ts` — добавить кейсы для multi-upload и смешанных ошибок.

**Файлы для создания:**
- Нет.

**Файлы для удаления:**
- Нет.

**Детальное описание:**
- В `MediaLibraryClient.tsx`:
  - У `input type="file"` включить `multiple`.
  - В `onUploadFile` обрабатывать `FileList` как массив и отправлять пачку.
  - После успешной загрузки обновлять список библиотеки одним `reloadKey`.
- В `POST /api/media/upload`:
  - Поддержать чтение нескольких файлов (`file`, `files[]`), не ломая старый контракт.
  - Для каждого файла выполнять текущие проверки: размер, MIME whitelist, magic-bytes.
  - Вернуть массив результатов (например, `uploaded[]`) и оставить совместимость с текущим `ok/mediaId/url` для случая одного файла.
- В тестах:
  - Проверить успешную загрузку 2+ валидных файлов.
  - Проверить поведение при смешанном батче (валидный + невалидный), зафиксировать ожидаемую стратегию (all-or-nothing или partial success).

**Тесты:**
- [ ] Unit API: multi-file `multipart/form-data` возвращает успешный список загруженных элементов.
- [ ] Unit API: при MIME/signature mismatch возвращается корректный код/ошибка и деталь по файлу.
- [ ] Manual UI: выбрать 5 файлов и убедиться, что после завершения они появились в списке.

**Критерии готовности:**
- [ ] В форме загрузки доступен множественный выбор файлов.
- [ ] API корректно принимает и валидирует несколько файлов.
- [ ] Ошибки батча отображаются в UI без «тихих» падений.
- [ ] `pnpm run ci` зелёный.

### Задача 1.2: Прогресс-бар загрузки (XHR `onprogress`)

**Цель:** врач видит понятный прогресс загрузки (процент/статус), а не только текст «Загрузка...».

**Предусловия:**
- Выполнена задача 1.1 (множественная загрузка).
- API upload стабилен и возвращает предсказуемый JSON.

**Файлы для изменения:**
1. `apps/webapp/src/app/app/doctor/content/library/MediaLibraryClient.tsx` — внедрить upload transport с прогрессом.

**Файлы для создания:**
1. `apps/webapp/src/app/app/doctor/content/library/uploadWithProgress.ts` — helper для XHR-отправки multipart и события прогресса.

**Файлы для удаления:**
- Нет.

**Детальное описание:**
- Перейти с `fetch` на XHR-обертку только для upload-сценария.
- Отображать:
  - общий прогресс батча (0–100%),
  - текущий файл (имя + статус),
  - финальный статус (успех/ошибка).
- При ошибке не «зависать» в состоянии `uploading=true`.
- Сохранить существующий UX фильтров и списка, не блокируя навигацию после завершения.

**Тесты:**
- [ ] Unit helper: корректно прокидывает `progress`, `success`, `error`.
- [ ] Manual UI: при загрузке файла >10MB прогресс изменяется постепенно, а не скачком.
- [ ] Manual UI: при сетевой ошибке прогресс сбрасывается и показывается читабельная ошибка.

**Критерии готовности:**
- [ ] Врач видит прогресс загрузки в процентах.
- [ ] Состояние загрузки корректно завершается в success/error.
- [ ] После upload список обновляется без ручного reload страницы.
- [ ] `pnpm run ci` зелёный.
