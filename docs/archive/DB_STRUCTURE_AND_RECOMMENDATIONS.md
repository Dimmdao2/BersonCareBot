# База данных как интегратор без «главной» интеграции

Целевая модель БД и правила добавления интеграций. Ядро не зависит от конкретного канала или провайдера; источник всегда задаётся контекстом (например `event.meta.source`).

## Целевая модель

### User

- Одна сущность «человек/пациент»; не привязана к каналу.
- Может существовать только в Max, только в SMS, только в RubiTime и т.д.
- Хранится в таблице `users` (id, created_at, updated_at).

### Identities

- Все «точки входа» пользователя: Telegram, Max, VK, RubiTime, SMS, email, будущие каналы.
- Один `user_id` может иметь несколько записей в `identities` с разными `resource`.
- Таблица: `identities` (id, user_id, resource, external_id, …). Уникальность: `(resource, external_id)`.
- Разрешение пользователя по каналу выполняется по паре **(resource, external_id)**, а не по «channelId» с подразумеваемым Telegram.

### Contacts

- Способы связи с пользователем (телефон, email), привязаны к `user_id`.
- Могут приходить из любого канала или из виджета записи, без обязательного Telegram.
- Таблица: `contacts` (user_id, type, value_normalized, …).

### Состояние и настройки канала

- На каждую identity (или пару resource + external_id): state, флаги уведомлений и т.п.
- Реализация: либо отдельная таблица на интеграцию (`telegram_state`, в будущем `max_state` и т.д.), либо одна обобщённая таблица по `identity_id` + тип данных.
- Контракты портов (`user.state.set`, `user.phone.link`, `notifications.update`, `notifications.settings`) принимают **resource** в params; инфраструктура маршрутизирует только в ту таблицу состояния, которая поддерживается для данного resource (сейчас только Telegram).

### Записи и внешние сущности

- Записи на приём, календари, платежи.
- **Booking (записи на приём):** ядро использует универсальный контракт `booking.*` (booking.upsert, booking.byExternalId, booking.activeByUser). Реализация может быть:
  - **Вариант A:** одна таблица с колонкой `provider` (например `booking_records(provider, external_id, user_id, …)`); миграция данных из `rubitime_records` с `provider='rubitime'`.
  - **Вариант B:** отдельные таблицы на провайдера (`rubitime_records`, в будущем другие); readPort/writePort по типу запроса и при необходимости по параметру provider вызывают нужный репозиторий.
- Выбор можно зафиксировать при появлении второго провайдера записей; контракт в ядре не меняется.

## Принцип «никакой интеграции как основной»

- Пайплайн: **webhook → нормализация (meta.source) → загрузка контекста по (source, id в канале) → выбор сценария по source + audience → исполнение**.
- Никакой интеграции не отводится роль «главной»; источник всегда задаётся событием.
- Ядро (kernel) не знает о telegram/rubitime/smsc; только универсальные контракты и параметры (resource, externalId и т.д.).
- Инфраструктура может ветвиться по `resource` только для выбора таблицы/репозитория (например `telegram_state` при `resource === 'telegram'`), без бизнес-логики в infra.

## Правила добавления новой интеграции

1. **Миграции:** добавить папку `src/integrations/<name>/db/migrations/`; миграции подхватываются автоматически, правки в ядре не требуются.
2. **Identity:** новая интеграция пишет в `identities` с своим `resource` и `external_id`.
3. **Состояние:** при необходимости — своя таблица состояния (например `max_state`) по identity или по (resource, external_id); контракты `user.state.set` / `notifications.*` уже передают `resource`, достаточно обработать его в writePort/readPort.
4. **Контент:** сценарии и шаблоны по `content/<source>/<audience>`; новый канал = новый source.
5. **Доставка:** новый адаптер с `canHandle(intent)`; добавление канала без изменения ядра.

## Текущие отступления и план устранения

| Область | Текущее состояние | План |
|--------|--------------------|------|
| Разрешение пользователя | Реализовано по `user.byIdentity` с (resource, externalId). | Готово. |
| User lookup | Поддержка любого resource в `user.lookup`. | Готово. |
| Состояние канала | В params передаётся `resource`; writePort/readPort применяют только к Telegram. | Готово; для других каналов — добавить таблицы состояния и ветвление по resource. |
| Записи на приём | Одна таблица `rubitime_records`; контракт `booking.*` общий. | При появлении второго провайдера — либо одна таблица с provider, либо отдельные таблицы с маршрутизацией в портах. |
| Админ-статистика | Подсчёт пользователей Telegram из `identities` (resource='telegram'), «с телефоном» — по `contacts`. | Реализовано. |
