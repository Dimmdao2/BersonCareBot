# Stage 6 - Бэкфилл исторических данных времени

## Цель этапа

Исправить уже сохраненные некорректные `record_at`/`slot_start`/`slot_end`, которые были записаны до нормализации входа.

## Scope (только Stage 6)

- Диагностика и критерии выборки ошибочных строк.
- Dry-run и безопасный backfill-скрипт.
- Применение в maintenance window.

Не включать:

- Новый функционал в runtime API.
- Изменение UX/форматов отображения.

## Предусловия

- Stage 1-5 завершены и в проде.
- Есть валидный backup перед изменениями.
- Есть согласованное окно применения.

## План реализации (детально)

### S6.T01 - Диагностика смещений

Собрать SQL-диагностику для таблиц:

- `rubitime_records`
- `appointment_records`
- `patient_bookings` (`source='rubitime_projection'`)

Цель:

- Найти строки, где `record_at` не соответствует ожидаемому instant, вычисленному из исходного payload + timezone филиала.
- Зафиксировать объем затронутых данных по диапазонам дат.
- Отдельно диагностировать строки с `record_at IS NULL`, где в payload есть raw datetime (кандидаты на восстановление/ручной разбор).

### S6.T02 - Backfill для `rubitime_records`

Подготовить SQL/скрипт с фильтрами:

- Ограничить по периоду до даты деплоя фикс-этапов.
- Исправлять только строки, соответствующие диагностическим критериям.
- Обновлять `updated_at`.

Нельзя:

- Делать «слепой» сдвиг всей таблицы.

### S6.T03 - Backfill для `appointment_records`

Аналогично:

- Выборка только явно испорченных строк.
- Корректировка через вычисленный instant.

### S6.T04 - Backfill для `patient_bookings`

Ограничения:

- Только для проекционных строк (`source='rubitime_projection'`).
- Не трогать native booking-слоты без диагностического подтверждения.

### S6.T07 - Обработка `record_at IS NULL`

Добавить отдельный шаг backfill-процесса:

- попытка восстановить UTC instant из сохраненного raw payload + timezone филиала на момент события;
- если восстановление невозможно (битый input), инцидент помечать как `unresolved` в хранилище инцидентов;
- подготовить выгрузку unresolved-кейсов для ручного операционного разбора.

### S6.T05 - Dry-run и отчеты

Обязательно прогнать:

- `BEGIN; ... UPDATE ...; SELECT count(...) ...; ROLLBACK;`
- Снять отчет: сколько строк потенциально обновится в каждой таблице.

### S6.T06 - Production apply

В maintenance window:

- Backup перед применением.
- Применить backfill.
- Проверить post-check запросами.
- Проверить ключевые пользовательские сценарии.

## Проверки и тесты

- Прогнать тесты скриптов/утилит, если есть.
- Прогнать `pnpm run ci` для репозитория.

Post-check:

- Сравнение `rubitime` vs `stored instant` дает `diffMin=0` для целевых записей.
- Контрольные записи из UI/бота показывают правильное локальное время.
- Для `record_at IS NULL` есть прозрачный итог: либо восстановлено, либо помечено как `unresolved` с операционным списком.

## Gate (обязателен для PASS)

- Backfill затронул только целевые строки.
- После backfill расхождения по целевому набору устранены.
- Для null-кейсов есть подтвержденный разбор (restored/unresolved).
- Backup и evidence сохранены.
- `pnpm run ci` зеленый.
