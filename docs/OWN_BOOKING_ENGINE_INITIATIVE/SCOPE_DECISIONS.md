# SCOPE_DECISIONS — границы, решения, сужения

Здесь фиксируются: границы scope, сознательные сужения (с причиной), открытые вопросы `[need-decision]` и их ответы, связь с Rubitime и смежными инициативами. **Любое** сужение объёма ТЗ должно появиться здесь — иначе считается невыполненным.

## 1. Связь с Rubitime (переходная модель)
- Rubitime остаётся **двусторонним мостом/зеркалом** на переходный период; источник правды — собственная БД.
- Мост изолирован в адаптере, управляется настройкой (`system_settings`), может быть отключён без потери функциональности ядра.
- Текущий вебхук-пайплайн (`RUBITIME_BOOKING_PIPELINE.md`) не ломаем: на этапе 1 добавляем канон + проекцию, на этапе 2 переключаем write на канон, далее постепенно снимаем зависимость.
- Решение о полном отключении Rubitime — отдельное, после этапов 1–4 и 8 (когда канон закрывает запись/перенос/отмену/календарь). Зафиксировать дату/решение здесь при наступлении.

## 2. Связь со смежными инициативами
- **ONLINE_CONSULT_REHAB_NUTRITION_BOOKING_INITIATIVE**: онлайн-консультации вне Rubitime — частный случай собственного движка. Эта инициатива **поглощает** её целевую модель: онлайн-запись по реабилитации/нутрициологии реализуется на каноне (этапы 1–2), а не через Rubitime-слоты. Симптом `slots_mapping_not_configured` устраняется переходом на канон, а не настройкой Rubitime-маппинга. Отдельную инициативу после согласования пометить как absorbed со ссылкой сюда.
- **LOGIN_REGISTER_NEW_LOGIC**: идентичность пациента — переиспользуем (`platform_users`, `phone_normalized`, merge). PHASE_07 (backfill историч. записей без `platform_user_id`) и PHASE_08 (mass-email) остаются deferred; этой инициативой не активируются по умолчанию (если этап 3/9 не потребует — тогда `[need-decision]`).
- **COURSES_INITIATIVE**: курс — отдельная сущность (не план лечения), потребитель платёжного/продуктового слоя (этап 7). Не строить параллельный курс-движок.
- **DOCTOR_PATIENT_CARD_TREATMENT_PROGRAM_INITIATIVE**: карточка клиента (этап 9) согласуется с этой инициативой; booking-история — источник событий для карточки.

- **Q3 (этап 8).** Готовый календарный компонент vs собственный; допустимые лицензии/зависимости. — _ответ: TBD._

- **Q5 (этап 1).** Глубина SaaS на старте: один tenant с заделом vs реальная мультиарендность в UI сразу. По умолчанию — один tenant + `organization_id` в схеме. — _ответ по умолчанию принят; пересмотр по запросу._

## 5. Этап 4 — осознанные ограничения MVP (2026-05-29)

Не отмена пунктов чек-листа, а зафиксированный объём первой поставки:

| Тема | Решение | Где расширять |
|------|---------|----------------|
| UI политик specialist/service/product | Только org-default в `BookingPoliciesSection`; уровни — через API `POST /api/admin/booking-engine/policies` | Отдельная UI-задача при необходимости |

## 6. Жёсткие инварианты (не сужать никогда без явного решения владельца)
- Услуги не дублируются под комбинации (ТЗ §22.1, §24.1).
- Собственная БД — канон (ТЗ §22.2, §24.2).
- Платёжные ключи — в `system_settings`, не в ENV (ТЗ §24.12).
- Анти-обход штрафа через перенос (ТЗ §8.4, §22.3, §24.10).
- Multi-tenant задел с этапа 1 (ТЗ §20, §24.20).
- UI-паритет во всех кабинетах (требование владельца).
