# STAGE_CHECKLISTS — обязательные результаты по этапам

Ядро инициативы. Для каждого этапа: **что должно быть на выходе** (DoD), **правильный способ реализации**, **что нельзя пропустить**, **критерии приёмки**, **ссылки на разделы ТЗ**.

Правила использования (для агента-исполнителя):
- Каждый пункт чек-листа — обязательство. Его нельзя молча пропустить. Допустимо только `[cancelled]` с записью причины в [`SCOPE_DECISIONS.md`](SCOPE_DECISIONS.md).
- Формулировки «опционально / по желанию / если успеем» **запрещены** (правило plan-authoring). Пункт либо в DoD, либо явно отменён, либо вынесен в backlog со ссылкой.
- Перед стартом этапа построить декомпозированный план `.cursor/plans/<stage>.plan.md` по [`AGENT_BRIEF.md`](AGENT_BRIEF.md).
- Сквозные требования C1–C10 (`MASTER_PLAN.md` §4) и UI-паритет ([`UI_SURFACES_CHECKLIST.md`](UI_SURFACES_CHECKLIST.md)) — часть DoD каждого применимого этапа.
- Тэг `[need-decision]` = требуется продуктовое решение владельца до реализации (зафиксировать ответ в `SCOPE_DECISIONS.md`).

Легенда: ТЗ §X = раздел в [`SOURCE_SPEC.md`](SOURCE_SPEC.md).

## Этап 1. Каноническая модель данных

**Цель:** собственная организационная модель + каноническое хранение записей + статусы + события; совместимость с Rubitime. (ТЗ §2, §3, §7, §16.2, §17, §20, §21, §22.1, §22.2, §24.1–3.)

### 1.1 Организационная модель (ТЗ §2, §21.1)
- [x] Сущность **Clinic/Organization** как верхнеуровневый tenant-контейнер; `organization_id` закладывается во **все** новые доменные таблицы (C1).
- [x] Сущность **Branch/Location** (город/адрес/точка приёма); филиал **не** хранит список услуг.
- [x] Сущность **Room** (кабинет) внутри филиала.
- [x] Сущность **Specialist**; связи **SpecialistLocation** (специалист×филиал) и **SpecialistRoom** (специалист×кабинет).
- [x] Сидирование одной дефолтной организации/клиники (текущая клиника Берсона) + перенос текущих филиалов/специалистов в новую модель (с маппингом на Rubitime-идентификаторы).

### 1.2 Услуги и гибкая доступность (ТЗ §3, §21.1, §22.1, §24.3)
- [x] Сущность **Service** — общая для клиники (название, длительность, стоимость, описание, активность, флаги: предоплата-применима, использование в абонементах, онлайн-оплата-применима, показ в публичном виджете). Финальный состав полей предоплаты — на этапе 5, но колонки-флаги закладываются здесь.
- [x] Связь **SpecialistServiceAvailability**: доступность услуги на уровнях специалист / специалист×филиал / специалист×кабинет / специалист×город.
- [x] Связь **ServiceLocationAvailability** (услуга×локация) — для фильтра «город → доступные услуги».
- [x] **Запрет дублей**: одна услуга «Приём 60 минут» — одна запись; разная доступность — только через связи (модель должна делать дубль-подход невозможным/ненужным).
- [x] Флаг видимости услуги: «для публичной записи» vs «только ручное назначение администратором» (ТЗ §3.2).

### 1.3 Каноническая запись и статусы (ТЗ §17, §22.2)
- [x] Каноническая сущность **Appointment** в собственной БД (организация, филиал, кабинет, специалист, услуга, пациент, время начала/конца, длительность, источник записи, ссылки на оплату/абонемент — nullable до этапов 5/6).
- [x] Единый **enum статусов записи** (ТЗ §17): создана; ожидает оплаты/предоплаты; оплачена; подтверждена; перенесена; отменена пациентом; отменена специалистом; поздняя отмена; неявка; завершена; посещение подтверждено; списана по абонементу; требует ручного решения.
- [x] Машина переходов статусов в service-слое (валидные переходы; нельзя из терминального в активный без явного правила). Документировать диаграмму переходов (`CANONICAL_MODEL.md`, `appointmentStatusFsm.ts`).
- [x] Поля для будущей анти-обход-логики: `original_start_at` (исходная дата), счётчик переносов — закладываются здесь (заполняются на этапе 4).

### 1.4 События и история (ТЗ §16.2, §21.6, C3)
- [x] Append-only **AppointmentEvent** (тип события, актор, время, payload) — пишется при каждом значимом изменении записи.
- [x] Заложены каркасы таймлайнов: **PatientTimelineEvent** (агрегатор), **AppointmentHistoryEvent**; `PaymentHistoryEvent`/`PackageHistoryEvent` — модель добавляется на этапах 5/6, но единый формат таймлайна определяется здесь.
- [x] Правило: запрещено хранить состояние без истории — каждая мутация ядра порождает событие.

### 1.5 Совместимость с Rubitime (ТЗ §1, §22.2, C8)
- [x] Маппинг канонических сущностей ↔ Rubitime (branch/service/specialist/record id) в адаптерном модуле, изолированном от ядра.
- [x] Проекция текущих `appointment_records`/`rubitime_records` на канонические `Appointment` (read-bridge) — чтобы данные не потерялись на переходе.
- [x] Решение по направлению истины на этапе 1: ядро — канон; read-bridge и маппинги; мост включается/отключается `booking_rubitime_bridge_enabled`. **Двусторонняя** синхронизация — этапы 2–4.
- [x] Не ломать текущий Rubitime-вебхук-пайплайн (`RUBITIME_BOOKING_PIPELINE.md`); добавить, а не заменить, пока этап 2 не переключит запись на канон.

### 1.6 Миграции и архитектура
- [x] Все новые таблицы — Drizzle schema + SQL-миграции `db/drizzle-migrations` (`bookingEngine.ts`); без `pool.query` в приложении.
- [x] Доменные модули с `ports.ts`; инфраструктура реализует порты; сборка в `buildAppDeps`.
- [x] Полиморфные ссылки без FK (валидация в service).
- [x] План совместимости/rollback для миграций (как существующие данные мигрируют).

### Способ реализации (этап 1)
- Сначала спроектировать ER-модель и enum статусов как документ внутри плана этапа, затем Drizzle-схема. Свериться с [`DATA_MODEL_REFERENCE.md`](DATA_MODEL_REFERENCE.md).
- `organization_id` — обязательная колонка; для текущего арендатора — фиксированный seed-UUID в `system_settings`/конфиге.
- Сохранять явные внешние идентификаторы Rubitime в отдельных bridge-колонках/таблице, не смешивая с каноном.

### Приёмка этапа 1
- [x] Можно создать (в БД) полную организационную структуру без дублирования услуг под комбинации.
- [x] Канонический `Appointment` со статусами и событиями существует; проекция Rubitime в него работает; текущий пайплайн не сломан.
- [x] Multi-tenant поля на месте; миграции `0086`–`0088` идемпотентны; целевые тесты + `pnpm run ci` зелёные.
- [x] UI-паритет: админ может просматривать/редактировать организацию/филиалы/кабинеты/специалистов/услуги и связи доступности (см. UI §A1–A5, §A14).

## Этап 2. Базовая запись пациента

**Цель:** создание записи в собственной БД из приложения и публичного входа; настраиваемые поля; проверка доступности; уведомления. (ТЗ §4, §5, §6, §7, §18.)

### 2.1 Создание записи в каноне (ТЗ §4.1–4.2, §22.2)
- [x] `createBooking` создаёт каноническую `Appointment` в собственной БД **без зависимости от Rubitime-ID**; Rubitime-синхронизация — побочный эффект через адаптер (мост), не блокирующий ядро.
- [x] Из приложения авторизованный пациент с заполненным профилем записывается без повторного ввода; недостающие обязательные поля запрашиваются точечно (ТЗ §4.2).
- [x] Идентификация: для внешнего входа — телефон как основной идентификатор; для приложения — текущая сессия/профиль (C5, ТЗ §4.4).

### 2.2 Настраиваемые поля записи (ТЗ §5, §21.3)
- [x] Модель **BookingFormField** (тип, обязательность, видимость пациенту, видимость админу/специалисту, порядок, placeholder/подсказка) на уровне клиники/записи.
- [x] Поддержка типов: имя, фамилия, телефон, email, комментарий, описание проблемы/жалобы, произвольный текст, кастомные поля (ТЗ §5.1, §5.3).
- [x] Комментарий к записи — настраиваемый (вкл/выкл, обязательность) (ТЗ §5.2).
- [x] Сохранение ответов как **BookingFormSubmission**, привязанных к записи (для карточки клиента — C3).
- [x] Валидация обязательных полей на сервере (не только в UI).

### 2.3 Доступность и слоты (ТЗ §6, §7)
- [x] Расчёт доступных слотов по: специалист, филиал, кабинет, услуга, длительность, рабочее расписание, существующие записи, блокировки, отсутствия, правила (предоплата/перенос/отмена — учитываются по мере появления), тех. зазоры между приёмами (если заведены) (ТЗ §7.1).
- [x] Фильтрация: пациент видит только реально доступные варианты; «город → доступные услуги» (ТЗ §3.3, §7.2).
- [x] **Несколько последовательных слотов подряд** (ТЗ §6): показ непрерывных интервалов; проверка реальной свободы цепочки; блокировка непрерывного периода; учёт длительности/кабинета/специалиста/филиала.
- [x] Защита от двойного бронирования (конкурентные брони на один слот/кабинет/специалиста) — на уровне БД (констрейнт/транзакция).

### 2.4 Расписание/рабочие часы (ТЗ §7.1, §21.2)
- [x] Модель **WorkingHours / AvailabilityRule / ScheduleBlock** (рабочее время специалиста по филиалу/кабинету, блокировки, отсутствия/отпуска).
- [x] Ручное создание брони/блока администратором (заготовка под календарь этапа 8).

### 2.5 Уведомления (ТЗ §18, C4)
- [x] Уведомление пациенту: запись создана/подтверждена (ТЗ §18.2).
- [x] Уведомление специалисту/админу: новая запись (ТЗ §18.1).
- [x] Интеграция с существующим reminder/notification пайплайном (не изобретать новый канал).

- [x] Обязательные поля настраиваются админом и валидируются на сервере.
- [x] Цепочка последовательных слотов бронируется корректно; двойное бронирование невозможно.
- [x] Rubitime-зеркало обновляется при включённом мосте; при выключенном — ядро работает.
- [x] UI: пациентский флоу записи (приложение); админ-настройка полей записи; врач/админ видят новую запись (UI §A6, §C-формы, §B-список записей).

## Этап 4. Переносы и отмены

**Цель:** самостоятельные перенос/отмена с гибкими политиками, ручные решения, защита от обхода, история. (ТЗ §8, §9, §17, §22.3, §22.5.)

### 4.1 Политики (ТЗ §8.3, §8.5, §9.1, §21.5)
- [x] Две отдельные политики: **CancellationPolicy** и **ReschedulePolicy** (не одна общая).
- [x] Уровни привязки и разрешение приоритета: клиника / специалист / услуга / продукт (C10, ТЗ §22.6).
- [x] Параметры отмены: срок бесплатной отмены; разрешена ли отмена; поведение при поздней отмене; возврат предоплаты; списание занятия по абонементу; требуется ли подтверждение; уведомления (ТЗ §9.1).
- [x] Параметры переноса: срок самостоятельного переноса; лимит числа переносов; разрешён ли перенос в другой филиал/город/специалиста/услугу; поведение при превышении лимита; подтверждение; уведомления (ТЗ §8.3).

### 4.2 Самостоятельный перенос (ТЗ §8.1–8.2)
- [x] Перенос — изменение существующей записи (не создание новой); сохранение связи с исходной записью, оплатой, абонементом, предоплатой.
- [x] UI переноса использует модуль записи в режиме «перенос» (выбор нового слота с учётом политик и доступности этапа 2).
- [x] Фиксация: история переносов, кто перенёс, был ли в бесплатном периоде, применённые правила.

### 4.3 Самостоятельная отмена (ТЗ §9.1)
- [x] Пациент отменяет, если разрешено; применяются правила бесплатной/штрафной отмены, возврат/удержание предоплаты, списание/несписание занятия (списание интегрируется на этапе 6).

### 4.4 Ручные решения специалиста/админа (ТЗ §9.2–9.3, §22.5)
- [x] Ручная отмена с выбором типа: бесплатная / штрафная / со списанием / без списания / удержание предоплаты / возврат предоплаты / индивидуальное решение.
- [x] Ручной перенос администратором; ручной override любой автоматики.

### 4.5 Защита от обхода штрафов (ТЗ §8.4, §22.3, C7)
- [x] Хранить: `original_start_at`, дату каждого переноса, актора, был ли перенос в бесплатном периоде, применённые штрафные правила, доступность бесплатной отмены после переноса, лимит переносов.
- [x] Право на бесплатную отмену пересчитывается **от исходной записи**, а не от новой даты: перенос не «восстанавливает» бесплатную отмену.

### 4.6 История (ТЗ §8.4, §9.4, C3)
- [x] Append-only события переноса (**AppointmentReschedule**) и отмены (**AppointmentCancellation**) со всеми полями ТЗ §9.4 (кто/когда/причина/бесплатная/штрафная/списание/удержание/возврат/комментарий/уведомления).

### Способ реализации (этап 4)
- Политики — данные (таблицы настроек) + чистый policy-resolver в service (вход: запись, история, политика; выход: разрешено/штраф/возврат).
- Анти-обход — инвариант в resolver, покрытый юнит-тестами на сценарий §8.4.
- Связь оплаты/абонемента сохраняется на всех переходах (C6); реальные деньги/занятия подключаются на этапах 5/6, но контракт переходов закладывается здесь.

### Приёмка этапа 4
- [x] Пациент переносит/отменяет по правилам; лимиты и сроки соблюдаются.
- [x] Сценарий §8.4 не позволяет обойти штраф (тест проходит).
- [x] Специалист/админ делает ручные решения с выбором типа; всё логируется в историю.
- [x] UI: настройки политик (админ); кнопки переноса/отмены (пациент); панель ручных решений (врач/админ) (UI §A8, §C-actions, §B-actions).
