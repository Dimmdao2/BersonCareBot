# UI_SURFACES_CHECKLIST — поверхности кабинетов (админ / врач / пациент / публичный вход)

Требование владельца: **все настройки и вся информация должны быть доступны в UI** соответствующих кабинетов. Этот файл — обязательный чек-лист поверхностей. Этап не закрывается (`done`), если данные/правила существуют в БД, но ими нельзя управлять или их нельзя увидеть в нужном кабинете.

Маркировка ролей: **A** = админ (настройки клиники/SaaS), **B** = врач/специалист (рабочее место, может пересекаться с админом в текущей моноклинике), **C** = пациент (личный кабинет), **P** = публичный вход (вне `/app`).

Соблюдать UI-правила проекта: `patient-ui-shared-primitives.mdc`, `ui-copy-no-excess-labels.mdc`, `ui-select-trigger-display-label.mdc`, `cms-unified-media-picker-layout.mdc` (где есть медиа). Платёжные ключи редактируются в admin Settings и хранятся в `system_settings`.

## A. Админ / настройки клиники (этап ввода в скобках)

- [x] **A1. Организация/клиника** — CRUD клиники, реквизиты, дефолтный tenant (этап 1). UI: `BookingEngineSection`, API `/api/admin/booking-engine/organizations`.
- [x] **A2. Филиалы/локации** — CRUD филиалов (город/адрес/точка) (этап 1). API `branches`, `branches/[id]`.
- [x] **A3. Кабинеты** — CRUD кабинетов внутри филиала (этап 1). API `rooms`, `rooms/[id]`.
- [x] **A4. Специалисты** — CRUD специалистов; привязка к клинике/филиалам/кабинетам (SpecialistLocation/SpecialistRoom) (этап 1). API `specialists`, `specialist-rooms`.
- [x] **A5. Услуги и доступность** — CRUD услуг (название, длительность, цена, описание, активность, флаги: предоплата/абонементы/онлайн-оплата/публичный виджет); матрица доступности `специалист × филиал × кабинет × город × услуга`; флаг «публичная запись vs только ручное назначение» (этап 1; флаги предоплаты заполняются на 5). API `services`, `availability`.
- [ ] **A6. Настройки полей записи** — конструктор BookingFormField: тип, обязательность, видимость пациенту/админу, порядок, placeholder; вкл/обязательность комментария; кастомные вопросы (этап 2).
- [ ] **A7. Публичный виджет** — генерация кода вставки (JS/iframe/popup/ссылка), параметры/UTM, привязка к клинике/филиалу/специалисту/услуге/акции; настройка мерджа пациентов (PatientMergeCandidate) (этап 3).
- [ ] **A8. Политики отмены и переноса** — раздельные CancellationPolicy/ReschedulePolicy; уровни (клиника/специалист/услуга/продукт); сроки, лимиты, разрешения (другой филиал/город/специалист/услуга), поведение при превышении, требование подтверждения, уведомления (этап 4).
- [ ] **A9. Платёжные провайдеры** — добавление/вкл/откл провайдера; ключи/токены/настройки (в `system_settings`, не ENV); выбор провайдера для сценария/страницы/продукта/услуги; несколько способов оплаты (этап 5).
- [ ] **A10. Предоплата** — PrepaymentPolicy по услуге (фикс/процент/полная/откл) (этап 5).
- [ ] **A11. Абонементы-продукты** — конструктор готовых абонементов (состав, цена, срок, правила списания) (этап 6).
- [ ] **A12. Продукты/акции/подписки/курсы** — CRUD продуктов и типов; акции/подарочные сертификаты; подписки; привязка курсов; правила доступа/оплаты/срок; покупка по ссылке (этап 7).
- [ ] **A13. Уведомления** — настройка адресатов/каналов событий (новая запись/перенос/отмена/оплата/покупка/запросы/ручное решение) (этапы 2,4–7).
- [x] **A14. Rubitime-мост** — переключатель вкл/откл; статус/маппинг; read-bridge проекция (этап 1). API `bridge`, `system_settings.booking_rubitime_bridge_enabled`. Двусторонняя синхронизация — этапы 2–4.

## B. Врач / специалист (рабочее место)

- [ ] **B-list. Список записей** — канонические записи (не проекция Rubitime) с фильтрами и статусами (этап 2).
- [ ] **B-calendar. Календарь** — по принципу GCal: записи/слоты/статусы/филиалы/кабинеты/специалисты/услуги; фильтры специалист/филиал/кабинет/день-неделя-месяц; создание/перенос/отмена; статус оплаты/абонемента/комментарии (этап 8).
- [ ] **B-actions. Ручные решения** — отмена с выбором типа (бесплатная/штраф/списать/не списывать/удержать/вернуть предоплату/индивидуально); ручной перенос; override автоматики (этап 4).
- [ ] **B-pay. Оплаты записи** — статус оплаты/предоплаты, история оплат записи, возвраты/удержания (этап 5).
- [ ] **B-package. Абонементы пациента** — состав, остаток, списания, истории посещений/отмен/переносов/оплат, комментарии/ручные решения; создание/назначение индивидуального абонемента + ссылка на оплату (этап 6).
- [ ] **B-products. Продукты пациента** — что куплено (акция/подарок/курс/подписка), оплачено/использовано, остаток визитов; инфо о покупке при записи (этап 7).
- [ ] **B-merge. Мердж пациентов** — обработка кандидатов на объединение профилей (этап 3).
- [ ] **B-card. Карточка клиента** — полная история (записи/посещения/отмены/переносы/оплаты/предоплаты/возвраты/абонементы/списания/продукты/акции/комментарии); комментарии в карточке и к записи; пометка «проблемный»; booking-блокировка (этап 9).

## C. Пациент (личный кабинет, `/app/patient/**`)

- [ ] **C-book. Запись** — выбор города/филиала/специалиста/услуги/слота; автоподстановка профиля; запрос только недостающих полей; несколько слотов подряд (этап 2).
- [ ] **C-actions. Перенос/отмена** — кнопки в своей записи; перенос как изменение существующей; применение политик/лимитов; понятные сообщения о штрафе/невозможности (этапы 4).
- [ ] **C-pay. Оплата/предоплата** — «нужна оплата/предоплата», оплата, статус, история своих оплат (этап 5).
- [ ] **C-package. Абонементы** — активные абонементы, состав, остаток/использовано, срок, история посещений по абонементу, будущие записи, отменённые/перенесённые, статус оплаты (этап 6).
- [ ] **C-products. Продукты/покупки** — купленные акции/подарочные абонементы, доступные продукты, статус оплаты, запись по продукту, остаток; доступ к курсам/материалам/подписке после покупки (этап 7).
- [ ] **C-history. История** — свои визиты/записи/оплаты (срез карточки для пациента) (этап 9).

## P. Публичный вход (вне `/app`, для Tilda)

- [ ] **P-page. Публичная страница записи** — без сессии; те же поля/слоты/доступность, что в приложении (этап 3).
- [ ] **P-widget. Встраивание** — JS-виджет/iframe/popup/ссылка; параметры (клиника/филиал/специалист/услуга/акция/город/услуга) и UTM; работает на стороннем сайте (этап 3).
- [ ] **P-pay. Оплата из публичного входа** — если услуга требует предоплаты, оплата доступна гостю (этап 5, после публичного входа).

## Кросс-роль чек (для каждой настройки/сущности)

Для любой новой настройки или сущности агент обязан подтвердить:
- [ ] Кто **создаёт/редактирует** (роль A или B) — поверхность есть.
- [ ] Кто **видит результат** (B и/или C и/или P) — поверхность есть.
- [ ] Тексты соответствуют `ui-copy-no-excess-labels.mdc` (без лишних пояснений) и patient-терминологии (`patient-lfk-means-rehab-program.mdc`, где применимо).
- [ ] Селекты с opaque-value используют `displayLabel`/`items` (`ui-select-trigger-display-label.mdc`).
- [ ] Patient-страницы используют shared-примитивы (`patient-ui-shared-primitives.mdc`); медиа-превью — статичная картинка по правилу.
