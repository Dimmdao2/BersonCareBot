## Detail

### C1 — Migration SQL correctness — PASS

`0131_patient_files_media_file_id_fk.sql`:

```sql
ALTER TABLE "patient_files"
  ADD COLUMN IF NOT EXISTS "media_file_id" uuid;

ALTER TABLE "patient_files"
  ADD CONSTRAINT "patient_files_media_file_id_fkey"
  FOREIGN KEY ("media_file_id")
  REFERENCES "media_files"("id")
  ON DELETE SET NULL;

CREATE INDEX IF NOT EXISTS "idx_patient_files_media_file_id"
  ON "patient_files" ("media_file_id")
  WHERE media_file_id IS NOT NULL;
```

- Column add is idempotent via `ADD COLUMN IF NOT EXISTS`. ✓
- Column type `uuid`, nullable — matches `media_files.id` PK type (`uuid` defaultRandom). ✓
- FK constraint name `patient_files_media_file_id_fkey` matches the schema-side
  `foreignKey({...}).name` exactly (`patientFiles.ts:66`), so drizzle's introspection
  view of the constraint stays in sync. ✓
- `ON DELETE SET NULL` matches schema `.onDelete("set null")`. ✓
- Partial index name `idx_patient_files_media_file_id`, predicate
  `WHERE media_file_id IS NOT NULL`, and column match the schema index
  (`patientFiles.ts:45-47`) exactly. ✓ Index create is idempotent via `IF NOT EXISTS`. ✓
- Journal entry added (`meta/_journal.json` idx 131, tag `0131_..._fk`, version "7"
  consistent with all prior entries). ✓

Minor note (non-blocking): the `ADD CONSTRAINT` statement does not use a guard
(`IF NOT EXISTS` is not valid Postgres syntax for `ADD CONSTRAINT` until PG16, and the
project does not use a `DO $$` block elsewhere for this). Re-running the migration on a
DB that already has the constraint would fail on the constraint step. This matches how
the rest of this migration directory handles FK adds (no guard), so it is consistent
with project convention; drizzle's `__drizzle_migrations` ledger prevents re-application
in practice. Not a defect against the DoD.

### C2 — Migration method vs DoD — ADVISORY (not FAIL)

The DoD line reads "migration generated by drizzle-kit (no hand raw SQL)". The migration
here is handwritten. I evaluated whether drizzle-kit *could* produce this:

- `pnpm drizzle-kit generate` from the worktree aborts with:

  drizzle-kit cannot run in CI/audit without a live DB connection, so the DoD as
  literally written is impractical to enforce here.
- The project has NOT maintained drizzle snapshots since migration 0035 — `meta/` holds
  only 24 `*_snapshot.json` files (last is `0035_snapshot.json`) for 131 migrations.
  This is conclusive evidence the project switched to handwritten migrations long ago;
  PFI-ST-04 follows the established, repo-wide convention rather than introducing a
  divergence.
- The handwritten SQL is byte-for-byte faithful to what drizzle-kit *would* emit for
  this schema delta (column, FK with matching name + onDelete, partial index with
  matching name + predicate). There is no incompatible difference.

Per the audit rubric: drizzle-kit cannot run (DATABASE_URL required) AND the SQL is
equivalent → ADVISORY only, not FAIL. The DoD wording is the divergence, not the code.
