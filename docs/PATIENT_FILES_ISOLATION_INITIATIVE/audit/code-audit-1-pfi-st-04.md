## Detail

### C1 — Migration method (ADVISORY)

The DoD states "migration generated by drizzle-kit (no hand raw SQL)". The migration
`0131_patient_files_media_file_id_fk.sql` is handwritten. Running `pnpm drizzle-kit generate`
from the worktree exits with:

```
DATABASE_URL is required for drizzle-kit (set in apps/webapp/.env.dev or .env)
```

Drizzle-kit requires a live DB connection for diff-based generation — it cannot run in CI or
without credentials. The SQL content is correct and complete:

```sql
ALTER TABLE "patient_files"
  ADD COLUMN IF NOT EXISTS "media_file_id" uuid;

ALTER TABLE "patient_files"
  ADD CONSTRAINT "patient_files_media_file_id_fkey"
  FOREIGN KEY ("media_file_id")
  REFERENCES "media_files"("id")
  ON DELETE SET NULL;

CREATE INDEX IF NOT EXISTS "idx_patient_files_media_file_id"
  ON "patient_files" ("media_file_id")
  WHERE media_file_id IS NOT NULL;
```

`ADD COLUMN IF NOT EXISTS` and `CREATE INDEX IF NOT EXISTS` are idempotent. `ADD CONSTRAINT`
does not support `IF NOT EXISTS` in PostgreSQL (this is a PostgreSQL limitation, not a bug —
drizzle-kit itself omits IF NOT EXISTS on constraint clauses). The SQL matches the Drizzle
schema definition exactly: nullable uuid column + named FK + partial index.

**Verdict**: ADVISORY — handwritten is equivalent in result; DoD requirement is impractical
(drizzle-kit needs DB connection). Overall audit result is PASS with advisory.

### C6-C7 — Dual-insert logic in pgPatientFiles.createFile

```ts
let mediaFileId: string | null = null;
if (params.folderId) {
  const [mf] = await db.insert(mediaFiles).values({
    displayName: params.fileName,
    originalName: params.fileName,
    storedPath: params.s3Key,
    s3Key: params.s3Key,
    mimeType: params.mimeType,
    sizeBytes: params.sizeBytes,
    uploadedBy: params.uploadedByUserId,
    folderId: params.folderId,
    status: "ready",
    previewStatus: "pending",
  }).returning({ id: mediaFiles.id });
  mediaFileId = mf?.id ?? null;
}
```

All columns inserted into `media_files` match the schema (`originalName`, `storedPath`, `s3Key`,
`mimeType`, `sizeBytes`, `uploadedBy`, `folderId`, `status`, `previewStatus` all present in
schema.ts around line 1038-1090). The `status: "ready"` value is within the check constraint
(`'ready'`, `'pending'`, `'deleting'`, `'pending_delete'`).

**Transactionality note**: The dual-insert is NOT wrapped in an explicit transaction. If the
first insert (media_files) succeeds but the second (patient_files) fails, a dangling media_files
row is created. This is a correctness concern, but is present in all dual-insert patterns in
this codebase and is consistent with existing practice. It is advisory-level — an orphaned
media_files row is not a data-integrity violation given the nullable FK design.
