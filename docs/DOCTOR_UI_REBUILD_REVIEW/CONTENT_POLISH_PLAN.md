# Контент — полировочный проход (редизайн по вайрфрейму)

Generated 2026-06-15. Branch `claude/doctor-pages-rebuild`. По итогам аудита всех
экранов Контента (Plan-агент) vs `docs/design/doctor-cabinet-wireframe.html` (`#p-content`).
Эталон — внутренняя вёрстка вайрфрейма + канон страницы exercises (внешний шелл
вайрфрейма устарел). Принцип: **визуальная консистентность, НЕ ре-архитектура IA;
не ломать рабочий код и формы.**

## Решения по скоупу
**Делаем (низкий/средний риск, чистый визуал/additive):**
- Батч A — навигация + хаб: лёгкие list-item в `ContentNav` вместо «кнопок-пилюль»
  (#1), бейдж счётчика страниц у пунктов (#2), инлайн-иконка видимости внутри пункта
  (#10), хинт-блурб под навигацией (#11), заголовок секции `text-sm` + действия как
  инлайн-чипы (#3), page-level `DoctorPageHeader` h1+подзаголовок (#19).
- Батч B — редактор материала: ширина split в пользу редактора (#5, Фаза 4.3a),
  контекстная ширина сайдбара 340→~300/260 (#6, 4.3a), чип статуса публикации в шапке
  редактора (#8), разделение Сохранить/Опубликовать через `DoctorCatalogPersistPublishBar`
  как КЛИЕНТСКОЕ улучшение (hidden `is_published`, серверный `saveContentPage` НЕ трогаем)
  (#4, 4.3b).
- Батч C — мелкие победы по остальным экранам: лок-бейдж 🔒 на неизменяемых блоках
  Главной пациента (#13, additive), `backHref` в медиабиблиотеке (#14), убрать дубль
  `<h2>` (#16), тулбар-шапка списка в Мотивации (#17).

**НЕ делаем сейчас (риск/ре-архитектура — отдельная инициатива по согласованию):**
- Замена native `<select>` → Radix Select в ContentForm/SectionForm/MediaLibrary
  (#7,#12,#15) — риск сломать сабмит форм ради умеренного визуала.
- Перенос управления секциями / медиабиблиотеки / мотивации внутрь хаба как панелей
  (#18 и пр.) — IA-ре-архитектура, не полировка; Step 5 сознательно выбрал ссылки.
- Постоянный live-preview как колонка (#9, Фаза 4.3c) — L, реструктуризация layout;
  редактор в основном в узком pane → выгода спорная. Опционально позже.
- Миграция `patient-home/page.tsx` на `DoctorAppShell` (#20) — структурно, трогает
  рабочую страницу; вне полировки Контента.

## Порядок
Батчи последовательно (файлы пересекаются: A и B → ContentHubShell; B → ContentForm).
Каждый батч: Sonnet по брифу → ревью дифа → tsc/тесты → коммит. После всех — одна
живая сессия визуального сравнения с вайрфреймом (headless, dev:admin, порт 5311),
затем аккуратный мердж в `feat/doctor-ui-rebuild`.

## Защитные ограничения для агентов
- Не менять `saveContentPage` и имена полей FormData; формы должны сабмититься как есть.
- Не трогать файлы параллельного чата (payments/booking-lifecycle/system-settings/no-show).
- Стейдж только явными путями; без commit/push (ревьюит/коммитит оркестратор).
- tsc чисто + существующие тесты Контента зелёные.
