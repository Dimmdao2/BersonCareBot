# Контент Шаг 4 — качественный инлайн-редактор материалов

Generated 2026-06-15. Branch `claude/doctor-pages-rebuild`. Owner directive: сделать
Шаг 4 «по полной» и **качественный** редактор (текущий — «говна кусок»). Источник
фактуры — аудит редактора (Plan-агент, 2026-06-15) + вайрфрейм `#cs-editor`
(`docs/design/doctor-cabinet-wireframe.html`, ~L1976-1993; внешний шелл вайрфрейма
устарел, эталон — только внутренняя вёрстка редактора).

## Архитектурное решение
**Рефактор `ContentForm` НА МЕСТЕ** (не дубль `ContentFormInline`). Один компонент,
параметризованный режимом отображения, используется И на полностраничном роуте
`/app/doctor/content/edit/[id]` (deep-link, фолбэк — НЕ удаляем), И в инлайн-pane
master-detail. Оба контекста получают качественный редактор, без дублирования логики
сохранения/валидации. Двухколоночная вёрстка из вайрфрейма (`1fr` контент / ~340px
мета-сайдбар) сворачивается в одну колонку в узком pane (адаптив по контейнеру).

Бэкенд НЕ трогаем: без новых портов/SQL/миграций. `saveContentPage` — как есть.
Единственное серверное добавление в 4.2 — тонкий экшен над существующим
`contentPages.getById`.

## Фазы (каждая — независимый коммит + живая проверка)

### Фаза 4.1 — качественный редактор (UI, проверяется на готовом edit-роуте)
Рефактор `ContentForm.tsx` (+ при необходимости мелкие правки `edit/[id]/page.tsx`,
`new/page.tsx`). Без master-detail — проверяется на `/app/doctor/content/edit/<id>`
(в dev-БД есть данные в системных страницах warmup-*).
1. **Двухколоночная вёрстка** (вайрфрейм-эталон): слева — Заголовок, ряд Раздел+Slug,
   Краткое описание, Содержимое (markdown), чекбоксы; справа мета-сайдбар (~340px):
   Картинка, Видео, Связан с курсом, и блок сохранения. На узком контейнере → одна
   колонка.
2. **Sticky save bar** в мета-сайдбаре (или sticky bottom на мобиле) — «Сохранить»
   всегда в зоне видимости; disabled когда нет изменений (dirty-state).
3. **Dirty-state + защита от потери**: вычислять `isDirty` (initial vs current);
   `beforeunload` + подтверждение при `← к списку`/уходе с несохранённым.
4. **Нейтрализовать кнопку картинки в тулбаре Toast UI** (`MarkdownEditorToastUiInner`):
   скрыть нативную кнопку image (CSS `[data-type="image"]`) ИЛИ переопределить на
   `MediaLibraryInsertDialog`. Сейчас она вставляет невалидируемые пути → битый markdown.
5. **`useEffect([recordKey])`-сброс** всех `useState` (title/slug/image/video и пр.)
   при смене записи — паттерн `ExerciseForm` (готовит компонент к инлайн-переиспользованию).
6. **a11y/полировка**: `Label`-примитивы вместо `<span>` (+ починить `id`/`htmlFor`
   у select Раздел и Связан с курсом); Раздел+Slug в один ряд; счётчик `summary.length/2000`.
7. Якорение/скролл к ошибке и баннеру «Сохранено» (сейчас они вверху длинной формы,
   вне зоны видимости после скролла).

**Проверка 4.1 (живьём)**: открыть `/app/doctor/content/edit/<warmup id>` — двухколоночный
редактор, save в сайдбаре, тулбар без «битой» картинки, медиа-вставка через библиотеку,
сохранение работает, баннер виден. tsc + затронутые тесты зелёные.

### Фаза 4.2 — инлайн master-detail
1. **Серверный экшен** `loadContentPageForInlineEdit(id)` (`"use server"`) над
   `deps.contentPages.getById` → editable shape для `ContentForm`. Загрузка по выбору
   (не отдаём bodyMd всех страниц сразу — раздувает RSC payload).
2. **`ContentMasterDetailPane`** (новый client-компонент): `CatalogSplitLayout`
   left = список(и) материалов в режиме выбора, right = `CatalogRightPane` →
   `ContentForm` с `key={selectedPageId}` (форс-ремаунт сбрасывает Toast UI),
   скелетон на время загрузки записи, шапка `← к списку`. Десктоп — асимметричный
   split (редактор шире, напр. `lg:grid-cols-[minmax(0,1fr)_minmax(0,1.4fr)]`),
   мобайл — slide через `mobileView`/`mobileBackSlot`.
3. **Режим выбора в `ContentPagesSectionList`**: строки (сейчас `<Link>`) и плитки
   (у `ContentPageTileCard` уже есть `onSelect`) умеют звать `onSelect(id)` вместо
   навигации, когда передан колбэк. DnD остаётся только в списочном режиме.
4. **Подключение в панях**: `ArticleSectionPane` (один список) и `SystemFolderPane`
   (стопка подразделов в общем left, один общий right-pane) рендерят
   `ContentMasterDetailPane`. Роут `/content/edit/[id]` остаётся рабочим (deep-link).

**Проверка 4.2 (живьём)**: Разминки → клик по материалу открывает редактор в правом
pane; переключение материалов корректно сбрасывает форму и редактор; сохранение
обновляет список; `← к списку`; мобильный slide. Без hydration-ошибок.

## Риски и как смягчаем
- **Ремаунт тяжёлого Toast UI при смене выбора** → `key={id}` + `dynamic(ssr:false)`
  уже даёт лоадер; если «мигание» критично — fallback на императивный
  `editor.setMarkdown()` через `useEffect` (заложено как опция, не делаем сразу).
- **Сброс `useState` формы при инлайн-переиспользовании** (B.5 аудита) — закрывается
  пунктом 4.1.5 (`useEffect([recordKey])`) И `key` на pane в 4.2 (двойная страховка).
- **Не сломать рабочий edit-роут** — ContentForm остаётся совместимым; 4.1 проверяется
  именно на нём до начала 4.2.
