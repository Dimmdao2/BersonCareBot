# ROADMAP — доработка записи

Статусы этапов: `pending` · `in_progress` · `blocked` · `done` · `cancelled`.

Эта инициатива не заменяет [`OWN_BOOKING_ENGINE_INITIATIVE`](../OWN_BOOKING_ENGINE_INITIATIVE/README.md). Она является следующим продуктовым слоем над уже созданным собственным booking engine: привести кабинет записи, абонементы и календарь к понятному рабочему состоянию для текущего продукта.

## 1. Контекст

Сейчас кабинет записи технически закрывает много внутренних чек-листов, но в продукте выглядит как набор низкоуровневых сущностей:

- `Кабинет` / room/resource показан как обычная сущность, хотя текущий продукт первично делается для одного специалиста.
- В админке одновременно видны канонический `be_*` слой, legacy `booking_*` слой и Rubitime-проекция.
- В интерфейсе появляются формулировки вроде `специалист × услуга × филиал × город × кабинет`, которые описывают таблицы/связи, а не пользовательскую задачу.
- Цена вводится как `priceMinor`, из-за чего пользователь вводит копейки вместо рублей.
- `Только вручную` не объясняет, что услуга скрыта от самостоятельной записи пациента, но остается доступной персоналу.
- Расписание показывает рабочие часы и блокировки как технические записи, без нормального редактирования дней, интервалов и перерывов.
- Форма записи выглядит как таблица внутренних ключей, а не как конструктор вопросов.
- Абонементы не дают нормальной работы с индивидуальными предложениями, комментариями, отвязкой сеансов и списанием при отмене.
- Календарь пока недостаточно интерактивен: нет полноценного перемещения/resize, детали неудобны, отмененные записи могут накладываться на новые записи в то же время.

## 2. Целевая продуктовая модель

Основной сценарий на текущем этапе:

```text
Один специалист
  → несколько локаций
  → общий список услуг
  → доступность услуг по локациям
  → расписание по локациям
  → записи клиентов
```

У специалиста может быть несколько локаций/филиалов. В разных локациях может быть разный набор услуг: например, в Москве доступны 40 и 60 минут, в Санкт-Петербурге — 60 и 90 минут. Расписание также может отличаться по локациям: разные дни, начало/конец рабочего дня, перерывы, буферы между приемами.

`Кабинет` не является основной продуктовой сущностью для текущего приложения. Если в БД остается `roomId`, он используется как внутренний ресурс/default room и не показывается в основном кабинете записи.

Rubitime на переходный период трактуется как внешний адаптер:

```text
BersonCare
  наша локация + наша услуга + один специалист

Rubitime
  rubitime-филиал + rubitime-специалист-дубль + rubitime-услуга
```

Rubitime-дубли специалистов не должны попадать в обычный кабинет записи. Они видны только на экране интеграционного маппинга.

## 3. Вне Scope

- Онлайн-направление консультаций не перерабатывается в этой инициативе.
- Новая SaaS-модель для клиники с несколькими самостоятельными специалистами не является целью текущего этапа.
- Полное удаление `roomId` / `be_rooms` из БД не выполняется в первом проходе. Сначала делается продуктовый UX-слой, который скрывает лишнее.
- Переписывание всех legacy таблиц `booking_*` с нуля не выполняется в первом проходе. Они могут оставаться скрытой переходной таблицей маппинга для Rubitime.

## 4. Главные решения

1. **Solo-specialist UI.** В кабинете записи пользователь не выбирает специалиста. Система использует одного default specialist.
2. **Локации вместо филиал/город/кабинет.** В обычном UI локация объединяет то, что нужно пользователю: название места приема, адрес/описание, активность, порядок.
3. **Услуги как единый каталог.** Услуга существует один раз и не дублируется под филиал/Rubitime-специалиста.
4. **Доступность как `услуга × локация`.** Пользователь включает или выключает услугу в конкретной локации.
5. **Расписание по локации.** В каждом дне может быть несколько рабочих интервалов; перерыв выражается разрывом между интервалами или отдельной сущностью `Перерыв` в UI.
6. **Rubitime-маппинг отдельно.** Связь нашей локации/услуги с Rubitime-филиалом/специалистом/услугой настраивается на отдельной вкладке интеграции.
7. **Абонементы списывают по реальной услуге независимо от источника записи.** Запись из BersonCare и запись из Rubitime должны одинаково определять услугу для резервирования/списания.
8. **Календарь становится рабочим инструментом.** Запись можно открыть, перенести, изменить длительность, увидеть отмены и историю конфликта времени.
9. **Полный UI-проход владельцем — обязательный gate.** Инициатива не закрывается без явного принятия нового интерфейса владельцем постановки.

## 6. Этап 0 — Инвентаризация и IA

### Цель

До реализации зафиксировать, какие текущие экраны остаются, какие заменяются, какие становятся advanced/integration-only, и какие контракты нельзя ломать в переходный период.

### Задачи

- Составить карту текущих вкладок `/app/doctor/admin/booking`:
  - `Обзор`
  - `Каталог`
  - `Доступность`
  - `Расписание`
  - `Форма`
  - `Правила`
  - `Оплата`
  - `Публичная`
  - `Операции`
  - `Интеграции`
- Для каждой вкладки определить:
  - что остается в основном UI;
  - что переносится в integration/advanced;
  - что переименовывается;
  - что удаляется из видимого UX без удаления из БД.
- Зафиксировать текущие источники данных:
  - canonical `be_*`;
  - legacy `booking_*`;
  - `patient_bookings`;
  - `appointment_records`;
  - `be_external_entity_mappings`;
  - `system_settings`.
- Отдельно описать текущую зависимость очной записи от `branchServiceId` и решение по переходному периоду:
  - первый этап: скрыть `branchServiceId` из UX;
  - дальнейший этап: перевести очную запись на канонические `{ branchId, serviceId }` и использовать Rubitime mapping adapter.

### Проверки

- `rg "branchServiceId" apps/webapp/src` — понять все runtime-зависимости.
- `rg "roomId|be_rooms|specialist-rooms" apps/webapp/src apps/webapp/db` — проверить, где room нельзя просто скрыть.
- `rg "booking_slots_read_source|booking_doctor_appointments_read_source" apps/webapp/src` — проверить read-source переключатели.
- Документ IA должен быть согласован до начала крупных UI-правок.

### Критерии Закрытия

- Есть согласованная карта новых вкладок.
- Есть список старых сущностей, которые скрываются из основного UX.
- Есть список контрактов Rubitime, которые сохраняются на переходный период.

## 7. Этап 1 — Новый Кабинет Записи: Solo UX

### Цель

Сделать основной кабинет записи понятным для работы одного специалиста с клиентами. Пользователь должен видеть не таблицы/связи, а рабочие задачи: локации, услуги, доступность, расписание, форма, проверка записи.

### Новая Навигация

Целевая структура вкладок:

1. `Обзор`
2. `Локации`
3. `Услуги`
4. `Доступность`
5. `Расписание`
6. `Форма`
7. `Правила`
8. `Оплата`
9. `Абонементы и продукты`
10. `Публичная запись`
11. `Операции`
12. `Интеграция Rubitime`

Текущая вкладка `Каталог` разделяется на `Локации` и `Услуги`. Текущая вкладка `Интеграции` становится местом для Rubitime-маппинга и технических переключателей.

### 7.1. Обзор

Обзор должен показывать рабочее состояние записи, а не технический readiness.

Показывать:

- включена ли запись;
- сколько активных локаций;
- сколько активных услуг;
- сколько услуг доступно пациентам;
- есть ли расписание на ближайшие дни;
- есть ли услуги без доступности;
- есть ли Rubitime-маппинги, которые не заполнены;
- есть ли конфликт read-source режимов;
- быстрые действия:
  - добавить локацию;
  - добавить услугу;
  - настроить расписание;
  - проверить слоты;
  - открыть Rubitime-маппинг.

Не показывать в основном блоке:

- `Канон` / `Rubitime` как непонятный режим без контекста;
- сырые mapping counts как основную информацию;
- `кабинеты` как обязательный readiness пункт.

### 7.2. Локации

Локация — пользовательская сущность для места приема.

Поля:

- название;
- адрес/описание;
- timezone;
- активна;
- порядок отображения;
- служебный city code скрывается или показывается только в advanced.

UI:

- список карточек или таблица локаций;
- создание/редактирование в одном стиле;
- активность через понятный переключатель `Показывать пациентам`;
- не показывать `Кабинет`.

Техническая реализация:

- Основной источник: `be_branches`.
- `be_rooms` не удалять, но не показывать в основном UI.
- Если текущий код требует room, использовать default internal room без ручной настройки пользователя.

### 7.3. Услуги

Услуга — общий каталог услуг специалиста.

Поля:

- название;
- длительность в минутах;
- цена в рублях;
- описание для пациента;
- активна;
- доступна для самостоятельной записи пациентом;
- доступна для абонементов;
- применима предоплата/онлайн-оплата, где уже используется платежный слой.

Обязательное исправление:

- UI вводит и показывает цену в рублях.
- В БД продолжает храниться `priceMinor`.
- Пользователь не должен вводить копейки.

Переименование:

- `Только вручную` заменить на понятный язык:
  - `Доступна пациентам`;
  - или `Скрыта от самостоятельной записи`.

Техническая реализация:

- Основной источник: `be_clinic_services`.
- При сохранении UI переводит рубли в `priceMinor`.
- При редактировании UI показывает `priceMinor / 100`.

### 7.4. Доступность

Текущую матрицу `специалист × услуга × филиал × город × кабинет` заменить на понятную матрицу `услуга × локация`.

Целевой вид:

```text
Услуга              Москва      Санкт-Петербург
Сеанс 40 минут      включено    выключено
Сеанс 60 минут      включено    включено
Сеанс 90 минут      выключено   включено
```

Правила:

- Специалист в solo-режиме подставляется автоматически.
- `Кабинет` не участвует в основном UI.
- `Город` отдельно не вводится, если уже выбрана локация.
- Связь `услуга × филиал` не показывается как отдельная сущность.
- Пользователь видит результат: где какая услуга доступна.

Техническая реализация:

- Использовать `be_service_location_availability` и `be_specialist_service_availability`.
- Для solo specialist при включении услуги в локации создавать/активировать необходимые canonical availability rows.
- При выключении деактивировать связь без удаления истории.
- В таблице связей заменить прочерки на явные scope labels:
  - `все локации`;
  - `без ограничения`;
  - `не задано`;
  - `внутренний ресурс`.

### 7.5. Расписание

Расписание должно быть экраном управления рабочим временем, а не списком технических строк.

Основной сценарий:

1. Пользователь выбирает локацию.
2. Видит дни недели.
3. В каждом дне задает несколько рабочих интервалов.
4. Перерыв задается разрывом между интервалами.

Пример:

```text
Москва
Пн: 09:00–13:00, 14:00–18:00
Вт: 09:00–13:00, 14:00–18:00
Ср: выходной
```

Обязательные возможности:

- добавить интервал;
- редактировать интервал;
- удалить/отключить интервал;
- копировать день на другие дни;
- видеть, для какой локации задано расписание;
- видеть fallback только как предупреждение, а не как нормальное состояние;
- проверить свободные слоты глазами пациента.

Исключения:

- текущие `Блокировки расписания` переименовать в `Исключения` или `Занятое время`;
- типы UI:
  - `Перерыв`;
  - `Отсутствие`;
  - `Занято`;
  - `Ручная запись`, если это требуется календарю;
- в основном сценарии обед/перерыв лучше выражать несколькими интервалами рабочего дня.

Буфер:

- добавить управление буфером между приемами:
  - 0;
  - 10;
  - 15;
  - 30 минут;
  - пользовательское значение при необходимости в advanced.

Минимальное время до записи:

- добавить настройку минимального notice period, например `не позднее чем за 12 часов`.
- Хранить как настройку в БД, не в env.

Техническая реализация:

- `be_working_hours` уже поддерживает несколько строк на один weekday.
- Нужен UI для редактирования существующих строк, а не только `Отключить`.
- `be_schedule_blocks` использовать для исключений, отсутствий и занятых интервалов.

### 7.6. Форма Записи

Форма должна стать конструктором вопросов.

Показывать:

- `Имя`;
- `Телефон`;
- `Email`;
- `Комментарий`;
- дополнительные вопросы.

Для каждого поля:

- показывать пациенту;
- обязательно;
- подпись;
- placeholder/подсказка;
- порядок через drag/drop или кнопки вверх/вниз.

Скрыть из основного UI:

- `fieldKey`;
- raw `fieldType`;
- `sortOrder`;
- `visibleToStaff`, если нет отдельного staff-сценария.

Добавление нового вопроса:

- пользователь вводит текст вопроса;
- выбирает тип ответа: текст, телефон, email, комментарий/многострочный текст;
- система сама генерирует технический ключ.

### 7.7. Проверка Слотов

Текущий технический probe заменить на `Проверка записи глазами пациента`.

Поля:

- локация;
- услуга;
- дата;
- результат: список доступных времен.

Задача блока — быстро понять, что пациент реально увидит при записи.

### Цель

Сделать Rubitime переходной интеграцией, а не языком основного продукта.

### Текущая Проблема

Очная запись сейчас завязана на `branchServiceId` из legacy `booking_branch_services`. Эта строка содержит:

- legacy branch;
- legacy specialist;
- legacy service;
- Rubitime branch id;
- Rubitime cooperator id;
- Rubitime service id.

Из-за этого в продуктовый UI протекают Rubitime-дубли специалистов и услуг.

### Целевой UX

Обычный кабинет:

```text
Наша локация → наша услуга → слот → запись
```

Вкладка `Интеграция Rubitime`:

```text
Наша локация | Наша услуга | Rubitime филиал | Rubitime специалист | Rubitime услуга
Москва       | 60 минут    | Москва          | Дмитрий Москва      | Сеанс 60 минут
СПб          | 90 минут    | СПб             | Дмитрий СПб         | Сеанс 90 минут
```

### Задачи

- Оставить legacy `booking_*` как переходный mapping source, но убрать его из основного UX.
- В `RubitimeSection` разделить:
  - справочник Rubitime;
  - таблицу связей нашей модели с Rubitime;
  - технические действия синхронизации.
- Для каждой пары `наша локация + наша услуга` показывать состояние:
  - замапплено;
  - не замапплено;
  - Rubitime-цель отключена;
  - конфликт длительности/цены;
  - отсутствует обратный mapping.
- Запись из BersonCare должна брать Rubitime ids через mapping adapter.
- Запись из Rubitime должна маппиться обратно в нашу локацию и нашу услугу.
- В обычном кабинете не показывать Rubitime specialists как реальных специалистов.

### Переходная Реализация

Первый безопасный шаг:

- пациентский/public flow может продолжать технически использовать `branchServiceId`;
- UI и админка перестают показывать `branchServiceId`;
- связь `наша локация + наша услуга → branchServiceId` настраивается в Rubitime-маппинге.

Следующий шаг:

- заменить вход очной записи на canonical `{ branchId, serviceId }`;
- `branchServiceId` оставить только во внутреннем Rubitime resolver;
- `booking_slots_read_source=canonical` становится целевым режимом;
- `booking_doctor_appointments_read_source=canonical` становится целевым режимом после проверки календаря и списков.

### Проверки

- Запись из нашего кабинета в Rubitime отправляет правильные `rubitimeBranchId`, `rubitimeCooperatorId`, `rubitimeServiceId`.
- Webhook/проекция из Rubitime создает/обновляет нашу запись с правильной локацией и услугой.
- В обычном кабинете нет дублей специалистов Rubitime.
- Все незамаппленные пары явно видны в интеграции.

## 9. Этап 3 — Абонементы: UX и Корректное Списание

**Статус:** `done` (2026-06-04, код + автотесты; ops smoke Rubitime — по [`ACCEPTANCE_STAGE3.md`](ACCEPTANCE_STAGE3.md)).

**Исполнение:** [`STAGE3_DECOMPOSITION.md`](STAGE3_DECOMPOSITION.md) (блоки 3.0–3.6) · приёмка: [`ACCEPTANCE_STAGE3.md`](ACCEPTANCE_STAGE3.md) · журнал: [`LOG.md`](LOG.md).

### Цель

Доработать абонементы так, чтобы они работали с реальными услугами текущей базы, корректно списывались независимо от источника записи и давали врачу понятное управление привязанными сеансами.

### 9.1. Индивидуальный Абонемент

Для индивидуального абонемента **не нужно название**.

Вместо названия важны:

- пациент;
- состав абонемента;
- цена;
- срок действия;
- комментарий;
- статус оплаты/активации;
- история списаний.

Комментарий:

- у абонемента клиента должно быть опциональное поле комментария;
- комментарий нужен и для индивидуального абонемента;
- комментарий нужен и при покупке готового шаблона;
- комментарий виден врачу/админу в карточке клиента и в блоке абонемента.

### 9.2. Позиции Абонемента

Позиции для абонемента должны браться из существующих позиций актуальной базы.

Текущий переходный период:

- запись может идти через Rubitime;
- сеансы/услуги могут приходить из Rubitime-каталога;
- canonical service должен быть определен независимо от источника записи.

Требование:

- если запись создана из BersonCare, абонемент определяет услугу по canonical service id;
- если запись пришла из Rubitime, абонемент определяет услугу через mapping Rubitime → canonical service;
- если mapping отсутствует, запись не должна молча списывать неправильную позицию;
- в UI должно быть видно, что списание невозможно из-за отсутствующего mapping.

### 9.3. Резервирование и Списание

Абонемент должен поддерживать понятные состояния:

- будущая запись зарезервировала позицию;
- запись завершена и позиция списана;
- запись отменена и резерв вернулся;
- поздняя отмена списала позицию как штраф;
- врач вручную решил вернуть позицию;
- врач вручную решил списать позицию как оказанную.

По умолчанию:

- при отмене сеанса отвязка/возврат резерва происходит автоматически;
- если отмена поздняя и политика требует штрафа, система применяет правило списания;
- врач видит итог и может выполнить разрешенное ручное действие.

### 9.4. Отвязка Записи от Абонемента

Текущий блок `отвязать` с вводом id записи неприемлем.

Нужно показывать список сеансов по абонементу:

- будущие записи;
- прошедшие записи;
- отмененные записи;
- статус связи с абонементом;
- что было списано/зарезервировано/возвращено;
- дата, время, услуга, локация, статус записи.

Основной вид:

- по умолчанию показывать только будущие записи;
- рядом с выбором записи должна быть галочка `Показать прошедшие`;
- прошедшие записи появляются только после включения галочки.

Действия:

- отвязать и вернуть в абонемент;
- отвязать и оставить списанной;
- списать как оказанную;
- открыть запись в календаре/деталях.

### 9.5. Защита от Ошибочной Отвязки Прошедших Записей

Для прошедших записей требуется двойное подтверждение.

Первое подтверждение:

```text
Точно отвязать эту запись и вернуть занятие в абонемент?
```

Второе подтверждение:

```text
Сеанс уже прошел. Проверьте, что выбрана правильная запись.
```

Если врач не подтверждает оба шага, действие не выполняется.

### 9.6. Настройка Врача

В настройках врача/админа нужен флаг:

```text
Разрешить отвязывать прошедшие записи от абонемента
```

Правила:

- если флаг выключен, прошедшие записи нельзя отвязать через UI;
- если флаг включен, прошедшие записи можно отвязать только с двойным подтверждением;
- настройка хранится в БД, не в env;
- ключ должен быть добавлен в разрешенные настройки, если используется `system_settings`.

### 9.7. Поздняя Ручная Отвязка

Если врач отвязывает запись после разрешенного периода, например позже чем за сутки до сеанса, система должна предложить выбор:

1. `Списать услугу из абонемента как оказанную`
2. `Вернуть занятие в абонемент`

Нельзя выполнять такое действие молча.

Правило периода должно быть настройкой политики, а не хардкодом.

### 9.8. UI В Карточке Клиента

В блоке абонемента клиента показывать:

- состав абонемента;
- остатки по каждой услуге;
- будущие привязанные записи;
- прошедшие списания;
- отмены и возвраты;
- комментарий;
- историю ручных решений.

### Проверки

- Создание индивидуального абонемента без названия.
- Создание/покупка готового шаблона с комментарием клиента.
- Создание записи из BersonCare резервирует правильную позицию.
- Запись из Rubitime через mapping резервирует/списывает правильную позицию.
- Отмена будущей записи возвращает резерв автоматически.
- Поздняя отмена применяет политику списания.
- Отвязка будущей записи работает из списка.
- Прошедшие записи скрыты по умолчанию.
- Прошедшие записи появляются после галочки `Показать прошедшие`.
- Отвязка прошедшей записи требует двойного подтверждения.
- Флаг врача запрещает/разрешает отвязку прошедших.

## 10. Этап 4 — Интерактивный Календарь

**Исполнение:** [`STAGE4_DECOMPOSITION.md`](STAGE4_DECOMPOSITION.md) (блоки 4.0–4.8) · **приёмка:** [`ACCEPTANCE_STAGE4.md`](ACCEPTANCE_STAGE4.md).
**Статус этапа:** `done` (код + targeted auto-checks, 2026-06-04). Ops: staging read-source cutover сделан; prod cutover — defer (см. [`LOG.md`](LOG.md)).

**Зафиксировано (2026-06-04):** читать записи только из нашей БД; фон — рабочее время и перерывы из канона (не слоты Rubitime), с настройкой **`booking_calendar_show_working_hours`** (можно скрыть); готовая calendar-библиотека с DnD/resize/long-press; при конфликте Rubitime — откат + обновление календаря (webhook/poll).

### Цель

Сделать календарь рабочим инструментом врача/админа, а не только read-only сеткой.

### 10.1. Основные Действия

Календарь должен поддерживать:

- открыть детали записи по клику/тапу;
- перенести запись drag/drop на другое время;
- изменить длительность перетягиванием за край;
- создать ручную запись;
- отменить запись;
- увидеть оплату/предоплату;
- увидеть абонемент/списание;
- увидеть комментарии staff;
- увидеть историю переносов и отмен.

### 10.2. Перемещение Записи

При drag/drop:

- показывать новое время до сохранения;
- проверять доступность слота;
- учитывать специалиста solo-mode;
- учитывать локацию;
- учитывать длительность;
- учитывать занятость и исключения;
- после сохранения писать lifecycle event;
- при включенном Rubitime sync обновлять Rubitime синхронно; при conflict откатывать локальный create/reschedule и возвращать понятную ошибку.

Если перенос нарушает правила:

- показывать понятную причину;
- не создавать новую запись;
- не терять старую запись.

### 10.3. Изменение Длительности

Resize за край должен:

- менять `endAt`;
- пересчитывать duration;
- проверять пересечения;
- учитывать услугу и правила длительности;
- показывать предупреждение, если новая длительность отличается от стандартной длительности услуги;
- писать событие в историю.

### 10.4. Детали Записи

При клике/тапе открывать side panel / dialog с:

- пациент;
- телефон/email;
- услуга;
- локация;
- дата/время;
- длительность;
- статус;
- источник записи;
- Rubitime id/manage URL, если есть;
- оплата/предоплата;
- абонемент и статус списания;
- комментарии;
- история действий.

### 10.5. Отмененные Записи и Наложения

Сейчас при отмене записи и последующей записи на то же время записи могут накладываться. Нужно отображать их рядом.

Правило отображения:

- основная активная запись занимает примерно 3/4 ширины ячейки/колонки;
- отмененная запись в то же время отображается как узкая полоска на оставшуюся 1/4;
- полоска имеет понятный признак отмены, например крестик;
- при клике/тапе на полоску открываются детали отмены;
- если отмен в этом времени несколько, детали показывают список всех отмен.

Нельзя скрывать отмены полностью, потому что врачу нужна история и контекст, почему время переиспользовано.

### 10.6. Шапка и Даты

Исправить шапку календаря:

- понятные даты в режиме дня;
- понятный диапазон в режиме недели;
- понятный месяц в режиме месяца;
- корректные подписи при переходе между периодами;
- единый формат дат на русском языке;
- отсутствие визуального перекоса/обрезки.

### 10.7. Мобильность

Для touch-устройств:

- tap открывает детали;
- перенос не должен случайно срабатывать от обычного скролла;
- если drag/drop на мобильном ненадежен, должно быть действие `Перенести` через dialog, но desktop drag/drop остается обязательным.

### Проверки

- Клик по записи открывает детали.
- Drag/drop переносит запись и пишет lifecycle.
- Resize меняет длительность и проверяет конфликты.
- Активная запись и отмененная запись на то же время отображаются рядом, а не друг на друге.
- Несколько отмен на одно время открываются списком.
- Шапка корректна в day/week/month.
- Календарь врача работает на canonical read-source; после smoke этапа 4 переключатель `booking_doctor_appointments_read_source` переводится в `canonical` для списка записей/KPI.

## 12. Риски

### Риск 1: `branchServiceId` остается runtime-контрактом

Пока очная запись принимает `branchServiceId`, Rubitime-дубли будут существовать под капотом.

Решение:

- в первом проходе скрыть это из UX;
- отдельным техническим этапом перевести очную запись на canonical `{ branchId, serviceId }`;
- оставить `branchServiceId` только как Rubitime mapping adapter.

### Риск 2: canonical и Rubitime read-source расходятся

Докторский календарь/список и пациентские слоты могут читать разные источники.

Решение:

- явно показывать режим в integration/diagnostics;
- не показывать технические режимы в основном UI;
- перед cutover на canonical провести отдельную проверку календаря, слотов и webhook-проекций.

### Риск 3: абонементы спишут неправильную услугу

Если запись пришла из Rubitime и mapping неполный, списание может быть неверным.

Решение:

- запрещать автоматическое списание при отсутствии canonical service mapping;
- показывать проблему врачу;
- требовать заполнения mapping.

### Риск 4: интерактивный календарь затронет много lifecycle-логики

Drag/drop и resize меняют запись и должны уважать политики.

Решение:

- использовать существующие lifecycle API там, где возможно;
- не делать прямой update записи в UI;
- писать события истории;
- покрыть перенос/resize targeted tests.
