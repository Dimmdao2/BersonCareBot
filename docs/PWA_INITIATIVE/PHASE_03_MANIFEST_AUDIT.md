# Фаза 3 — Аудит манифеста после лендинга

**Цель:** убедиться, что Web App Manifest и иконки согласованы с продакшен-раздачей и сценарием «открыть с иконки → `/app/patient`».

**Пререквизиты:** [фаза 1](PHASE_01_ROOT_LANDING.md) (корень больше не редиректит молча в `/app`). Можно выполнять **до** SW из фазы 2.

## Чеклист

### Код и статические файлы (закрыто)

- [x] Поля в **`manifest.ts`**: `name`, `short_name`, `start_url`, `scope`, `display`, `theme_color`, иконки — согласованы с продуктом.
- [x] **`purpose: "maskable"`** и отдельные maskable‑PNG **не вводили** (нет ассетов в репо); при появлении набора — отдельный PR (см. `LOG.md`).
- [x] **`scope: "/"`** покрывает и **/**, и **`/app/*`**.
- [x] **`start_url: "/app/patient"`** — сценарий «иконка → кабинет пациента» задан в манифесте.

### На стенде (после деплоя)

- [ ] **Chrome DevTools → Manifest** — без ошибок; визуально совпадает с ожиданиями продукта.
- [ ] После установки с иконки: открытие ведёт в **`/app/patient`**; откат на **`/`** только из‑за явных редиректов auth/guards (ожидаемое поведение).
