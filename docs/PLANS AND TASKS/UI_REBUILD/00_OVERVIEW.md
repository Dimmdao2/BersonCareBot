# UI Rebuild: обзор этапов

Перестройка клиентского webapp: UI пациента, определение ролей, контент-менеджмент.

## Приоритеты

- **P1:** Восстановить определение администратора/специалиста (этап 8) — КРИТИЧНО, делать ПЕРВЫМ.
- **P2:** Базовая структура контента в БД (этап 9).
- **P3:** Редактирование контента администратором (этап 10).
- **UI:** Этапы 1–7 — перестройка интерфейса пациента.

## Зависимости

```
Этап 8 — ПЕРВЫЙ, независимый. Не блокирует UI-этапы, но критичен для прод.

UI-этапы:
  Этап 1 → Этап 2 (drawer ссылается на /app/patient/profile)
  Этап 1 → Этап 3 (drawer ссылается на /app/patient/notifications)
  Этап 2 → Этап 3 (профиль убирается из settings)
  Этап 4, 5, 6, 7 — параллельны после 1–3

Контент:
  Этап 9 — независим (можно параллельно с UI-этапами)
  Этап 9 → Этап 10 (редактор требует таблицу в БД)
  Этап 7 (видеоплеер) + Этап 9 (данные из БД) — дополняют друг друга
```

## Рекомендуемый порядок выполнения

1. **Этап 8** — фикс ролей (P1, критично).
2. **Этапы 1–3** — шапка, drawer, профиль, уведомления (последовательно).
3. **Этап 9** — контент в БД (параллельно с 4–7).
4. **Этапы 4–7** — кабинет, дневники, кнопка, видео (параллельно).
5. **Этап 10** — редактор контента (после 9).

## Общие правила для агента

1. Перед каждым изменением — прочитать целевой файл.
2. После каждого этапа — `pnpm run ci`.
3. Один коммит на каждый завершённый этап.
4. Не менять файлы врачебного кабинета (`/app/doctor/*`) если не указано явно.
5. Не менять бэкенд (integrator) если не указано явно.
