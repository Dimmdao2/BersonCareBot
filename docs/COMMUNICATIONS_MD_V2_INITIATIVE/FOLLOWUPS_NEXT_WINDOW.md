## 1. City-подписка на рассылки (Москва / Петербург из настроек пациента) — ГЛОБАЛЬНОЕ

**Что хотим:** в форме рассылки — опции «Москва» / «Петербург»; доставка идёт тем, кто **подписан на
этот город в своих настройках**. Не зависит от того, в каком филиале человек был/записывался.

**Находки (код + прод-дамп):**
- Каталог городов **реален и канонический**: `booking_cities` = `moscow` «Москва», `spb` «Санкт-Петербург».
- В данных есть **дормантные** city-топики: `user_notification_topics.topic_code` ∈
  {`booking_msk`, `booking_spb`, `bookings`, `booking_online`} — но всего у **12 пользователей**.
- ⚠️ Эти топики **НЕ в каноническом реестре** `DEFAULT_NOTIFICATION_TOPICS`
  (`modules/patient-notifications/notificationsTopics.ts`) и в коде на них **только одна ссылка**
  (label-карта аналитики). То есть это **остатки данных, не живая фича**.
- Legacy mailing-механизм (`mailing_topics`/`user_subscriptions`/`*_webapp`) — **пустой** (мёртв).
- Бродкаст сейчас город **не учитывает**: web-push гейтится по категории →
  `important_broadcasts`/`patient_news` (`broadcastNotificationTopicCode`).
- Резолвер «город пациента» как концепт уже есть для «Записи»/help: `patientHelpAddressLink.ts`
  (`BookingCatalogCityCode = "moscow"|"spb"`, город из ближайшей записи / `booking_cities.code`).

**Что нужно сделать (это не «тумблер», а фича):**
1. Внести city-топики в канонический реестр тем пациента.
2. UI настроек пациента: тогглы «получать рассылки: Москва / Петербург» (`user_notification_topics`
   + `user_notification_topic_channels`).
3. Форма рассылки врача: опция города → доставка фильтруется по подписке на этот city-топик.
4. Дефолты при привязке/онбординге (по аналогии с `upsertBroadcastDefaultsAfterChannelBind`).

## 2. Сегмент «давно не открывал приложение/программу N дней» — ГЛОБАЛЬНОЕ (но ограниченное)

**Что хотим:** срез аудитории по **неактивности в приложении** (не открывал программу/видео N дней) —
для re-engagement. Параметрический (выбрать N). ⚠️ Это НЕ текущий `inactive` (тот — про приёмы 90+ дней).

**Находки:** активность пишется в `program_action_log` (timestamps, last opened). Данные есть.
**Подход:** новый параметрический срез в модуле рассылок, читающий последнюю активность; аддитивно.

## 3. Журнал рассылок B.4 — бэкенд-доработки

- **«Создать на основе»** — дёшево и полезно: префилл формы из записи журнала (колбэк/URL-параметр).
  Хороший кандидат на near-term.
- **Разбивка «Доставка по каналам» с причинами/именами** — «вперёд»: исторические рассылки не хранят
  per-recipient/per-reason разбивку; нужно начать писать её при отправке, затем показывать.
- **«Повторить N неудачным» — НЕ делаем.** Модель подтверждена в коде:
  - `blockedRecipientCount` — отдельный счётчик, **не ошибка** (в журнале нейтрально «Бот заблокирован: N»,
    красным только `errorCount`).
  - Недоставка **не уводит систему в `degraded`** (degraded — про cron/integrator-push).
  - Заблокированных **не исключаем навсегда** — на следующей рассылке им снова создаётся задача
    (персистентного «блокирующего» флага нет — что и логично для «ретраим в следующий раз»).
  - Основной кейс недоставки — заблокированный бот, ручной повтор почти бессмыслен.

## 4. Read-only «Прошлые программы» в комментариях (state B) — низкий приоритет

**Что хотим:** информационная история комментариев по прошлым программам — **только просмотр**
(без ответа/отметок прочитанным). Без модалки и без поиска (решение владельца).
**Подход:** в правом пейне state B — сворачиваемая секция «Прошлые программы» (read-only тред:
поле ответа скрыто, «отметить прочитанным» нет).

⚠️ **Бэкенд НЕ готов (проверено 2026-06-13).** Документ ранее ошибочно утверждал обратное. Загрузчик
`loadDoctorPatientExercisesWithComments` принимает `includePastPrograms`, но этот флаг лишь меняет
**fallback**, когда активной программы нет (берёт самый свежий инстанс по `updatedAt`). Он всё равно
агрегирует **ровно один инстанс** (`targetInstance`), а не прошлые программы РЯДОМ с активной —
см. реализацию `loadDoctorPatientExercisesWithComments.ts` (выбор `targetInstance`) и её же док-комментарий
(«Загрузка прошлых программ — опциональный вторичный путь… не реализован в основном потоке»).
**Объём §4 = backend-расширение:** агрегировать прошлые инстансы (отдельные группы с пометкой
программы), изменить форму результата (`PatientExercisesWithCommentsResult`) + тесты; и только потом
read-only UI-секция. Это **самостоятельная фича, не «только UI»**.

## 5. Миграция исторических данных Rubytime — ВАЖНО при финальном переезде с коннектора

**Находки (прод-дамп `bcb_webapp_dev`, 2026-06-13):**

  `be_appointments` (231), `patient_bookings` (209). Глубже — пусто.
- `rubitime_records`/`rubitime_events`/`rubitime_*` справочники — **0 строк** (сырой слой очищен).
- `be_appointments`: **225/231 = `source='rubitime_projection'`**, `branch_id` заполнен на 100% →
  `be_branches.city_code` (`moscow`/`spb`). Legacy `appointment_records.branch_id` — 74%, `branches.name` = город.
- **Вывод:** глубокого исторического архива Rubytime в webapp-БД **нет** — он живёт в Rubytime.
  При окончательном переезде с коннектора **нужен осознанный экспорт/бэкфилл из Rubytime**, иначе
  до-2026 история теряется. Фильтрацию по филиалу/городу для исторических записей строить по
  `be_appointments` (projection) — там branch на 100%.
- ⚠️ Объём дампа (33 уник. пользователя / ~240 записей) выглядит как **урезанный dev-сабсет**, а не
  полный прод. «Сколько истории в проде» по нему измерить нельзя — проверить на реальном проде.

**Правило:** боевую SMTP-отправку проверять **только в песочнице с тестовым получателем и тестовым SMTP**,
**никогда** против прод-дампа. Гард: если `smtp_outbound` пуст — фанаут это no-op (skip всех).

**Что уже проверено live (2026-06-13, read-only, без отправки):** все 4 вкладки рендерятся (HTTP 200,
