# COMMUNICATIONS_MD_V2_INITIATIVE — ТЗ

Доработка раздела **«Коммуникации»** кабинета врача (`/app/doctor/communications`) по итогам
живого ревью всех четырёх вкладок. Документ самодостаточный: рассчитан на агента, который входит
в задачу «с холода». Сначала прочитай раздел **0. Контекст и правила**, затем работай по фазам.

- **Референс-вайрфрейм:** `docs/design/doctor-cabinet-wireframe.html` — секция `#p-comms`
  (cpane-блоки `ct-chats`, `ct-intake`, `ct-comments`, `ct-broadcasts`).
- **Канон UI:** `docs/ARCHITECTURE/DOCTOR_APP_UI_STYLE_GUIDE.md`.
- **Текущее описание модуля:** `apps/webapp/src/app/app/doctor/communications/communications.md`.
- **Эталон split-layout / каталога:** страница упражнений (`exercises`).

> ⚠️ Где вайрфрейм и этот ТЗ расходятся — **приоритет у ТЗ** (вайрфрейм рисовался до ревью).
> Явно отмеченные расхождения: порядок вкладок и редизайн комментариев (Фаза B).

## 0. Контекст и правила

### 0.1 Где это живёт

| Что | Путь |
|-----|------|
| Серверная страница | `apps/webapp/src/app/app/doctor/communications/page.tsx` |
| Клиентский шелл (табы, URL-sync) | `.../communications/DoctorCommunicationsShell.tsx` |
| Конфиг/нав табов | `.../communications/doctorCommunicationsTabs.ts`, `DoctorCommunicationsTabsNav.tsx`, `communicationsTabRegistry.ts` |
| Вкладки | `.../communications/tabs/{ChatsTab,IntakeTab,CommentsTab,BroadcastsTab}.tsx` |
| Архив ошибок рассылок | `apps/webapp/src/app/app/doctor/broadcasts/BroadcastDeliveryArchiveClient.tsx` |
| Канон-примитивы | `apps/webapp/src/shared/ui/doctor/` — `doctorVisual.ts`, `DoctorSection.tsx`, `DoctorEmptyState.tsx`, `doctorWorkspaceLayout.ts` |
| Split-layout эталон | примитивы каталога (`DoctorCatalogPageLayout`, `CatalogSplitLayout`) — найди по использованию на `exercises` |

Шелл работает по паттерну **keepMounted**: таб монтируется при первом открытии и скрывается (не
размонтируется). URL-sync — через `history.replaceState`/`popstate` (`?tab=` + deep-link ключи).
Не ломай этот контракт.

### 0.2 Жёсткие правила репозитория (ОБЯЗАТЕЛЬНО)

1. **Clean Architecture слои.** Прямые импорты `@/infra/*` в `page.tsx`/UI запрещены — данные тянем
   через app-layer loader/сервис, страницы тонкие. (Перед работой прочитай `AGENTS.md` + `.cursor/rules`.)
2. **Изоляция зон UI.** В doctor-зоне нельзя `@/components/ui/**` и `@/shared/ui/patient/**`.
   Импорты примитивов — только из `@/shared/ui/doctor/primitives/*`.
3. **Канон секций и пустых состояний.** Карточки-секции — через `DoctorSection`
   (`doctorSectionCardClass`), **не** копипастой `rounded-xl border bg-card p-3`. Пустые состояния —
   `DoctorEmptyState` (есть проп `size: "sm" | "xs"`), **не** ad-hoc `<p class="text-sm text-muted-foreground">`.
4. **Плотность контролов §B.2.** input/select/button — `h-8`. Никаких `h-9`/`h-10`.
5. **Типографика.** Заголовки секций — `doctorSectionTitleClass`; запрещены `text-lg/xl/3xl`,
   `text-[13px]`, `rounded-2xl`, теней на page-level секциях.
6. **Select** — всегда с `displayLabel`/явным `SelectValue` (никаких сырых uuid в триггере).

### 0.3 Definition of Done (для каждой фазы)

- `npx tsc --noEmit` — 0 ошибок (из `apps/webapp`).
- `npx eslint <изменённые файлы>` — 0.
- `npx vitest run <релевантные .test>` — зелёные; новые тесты на новую логику.
- Ручная проверка на dev-логине доктора (см. 0.4).
- Вести `docs/COMMUNICATIONS_MD_V2_INITIATIVE/LOG.md` (создать) по образцу других инициатив.

### 0.4 Live-проверка (dev-логин доктора)

Авторизованным doctor-страницам нужна реальная сессия. Используй **dev-bypass** (работает только при
`ALLOW_DEV_AUTH_BYPASS=true` в `apps/webapp/.env.dev`, никогда не прод). Хост — **`127.0.0.1:5200`**,
не `localhost`. Старт дев-сервера: `pnpm dev` в `apps/webapp` (стоп — `pnpm dev:stop`).

```
curl -s -c /tmp/c.cookies -b /tmp/c.cookies -L \
  "http://127.0.0.1:5200/api/auth/dev-bypass?token=dev%3Adoctor&next=/app/doctor/communications"
curl -s -b /tmp/c.cookies "http://127.0.0.1:5200/api/me"   # подтвердить роль
```

Токены: `dev:doctor` (кабинет врача), `dev:admin` (врач + admin-режим), `dev:client` (пациент).
Не сканируй `.env` ради кредов — используй этот bypass.

## 3. Фаза A — Быстрые правки (низкий риск, можно делать пакетом)

### A0. Общий шаблон вкладок

- **Порядок вкладок** (правка `doctorCommunicationsTabs.ts` / `communicationsTabRegistry.ts` +
  `DoctorCommunicationsTabsNav`): **Чаты → Комментарии → Заявки → Рассылки.**
  ⚠️ Это **расходится с вайрфреймом** (там chats → intake → comments → broadcasts) — делаем по ТЗ.
  Редиректы со старых URL и deep-link ключи не менять.
- Независимый скролл по разделу 2.

### A1. Чаты (`ChatsTab`)

Работают хорошо: поиск, «Непрочитанные», «★ На сопровождении». Правки косметические:

- **Ширины пейнов:** сейчас список слишком широкий, окно чата узкое. Выровнять так, чтобы тред
  читался комфортно (окно чата — не уже списка; ориентир — равные или чат шире). Независимый скролл
  у списка и у треда.
- **Сообщения в треде:** добавить горизонтальный padding от края пейна (сейчас прижаты к краю) и
  больший вертикальный отступ между сообщениями.
- **Превью в списке чатов:** обрезать до **одной строки** с ellipsis (как в вайрфрейме:
  `white-space:nowrap; overflow:hidden; text-overflow:ellipsis`). Сейчас слишком шумно/многострочно.
- Пустое правое состояние («Выберите чат слева») — через `DoctorEmptyState`.

### A2. Заявки (`IntakeTab`)

- **Фильтры статусов → тоггл-кнопки** (как в чатах): клик включает/выключает; **убрать явную «Все»**;
  пустой выбор = показать все. (Сейчас в вайрфрейме это single-select табы Новые/В работе/Записанные/
  Отказанные/Все — меняем на мультитоггл без «Все».)
- **Независимый скролл** левого списка и правой карточки (split-layout).
- **Статистику заявок не трогаем** — она по периодам (7/30/90/год), не зависит от фильтра. Так и
  задумано (решение пользователя).

### A3. Комментарии — быстрые правки (структуру см. в Фазе B)

- **Фильтры → тоггл-кнопки.** Сейчас «★ Только на сопровождении» не включается/выключается — это баг.
  Поведение как в чатах (клик вкл/выкл, пустой выбор = все).
- **Поиск (адаптивная раскладка):** на широком экране может стоять слева; на узком — уезжает
  **отдельной строкой на всю ширину сверху**, **фильтры под ним**. Никогда не сжимается в одну строку
  с фильтрами с обрезкой текста. Плейсхолдер — просто «Поиск».
- Поиск многополевой (как на странице пациентов): имя, фамилия (вкл. скрытую), отображаемое имя,
  телефон, email, telegram, MAX.

### A4. Рассылки (`BroadcastsTab` → `BroadcastsMainView`)

Форма «Новая рассылка» — порядок полей: **Аудитория → Категория → Каналы → Заголовок → Текст →
кнопки**. (Категорию поднять **над** каналами.)

- **Аудитория · кому** — заменить уродливый нативный `<select>` на дропдаун-паттерн со страницы
  упражнений. **Один срез** (single-select). Состав см. раздел 5.1. Опция **«Выбрать вручную»**
  открывает диалог со списком пациентов (поиск + чекбоксы). ⚠️ Аудитория — это **«кому»**, НЕ каналы;
  убрать из неё каналы (Max и т.п.) и **SMS** (SMS — это канал).
- **Категория** — 4 тоггл-чипа в порядке: **Организационное · Важное · Сервисное · Рекламное**,
  по умолчанию выбрано **Организационное** (раздел 5.2). Старое деление убрать.
- **Каналы · куда** — 5 чекбоксов: **Telegram · MAX · Push · SMS · Email**, под каждым — **точная
  цифра подключений из БД** (раздел 5.3). По умолчанию отмечены **Telegram + MAX + Push**
  (SMS и Email — выкл). Push сейчас показывает 0 и не отмечен — это баг, нужна реальная цифра и дефолт-вкл.
- **Отступ** у заголовка «Новая рассылка» слишком большой — уменьшить.
- Compose (заголовок / текст / предпросмотр / черновик / отправить) работает — не трогать логику.
- **Независимый скролл** (общий шаблон).

Журнал рассылок (правый пейн) в целом нравится — структурные правки журнала вынесены в Фазу B.4
(не блокируют Фазу A).

## 4. Фаза B — Редизайн комментариев + журнал рассылок

> ⚠️ **Полностью заменяет** текущую плоскую раскладку вкладки «Комментарии» и её отрисовку в
> вайрфрейме (`ct-comments`). Вайрфрейм для комментариев — НЕ ориентир по структуре, только по стилю
> отдельных элементов (карточка упражнения, поле ответа).

### B.1 Модель: drill-down с тремя состояниями правого пейна

**Левый пейн — всегда список ПАЦИЕНТОВ** (а не отдельных комментариев). Строка пациента = имя +
счётчик непрочитанных комментариев + «★» если на сопровождении. Сверху — поиск + тоггл-фильтры
(раздел A3). Поиск/фильтры **единые** и фильтруют **оба** пейна (лента справа — это тот же левый
список пациентов, развёрнутый в плоский вид).

**Правый пейн живёт в трёх состояниях:**

- **(A) Лента всех комментариев** — когда пациент не выбран (дефолт/пустое состояние). Плоский список
  всех комментариев: имя пациента + **первые две строки** комментария (по принципу сообщения в чате).
  Без собственной группировки/поиска/фильтров — применяются верхние (единые) и меняют оба блока.
- **(B) Упражнения пациента** — выбран пациент слева. Список **только упражнений, у которых есть
  комментарии** (по умолчанию — активной программы; см. ниже про прошлые). Строка упражнения:
  маленькая превью-миниатюра + название + бейдж «всего / новых». Высота строки компактная.
  - **Группировка по этапам:** упражнения активного этапа сверху; **закрытые этапы свёрнуты**
    (отдельные сворачиваемые блоки внизу списка, под активным этапом).
  - **Сортировка:** от новых к старым (по дате последнего комментария).
  - **Прошлые программы:** по умолчанию — только активная. Возможность показать упражнения прошлых
    программ — вторичная, «по необходимости» (опциональная кнопка/раскрытие). НЕ основная задача.
- **(C) Чат по упражнению** — выбрано упражнение. В том же правом пейне: карточка упражнения сверху,
  ниже — чат-тред комментариев по этому упражнению с возможностью **ответить на каждый комментарий**
  и **отметить прочитанным**. Переиспользовать существующую механику комментариев/ответа
  (найди текущий компонент комментария/модалку в коде), **но привести под текущий стиль/дизайн**.

**Навигация и закрытие:**

| Действие | Переход |
|----------|---------|
| Клик по пациенту (слева) | A → B |
| Клик по упражнению (справа) | B → C |
| Кнопка-текст **«Закрыть»** в чате (НЕ «×») | C → B |
| **«×»** в шапке выбранного пациента | B → A (сброс выбора, лента всех комментариев) |

(Текст «Закрыть» в чате — намеренно не «×», чтобы не дублировать и не путать с «×» сброса пациента.)

> Чат упражнения — **в правом пейне, НЕ модалка** (по принципу единого правого блока со сменой
> содержимого). Плоский режим «все сообщения» отдельным переключателем пока **не делаем** — состояние
> (A) уже даёт плоскую ленту; вернёмся, если в бою будет не хватать.

### B.2 Шапка правого пейна

- В состоянии **B**: имя пациента + статус «★ на сопровождении» + счётчик **всего / новых
  комментариев** (не «N упражнений»).
- В состоянии **C**: хлебная крошка пациент → упражнение, чтобы было видно уровень.
- **Клик по имени пациента** → открывает **дашборд/обзор пациента** (страница пациента). Тогда
  отдельная ссылка «открыть карточку» не нужна. Опционально — «Открыть программу пациента».
- **Убрать** ссылку «Открыть упражнение →» (не нужна).

### B.3 Микро-график статистики упражнения (в шапке чата состояния C)

Отвечая на комментарий, врач должен сразу видеть свежую динамику по упражнению — **за последнюю
неделю** минимум: повторения / (подходы) / вес / насколько тяжело (отметки «легко/тяжело»).
Реализовать компактным микро-графиком (полоски) в шапке чата.

- Делать **гибким по метрикам:** рисуем то, что уже пишется в модели выполнения упражнения
  (повторения / вес / тяжесть — свериться, что реально доступно). Подходы (sets) сейчас **не
  пишутся** — см. Фазу C; график должен подхватить их, когда появятся, без переделки.

### B.4 Журнал рассылок (правый пейн вкладки «Рассылки») — целевой вид по вайрфрейму

Аккордеон, раскрывается **одна** строка за раз. Внутри строки:

- Сводка: **Категория · Аудитория · Каналы**.
- Полный текст.
- **Доставка по каналам** — вложенный аккордеон на канал: `N доставлено · M ошибок`, внутри —
  причины с именами пациентов (напр. «Пользователь заблокировал бота — 3 (…)», «Telegram ID не
  привязан», «Сетевая ошибка», «Push отключены пользователем», «Токен устройства устарел»).
- Действия: «Повторить N неудачным», «Открыть ошибки →», «Создать на основе».

**Приоритет — низкий.** Разделение причин недоставки (заблокирован бот vs ошибка отправки)
исторические рассылки не содержат; делать «по возможности», не блокирует Фазы A/B комментариев.

## 5. Разрешённые решения (из вайрфрейма)

### 5.1 Сегменты аудитории (single-select дропдаун)

Источник — «Основные сегменты» страницы пациентов (вайрфрейм `#p-patients`). Цифры — пример:

| Сегмент | Значение |
|---------|----------|
| **Все активные** (дефолт) | сброс среза |
| На сопровождении | флаг сопровождения |
| С программой | есть активная программа |
| Приём в этом месяце | есть запись в текущем месяце |
| С абонементами | активный абонемент |
| Подписчики | никогда не было записи |
| Новые | есть будущая запись, приёма ещё не было |
| Бывшие | были визиты, сейчас нет активной записи |
| С отменами | есть отменённые записи |
| **Выбрать вручную** | диалог: список пациентов, поиск, чекбоксы |

(Формулировки сегментов финализировать при реализации; набор — отсюда.)

### 5.2 Категории (порядок и дефолт)

`Организационное` (дефолт) → `Важное` → `Сервисное` → `Рекламное`. Категории «Опрос»,
«Расписание приёмов» — **не добавлять** (появятся, когда будет логика).

### 5.3 Каналы

5 чекбоксов: `Telegram` · `MAX` · `Push` · `SMS` · `Email`. Под каждым — **точное число
подключений из БД** (не «примерно»): у кого привязан TG, кто в MAX, **у кого включены push-
уведомления**, у кого есть телефон для SMS, у кого есть email. По умолчанию отмечены **Telegram +
MAX + Push** (SMS и Email — выкл). Дублирование (один человек получит и TG, и SMS) — **by design**,
не дедуплицировать.

## 6. Фаза C — Зависимости (глубже: патиент-апп + схема)

> Трогает приложение пациента и схему выполнения упражнений — за пределами doctor-UI. Пересекается с
> инициативой страницы пациента; согласовать порядок отдельно. **Не блокирует Фазы A/B.**

1. **Подходы (sets) в отметке выполнения.** Сейчас в отметке выполнения упражнения подходы не
   фиксируются. Добавить: схема выполнения + UI отметки в приложении пациента. После этого микро-
   график (B.3) подхватит метрику.
2. **Статистика упражнения** как переиспользуемый компонент (микро-график + возможно расширенный вид)
   — спроектировать так, чтобы переиспользовать и на странице пациента.

---

## 7. Открытые вопросы

На момент написания критичных открытых вопросов нет — порядок категорий, состав аудитории и дефолты
каналов разрешены из вайрфрейма (раздел 5). Если при реализации B.1/B.3 всплывут продуктовые развилки
(точная разбивка «всего/новых», поведение прошлых программ, набор метрик графика) — зафиксировать в
`LOG.md` и спросить владельца, не угадывать.

---

## 8. Вне scope

- Логику compose-отправки рассылок, редиректы со старых URL, контракт keepMounted/URL-sync — не менять.
- Зоны вне канона (`admin/booking`, `admin/*`) — не трогать.
- Полноценный «плоский режим» комментариев отдельным переключателем (состояние A покрывает).

---

## 9. Рекомендуемый порядок работ

1. **Фаза A** целиком (сквозной независимый скролл + порядок вкладок + чаты/заявки косметика +
   рассылки: аудитория-дропдаун, категории, каналы с точными счётчиками, отступы). Низкий риск.
2. **Фаза B.1–B.3** — редизайн комментариев (самостоятельный кусок; здесь основная сложность —
   бэкенд-агрегация: пациенты с непрочитанными, упражнения с комментариями по этапам вкл. прошлые,
   статистика упражнения для графика).
3. **Фаза B.4** — журнал рассылок (по возможности).
4. **Фаза C** — согласовать со страницей пациента, делать отдельно.

Старт каждой фазы — обновить `LOG.md`. Перед кодом — прочитать `AGENTS.md`, `.cursor/rules` и
`DOCTOR_APP_UI_STYLE_GUIDE.md`; держать диффы тонкими и в рамках канона (раздел 0.2).
