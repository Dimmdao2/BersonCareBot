# Модели декомпозиции и выполнения задач

---

## 1. Роли агентов

| Роль | Модель | Задача | Input | Output |
|---|---|---|---|---|
| **Декомпозитор** | GPT 5.3 / Sonnet 4.6 Opus | Разбить фазу на атомарные задачи с полной спецификацией | PLAN.md + код проекта | Список задач с файлами, изменениями, тестами |
| **Исполнитель** | Composer 1.5 (auto) | Реализовать атомарную задачу по спеку | Спек задачи + код проекта | Код, тесты |
| **Аудитор** | GPT 5.3 Codex | Code review, проверка соответствия спеку | Код до/после + спек | Замечания / approve |
| **Фиксер** | Composer 1.5 (auto) | Исправить замечания аудитора | Замечания + код | Исправленный код |

---

## 2. Шаблон атомарной задачи (для декомпозитора)

Декомпозитор генерирует для каждой задачи блок в следующем формате:

```markdown
### Задача [ID]: [Название]

**Цель:** Одно предложение — что должно измениться для пользователя или системы.

**Предусловия:**
- Задача X.Y выполнена
- Файл Z существует

**Файлы для изменения:**
1. `path/to/file.tsx` — описание изменений
2. `path/to/other.ts` — описание изменений

**Файлы для создания:**
1. `path/to/new-file.tsx` — назначение

**Файлы для удаления:**
- `path/to/obsolete.tsx` — причина

**Детальное описание:**
Пошаговое описание того, что нужно сделать. Не «что», а «как именно»:
- В файле X найти компонент Y
- Заменить блок Z на новую реализацию
- Добавить props A, B
- и т.д.

**Тесты:**
- [ ] Тест 1: описание + ожидаемый результат
- [ ] Тест 2: описание + ожидаемый результат

**Критерии готовности:**
- [ ] Пользователь видит / не видит X
- [ ] API возвращает Y
- [ ] `pnpm run ci` зелёный
```

## 3. Шаблон аудита (для аудитора)

```markdown
### Аудит задачи [ID]: [Название]

**Файлы проверены:**
- `path/to/file.tsx` ✅ | ⚠️ | ❌

**Соответствие спеку:** Да / Частично / Нет

**Замечания:**
1. [severity: critical/major/minor] Описание проблемы
   - Файл: `path`
   - Строка: N
   - Что не так: ...
   - Как исправить: ...

2. [severity: minor] ...

**Тесты:**
- Покрытие: достаточное / недостаточное
- Отсутствующие тесты: ...

**CI:** green / red (причина)

**Решение:** approve / rework (список замечаний для исправления)
```

## 4. Цикл выполнения фазы

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  1. ДЕКОМПОЗИЦИЯ (GPT 5.3 / Sonnet 4.6)                │
│     Input: PLAN.md фаза N + код проекта                 │
│     Output: список атомарных задач со спеками            │
│                                                          │
│  2. РЕАЛИЗАЦИЯ (Composer 1.5 auto) — по каждой задаче:  │
│     Input: спек задачи                                   │
│     Output: изменения в коде + тесты                     │
│     → Запустить pnpm run ci                              │
│     → Записать в AGENT_LOG.md                            │
│                                                          │
│  3. АУДИТ БЛОКА (GPT 5.3 Codex)                         │
│     Input: diff блока задач + спеки                      │
│     Output: замечания или approve                        │
│     → Если rework — вернуться к п.2 (Composer 1.5)      │
│                                                          │
│  4. АУДИТ ФАЗЫ (GPT 5.3 Codex / 5.4 при необходимости) │
│     Input: весь diff фазы + план                         │
│     Output: финальные замечания                          │
│     → Записать результат в AGENT_LOG.md                  │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## 5. Правила выбора модели для аудита

| Ситуация | Модель | Обоснование |
|---|---|---|
| Стандартный code review (UI, routing, CSS) | GPT 5.3 Codex | Достаточное качество, низкая цена |
| Сложная бизнес-логика (booking, sync) | GPT 5.3 Codex | Обычно хватает; 5.4 при повторных багах |
| Критический путь (auth, payments, data loss) | GPT 5.4 | Максимальное качество, минимум рисков |
| Повторный аудит после rework | GPT 5.3 Codex | Проверка конкретных замечаний |
| Финальный аудит фазы целиком | GPT 5.3 Codex → 5.4 если > 2 rework | Эскалация при проблемах |

**Принцип:** начинать с 5.3, эскалировать на 5.4 только если количество переделок > 2 на один блок.

## 6. Размер атомарной задачи

### Хорошая задача

- Затрагивает 1–3 файла
- Реализуется за 1 промпт Composer'у
- Имеет чёткие критерии готовности
- Независима или зависит от одной предыдущей

### Слишком большая задача (надо разбить)

- Затрагивает > 5 файлов
- Включает и backend, и frontend
- Требует несколько итераций промптов
- Критерии готовности размыты

### Слишком мелкая задача (объединить с соседней)

- Одна строка изменений
- Чисто стилистическое изменение
- Нет пользовательского эффекта

## 7. Соглашения по коммитам

```
[phase-N.taskID] краткое описание

Пример:
[0.6] fix DashboardTile duplicate border classes
[1.5] add grid view mode to MediaLibraryClient
[2.10] redesign patient cabinet page with native booking
```

## 8. Чеклист перед мержем фазы

- [ ] Все задачи фазы в статусе `done` в AGENT_LOG.md
- [ ] `pnpm run ci` зелёный
- [ ] Аудит фазы завершён, все замечания закрыты
- [ ] Нет TODO/FIXME/HACK в новом коде (кроме осознанных с issue-ссылкой)
- [ ] Нет console.log/console.warn в production-коде
- [ ] Новые API endpoints документированы
- [ ] Навигация проверена на mobile и desktop
