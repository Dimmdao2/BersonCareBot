# Фаза 3 — атомарные задачи (UX кабинета врача)

Источник: `docs/BRANCH_UX_CMS_BOOKING/PLAN.md` (раздел «Фаза 3»), `docs/BRANCH_UX_CMS_BOOKING/DECOMPOSITION_MODEL.md`, текущий код doctor-кабинета в `apps/webapp/src/app/app/doctor/*`.

Цель фазы: улучшить ежедневный UX врача без изменения продуктовой роли экранов.

### Задача 3.1: Объединить «Клиенты» и «Подписчики» в единый список с фильтром

**Цель:** врач работает с одним списком людей, переключая режим отображения фильтром, без дублирования двух почти одинаковых страниц.

**Предусловия:**
- Фаза 0 завершена (меню очищено от нерелевантных разделов).
- Текущие страницы `clients/page.tsx` и `subscribers/page.tsx` используют один и тот же `DoctorClientsPanel`.

**Файлы для изменения:**
1. `apps/webapp/src/app/app/doctor/clients/page.tsx` — добавить режим списка (`scope`) и загрузку данных в зависимости от режима (все клиенты vs только с appointment records).
2. `apps/webapp/src/app/app/doctor/clients/DoctorClientsPanel.tsx` — добавить переключатель режима списка и синхронизацию `scope` в URL.
3. `apps/webapp/src/shared/ui/DoctorHeader.tsx` — убрать отдельный пункт «Подписчики», оставить единый entrypoint в список клиентов.
4. `apps/webapp/src/shared/ui/doctorScreenTitles.ts` — актуализировать title-map (единый экран списка).
5. `apps/webapp/src/app/app/doctor/page.tsx` — обновить deep links плиток (в «общий список» и в «клиенты с записями») через новый `scope`.

**Файлы для создания:**
- Нет.

**Файлы для удаления:**
1. `apps/webapp/src/app/app/doctor/subscribers/page.tsx` — заменить отдельный master-list на redirect в `/app/doctor/clients?scope=all` или удалить после редиректа.

**Детальное описание:**
- Ввести query-параметр `scope`:
  - `scope=all` — бывшие «Подписчики» (все `role=client`),
  - `scope=appointments` — бывшие «Клиенты» (только с записями на прием).
- В `DoctorClientsPanel` добавить UI-переключатель режима (segmented control или пара кнопок), который:
  - не сбрасывает активные фильтры `telegram/max/appointment/visitedMonth`,
  - сохраняет `selected` при desktop master-detail сценарии.
- Для `visitedMonth` оставить доступность только в `scope=appointments`.
- Перенаправить старый маршрут `/app/doctor/subscribers` в единый список, чтобы не ломать старые ссылки.
- Обновить навигацию/тайтлы так, чтобы в меню был один вход в список людей.

**Тесты:**
- [ ] Unit: `doctorScreenTitles` возвращает корректный заголовок для нового единого экрана.
- [ ] Manual: переключение `scope=all`/`scope=appointments` меняет состав списка без дублирования страниц.
- [ ] Manual: старый URL `/app/doctor/subscribers` не падает и открывает единый список в режиме `all`.

**Критерии готовности:**
- [ ] У врача один экран списка людей вместо двух почти одинаковых.
- [ ] Переключение режимов работает через URL и совместимо с текущими фильтрами.
- [ ] Навигация doctor-кабинета не содержит отдельного пункта «Подписчики».
- [ ] `pnpm run ci` зелёный.

### Задача 3.3: Добавить пагинацию в журнал сообщений врача

**Цель:** журнал сообщений масштабируется и не ограничивается фиксированными 50 записями без навигации.

**Предусловия:**
- На `doctor/messages/page.tsx` сейчас вызывается `listAllMessages(50)`.
- `MessageLogPort` не поддерживает `offset`/paging metadata.

**Файлы для изменения:**
1. `apps/webapp/src/modules/doctor-messaging/ports.ts` — расширить контракт listAll/listByUser поддержкой пагинации (`limit`, `offset`, `total`).
2. `apps/webapp/src/infra/repos/pgMessageLog.ts` — реализовать SQL пагинацию с `LIMIT/OFFSET` и отдельным `COUNT(*)`.
3. `apps/webapp/src/infra/repos/inMemoryMessageLog.ts` — синхронизировать поведение in-memory реализации с новым контрактом.
4. `apps/webapp/src/modules/doctor-messaging/service.ts` — добавить use-case списка журнала с метаданными страниц.
5. `apps/webapp/src/app/app/doctor/messages/page.tsx` — читать `page`/`pageSize` из `searchParams` и рендерить контролы страниц.

**Файлы для создания:**
1. `apps/webapp/src/app/app/doctor/messages/DoctorMessagesLogPager.tsx` — UI-компонент пагинации (предыдущая/следующая, текущая страница, total).

**Файлы для удаления:**
- Нет.

**Детальное описание:**
- Ввести базовую модель пагинации:
  - query: `page` (1-based), `pageSize` (например 20/50),
  - response/service: `items`, `total`, `page`, `pageSize`, `hasNext`.
- На странице журнала:
  - заменить фиксированный вызов `listAllMessages(50)` на paged use-case,
  - добавить ссылки/кнопки навигации страниц без потери существующего блока «Новое сообщение».
- Сохранить обратную совместимость: если `page` не передан, открыть первую страницу.

**Тесты:**
- [ ] Repo test (PG): `offset` корректно смещает набор записей и возвращается корректный `total`.
- [ ] Service test: `page/pageSize` маппятся в корректные `limit/offset`.
- [ ] Manual: переход на страницу 2 действительно показывает следующий набор сообщений.

**Критерии готовности:**
- [ ] Журнал сообщений не упирается в один фиксированный лимит.
- [ ] Пользователь может листать страницы вперед/назад.
- [ ] Пагинация одинаково работает на PG и in-memory реализации.
- [ ] `pnpm run ci` зелёный.
