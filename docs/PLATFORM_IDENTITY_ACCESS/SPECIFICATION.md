# Спецификация: платформенная идентичность и access-tier

Нормативный текст для реализации и ревью. Версия согласована с архитектурной дискуссией (2026); уточнения по порядку резолва канона, trusted-источникам и onboarding без бизнес-действий.

---

## 1. Назначение

Единая модель для **интегратора**, **веб-приложения** (`apps/webapp`), **Telegram**, **Max** и будущих каналов. Вопросы «кто это» и «что разрешено» сводятся к **одному каноническому пользователю платформы** и его **актуальному состоянию в БД**.

---

## 2. Канонический пользователь и данные

- **Канонический пользователь** — одна живая логическая сущность: строка `platform_users` в webapp-БД с учётом цепочки `merged_into_id` и резолва канонического id (см. `docs/ARCHITECTURE/PLATFORM_USER_MERGE.md`).
- **Привязки каналов** — таблица `user_channel_bindings`: внешний identity (`telegram` / `max` / `vk`) → `user_id` канона.
- **Телефон** хранится на уровне **`platform_users.phone_normalized`**, не на уровне binding.
- **Merge** (`apps/webapp/src/infra/repos/pgPlatformUserMerge.ts` и связанные пути) — подстраховка при дублях и проекциях; **не** основной сценарий жизненного цикла. Новые входы по возможности **сначала** сопоставляют канал с существующим каноном (binding, `integrator_user_id`, доверенные события интегратора).

**Инвариант после merge:** любой доступ и запись от имени пользователя должны опираться на **канонический** `user_id` после резолва; использование «исходного» id проигравшей строки без резолва запрещено (иначе фантомные баги).

---

## 3. Роль в БД и вычисляемый tier

**В БД** сохраняется роль: **`client` | `doctor` | `admin`** (`platform_users.role`, типы в `apps/webapp/src/shared/types/session.ts`).

**Access context** на сервере: **`dbRole`** + **`tier`**, где **`tier` имеет смысл только для `client`**. Для **`doctor` / `admin`** доступ задаётся **ролью** (и существующими правилами их зон), а не guest/onboarding/patient.

**Tier** (только контур **`client`**) — **вычисляемый** уровень для access-политики. Отдельная бизнес-роль `onboarding` в БД **не вводится**.

| Tier | Смысл (только при `dbRole === 'client'`) |
|------|------------------------------------------|
| **guest** | Нет сессии или пользователь не идентифицирован для закрытых зон. |
| **onboarding** | Канон известен, но нет **активации пациента** по §5. |
| **patient** | Канон + **доверенный телефон** у канона (§5). |

При **`doctor` / `admin`** поле `tier` в типах может быть **нейтральным / N/A** или не использоваться в политике patient-зоны.

---

## 4. Onboarding для `client` (жёсткое определение)

**Сейчас** tier **onboarding** для роли **`client`** означает: у **канонического** пользователя **нет** телефона, засчитанного как **активация** по §5.

**Иные условия активации** (согласия, профиль и т.д.) в эту спецификацию **не входят** и добавляются отдельным решением с обновлением политики tier.

**Граница доступа:** в **onboarding** пользователь **не** выполняет **бизнес-действия** (запись на приём, дневники, сообщения врачу, оплаты и т.п.) **ни в UI, ни на сервере** (включая **API** и **server actions**), кроме явно отнесённых к **активации**. Whitelist активации и публичных исключений — **серверный** (общий для маршрутов и API), а не только «скрыть кнопку в UI».

**RSC и БД:** React Server Components, которые при рендере на сервере читают **персональные** данные пациента из БД по `userId`, обязаны применять **тот же** критерий tier, что patient-business API (в webapp: `patientClientBusinessGate`, обёртка для страниц — `patientRscPersonalDataGate` в `apps/webapp/src/app-layer/guards/requireRole.ts`). Наличие телефона только в snapshot cookie **не** заменяет эту проверку. Примеры маршрутов и виджетов: раздел **`/app/patient/sections/warmups`** (правила напоминаний), страница **`/app/patient/purchases`** (основной контент при tier **patient**). Карта: [`SCENARIOS_AND_CODE_MAP.md`](SCENARIOS_AND_CODE_MAP.md) §7.

## 5. Телефон и активация patient

**Patient** для `client` только если у канона есть телефон, принятый системой как **достаточный для активации**.

Это **не** эквивалентно проверке «`phone_normalized IS NOT NULL` в произвольной записи».

**Перечень доверенных источников телефона** — **закрытый** и **централизован в коде** (одна точка принятия решения «телефон засчитан для tier patient»), чтобы разные модули не трактовали активацию по-разному. **Не** любой код, который пишет `phone_normalized` в БД, автоматически расширяет этот перечень — только явная регистрация в модуле trusted phone policy.

Примеры классов источников (конкретный enum/список — в реализации):

- подтверждение OTP (`apps/webapp/src/modules/auth/phoneAuth.ts`, `createOrBind` в `pgUserByPhone.ts`);
- доверенные обновления из интегратора (`user.upserted`, `contact.linked`, `ensureClientFromAppointmentProjection` в `pgUserProjection.ts`, события в `apps/webapp/src/modules/integrator/events.ts`).

## 6. Порядок операций: резолв канона и session

**Резолв канонического пользователя** должен выполняться **до** (или в тот же момент, что и) **запись session cookie** с `userId`, так чтобы в cookie попадал **уже канонический** id, когда он определим.

- Допустима **минимальная** сессия для цепочки активации, если она **не** даёт patient-доступ и явно ограничена политикой.
- **Недопустимо** полагаться только на guard/layout как на **первое** место «исправления» неправильного `user_id` в cookie после логина.

**Session/cookie** — транспорт идентичности; **tier** для `client` **всегда** пересчитывается на сервере по **актуальному канону из БД** после резолва (не по устаревшему snapshot в cookie). Решения о доступе опираются на БД; cookie может отставать до следующей легальной перезаписи (ограничения Next.js Server Components учитываются в реализации).

**Legacy `tg:…` / не-UUID в `userId`:** архитектурное решение зафиксировано в `apps/webapp/src/modules/auth/sessionCanonicalUserIdPolicy.ts`: такой транспорт **не** является каноническим ключом; для `client` политика tier даёт только **onboarding** (см. `legacy_non_uuid_session` в `resolvePlatformAccessContext`). Основные login flows с PostgreSQL пишут в cookie UUID канона; совместимость с префиксами — временный onboarding-only режим (тесты, dev bypass, пути без БД).

## 7. Первый вход через мессенджер

При первом входе через мессенджер **допускается** создание:

- **технической** записи в `platform_users` (минимально необходимой для binding);
- строки **`user_channel_bindings`** для данного канала.

Это **техническая запись до полной канонизации** (якорь для identity): она **не** означает tier **patient** и **не** означает «полноценный клиент в продуктовом смысле».

Дальнейшая логика **обязана** уметь привести сущность к канону (интегратор, OTP, merge). Пока у канона нет доверенного телефона по §5 — tier **onboarding** для `client`.

Реализация первичного создания по каналу сегодня: `findOrCreateByChannelBinding` в `apps/webapp/src/infra/repos/pgIdentityResolution.ts`.

---

## 8. Веб-вход

Веб опирается на OAuth, SMS и связанные маршруты (`apps/webapp/src/app/api/auth/oauth/`, `apps/webapp/src/app/api/auth/phone/`).

**Формулировка:** не создавать **лишние отдельные канонические записи** без необходимости; объединять с существующим каноном, когда это следует из потока.

Если после OAuth у канона нет доверенного телефона — обязательный пост-логин сценарий активации; tier **onboarding** до завершения.

---

## 9. Интегратор

Интегратор создаёт и обогащает канона, передаёт телефон и `integrator_user_id`, связывает канал, инициирует merge через проекции. Вход: webhook событий (маршрутизация в `apps/webapp/src/modules/integrator/events.ts`, порты в `buildAppDeps`).

Если канон уже активирован по §5, повторный bind-phone **не** запускается при **доверенной** привязке канала к этому канону.

---

## 10. Multi-channel и отсутствие «угадывания»

Если новый канал **ещё не** сопоставлен с каноном **доверенным** способом, система **не обязана** и **не должна** «угадывать» личность по косвенным признакам на уровне UI без явного доверия (защита от ложных автосвязок).

Доверенные механизмы сопоставления задаются явно в реализации (события интегратора, подписанный webapp-entry token с контекстом, OTP-привязка и т.д.).

**Проверка целостности (сценарий):** канон с телефоном из интегратора; новый канал без binding → пользователь в onboarding → OTP / bind → merge при необходимости → patient. Спецификация этот путь покрывает как безопасный.

---

## 11. Bind-phone

Не роль и не постоянный access-level. Узкий набор маршрутов для перевода `client` из **onboarding** в **patient** через появление у канона доверенного телефона (§5). Запуск только если у канона **нет** такого телефона по централизованной проверке.

---

## 12. Legacy

Старые записи **`client`** без телефона, созданные по прежней схеме, **сохраняются** в БД. По политике доступа они трактуются как **onboarding**: рабочий **patient**-доступ не выдаётся, пока у канона не появится телефон по §5 через доверенный сценарий.

---

## 13. Единый инвариант

Перед выдачей рабочего клиентского доступа сервер **резолвит канона**, читает БД, вычисляет tier; не по локальному каналу в вакууме и не по устаревшему snapshot cookie в обход резолва и БД.

---

## 14. Наблюдаемость

Реализация должна позволять ответить, почему пользователь в **onboarding** при ожидании **patient**: структурированные логи (или эквивалент) с **причиной** итогового tier, признаком **trusted** резолюции идентичности там, где это уместно, и учётом критичных событий **merge** / **phone_bind** / проекций на входах. Уровень детализации и отсутствие утечки PII — по правилам проекта. См. DoD в [`MASTER_PLAN.md`](MASTER_PLAN.md) §3 п.8.

---

## 15. Итоговая формула

- **guest** — только явно разрешённые публичные маршруты.  
- **onboarding** — `client` без доверенной активации по телефону у канона; без бизнес-действий вне активации и публичного whitelist (включая сервер).  
- **patient** — `client` + канон + доверенный телефон у канона.  
- **doctor / admin** — доступ по **роли**, не по guest/onboarding/patient.  

**Принципы:** один канон для интегратора, веба и каналов; телефон через централизованный перечень доверенных путей; merge — подстраховка; bind-phone — только при отсутствии доверенного телефона у канона; без доверенного сопоставления канал ↔ канон система не угадывает identity.
