# COURSES_INITIATIVE — самостоятельные курсы (геткурс-модель)

**Статус:** документация / strawman. Код не менялся.
**Дата старта:** 2026-05-03.
**Назначение:** собрать в одном месте принципиальные решения о том, как устроены **курсы** в продукте — независимо от плана лечения.

> **Очерёдность:** последняя инициатива в очереди реструктуризации. Стартует **после** ядра пациентского `PROGRAM_PATIENT_SHAPE_PLAN` и появления оплаты. До этого — только документация.

## 1. Что такое курс (продуктовое определение)

Курс — это **самостоятельный self-paced образовательный продукт** на витрине, который пациент проходит сам:
- содержит **уроки** (видео + материалы) с условиями открытия;
- может содержать **тесты-проверки** как часть условия открытия следующего урока;
- имеет цену / условия доступа (бесплатный / по подписке / разово куплен / выдан врачом);
- имеет вводный материал (intro lesson), доступный без покупки.

**Курс — НЕ план лечения.** Это разные сущности с разной природой:

| Сущность | Назначает | Что внутри | Гибкость прохождения | Клиническая роль |
|---|---|---|---|---|
| **План лечения** (`treatment_program_instance`) | врач | этапы с упражнениями / комплексами / тестами / **рекомендациями** / уроками | следующий этап разлочивает врач (gate); цели/задачи/сроки этапа | первичная — индивидуальная реабилитация |
| **Курс** | пациент сам берёт с витрины (или приобретает) | уроки + опционально проверочные тесты, с правилами unlock | пациент проходит сам; правила определены курсом (по расписанию / после теста / все сразу) | вторичная — обучение, профилактика, маркетинг |

В курсе **нет рекомендаций-режима**, нет «отслеживания динамики симптомов», нет групп упражнений с расписаниями ЛФК. Если врачу нужны эти артефакты — он назначает план лечения, а не курс.

## 2. Отказ от §9 текущего SYSTEM_LOGIC_SCHEMA

Архивный документ [`../archive/2026-05-initiatives/TREATMENT_PROGRAM_INITIATIVE/SYSTEM_LOGIC_SCHEMA.md`](../archive/2026-05-initiatives/TREATMENT_PROGRAM_INITIATIVE/SYSTEM_LOGIC_SCHEMA.md) §9 описывает курс как «обёртку над `treatment_program_template`» — при покупке создаётся `treatment_program_instance`. Это решение **снимается** в продуктовой модели:
- курс не имеет смысловых рекомендаций / шкал тяжести / clinical-комментариев врача;
- FSM этапов программы (linear locked → available → completed) не подходит для курса с правилами «после теста», «через N дней», «все сразу»;
- натягивать программный движок на курс ведёт к двум полу-движкам и постоянным компромиссам.

→ Курс — **отдельная сущность с отдельной коллекцией уроков и отдельным движком unlock-rules**.

Замена для §9: см. §3 ниже.

## 3. Целевая модель данных (high-level)

```
courses
  ├── id, title, description, cover_media_id
  ├── intro_lesson_page_id  (опц.)              ← бесплатный вводный материал
  ├── access_kind: 'free' | 'subscription' | 'paid' | 'doctor_assigned'
  ├── price_minor, currency  (для 'paid')
  ├── status: 'draft' | 'published' | 'archived'
  └── created_at, updated_at

course_lessons
  ├── id, course_id FK
  ├── content_page_id FK → content_pages       (урок = страница CMS, переиспользуется)
  ├── title, sort_order
  ├── unlock_rule JSONB:
  │     { kind: 'manual' | 'all_open' | 'after_lesson' | 'after_test' | 'after_days' | 'after_purchase',
  │       params: { lesson_id?, test_id?, days?, ... } }
  └── is_required BOOLEAN  (учитывается ли в прогрессе курса)

course_lesson_tests   (опц., если ставим проверки)
  ├── id, course_lesson_id FK
  ├── test_id FK → tests
  └── passing_threshold

course_enrollments
  ├── id, user_id FK, course_id FK
  ├── status: 'active' | 'completed' | 'expired' | 'refunded'
  ├── access_source: 'subscription' | 'purchase' | 'doctor' | 'free'
  ├── started_at, completed_at NULL, expires_at NULL
  └── created_at

course_progress
  ├── id, enrollment_id FK
  ├── lesson_id FK
  ├── status: 'locked' | 'available' | 'completed'
  ├── completed_at NULL
  └── last_viewed_at

course_purchases  (после интеграции платежа)
  ├── id, user_id FK, course_id FK
  ├── provider, provider_payment_id, amount_minor, currency
  ├── status: 'pending' | 'succeeded' | 'failed' | 'refunded'
  └── timestamps
```

### 3.1 Уроки и переиспользование контента

Урок = строка `content_pages` (та же таблица, что и для статей/уроков плана лечения). Один урок может быть подключён к нескольким курсам **через разные `course_lessons` строки** (порядок и unlock-правила у каждого курса свои). Файлы (видео/изображения) **не дублируются** — одна страница CMS со своим набором медиа.

«Страница в странице» (embed одной CMS-страницы внутри другой) **не делается** — это новый редактор, не нужен. Если в одном уроке нужно несколько видео — это несколько медиа-блоков на одной странице.

CMS-`kind` для таких страниц — **`course_lesson`** (см. roadmap CMS-типизации, этап 2). Жёсткой привязки одной страницы к одному курсу нет.

### 3.2 Unlock-правила

Стартовый набор:
- `manual` — врач/админ открывает урок вручную (для курсов «по записи на приём»);
- `all_open` — все уроки открыты сразу;
- `after_lesson` — открывается после `course_progress.completed` указанного урока;
- `after_test` — открывается после прохождения теста с порогом;
- `after_days` — открывается через N дней после `enrollment.started_at`;
- `after_purchase` — открывается после `course_purchases.status = 'succeeded'`.

Расширение позднее (комбинации правил, OR/AND) — backlog.

### 3.3 Доступ к курсам

`courses.access_kind` определяет, как пациент получает доступ:
- `free` — все авторизованные;
- `subscription` — через активную подписку (модель подписки уже намечена на витрине пациента);
- `paid` — через `course_purchases`;
- `doctor_assigned` — врач выдаёт enrollment вручную (например, как часть лечения, без перехода в план).

Этот выбор не блокирует A1–A5 пациентского плана — это отдельный продукт.

## 4. UX пациента — целевая модель

### 4.1 Каталог курсов `/app/patient/courses`

- Список курсов с обложкой, названием, описанием, бейджем доступа (`Бесплатно` / `По подписке` / `1 990 ₽` / `Назначен врачом`).
- Фильтры (категории, длительность) — по мере роста каталога.

### 4.2 Карточка курса `/app/patient/courses/[id]`

- Hero (обложка + описание), вводный урок доступен сразу (`intro_lesson_page_id`), список уроков с состоянием (`locked`/`available`/`completed`).
- CTA «Записаться» / «Купить» / «Пройти бесплатно» / «Подписаться, чтобы получить доступ» — по `access_kind`.

### 4.3 Прохождение урока

- Открыть урок → читает страницу `content_pages` → отметка «Изучил».
- Если у урока есть `course_lesson_tests` — после изучения предлагает пройти тест, при достижении порога — следующий урок становится `available`.

## 5. UX врача / админа — каталог курсов

`/app/doctor/courses` (после реализации):
- список / фильтры / master-detail;
- редактор уроков курса (drag-n-drop, выбор `content_page` из CMS, настройка unlock-правил);
- управление доступом (`access_kind`, цены, intro lesson);
- статистика по записям / прогрессу.

Каталог `Курсы` остаётся **отдельным пунктом меню врача** (в кластере «Назначения» или «Контент» — выбирается по итогам §6 ниже).

## 8. Связь с другими инициативами

- **`PROGRAM_PATIENT_SHAPE_PLAN`** — независим. Курсы не меняют план лечения и наоборот.
- **`CMS_RESTRUCTURE_PLAN`** (вариант C) — `kind=course_lesson` уже есть; курсы используют его как источник.
- **`RECOMMENDATIONS_AND_ROADMAP` этап 7** — переформулирован: **курсы — отдельный продукт по геткурс-модели**, не «объединение с шаблонами программ».
- **`TARGET_STRUCTURE_DOCTOR.md` §6.3** (объединение курсов и шаблонов программ) — снимается; курсы остаются как отдельный каталог.
- **`TARGET_STRUCTURE_PATIENT.md` §4.2** (план как абсорбент `/courses`) — снимается; курсы остаются отдельной вкладкой/разделом пациента (`/courses`).
