## 1. Общая цель

Перейти с внешней логики RubyTime на собственный движок записи пациентов с каноническим хранением данных в собственной базе данных.

На первом этапе допускается сохранение интеграции/совместимости с RubyTime, но целевая модель должна быть собственной: записи, услуги, филиалы, кабинеты, специалисты, абонементы, оплаты, переносы, отмены и история пациента должны храниться и управляться внутри собственной системы.

Система должна проектироваться не только под текущий личный сценарий, но и с заделом под будущую SaaS-платформу для специалистов и клиник.

## 2. Каноническая модель организаций, клиник, филиалов, кабинетов и специалистов

### 2.1. Клиники / организации
Заложить сущность клиники / организации — верхнеуровневый контейнер, внутри которого: филиалы / локации; кабинеты; специалисты; услуги; настройки записи; настройки отмены и переноса; настройки обязательных полей; настройки оплаты; продукты, абонементы, подписки; история записей и посещений. В будущем одна система должна обслуживать разные клиники или частных специалистов.

### 2.2. Филиалы / локации
Переработать текущую систему филиалов. Филиал описывает физическую или логическую локацию (СПб; Москва; другой город; конкретный адрес; отдельная точка приёма). Филиал **не** должен жёстко определять список услуг сам по себе. Услуги подключаются через гибкую модель связей: специалист → филиал / кабинет → доступные услуги.

### 2.3. Кабинеты
Внутри филиала — кабинеты. Кабинет — место приёма. Сценарии: один специалист в одном кабинете; один специалист в разных кабинетах; 2–3 специалиста в одном кабинете; разные специалисты используют разные услуги в одном кабинете; один и тот же специалист в разных кабинетах может иметь разный набор услуг. Кабинет **не** должен быть единственным центром привязки услуги, как сейчас в RubyTime.

### 2.4. Специалисты
Заложить сущность специалиста. Специалист может: работать в одной клинике; работать в нескольких филиалах одной клиники; принимать в разных городах; принимать в разных кабинетах; иметь разные услуги в разных филиалах / кабинетах; иметь индивидуальное расписание; иметь индивидуальные настройки доступности услуг.

Пример: Дмитрий принимает в СПб и Москве. Услуги: приём 40 / 60 / 90 минут. В Москве — только 40 минут; в СПб — только 60 и 90. При этом **не** должно быть необходимости создавать дубли услуг вида «Приём 60 минут — Дмитрий — СПб». Одна услуга — общая сущность, доступность настраивается через связи.

## 3. Услуги

### 3.1. Общая модель услуги
Услуга — самостоятельная сущность внутри клиники, не привязана жёстко к одному филиалу/кабинету/специалисту. Примеры: приём 40/60/90 минут; первичная консультация; повторный приём; парный приём; другое занятие/процедура. Параметры минимум: название; длительность; стоимость; описание; активность; настройки предоплаты; возможность использования в абонементах; возможность онлайн-оплаты; возможность отображения в публичном виджете записи.

### 3.2. Назначение услуг специалистам
Гибкая настройка доступности: услугу можно назначить специалисту в целом; специалисту в конкретном филиале; специалисту в конкретном кабинете; специалисту в конкретном городе/локации. Пример: «Приём 40 минут» существует один раз — включить Дмитрию в Москве; не включать в СПб; включить другому специалисту только в одном кабинете; отключить для публичной записи, но оставить для ручного назначения администратором.

### 3.3. Разные услуги в разных филиалах
Пациент при выборе города/филиала видит только доступные в выбранной локации услуги. Москва — приём 40 минут; СПб — приём 60 и 90 (без 40).

## 4. Запись пациента

### 4.1. Каналы записи
1) внутри приложения / личного кабинета; 2) через внешний сайт; 3) через публичную страницу записи; 4) через виджет, встроенный на сайт (например, Tilda).

### 4.2. Запись внутри приложения
Авторизованный пациент с заполненными данными записывается без повторного ввода. Телефон/email и др. обязательные поля подставляются автоматически. Недостающие — запрашиваются только недостающие.

### 4.3. Запись через внешний сайт
На сайте — виджет или публичная страница. Варианты: JS-виджет; попап; iframe; отдельная публичная страница; ссылка с параметрами; ссылка «Записаться». Tilda: вставляется код виджета/ссылка → «Записаться» → интерфейс записи, который подтягивает данные из собственной системы.

### 4.4. Идентификация пациента при внешней записи
Основной идентификатор — телефон. Учесть: у пациента есть аккаунт без телефона; на сайте он вводит телефон; система может не понять, что это тот же человек — это допустимо. Предусмотреть последующее ручное/полуавтоматическое объединение профилей (мердж). Идентификация по телефону — основная для публичной записи; система не ломается, если ранее был только email.

## 5. Настраиваемые поля при записи

### 5.1. Обязательные и необязательные поля
В настройках клиники/записи — какие поля пациент обязан заполнить. Примеры: имя; фамилия; телефон; email; комментарий; описание проблемы; суть задачи; жалобы; произвольный текстовый вопрос; кастомные поля. Для каждого поля: обязательное/необязательное; отображается/нет; тип; порядок; подсказка/placeholder; видимость для пациента; видимость для администратора/специалиста.

### 5.2. Комментарий к записи
Пациент может оставить комментарий (если поле включено); обязательность зависит от настроек.

### 5.3. Кастомные вопросы
Произвольные вопросы перед записью: «Опишите вашу проблему»; «С чем хотите обратиться?»; «Есть ли острая боль?»; «Были ли операции/травмы?»; «Какой формат приёма нужен?».

## 6. Выбор нескольких сеансов подряд
Пациент может выбрать несколько последовательных слотов подряд. Сценарии: запись на два часа; приехать вдвоём; удлинённый приём; администратор вручную ставит несколько слотов. Система умеет: показывать доступные последовательные интервалы; проверять, что несколько слотов подряд свободны; блокировать выбранный непрерывный период; учитывать длительность услуги; занятость кабинета; занятость специалиста; филиал/город.

## 7. Расписание и доступные слоты

### 7.1. Общая логика
Доступные слоты считаются на основе: специалиста; филиала; кабинета; услуги; длительности услуги; рабочего расписания; существующих записей; блокировок; отпусков/отсутствий; правил переноса и отмены; правил предоплаты; технических ограничений по времени между приёмами (если будут добавлены).

### 7.2. Фильтрация доступности
Пациент видит только реально доступные варианты. Услуга только в конкретном городе — показывается только там. Специалист в нескольких городах — пациент выбирает город и видит доступные слоты в нём. При переносе: если разрешён другой город — показать слоты и в другом городе; если ограничен — только текущий город.

## 8. Перенос записи

### 8.1. Самостоятельный перенос пациентом
Если разрешено настройками. Сценарий: открыть запись → «Перенести» → доступные слоты → выбрать новый → перенос → уведомление специалисту/администратору. Перенос — именно перенос существующей записи, не создание новой.

### 8.2. Интерфейс переноса
Можно использовать обычный модуль записи в режиме переноса. Разница: новая запись не создаётся с нуля; сохраняется связь с исходной записью; фиксируется история переноса; учитывается число выполненных переносов; учитываются правила бесплатного/штрафного переноса; сохраняется связь с оплатой/абонементом/предоплатой.

### 8.3. Настройки переноса
За сколько времени можно самостоятельно перенести; сколько раз; разрешён ли перенос в другой филиал/город; к другому специалисту; на другую услугу; что делать после превышения лимита; требуется ли подтверждение специалиста; уведомления специалисту/пациенту. Пример: перенос не позднее чем за 3 суток; только 1 самостоятельный перенос; повторный — через запрос специалисту/администратору.

### 8.4. Защита от обхода правил отмены через перенос
Исключить лайфхак: пациент уже не может бесплатно отменить (осталось меньше срока) → переносит на более позднюю дату → пытается бесплатно отменить новую дату. Система учитывает исходную дату и историю переноса. Если запись перенесена в период, когда бесплатная отмена исходной уже была невозможна — бесплатная отмена новой записи не должна автоматически становиться доступной. Хранить: исходную дату; дату каждого переноса; кто перенёс; был ли перенос в бесплатном периоде; какие штрафные правила применялись; доступна ли бесплатная отмена после переноса; лимит переносов.

### 8.5. Разные правила для отмены и переноса
Отдельные настройки для отмены и переноса. Пример: отмена без штрафа — за 3 суток; перенос — за 2 суток; но после такого переноса бесплатная отмена не становится доступной автоматически. На первом этапе можно одна общая настройка, но архитектурно заложить две политики: `cancellationPolicy`; `reschedulePolicy`.

## 9. Отмена записи

### 9.1. Самостоятельная отмена пациентом
Если разрешено. Настройки: за сколько до приёма бесплатная отмена; разрешена ли отмена вообще; что делать при поздней отмене; возвращается ли предоплата; списывается ли занятие по абонементу; требуется ли подтверждение специалиста; какие уведомления.

### 9.2. Отмена специалистом / администратором
Ручная отмена с выбором типа: бесплатная; штрафная; со списанием занятия по абонементу; без списания; с удержанием предоплаты; с возвратом предоплаты; индивидуальное решение. Пример: пациент написал за сутки — специалист решает (без штрафа / перенести / списать / удержать предоплату / штрафная).

### 9.3. Правила поздней отмены
Варианты: штраф; списание занятия по абонементу; удержание предоплаты; ручное решение специалиста; индивидуальная пометка в истории пациента.

### 9.4. Хранение истории отмен
Кто отменил; когда; причина (если есть); была ли бесплатной; штрафной; списано ли занятие; удержана ли предоплата; был ли возврат; комментарий администратора/специалиста; отправленные уведомления.

## 10. Предоплата

### 10.1. Настройки предоплаты
Включать предоплату для услуг. Виды: фиксированная сумма; процент; полная оплата; отключена. Примеры: 1000/2000/3000 ₽; 50%; полная.

### 10.2. Предоплата по видам приёмов
Настраивается для каждой услуги/вида приёма. Пример: 40 мин — 1000 ₽; 60 мин — 2000 ₽; 90 мин — 3000 ₽; первичный — полная; повторный — без предоплаты.

### 10.3. Предоплата и отмена
Правила возврата/удержания. Раньше чем за 3 суток — возврат или на баланс; позже — не возвращается. Настраиваемое правило.

### 10.4. Предоплата и перенос
При переносе сохранять связь с оплатой. Корректный перенос — предоплата к новой дате. Поздний/нарушающий перенос — применять политику.

## 11. Абонементы

### 11.1. Общая логика
Система абонементов/пакетов занятий. Гибкий состав: произвольный набор услуг. Примеры: абонемент на 5 / 6 занятий; состав: 2×90 мин; 4×60 мин; 1×40 + 3×60; любой набор. Абонемент **не** только «N одинаковых занятий».

### 11.2. Ручное создание абонемента специалистом
Создать индивидуальный абонемент для пациента: выбрать пациента; состав; разные услуги; количество каждой; цену; срок действия; правила списания; создать ссылку на оплату; отправить предложение/ссылку; показать абонемент в кабинете пациента. Пример: 2×90 + 4×60, произвольная цена, ссылка на оплату в кабинете.

### 11.3. Готовые абонементы / продукты
Будущие готовые продукты: программа восстановления; курс; пакет консультаций; подарочный абонемент; акция; комплексная программа. Готовый продукт — состав задан заранее администратором/специалистом, пациент не компонует сам.

### 11.4. Списание занятий по абонементу
После приёма занятие списывается: автоматически после подтверждения факта посещения; либо вручную специалистом/администратором. При отмене зависит от правил: не списывается; списывается как штраф; решение вручную; переносится на новую дату.

### 11.5. Абонемент и запись
При записи пациент видит доступные по абонементу услуги. Система понимает: какие услуги входят; сколько занятий осталось; что использовано; какие записи забронированы; что списано; что отменено; что перенесено.

### 11.6. Личный кабинет пациента по абонементу
Видит: активные абонементы; состав; сколько осталось; сколько использовано; срок действия; историю посещений по абонементу; будущие записи по абонементу; отменённые/перенесённые занятия; статус оплаты.

### 11.7. Кабинет специалиста по абонементу
Видит: какие абонементы у пациента; что входит; сколько осталось; что списано; историю посещений; историю отмен; историю переносов; историю оплат; комментарии и ручные решения по спорным ситуациям.

## 12. Акции, подарочные предложения и специальные покупки

### 12.1. Акционные продукты
Создавать акционные продукты: акция на 8 марта; подарочный приём; приём по скидке; подарочный абонемент; спецпредложение для категории клиентов.

### 12.2. Связь покупки с пациентом
При покупке акционного продукта пациент указывает телефон. Система связывает покупку с пациентом по телефону (если возможно). При последующей записи с тем же номером система понимает, что есть купленный продукт/абонемент/акция.

### 12.3. Отображение информации о покупке
У специалиста — информация, что пациент записался по купленному продукту: что куплено; оплачен ли; использован или нет; сколько визитов осталось (если пакет).

### 12.4. Отображение в кабинете пациента
Видит: купленные акции; подарочные абонементы; доступные продукты; статус оплаты; возможность записаться по продукту; остаток занятий/визитов.

## 13. Оплаты

### 13.1. Общий платёжный слой
Поддерживает: оплату услуг; предоплату; оплату абонементов; готовых продуктов; курсов; подписок; индивидуальных предложений; историю оплат; возвраты; частичные удержания; связь оплаты с записью/пациентом/продуктом.

### 13.2. Провайдеры оплаты
Механизм подключения разных платёжных провайдеров. **Настройки провайдеров — в БД, не в ENV.** Возможность: добавить провайдер; хранить ключи/токены/настройки в БД; включить/отключить; выбрать провайдера для сценария / страницы / продукта / услуги; менять активную интеграцию без изменения кода.

### 13.3. Несколько способов оплаты
Разные способы для обычных приёмов / абонементов / курсов / спецпродуктов. Возможность выбрать способ для конкретного продукта/услуги/страницы.

### 13.4. Продукты
Модель создания продуктов: разовый приём; абонемент; подарочный сертификат; акция; курс; подписка; доступ к материалам; индивидуальное предложение. Продукт описывает: название; цену; состав; тип; правила доступа; правила оплаты; срок действия; связь с услугами/материалами/курсами; отображение в кабинете пациента; возможность покупки по ссылке.

### 13.5. Индивидуальная ссылка на оплату
Специалист/администратор создаёт индивидуальное предложение: создать продукт/абонемент/предложение → назначить пациенту → указать цену → система создаёт ссылку → пациент видит ссылку в кабинете → после оплаты продукт активируется.

## 14. Подписки, курсы и доступ к материалам

### 14.1. Подписки
Будущая платная подписка — доступ к материалам; урокам; программам; курсам; закрытым разделам приложения; доп. функциям.

### 14.2. Курсы
Продукт «курс»; покупка даёт доступ к материалам курса; связан с оплатами и личным кабинетом.

### 14.3. Доступы
После покупки система открывает доступ: к курсу; программе; материалам; подписке; записи по абонементу.

## 15. Календарь специалиста / администратора

### 15.1. Нужен календарь
По принципу Google Calendar. Видны: записи; свободные/занятые слоты; филиалы; кабинеты; специалисты; услуги; статусы; переносы; отмены; оплаты/предоплаты; абонементные визиты; ручные блокировки времени.

### 15.2. Возможность использовать готовый виджет
Рассмотреть готовый виджет/компонент: встроенный Google Calendar; готовый React-календарь; календарный компонент для Next.js/React; собственный, если готовые не покрывают логику.

### 15.3. Интеграция с Google Calendar
Синхронизация уже есть — рассмотреть использование. Важно: GCal — отображение/синхронизация, не единственный источник правды; канон — собственная БД; переносы/отмены/абонементы/оплаты/правила — внутри собственной системы.

### 15.4. Возможности календаря
Просматривать записи; создавать вручную; переносить; отменять; видеть статус оплаты; статус абонемента; комментарии; пациента; услугу; фильтровать по специалисту/филиалу/кабинету/дню/неделе/месяцу.

## 16. История пациента и карточка клиента

### 16.1. Общая история
Максимально полная: история записей; посещений; отмен; переносов; оплат; предоплат; возвратов; абонементов; списаний занятий; купленных продуктов; акций; комментарии специалиста/администратора.

### 16.2. Связь с будущей переработкой карточки клиента
Карточка клиента — отдельный крупный этап. Уже сейчас события должны сохраняться так, чтобы потом полноценно показать. Нельзя хранить только текущее состояние без истории.

### 16.3. История оплат
Дата; сумма; способ; провайдер; статус; назначение; связь с записью; с абонементом; с продуктом; возвраты; удержания; комментарии.

### 16.4. История посещений
Дата/время визита; специалист; филиал; кабинет; услуга; длительность; статус; был ли по абонементу; какое занятие списано; была ли предоплата; итоговая оплата; комментарий.

## 17. Статусы записей
Заложить статусную модель: создана; ожидает оплаты/предоплаты; оплачена; подтверждена; перенесена; отменена пациентом; отменена специалистом; поздняя отмена; неявка; завершена; посещение подтверждено; списана по абонементу; требует ручного решения. Статусы управляют: оплатами; абонементами; штрафами; предоплатами; переносами; отменами; уведомлениями; историей пациента.

## 18. Уведомления

### 18.1. Специалисту / администратору
Новая запись; перенос; отмена; поздняя отмена; оплата; предоплата; покупка абонемента; покупка акции; запрос на перенос; запрос на отмену; требуется ручное решение.

## 22. Основные бизнес-правила
1. Услуга не дублируется под каждый филиал/комбинацию.
2. Канонический источник — собственная база (даже при GCal/RubyTime на переходе).
3. Пациент не должен обходить штрафные правила через перенос.
4. Абонемент может быть составным.
5. Специалист принимает решение в спорных ситуациях (ручные override).
6. Настройки гибкие: клиника / специалист / услуга / продукт.

## 24. Критичные требования к проектированию
1. Не повторять модель RubyTime с дублированием услуг под каждую комбинацию.
2. Собственная база — канонический источник.
3. Услуги гибко назначаемы специалистам в разных филиалах/кабинетах.
4. Запись из приложения и с внешнего сайта.
5. Виджет пригоден для Tilda.
6. Обязательные поля записи настраиваются.
7. Поддержка нескольких последовательных слотов.
8. Самостоятельный перенос, если разрешён.
9. Гибкие правила и лимиты отмены/переноса.
10. Исключить обход штрафных правил через перенос.
11. Предоплата настраивается по услугам.
12. Платёжные провайдеры и ключи — в БД, не в ENV.
13. Абонементы составные и произвольные.
14. Списание занятия — авто/вручную по подтверждению.
15. При отмене — ручное решение (списать/не списать/удержать/вернуть и т.д.).
16. Акции/подарочные продукты связаны с пациентом и видны специалисту.
17. Пациент видит абонементы, остатки, покупки, историю.
18. Специалист видит всю историю.
19. Календарь удобный, желательно готовый компонент.
20. Архитектура учитывает SaaS-модель.
