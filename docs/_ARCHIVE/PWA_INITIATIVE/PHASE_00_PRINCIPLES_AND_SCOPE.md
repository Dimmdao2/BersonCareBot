# Фаза 0 — принципы, границы и DoD первой волны

Общий контекст: один origin (`bersoncare.ru`), один пакет `apps/webapp`. Детальные планы по шагам: [фаза 1](PHASE_01_ROOT_LANDING.md) · [фаза 2](PHASE_02_INSTALL_FLOW.md) · [фаза 3](PHASE_03_MANIFEST_AUDIT.md). [Backlog](BACKLOG.md).

## Продуктовые решения (не менять без явного решения)

1. **Главная `/`** — маркетинговая **обложка**: блоки про приложение (разминки, видео, советы, офис, острая боль и т.д.) и блок «обо мне»; ссылки на **внешний** сайт реабилитолога. **Без** кнопки «Войти» и **без** авто‑редиректа в `/app`.
2. **Цель главной** — довести пользователя до **установки PWA**; дизайн сначала **статичный**, визуальная полировка отдельным этапом.
3. **`start_url` в манифесте** — старт из иконки на **`/app/patient`** (как сейчас); на главной **не обязательна** ссылка на этот путь — установка и `start_url` решают вход в кабинет после установки.
4. **`scope` в манифесте** — **`/app`**: маркетинговый **`/`** и пути вне **`/app/*`** (например **`/legal/terms`**, **`/legal/privacy`**) **не** входят в URL‑оболочку установленного приложения. Регистрация **`/sw.js`** с лендинга с тем же **`scope: "/app"`** (лендинг **`/`** не попадает под контроль SW).
5. **Врач** — только **прямой** заход на **`/app/doctor`** (закладка, ссылка, QR). Отдельного второго логина не делать: общая точка **`/app`** после редиректа с guard’ов при отсутствии сессии.
6. **iOS** — нет программной установки как в Chrome; на главной — **текстовая инструкция** (Поделиться → На экран «Домой»). Принудительно «оболочку Chrome» для PWA на iOS **нельзя**.
7. **Корень `/`:** маркетинговый лендинг в **`apps/webapp/src/app/page.tsx`** (без безусловного редиректа в `/app` для обычного GET). Отдельный авто‑редирект с nginx на главную для этой задачи **не требуется**.

## Границы scope

### В scope

- Корневой маршрут `apps/webapp/src/app/page.tsx` (лендинг).
- Компоненты/стили лендинга в `apps/webapp/src/app/**` или `apps/webapp/src/shared/ui/**`.
- Manifest и установка (`apps/webapp/src/app/manifest.ts`, клиентский install-flow, при необходимости минимальный SW).
- Документация инициативы в `docs/PWA_INITIATIVE/**`.

### Вне scope

- Переименование URL/ролей `patient` → `client`.
- Полный offline-кэш и фоновая синхронизация данных.
- Полный Web Push контур (таблицы, API, фоновые отправки).
- Рефактор auth/guards вне необходимого минимума для сохранения текущего поведения.

## Definition of Done (первая волна)

Реализация в репозитории закрыта; **проверки в реальных браузерах и на продакшен‑origin** вынесены в блок ниже «Верификация на стенде» и в §«Ручные проверки» фаз 1–3.

- [x] Лендинг на **/** и UI **установки PWA** (Chrome + текст для **iOS**) реализованы в коде.
- [x] **Нет** безусловного редиректа с **/** в `/app` для обычного GET (см. `page.tsx`).
- [x] **`manifest`** и пути иконок в коде согласованы с сценарием «иконка → `/app/patient`»; **DevTools / HTTP 200 на развёрнутом хосте** — чеклист [фазы 3](PHASE_03_MANIFEST_AUDIT.md).
- [x] **Мини‑апп** Telegram/MAX: SW **не** регистрируется при `isMessengerMiniAppHost()` — см. [фазу 2](PHASE_02_INSTALL_FLOW.md); **регресс в WebView** — ручной пункт [фазы 2](PHASE_02_INSTALL_FLOW.md).
- [x] Запись в **`LOG.md`** о фазах и отложенном (maskable, push и т.д.).
- [x] Прямые маршруты приложения не переписывались; **дымовой прогон в клиентах** — ручной пункт [фазы 1](PHASE_01_ROOT_LANDING.md).

## Порядок исполнения (рекомендуемый)

1. [Фаза 1](PHASE_01_ROOT_LANDING.md) — лендинг `/` + локальный smoke маршрутов.
2. [Фаза 2](PHASE_02_INSTALL_FLOW.md) — install UX + только при необходимости минимальный SW.
3. [Фаза 3](PHASE_03_MANIFEST_AUDIT.md) — аудит manifest и иконок после первых двух фаз.
4. Финальная запись в `LOG.md` с результатами и тем, что намеренно отложено.

Примечание: быстрый pre-check из [фазы 3](PHASE_03_MANIFEST_AUDIT.md) допустим после фазы 1, если нужно рано поймать ошибки manifest.
