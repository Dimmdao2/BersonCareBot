## 0.1 Явные переопределения референса (заказчик, 2026-06-12)

Только эти пункты отклоняются от вайрфрейма; всё остальное — строго по референсу.

1. **«График работы» — классическая раскладка в две колонки:** слева — сетка дней, справа — настройка
   часов, **ниже** — настройка шаблонов. (Референс держит сетку на всю ширину сверху, часы+шаблоны в ряд
   снизу — заменяем.)
2. **Несколько перерывов в дне** (референс — один «обед»). Нужна поддержка N перерывов + строчная
   раскладка слотов: `начало–конец / перерыв 1 / перерыв 2 / (+ перерыв)`.
3. **Короткие названия филиалов** (Москва → «Мск», Петербург → «СПб») — настраиваемые, отдельное поле
   у филиала. (Термин — «короткое название / аббревиатура».)
4. **Карточки дней «График работы» — у́же по ширине**, **текст в карточке (время) — крупнее**.

## 2. Целевое поведение «Записи» (свод из вайрфрейма + пояснений)

> Полные формулировки — в «Пояснениях по поведению» и `#p-schedule`. Здесь — то, что определяет код.

1. **Две независимые оси.** Вкладки раздела (Записи · График работы · Настройки) и переключатель периода
   внутри «Записи» (3 дня · Неделя · Месяц · Лента) не пересекаются. Период сохраняется при уходе на
   work/setup и возврате.
2. **Период — единый источник истины.** Стрелки ◀▶, лейбл периода и KPI-ряд привязаны к **одному**
   видимому диапазону. Отдельных селекторов периода для метрик нет. Листание назад = история.
3. **Фильтры Локация/Услуга** применяются и к календарю, и к KPI одновременно.
4. **KPI-ряд (9 карточек)** — строго по видимому периоду и фильтрам; нули как нули; «Прошло/Впереди» —
   относительно текущего момента; карточки кликабельны (`cursor:pointer`, фильтрация — следующая
   итерация). Неявок в ряду нет.
5. **Режимы:** 3 дня (дефолт, сетка сегодня+2), Неделя (сетка пн–вс), Месяц (плашка=строка, фамилия;
   сегодня — жёлтая подсветка), Лента (бесконечный поток, **KPI и стрелки скрыты**).
6. **День — drill-down,** не режим: клик по заголовку дня (3 дня/Неделя) или числу (Месяц); внутри
   «← Назад» в исходный режим; KPI скрыт; стрелки листают по дням.
7. **Правая панель:** запись → карточка; пусто → заглушка «Запись не выбрана» + CTA «+ Создать запись» +
   подсказка ближайшего свободного окна сегодня.
8. **Дублирование «+ Создать запись»** (тулбар + заглушка) — намеренное.

## 3. Архитектурное решение «Записи»

### 3.1 KPI переезжает из шелла в таб «Записи» (метрики — ТОЛЬКО здесь)
KPI должны: (а) считаться по видимому диапазону, (б) реагировать на Локацию/Услугу, (в) скрываться в
Ленте и Дне, (г) **не появляться на табах «График работы»/«Настройки»** (сейчас висят над всеми табами —
баг со скрина). Всё это — состояние таба «Записи», а не шелла.

- `DoctorScheduleShell` **теряет**: `KpiRow`, `loadKpis`, состояние `period`, `?period`,
  `initialKpis`/`initialPeriod`. Остаётся: `DoctorAppShell` + `ScheduleTabsNav` + реестр + URL-sync.
- Таб «Записи» владеет: `view`, `date`, `location`, `service`, `appt`, `from`; по смене диапазона/фильтров
  — **параллельно** фид + KPI (один диапазон+фильтры).
- SSR: `page.tsx` предзагружает KPI+фид для дефолта «3 дня», отдаёт в таб как `initialData`.

### 3.2 Единый расчёт диапазона `visibleRange(view, anchor, tz)`
| view | rangeStart | rangeEnd | KPI |
|------|-----------|----------|-----|
| `3days` | сегодня/anchor | +2 дня | = фид |

## 4. Бэкенд-дельта

### 4.1 KPI: 9 метрик + фильтры + произвольный диапазон
**Тип** (`modules/doctor-appointments/ports.ts`, расширить `ScheduleKpis`):
```ts
export type ScheduleKpis = {
  recordsInPeriod: number;        // Записей: неотменённые, start_at в окне
  pastInPeriod: number;           // Прошло: из records, start_at < now()
  futureInPeriod: number;         // Впереди: из records, start_at >= now()
  bySubscriptionInPeriod: number; // По абонементу: records, package_usage_ref IS NOT NULL
  firstVisitInPeriod: number;     // Первичных: у пациента нет более ранней неотменённой записи
  repeatVisitInPeriod: number;    // Повторных: records − firstVisit
  uniquePatientsInPeriod: number; // Уникальных: COUNT(DISTINCT platformUserId)
  cancellationsInPeriod: number;  // Отмены (см. §13)
  reschedulesInPeriod: number;    // Переносы (см. §13)
};
```
Инварианты: `past+future = records`, `first+repeat = records` (сверка вайрфрейма: 2+7=9, 1+8=9).
Снять `newPatientsInPeriod` после `rg`-проверки (карточки «Новых» больше нет).

**Запрос** — новый тип: `ScheduleKpisQuery { from; to; branchId?; serviceId? }`. Сигнатура порта:
`getScheduleKpis(query, audience?)`.

**pg** (`pgDoctorAppointments.ts`) — **Drizzle**, переиспользовать window-хелперы; фильтры branch/service
в `WHERE` всех метрик (join к `appointment_records` для отмен/переносов). **inMemory** — паритет.
**Роут** `schedule-kpis` — GET `{from,to,branchId?,serviceId?}` (Zod), `requireDoctorAccess`.

### 4.2 Календарный фид: произвольные диапазоны + часы ±1ч + short-title
- `getCalendar` уже принимает `rangeStart/rangeEnd` — переиспользовать для `3days`/`day`/`feed`; **новых
  контрактов не плодить**.
- **Часы сетки** часовых видов — из `working`-событий видимых дней ±1ч; fallback — дефолт. Считать на
  клиенте из фида либо отдавать `workingBounds` в ответе (решить, LOG).
- `feed`: сервер отдаёт записи за окно; бесконечный скролл — клиент доращивает `from/to`, склеивает,
  пустые дни отбрасывает.
- В `CalendarFilterOption` (branches) добавить `shortTitle` (для коротких меток в фильтрах/сетке).

### 4.3 «Ближайшее свободное окно сегодня» (заглушка правой панели)
- Источник — слот-движок `modules/booking-scheduling`. В фиде свободные окна не отдаются → отдельный read
  `nearestFreeWindow(dateKey, scope) → {from,to}|null` (порт + pg/inMemory + тонкий роут или поле фида).
- Деградирует gracefully: нет окна/недоступно → CTA без подсказки, ребилд не блокируется.

### 4.4 Несколько перерывов в дне (переопределение §0.1.2) — модель + миграция
Сейчас `be_working_days`/`be_schedule_templates` хранят **один** перерыв
(`break_start_minute`/`break_end_minute`). Нужна поддержка **N перерывов**.

- **Схема (Drizzle):** добавить колонку `breaks jsonb` со `$type<Array<{startMinute:number; endMinute:number}>>`,
  default `'[]'`, в `beWorkingDays` и `beScheduleTemplates`. Миграция `0116_*` через `drizzle-kit generate`
  + ревью SQL; **бэкфилл** из существующих одиночных break-колонок в массив; легаси-колонки оставить
  nullable (читаем `breaks`, при пустом — fallback на легаси), снять/ослабить старые break-`check`
  (валидация массива — в Zod/сервисе, не в SQL-check).
- **Валидация (Zod/сервис):** каждый перерыв ⊂ `[start,end]`, не пересекаются, отсортированы по началу,
  `count ≤ N` (напр. ≤6). `assertUuid`, минуты 0..1440 — как сейчас.
- **Слот-движок** `computeSlots.ts:workingIntervalsForDate(...perDayRow?)`: вместо «один break → 2
  интервала» — вычесть **все** перерывы (отсортировать, получить N+1 интервалов). Юнит-тесты: 0/1/2/3
  перерыва, граничные, closed, fallback на weekday.
- **Порты/типы** `booking-scheduling`: `WorkingDayRecord.breaks`, `UpsertWorkingDaysInput.breaks`,
  `ScheduleTemplateRecord.breaks`; pg upsert пишет `breaks`; backward-compat для существующих вызовов.

### 4.5 Короткое название филиала (переопределение §0.1.3) — поле + место настройки
- **Схема (Drizzle):** `be_branches.short_title text` (nullable). Миграция `0117_*`. Бэкфилл — не нужен
  (nullable; отображение — `shortTitle ?? title`).
- **Чтение:** прокинуть `shortTitle` в booking-engine overview/branch-листы, в `CalendarFilterOption`
  (§4.2) и в источник списка филиалов для «Графика работы».
- **Настройка:** добавить одно поле «Короткое название (СПб, Мск)» в `BookingSoloLocationsSection`
  (Настройки → Локации) — точечная правка существующей секции (PATCH ветки branch). Зод-валидация:
  trim, ≤ 12 символов.
- **Отображение:** короткое имя — в карточках дней «Графика работы», в цветовой легенде филиалов, в
  компактных метках месяца «Записей». Где места достаточно — полный `title`.

## 6. Фронт «График работы» (`ScheduleWorkTab`) — ребилд

> База — текущий `ScheduleWorkTab.tsx` (месячная сетка, мультивыбор, панель часов, шаблоны). Перекладываем
> раскладку и допиливаем под §0.1 + fidelity.

### 6.1 Раскладка — две колонки (переопределение §0.1.1)
- **Слева:** месячная сетка дней (мультивыбор click/Shift/Cmd) + над ней — месяц-навигация `◀ Июнь 2026 ▶`
  и легенда филиалов.
- **Справа:** панель «Настройка часов» для выбранных дней (появляется при ≥1 дне; иначе — компактная
  заглушка «Выберите дни»).
- **Снизу (на всю ширину):** «Шаблоны расписаний» (список + Применить/удалить + «+ Создать»).
- Грид: `lg:grid-cols-[<дни>_<часы>]` слева-направо, шаблоны — отдельным рядом снизу. На узких — стэк.

### 6.2 Карточки дней (переопределение §0.1.4 + fidelity)
- **У́же по ширине** (компактнее, чтобы сетка помещалась в левой колонке).
- **Время — крупнее** (сейчас `text-[9px]` — мелко): поднять размер строки часов в карточке.
- Цвет ячейки — по локации дня (как сейчас, детерминированный маппинг), но метка — **короткое имя**
  филиала (§4.5): «СПб 11–19», «Мск 12–20»; перерывы — «обед 13–14» (или «2 перерыва» при N>1).
- Сегодня — жёлтая подсветка; выбранные — рамка primary; закрытые — серые «выходной».

### 6.3 Кнопки филиалов над днями — реальный фильтр сетки (✅ §13.2)
Сейчас верхние кнопки = «локация для назначения» (`activeBranchId`→`panelBranchId`) и сетку **не
фильтруют** — отсюда жалоба «фильтрация не работает». **РЕШЕНО:** сделать их **реальным фильтром сетки**.
- Над сеткой — переключатель филиалов с опцией **«Все»** (дефолт). Выбор филиала → сетка показывает
  дни-расписания **только этого филиала**; пустые/невыставленные дни видны всегда (чтобы можно было
  назначать). Фильтр прокидывается в чтение `working-days` (`branchId`) либо фильтруется на клиенте по
  `record.branchId` (решить по объёму — бэкенд-фильтр предпочтительнее, LOG).
- **Локация назначения** — отдельно, в правой панели часов (селект «Локация»); дефолт = активный
  филиал-фильтр (если выбран конкретный; иначе — первый активный).
- Кнопки фильтра и карточки дней — цвет филиала + **короткое имя** (§4.5).

### 6.4 Панель «Настройка часов» — строчная раскладка + несколько перерывов (§0.1.2)
Слоты — **построчно**, не в одну линию:
```
Начало [11:00]   Конец [19:00]
Перерыв 1  [13:00] – [14:00]   [×]
Перерыв 2  [15:00] – [15:30]   [×]
[ + перерыв ]
Локация: [ ▾ ]
[Сохранить] [Закрыть выбранные дни] [Очистить выбор]
```
- Несколько перерывов: добавление/удаление строк, отправка массива `breaks` в `PUT working-days`
  (§4.4). Клиентская валидация (перерыв ⊂ дня, без пересечений) перед сохранением.
- Заголовок панели — «Задать расписание для N выбранных дней (даты)» как сейчас.
- «Закрыть выбранные дни» = `action:"close"`; «Очистить выбор» — локально.

### 6.5 Шаблоны
- Поддержать **несколько перерывов** в шаблоне (форма создания: те же строчные перерывы); `breaks` в
  `POST templates`. Применение — `?action=apply` на выбранные дни (как сейчас).
- Список — короткое имя филиала в подписи; «Применить» доступна при ≥1 выбранном дне.
