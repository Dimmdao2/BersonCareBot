# Структура кабинета специалиста BersonCare

**Версия:** базовая продуктовая структура  
**Назначение:** состав экранов, функций и принципов интерфейса кабинета специалиста для вебаппа (мессенджеры, мобильный браузер, десктоп).

---

## 1. Общая логика продукта

Кабинет специалиста — **не** техническая админка платформы. Техническая админка будет отдельным продуктом на отдельном поддомене.

**Кто работает в кабинете:**
- специалист;
- старший специалист или управляющий;
- операционный администратор (доступ к клиентам, записям, коммуникациям, рассылкам).

**Основная задача кабинета:**
- видеть клиента целиком;
- быстро понимать, что с ним происходит;
- управлять ближайшими действиями;
- связываться с клиентом в нужных каналах;
- видеть историю записей, отмен, дневников и коммуникаций.

**Один интерфейс, разная плотность:**
- **Мобильный режим:** основной сценарий, второстепенные блоки — под вкладки и раскрывающиеся секции.
- **Десктопный режим:** те же маршруты, больше блоков одновременно, двух- и трёхколоночные компоновки.

---

## 2. Главные принципы проектирования

### 2.1. Один клиент — одна центральная карточка

Почти все действия специалиста начинаются из карточки клиента. Карточка — главный центр работы, а не просто страница с контактами.

### 2.2. Главное не прятать

На первом экране клиента всегда видны:
- контакты и все привязанные каналы;
- ближайшая запись;
- последние изменения по дневникам;
- история отмен и переносов;
- быстрые действия: написать, открыть записи, открыть дневники.

### 2.3. Десктоп — расширенный режим, а не отдельный продукт

Показывать больше панелей одновременно: список клиентов, выбранный клиент, ближайшие записи, быстрые фильтры, панель действий. Отдельной структуры навигации не создавать.

### 2.4. Коммуникации — системная часть кабинета

Отправка сообщения клиенту и рассылки — базовый сценарий, не «дополнительная функция».

### 2.5. Массовые действия — проверяемые

Для рассылок обязательно:
- категория рассылки;
- выбор аудитории;
- предпросмотр;
- фиксация автора;
- история отправки;
- статус по каналам.

## 3. Основные разделы кабинета

| № | Раздел | Назначение |
|---|--------|------------|
| 1 | **Обзор** | Задачи на сегодня, ближайшие записи, клиенты с рисками, последние события |
| 2 | **Клиенты** | Список, поиск, фильтры, карточка клиента, история, дневники, каналы |
| 3 | **Записи** | Ближайшие записи, отмены, переносы, статусы, переход в клиента |
| 4 | **Сообщения** | Индивидуальная коммуникация, шаблоны, статусы отправки, журнал |
| 5 | **Рассылки** | Массовые уведомления по категориям и сегментам (по правам доступа) |
| 6 | **Статистика** | Записи, отмены, загрузка, активность клиентов, каналы, дневники |
| 7 | **Настройки** | Личные настройки специалиста, уведомления, шаблоны, отображение |

В навигации также доступны, среди прочего, упражнения, шаблоны ЛФК, **справочники** (`/app/doctor/references`) и **контент (CMS)** (`/app/doctor/content`). Структура хаба CMS, фильтр по разделам и журналирование сбоев загрузки данных: [`DOCTOR_CMS_AND_RUNTIME.md`](DOCTOR_CMS_AND_RUNTIME.md).

Таблица значений справочника (`/app/doctor/references/[categoryCode]`): строки загружаются одним набором; поиск по названию/коду и фильтр «активные / архивные / все» выполняются **на клиенте** без смены URL; активность строки переключается иконкой глаза (как в CMS); **удалить** помечает строку `deleted_at` (soft delete), отдельно от флага архива `is_active`.

Экран упражнений (`/app/doctor/exercises`) поддерживает desktop-режимы:
- `tiles`: компактная сетка карточек с квадратным превью;
- `list`: master-detail компоновка (левый список + правая панель создания/редактирования);
- мобильный режим сохраняет упрощённый карточный сценарий без split-layout.

## 4. Раздел «Обзор»

**Цель:** ответ на вопрос «что требует внимания прямо сейчас».

### 4.1. Основные блоки

**Мой день**
- количество записей на сегодня;
- ближайшая запись;
- отмены и переносы за сегодня;
- клиенты без подтверждения / кому нужно напоминание.

**Ближайшие записи**
- сегодня и завтра: время, клиент, тип, статус;
- быстрый переход в карточку клиента.

**Требуют внимания**
- частые отмены;
- давно не выходил на связь;
- ухудшение по дневнику симптомов;
- пропуски дневника;
- нет активных каналов;
- не доставляются сообщения.

**Последние события**
- новая запись, отмена, перенос;
- новая привязка канала;
- запись в дневнике;
- отправлено сообщение / ошибка доставки.

**Быстрые действия**
- найти клиента;
- открыть записи на сегодня;
- создать сообщение;
- рассылки;
- статистика.

### 4.2. Десктоп

- Верхняя строка — KPI.
- Левая колонка — ближайшие записи.
- Центр — требуют внимания.
- Правая колонка — последние события и быстрые действия.

### 4.3. Мобильный

Сверху вниз: поиск → KPI → ближайшие записи → «требуют внимания». Последние события — под «Показать ещё».

### 4.4. KPI и deep links (реализация webapp)

Числовые определения плиток обзора и соответствие ссылок спискам зафиксированы в **DOCTOR_DASHBOARD_METRICS.md**. Историю изменений — **MIGRATION/DOCTOR_DASHBOARD_METRICS_CHANGELOG.md**.

## 5. Раздел «Клиенты»

Главный рабочий раздел.

### 5.1. Экран списка

- Строка поиска;
- быстрые фильтры;
- сортировка;
- список клиентов с признаками важных состояний в строке.

### 5.2. Поиск

По: имени, телефону, Telegram id, MAX id, внутреннему userId, части номера, части имени.

### 5.3. Фильтры (минимальный набор)

- есть / нет ближайшей записи;
- были отмены / отмены за 30 дней;
- есть активный Telegram / MAX / только SMS;
- нет каналов для уведомлений;
- есть новые записи в дневниках / давно не было дневников;
- новый / неактивный клиент.

### 5.4. Строка клиента в списке

Имя, телефон, иконки каналов, ближайшая запись, короткий статус, бейдж риска, число отмен за период.

### 5.5. Десктоп

Master-detail: слева список, справа карточка выбранного клиента, сверху фильтры и поиск.

### 5.6. Мобильный

Список → по нажатию отдельный экран клиента. Фильтры и сортировка — через верхнюю панель.

## 6. Карточка клиента

Ключевой экран кабинета.

### 6.1. Верхний блок

ФИО (отображаемое имя), телефон, внутренний id, дата первого контакта, статус, активный/неактивный, ответственный (если появится).

### 6.2. Контакты и привязки

- Телефон, Telegram, MAX, другие каналы;
- какие привязаны, разрешены для уведомлений, для сообщений;
- последний успешный канал.

**Действия:** отправить во все / в выбранные каналы / только SMS; история доставок; предпочтительный канал.

### 6.3. Ближайшие записи

Ближайшая — крупно; далее список: дата, время, тип, статус, ссылка/id внешней системы, комментарий.  
Действия: все записи клиента, отмены и переносы, копировать ссылку.

### 6.4. История записей

Прошедшие, отменённые, перенесённые, no-show (если есть); инициатор и причина отмены (если источник передаёт).

**Статистика:** всего записей, всего отмен, отмен за 30 дней, доля отмен, последняя отмена, последний/следующий визит.

### 6.5. Дневники

- Дневник симптомов и ЛФК;
- последние записи, дата последнего заполнения, динамика, пропуски, отклонения.

**Минимум:** список симптомов, последние оценки, последние отметки ЛФК, частота заполнения, «давно не заполнял».

### 6.6. Коммуникации

Последние сообщения клиенту, каналы, статусы доставки, ошибки, шаблоны, дата последнего контакта.  
Действия: написать, повторить, журнал, отправить шаблон.

### 6.7. Внутренние заметки

Короткие заметки специалиста, флаг клиента, напоминание; не показывать клиенту. Заложить с самого начала.

### 6.8. События клиента

Единая временная шкала: запись создана/отменена/перенесена, дневник заполнен, сообщение отправлено, привязка канала, смена телефона. Важно для контекста без перехода по экранам.

### 6.9. Реализация в webapp (ориентир)

- **Карточка** — `ClientProfileCard`: блоки как **аккордеон** (одна открытая секция; по умолчанию «Контакты»). На `/app/doctor/clients` и на `/app/doctor/clients/[userId]` один и тот же паттерн.
- **Admin + admin mode:** секции «Объединение учётных записей» и «История операций» не дергают тяжёлые API, пока секция закрыта (ленивая загрузка).
- **Редактирование профиля клиента админом** (ФИО, email, телефон): `PATCH /api/admin/users/:userId/profile` и панель `AdminClientProfileEditPanel`; доверие телефона — trusted source `admin_manual_profile_patch` (см. `PLATFORM_IDENTITY_SCENARIOS_AND_CODE_MAP.md` §8).

## 7. Раздел «Записи»

Отдельный рабочий инструмент.

### 7.1. Список

Сегодня, завтра, неделя; фильтры по статусу, специалисту, типу записи.

### 7.2. Строка записи

Время, клиент, тип, статус, контактный статус, канал напоминаний, число прошлых отмен, быстрые действия. Для записей из проекции Rubitime: если имя в `payload_json.name` отличается от подписи профиля (`platform_users`), под основной строкой клиента показывается краткая подсказка «В Rubitime: …» (см. [`DOCTOR_DASHBOARD_METRICS.md`](DOCTOR_DASHBOARD_METRICS.md), [`RUBITIME_BOOKING_PIPELINE.md`](RUBITIME_BOOKING_PIPELINE.md)).

### 7.3. Быстрые действия

Открыть клиента, напоминание, уточнение, отметить как требующую внимания, история записей клиента.

### 7.4. Представления

Ближайшие записи, отмены, переносы, пропущенные визиты, неподтверждённые.

### 7.5. Десктоп

Слева список записей, справа детали выбранной записи и краткая карточка клиента.

## 8. Раздел «Сообщения»

Персональная коммуникация.

### 8.1. Сценарии

- Индивидуальное сообщение клиенту;
- во все / выбранные каналы или только SMS;
- по шаблону;
- история отправок.

### 8.2. Экран создания сообщения

Получатель, список каналов, чекбоксы каналов, тип/причина сообщения, текст, превью, предупреждение при отсутствии каналов.

### 8.3. Категории сообщений (заложить сразу)

Организационное, напоминание, уточнение по записи, просьба заполнить дневник, обратная связь после приёма, сервисное.

### 8.4. Шаблоны

Библиотека, переменные, по категории, личные и общие шаблоны организации.

### 8.5. История отправок

Кто, когда, текст, каналы отправки/доставки/ошибок, тип сообщения.

---

## 9. Раздел «Рассылки»

Операционная работа внутри продукта, не техническая админка.

### 9.1. Идея

Массовое уведомление пользователям по разрешённым каналам.

### 9.2. Обязательно

Категория рассылки, сегмент аудитории, текст, выбор каналов, предпросмотр, подтверждение, лог автора, журнал результатов.

### 9.3. Категории (обязательный выбор)

Сервисная, организационная, маркетинговая, важное уведомление, изменение расписания, напоминание, обучение, опрос.

### 9.4. Аудитории

Всем; активным; с ближайшей записью / без записи; с Telegram / MAX / только SMS; по статусу; давно неактивным.

### 9.5. Доставка

Во все разрешённые для категории каналы у пользователя (или в выбранные вручную). Фиксировать: размер выборки, число отправок, попытки по каналам, успех/ошибки.

### 9.6. Безопасность

Права на рассылки, предпросмотр, защита от случайной отправки всем, журнал действий, сохранение текста.

### 9.7. Реализация в webapp (2026-05)

Каноническое описание маршрута **`/app/doctor/broadcasts`**: предпросмотр (в т.ч. список имён, учёт **`dev_mode`** / **`test_account_identifiers`**), подтверждение, запись в **`broadcast_audit`**, постановка заданий в **`public.outgoing_delivery_queue`** (`kind = doctor_broadcast_intent`), доставка **воркером integrator** через **`dispatchOutgoing`**, опция **«Прикрепить / обновить меню»** — см. **[`DOCTOR_BROADCASTS.md`](DOCTOR_BROADCASTS.md)**.

## 10. Раздел «Статистика»

Для операционной работы, не BI.

### 10.1. По записям

Записи за период, отмены, переносы, доля отмен, среднее на клиента, лидеры по отменам, по дням/типу, последний/следующий визит.

### 10.2. По коммуникациям

Объём отправок по каналам, эффективность, ошибки, кто не получает / давно не получал.

### 10.3. По дневникам

Кто ведёт дневники, кто давно не заполнял / регулярно, негативная динамика, пропуски.

### 10.4. По базе

Новые/активные/неактивные клиенты, без каналов, с одним/несколькими каналами.

## 11. Раздел «Настройки»

**Личные:** имя, роль, уведомления в кабинете, вид главной, шаблоны, подпись, фильтры по умолчанию.  
**Операционные:** категории сообщений по умолчанию, KPI, видимость блоков, быстрые шаблоны, напоминания.

## 12. Мобильный и десктоп

### Единый каркас страниц (реализация webapp)

Кабинет специалиста (`/app/doctor/*`) и страница **`/app/settings`** для ролей врач/админ используют одну визуальную колонку на десктопе:

- **Фиксированная шапка** — компонент `DoctorHeader` (`apps/webapp/src/shared/ui/DoctorHeader.tsx`); внутренний ряд ограничен **`max-w-7xl`** и теми же горизонтальными отступами, что и контент (**`px-4` / `md:px-6`**), чтобы края контента совпадали с баром шапки.
- **Отступ под шапку** — общий класс `DOCTOR_WORKSPACE_TOP_PADDING_CLASS` (высота шапки, safe-area, небольшой зазор); для `/app/doctor` он задаётся в `apps/webapp/src/app/app/doctor/layout.tsx`, для `/app/settings` — на той же странице.
- **Контент страниц** — контейнер **`DOCTOR_PAGE_CONTAINER_CLASS`** (`max-w-7xl`, те же `px`, `pb-8`): либо через **`AppShell` с `variant="doctor"`** (основной путь для экранов кабинета, включая онлайн-заявки), либо напрямую на `/app/settings`.
- **Источник правды по классам** — `apps/webapp/src/shared/ui/doctorWorkspaceLayout.ts` (при смене максимальной ширины кабинета править там и проверить шапку и контейнер).
- **Настройки `/app/settings` (вкладки admin):** вертикальный список вкладок (`TabsList` с `orientation="vertical"`); порядок и названия: **«Режимы»** (бывш. «Админ: режим»), «Здоровье системы», «Параметры приложения», «Авторизация», «Интеграции», «Каталог записи», «Лог операций» — без отдельной вкладки вайтлистов ролей (2026-05-02). Содержимое справа — `AdminSettingsTabsClient`.

Пациентский раздел (`variant="patient"`, узкая колонка) на это не распространяется.

### Одинаково

Маршруты, сущности, права доступа, сценарии, структура данных.

### Меняется

Количество блоков, форма навигации, плотность таблиц, правая панель, число видимых фильтров, второстепенные виджеты.

### Только десктоп

Master-detail по клиентам, боковая панель действий, расширенные таблицы записей, несколько блоков статистики, расширенный журнал событий, клиент + ближайшие записи рядом.

### Упрощать на мобильном

Большие таблицы, длинные фильтры, расширенные журналы, многоколонки, второстепенные KPI, технические статусы.

### Принцип мобильного

Один экран — один главный вопрос: кто клиент, что ближайшее, что сделать сейчас, как быстро написать.
