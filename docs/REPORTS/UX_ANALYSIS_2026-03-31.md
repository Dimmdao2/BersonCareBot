## 2. Ошибки UX

### 2.1. Критические

**2.1.1. Мок-данные на продакшн-экране «Мои покупки»**

Страница `/app/patient/purchases` показывает пациенту захардкоженные `MOCK_ITEMS` с конкретными названиями курсов и датами (`до 15.06.2025` — уже прошедшая дата). Пациент видит несуществующие покупки. Это вводит в заблуждение и подрывает доверие.

**Решение:** убрать страницу из навигации или заменить мок пустым состоянием с пояснением «Раздел будет доступен позже».

**2.1.2. Неработающие PlaceholderPage доступны из навигации**

`/app/patient/help`, `/app/patient/install`, `/app/doctor/references` — пользователь может перейти на них и увидеть «Раздел в разработке» без контекста и пояснения, когда он будет готов. Для пациента это выглядит как сломанное приложение.

**Решение:** скрыть из навигации всё, что показывает PlaceholderPage, или заменить содержательным empty-state.

**2.1.3. Адрес кабинета — внешний iframe без fallback**

`/app/patient/address` загружает `https://dmitryberson.ru/adress` в iframe. Если сайт недоступен, пациент видит белый/ошибочный экран. Нет loading-state, нет fallback-сообщения, нет обработки iframe-ошибок.

### 2.2. Существенные

**2.2.1. Медиабиблиотека — таблица min-width 52rem на мобильном**

`MediaLibraryClient` использует `<table>` с `min-w-[52rem]`. На мобильном экране (а у врача мобильный — основной рабочий инструмент) таблица уходит за экран, горизонтальный скролл неудобен. Фильтры тоже перегружены для мобильного (4 контрола + кнопка загрузки в одну строку).

**2.2.2. Медиабиблиотека — кнопка загрузки файла замаскирована**

Кнопка «Загрузить файл» — это `<label>` со скрытым `<input type="file">`, стилизованная как обычный outline-button. Нет визуального разделения от фильтров. Нет drag-and-drop зоны. Нет индикации прогресса загрузки (только текст «Загрузка...»). Нет множественной загрузки. Не поддерживает ни камеру мобильного, ни capture-режимы.

**2.2.3. Picker-диалог медиафайлов — не настоящий диалог**

`MediaLibraryPickerDialog` — это не модальный Dialog/Sheet, а inline-блок, который разворачивается внутри формы. На мобильном это создаёт прокрутку, перекрывает форму контентом, выглядит как сломанная страница. Таблица picker'а тоже min-width 36rem.

**2.2.4. CMS ContentForm — нет предпросмотра контента**

Форма редактирования контент-страницы не показывает, как будет выглядеть результат для пациента. Маркдаун-редактор есть, но preview для всей страницы (с картинкой, видео) отсутствует.

**2.2.5. Статистика врача — сырые данные без визуализации**

Страницы `/app/doctor/stats` и блок статистики в `/app/doctor/appointments` — это простые `<ul><li>` со строками «Записей: N», «Отмен: N». Нет графиков, нет трендов, нет сравнения с прошлым периодом. Для врача это бесполезная стена чисел.

**2.2.6. Журнал сообщений врача — нет пагинации**

`/app/doctor/messages` загружает `listAllMessages(50)` без пагинации. При росте объёма врач не увидит старые сообщения. Нет фильтрации по клиенту, по дате, по категории.

**2.2.7. Doctor Dashboard — дублирование border-классов**

В `DashboardTile` есть `rounded-2xl border border-border bg-card p-4 shadow-sm` **и** `rounded-xl border border-border/60 p-4 shadow-sm` — двойной border, двойной padding, конфликтующий border-radius. Визуально может рендериться некорректно.

### 2.3. Нерациональные решения

**2.3.1. Два отдельных списка: Клиенты и Подписчики**

`/app/doctor/clients` и `/app/doctor/subscribers` — фактически один и тот же интерфейс (`DoctorClientsPanel`) с разным источником данных. «Клиенты» = пользователи с записями, «Подписчики» = все client-роли. Для врача с 50-200 пациентами это избыточное разделение, создающее путаницу: «Где мне искать пациента?».

**Рекомендация:** объединить в один список с фильтром «Только с записями / Все».

**2.3.2. Кнопки «Быстрые действия» на дашборде врача дублируют навигацию**

Блок «Быстрые действия» на `/app/doctor` содержит ссылки (Подписчики, Клиенты, Записи, Сообщения, Статистика), которые 1:1 дублируют пункты DoctorHeader Sheet-меню. Экранное место расходуется без пользы.

**2.3.3. Два маршрута news + motivation в CMS**

`/app/doctor/content/news` и `/app/doctor/content/motivation` — отдельные страницы для двух конкретных типов контента, хотя контент-модель уже работает через разделы (sections). Это частный случай, ставший отдельным экраном.

**Рекомендация:** управлять новостями и мотивацией через общий интерфейс разделов, убрав отдельные маршруты.

**2.3.4. Поле «ID области (справочник)» в упражнениях — UUID вручную**

На `/app/doctor/exercises` фильтр «ID области» принимает UUID. Врач не знает UUID-ы. Это техническое поле, а не пользовательский интерфейс.

**Решение:** заменить на dropdown/autocomplete из справочника областей тела.

**2.3.5. Сортировка контент-страниц — числовое поле в форме**

В `ContentForm` порядок сортировки — числовой input `sort_order`. При этом в списке страниц уже есть drag-and-drop (`ContentPagesSectionList`). Два механизма сортировки конфликтуют: drag-and-drop сохраняет порядок через API, а числовое поле в форме — через отдельное сохранение.

## 3. Медиабиблиотека и CMS — детальный анализ и план

### 3.1. Текущее состояние

Что есть:
- API: `/api/admin/media` (GET список, DELETE), `/api/media/upload` (POST)
- Компонент `MediaLibraryClient`: таблица файлов, фильтры (тип, сортировка, поиск), загрузка одного файла, удаление
- Компонент `MediaLibraryPickerDialog`: inline-picker для выбора файла из библиотеки при редактировании контент-страницы
- Превью: изображения (img), видео (video), аудио (audio), файлы (ссылка)

Что не хватает:
1. **Множественная загрузка** — можно загрузить только 1 файл за раз
2. **Drag-and-drop загрузка** — нет зоны перетаскивания
3. **Прогресс-бар загрузки** — только текст «Загрузка...»
4. **Камера / capture с мобильного** — `<input type="file">` не использует `accept` и `capture`
5. **Grid-режим просмотра** — только таблица, неудобная на мобильном
6. **Полноэкранный просмотр** — превью маленькие (max-h-16), нет lightbox
7. **Копирование URL** — нет кнопки «скопировать ссылку»
8. **Пагинация** — грузит всё одним запросом
9. **Rename / метаданные** — нельзя переименовать файл, нет alt-текста
10. **Модальный picker** — inline-разворачивание ломает UX формы

### 3.2. Целевое состояние медиабиблиотеки

**Мобильный (основной сценарий врача):**
- Карточки/плитки вместо таблицы, с крупными превью
- Кнопка «+» внизу экрана для загрузки
- При нажатии — выбор: «Камера», «Фото и видео», «Файлы»
- Загрузка нескольких файлов одновременно с прогресс-индикаторами
- Swipe-действия на карточках (удалить, скопировать URL)

**Десктоп:**
- Два режима: плитки (grid) и таблица (list) — переключатель
- Drag-and-drop зона загрузки по всей странице
- Множественная загрузка с прогрессом
- Lightbox-просмотр по клику на превью
- Кнопка «Скопировать URL» в каждой строке/карточке
- Модальный picker при вставке в контент-форму (Dialog/Sheet, а не inline)

## 5. Нелогичные решения

### 5.1. Разделение Клиенты / Подписчики

Два почти идентичных экрана с одной и той же `DoctorClientsPanel` и разным источником данных. Врачу непонятна разница. Один список + фильтр — проще и понятнее.

### 5.2. Двойная сортировка контента (drag-and-drop + числовое поле)

Drag-and-drop в списке и `sort_order` input в форме — два конкурирующих механизма. Сохранение из формы перезаписывает порядок, заданный через drag. Оставить только drag-and-drop, убрать ручное поле.

### 5.3. Текстовая заглушка рассылок с вызовом бэкенда

`/app/doctor/broadcasts` вызывает `getCategories()` и `listAudit(5)`, но показывает только «Массовые рассылки во втором релизе». Бесполезный вызов API на каждый визит страницы.

### 5.4. Legacy-redirect'ы без 301

`/app/patient/lessons` и `/app/patient/emergency` делают `redirect()` (302) на новые URL. Для SEO и кеширования лучше 301, но важнее — убрать старые маршруты из навигации, если ссылки на них нигде не видны пользователю.

### 5.5. Iframe-адрес без обработки ошибок

`/app/patient/address` — iframe на внешний домен без onError, без loading-спиннера, без fallback-контента.

## 9. Рекомендации по архитектуре UI

1. **Убрать PlaceholderPage как паттерн.** Если экран не готов — его не должно быть в навигации. Компонент PlaceholderPage поощряет создание пустых маршрутов.

2. **Адаптивные таблицы → карточки на мобильном.** Все таблицы в проекте (медиабиблиотека, журнал сообщений, списки записей) используют `min-w-[Nrem]` и горизонтальный скролл. На мобильном лучше переключаться на карточный layout.

3. **Использовать Dialog/Sheet из shadcn/ui для picker'ов.** В проекте уже есть эти компоненты. `MediaLibraryPickerDialog` должен быть полноценным Sheet на мобильном и Dialog на десктопе.

4. **Один механизм сортировки контента.** Drag-and-drop через dnd-kit уже реализован и работает. Числовое поле `sort_order` в форме — атавизм.

5. **Один список пользователей с фильтрами.** Разделение на «клиентов» и «подписчиков» — архитектурное, не пользовательское. Для врача с десятками пациентов нужен один список.
