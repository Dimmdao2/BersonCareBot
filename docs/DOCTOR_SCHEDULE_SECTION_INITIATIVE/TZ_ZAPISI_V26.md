## 2. Целевое поведение (свод из вайрфрейма + пояснений)

> Полные формулировки — в «Пояснениях по поведению» и `#p-schedule`. Здесь — то, что определяет код.

1. **Две независимые оси.** Вкладки раздела (Записи · График работы · Настройки) и переключатель
   периода внутри «Записи» (3 дня · Неделя · Месяц · Лента) не пересекаются. Состояние периода
   сохраняется при уходе на work/setup и возврате.
2. **Период — единый источник истины.** Стрелки ◀▶, лейбл периода и KPI-ряд привязаны к **одному**
   видимому диапазону. Отдельных селекторов периода для метрик нет. Листание назад = история (отдельной
   «Истории» нет).
3. **Фильтры Локация/Услуга** применяются и к календарю, и к KPI одновременно.
4. **KPI-ряд (9 карточек)** — строго по видимому периоду и активным фильтрам; нули показываем как нули;
   «Прошло/Впереди» — относительно текущего момента; каждая карточка кликабельна (`cursor:pointer`,
   поведение фильтрации — следующая итерация). Неявок в ряду нет.
5. **Режимы:** 3 дня (дефолт, часовая сетка сегодня+2), Неделя (часовая сетка пн–вс), Месяц (плашка=строка,
   фамилия; сегодня — жёлтая подсветка), Лента (бесконечный поток, **KPI и стрелки скрыты**).
6. **День — drill-down,** не режим: открывается кликом по заголовку дня (3 дня/Неделя) или числу (Месяц);
   внутри «← Назад» в исходный режим; KPI скрыт; стрелки листают по дням.
7. **Правая панель:** выбрана запись → карточка; ничего не выбрано → заглушка «Запись не выбрана» + CTA
   «+ Создать запись» + подсказка ближайшего свободного окна сегодня.
8. **Дублирование «+ Создать запись»** (тулбар + заглушка) — намеренное.

## 3. Архитектурное решение

### 3.1 KPI переезжает из шелла в таб «Записи»
Причина: KPI должны (а) считаться по видимому диапазону календаря, (б) реагировать на Локацию/Услугу,
(в) скрываться в Ленте и Дне. Всё это — состояние таба «Записи», а не шелла.

- `DoctorScheduleShell` **теряет**: `KpiRow`, `loadKpis`, состояние `period`, URL-параметр `?period`,
  пропсы `initialKpis`/`initialPeriod`. Шелл остаётся: `DoctorAppShell` + `ScheduleTabsNav` + реестр
  табов + URL-sync `?tab` и под-параметров.
- Таб «Записи` становится владельцем: режим (`view`), якорь периода (`date`), фильтры
  (`location`/`service`), выбранная запись (`appt`), drill-down-источник (`from`). По смене
  диапазона/фильтров — **параллельно** тянет фид календаря и KPI (оба по одному диапазону+фильтрам).
- SSR: `page.tsx` предзагружает KPI и фид для **дефолтного периода «3 дня»** и передаёт в таб как
  `initialData` (без мигания). Клиентские смены — через API-роуты.

### 3.2 Диапазоны видов (единый расчёт `visibleRange(view, anchor, tz)`)
| view | rangeStart | rangeEnd | KPI-диапазон |
|------|-----------|----------|--------------|
| `3days` | сегодня (или anchor) | +2 дня | = фид |
| `weekgrid` | пн недели anchor | вс недели | = фид |
| `month` | 1-е число месяца anchor | последнее число | = календарный месяц (серые дни соседних месяцев в KPI **не** входят) |
| `feed` | anchor − N дней | anchor + N дней (ленивое расширение) | — (KPI скрыт) |
| `day` (drill-down) | конкретный день | тот же день | — (KPI скрыт) |

Один и тот же диапазон используется и для фида, и для KPI — буквальная реализация «единого источника».

## 4. Бэкенд-дельта

### 4.1 KPI: 9 метрик + фильтры + произвольный диапазон

**4.1.1 Тип результата** (`modules/doctor-appointments/ports.ts`, расширить `ScheduleKpis`):

```ts
export type ScheduleKpis = {
  recordsInPeriod: number;       // Записей: неотменённые, start_at в окне
  pastInPeriod: number;          // Прошло: из records, start_at < now()
  futureInPeriod: number;        // Впереди: из records, start_at >= now()
  bySubscriptionInPeriod: number;// По абонементу: из records, package_usage_ref IS NOT NULL
  firstVisitInPeriod: number;    // Первичных: записи, у пациента нет более ранней неотменённой записи
  repeatVisitInPeriod: number;   // Повторных: records − firstVisit
  uniquePatientsInPeriod: number;// Уникальных: COUNT(DISTINCT platformUserId)
  cancellationsInPeriod: number; // Отмены (см. §11 — решение по семантике)
  reschedulesInPeriod: number;   // Переносы (см. §11)
};
```
Инвариант: `pastInPeriod + futureInPeriod === recordsInPeriod`;
`firstVisitInPeriod + repeatVisitInPeriod === recordsInPeriod`. Сверка вайрфрейма: 2+7=9, 1+8=9. ✓
Существующее поле `newPatientsInPeriod` (уникальные новые пациенты) в ряду v26 не используется —
снять из `ScheduleKpis` (UI-карточки «Новых» больше нет), либо оставить как internal только если
переиспользуется в другом месте (проверить `rg newPatientsInPeriod` перед удалением).

**4.1.2 Входной запрос** — новый тип (не загрязнять общий `DoctorAppointmentStatsFilter`):
```ts
export type ScheduleKpisQuery = {
  from: string;            // ISO, начало окна (бизнес-таймзона)
  to: string;              // ISO, конец окна (эксклюзивно)
  branchId?: string | null;
  serviceId?: string | null;
};
```
Сигнатура порта: `getScheduleKpis(query: ScheduleKpisQuery, audience?: DoctorAppointmentsAudience)`.

**4.1.3 pg-реализация** (`infra/repos/pgDoctorAppointments.ts`) — **Drizzle**, переиспользовать
`resolveAppointmentStatsBounds`-стиль окна и существующие condition-хелперы. Все 9 метрик одним
запросом или серией параллельных (как уже сделано в `getDashboardAppointmentMetrics`). Фильтры
`branchId`/`serviceId` добавляются в `WHERE` для всех метрик-по-записям; для Отмен/Переносов —
join к `appointment_records` для согласованного применения фильтров. `__none__`/null-семантика
фильтров — как в существующих роутах.

**4.1.4 inMemory-реализация** (`inMemoryDoctorAppointments.ts`) — паритет: те же 9 чисел на тестовом
наборе; read-switch (`doctorAppointmentsReadSwitch.ts`) обновить.

**4.1.5 Роут** `api/doctor/schedule-kpis/route.ts` — GET с Zod-схемой
`{ from, to, branchId?, serviceId? }` (вместо `?period=`); `requireDoctorAccess`; через
`deps.doctorAppointments.getScheduleKpis`; вернуть `{ ok, kpis }`. Лог ошибки через `logger.error`.

### 4.2 Календарный фид: произвольные диапазоны + часы ±1ч
- Фид `getCalendar` уже принимает `rangeStart`/`rangeEnd` (`CalendarFilters`) — переиспользовать для
  `3days` (today..+2), `day` (один день), `feed` (окно ±N с расширением). **Новых контрактов не плодить.**
- **Часы сетки** для часовых видов (`3days`/`weekgrid`/`day`): вычислять `slotMinTime`/`slotMaxTime` из
  рабочих интервалов видимых дней (события `kind:"working"` уже в фиде) **минус/плюс 1 час**; fallback —
  текущий дефолт. Реализовать на клиенте из событий фида (без нового бэкенда), либо отдавать
  `workingBounds` в ответе фида, если так чище (решить при реализации, зафиксировать в LOG).
- `feed` (Лента): сервер отдаёт записи за запрошенное окно; **бесконечный скролл** — клиент
  доращивает окно (`from`/`to`) в обе стороны и склеивает; пустые дни отбрасываются на клиенте; границ
  недель нет.

### 4.3 «Ближайшее свободное окно сегодня» (для заглушки правой панели)
- Источник: слот-движок `modules/booking-scheduling` (есть `computeSlots`). Свободные окна в текущем
  фиде **не** отдаются (`freeSlot` объявлен в типах, но не заполняется), поэтому нужен отдельный read:
  по Графику работы на сегодня минус существующие записи → ближайший свободный интервал.
- Реализация: метод порта `booking-scheduling` (напр. `nearestFreeWindow(dateKey, scope)`), pg/inMemory,
  тонкий роут `api/doctor/schedule/nearest-free-window` (или включить в ответ фида — решить при
  реализации). Возврат: `{ from, to } | null`. Лог на сервисе.
- Если решения по этому read нет к моменту реализации фронта — заглушка показывает CTA без подсказки
  окна (graceful), метрика не блокирует ребилд.

## 5. Фронтенд-дельта (таб «Записи», `ScheduleCalendarTab`)

Эталон примитивов — `shared/ui/doctor/**` (Button/Badge/DoctorSection/DoctorEmptyState),

### 5.1 Тулбар (`wf` строка вайрфрейма)
- Переключатель периода: `3 дня · Неделя · Месяц · Лента` (role=group). **Без «День».**
- Лейбл периода `◀ 10–12 июня ▶`: стрелки + лейбл — рядом, привязаны к видимому диапазону;
  в `feed` — **скрыты**.
- Фильтры `Локация ▾ · Услуга ▾` (`DoctorCalendarToolbarFilter`).
- Кнопка `+ Создать запись` (primary, справа) — постоянная (открывает диалог создания, как в текущем
  `DoctorCalendarEventPanel`-флоу).

### 5.2 KPI-ряд (9 карточек) — внутри таба, под тулбаром
- Сетка `repeat(9,1fr)` (адаптив на узких — перенос), классы из `doctorVisual.ts`.
- Карточки: Записей · Прошло · Впереди · По абонементу · Первичных · Повторных · Уникальных · Отмены ·
  Переносы. `data-testid="kpi-<key>"`, `cursor-pointer` (клик-фильтрация — следующая итерация, обработчик
  пока no-op/заглушка).
- **Скрыт** в `feed` и `day`. Нули рендерим как `0`, не как `—` (после первой загрузки).

### 5.3 Виды
- **3 дня (`3days`, дефолт):** часовая сетка, 3 колонки (сегодня+2). Клик по заголовку колонки →
  drill-down `day`. Часы — ±1ч (§4.2). Красная линия «сейчас».
- **Неделя (`weekgrid`):** та же часовая сетка, пн–вс. Клик по заголовку дня → drill-down `day`.
- **Месяц (`month`):** месячная сетка; плашка записи = **одна строка, только фамилия**;
  сегодня — мягкая жёлтая подсветка (`#fff8e6`); `+N` при переполнении; клик по числу → drill-down `day`.
- **Лента (`feed`):** вертикальный поток карточек-дней (дата + N записей + локация), бесконечный скролл
  в обе стороны, пустые дни пропущены, без границ недель. KPI/стрелки скрыты.
- **День (`day`, drill-down):** дневная часовая сетка; «← Назад» возвращает в исходный режим
  (запомнить через `from`); стрелки листают по дням; KPI скрыт.

> Часовые виды (3 дня/Неделя/День) — один компонент сетки с параметром числа колонок (переиспользовать
> FullCalendar timeGrid там, где он уже подключён, либо общий кастомный grid — выбрать минимальный по коду
> путь, зафиксировать в LOG). Месяц — FullCalendar `dayGridMonth` с кастомным `eventContent` (фамилия),
> `dayCellClassNames` (сегодня), `dateClick` (drill-down), либо кастомная сетка как в `cv-month`.

### 5.4 Правая панель
- Запись выбрана → `DoctorCalendarEventPanel` (карточка записи) — как сейчас.
- Ничего не выбрано → **заглушка**: иконка, «Запись не выбрана», подсказка «Кликни запись… или создай
  новую», CTA `+ Создать запись`, строка «Ближайшее окно сегодня: HH:MM–HH:MM» (из §4.3; скрыть строку,
  если окна нет/недоступно).

### 5.5 Deep-link / URL-sync
Реестр (`scheduleTabRegistry.ts`) для `cal`: ключи `view`, `date`, `location`, `service`, `appt`, **+`from`**
(источник drill-down). Значения `view`: `3days|weekgrid|month|feed|day`. Шелл синхронизирует их в URL
через `history.replaceState` (как сейчас); `?period` удалить.

### 5.6 Левое меню: пункт «Расписание» — аккордеон → одна ссылка
Сейчас (`doctorNavLinks.ts`, id `schedule`) пункт — **устаревший аккордеон** с под-пунктами

## 11. Решения по умолчанию (подтвердить/переопределить)
1. **Семантика «Отмены»/«Переносы» в KPI-ряду.** Рекомендация: привязка к **видимому периоду по дате
   визита** (`start_at` в окне; для Отмен — записи со статусом отмены; для Переносов — `rescheduleCount>0`),
   чтобы все 9 метрик считались «строго по видимому периоду» единообразно. Это **отличается** от текущей
   реализации (`be_appointment_cancellations/reschedules` по `created_at` действия). Если нужна семантика
   «действий за период» — скажите, поменяем определение.
2. **KPI-диапазон для «Месяца»** — календарный месяц (1-е…последнее), серые overflow-дни сетки **не**
   входят. Альтернатива — весь видимый 42-дневный грид; по умолчанию берём календарный месяц.
3. **`feed` ширина окна** N (старт ±N дней) и шаг догрузки — подобрать при реализации (старт: ±30 дней,
   шаг 30); зафиксировать в LOG.
4. **«Ближайшее окно»** — отдельный роут vs поле в ответе фида; выбрать минимальный по связности путь
   при реализации.
