# Master plan — TEST_AND_API_DI_OPTIMIZATION

**Статус:** активная инициатива (треки A/B). Discovery и аудиты закрытого цикла 2026-04 лежат в **`docs/archive/2026-04-docs-cleanup/test-api-di-optimization/`**. **Дата привязки:** 2026-04-16.

## Общая цель

- Уменьшить стоимость прогона и сопровождения тестов (**трек A**), не снижая критичное поведенческое покрытие.
- Сузить и формализовать импорты в **`apps/webapp/src/app/api/**/route.ts`**: бизнес-логика и низкоуровневый wiring — через слой композиции и модули (**трек B**).

## Независимость треков (обязательное разделение)

| Трек | Меняет прод-код? | Меняет тесты? | Типичный эффект на метрики |
|------|------------------|---------------|----------------------------|
| A — test optimization | Нет (в идеале только тесты/скрипты) | Да | Wall time `pnpm test:webapp` / состав suite |
| B — API DI boundary | Да (`route.ts`, app-layer, модули) | Обычно да (параллельно route tests) | Не использовать как объяснение выигрыша трека A |

**Почему нельзя один «слитный» рефактор:** смешение ломает причинно-следственный разбор (регрессия vs случайный флейк), усложняет откат и нарушает требование «не уменьшать critical behavioral coverage» без явного mapping.

## Порядок фаз (зачем именно так)

Сначала **трек A (тесты)**, затем **трек B (DI / import-boundary в API)**.

- **A не трогает прод-код:** выигрыш по времени и упрощение suite измеряются отдельно; откат проще.
- **B меняет маршруты и композицию:** после A на руках стабильный контур тестов и зафиксированный baseline; меньше шансов спутать регрессию рефакторинга с эффектом удаления дубликатов.
- **Между A и B** — явный checkpoint (см. ниже): не смешивать незакоммиченные правки тестов и DI в одном PR.

## Рекомендуемый порядок работ (детализация)

1. **Discovery / baseline** — команды, окружение, цифры (см. оба `BASELINE.md`). *На момент создания инициативы: замеры `pnpm test` / `pnpm test:webapp` есть; полный `pnpm run ci` для baseline — по желанию исполнителя (дорого), но **один раз** перед первым **пушем** после существенных изменений обязателен — см. `.cursor/rules/pre-push-ci.mdc`.*
2. **Трек A — test optimization** — инвентарь, решения по дубликатам, малые коммиты. **Проверки между коммитами** — по `.cursor/rules/test-execution-policy.md`: после точечных правок **step** (таргетированный Vitest и т.п.), после закрытого пакета работ по webapp — **phase** (`pnpm test:webapp` при необходимости); **не** гонять полный `pnpm run ci` после каждого микрошага.
3. **Checkpoint между A и B** — ветка/набор коммитов только по A; в `test-optimization/LOG.md` зафиксированы итоговые метрики (если измерялись); смешанных с B изменений нет. Сравнение с `.github/workflows/ci.yml` нужно **только если** кто-то менял workflow (в рамках инициативы workflow **не меняем** — см. раздел «GitHub CI и деплой»).
4. **Трек B — API DI-boundary normalization** — кластеры маршрутов, parity по контрактам, малые коммиты; между коммитами те же уровни step / phase.
5. **Final verification** — один полный `pnpm install --frozen-lockfile && pnpm run ci` перед **пушем** финального состояния (или по запросу «как в CI»).
6. **Docs sync** — обновить перечисленные ниже документы так, чтобы они совпадали с кодом, а не с абстракцией.

## GitHub CI и деплой (явный запрет на «починку» пайплайна)

- **Не изменять** `.github/workflows/ci.yml`, не перестраивать jobs **Deploy** и связанный с ним поток без отдельного решения команды: текущий GitHub CI → deploy считается **уже выверенным**.
- Инициатива **не** ставит задачу править инфраструктуру релиза; любые изменения workflow в PR с оптимизацией тестов / DI — **вне scope** и создают шум на review.

## Критерии входа / выхода

### Трек A

- **Вход:** заполнены `test-optimization/INVENTORY.md` и `BASELINE.md` (команды и хотя бы одно измерение на том же агенте/машине).
- **Выход:** нет удалённых тестов без строки в mapping «old → replacement»; контракты из `test-optimization/PLAN.md` остаются покрытыми; перед **пушем** зелёный полный CI (включает оба пакета тестов); отчётность в `LOG.md`. Между коммитами достаточно step/phase по test-execution-policy.

### Трек B

- **Вход:** завершён checkpoint после A; заполнены `api-di-boundary-normalization/INVENTORY.md` и `BASELINE.md`; политика импортов согласована с `EXECUTION_RULES.md`.
- **Выход:** для каждого изменённого endpoint задокументирован parity (статусы/ключи JSON/идемпотентность/подпись); исключения из политики — явный список; перед **пушем** — `pnpm run ci` зелёный; workflow GitHub не меняли.

## Обязательные checkpoints

| После | Проверка |
|-------|----------|
| Конец трека A | Сравнить длительность `pnpm test:webapp` и число files/tests (Vitest summary) с baseline **когда цель — оценка выигрыша**; убедиться, что skipped tests не «спрятали» обязательный сценарий. Полный `pnpm run ci` — перед **пушем**, не обязателен после каждого локального коммита. |
| Перед стартом B | Закрыт трек A (отдельные коммиты/ветка); перед **пушем** в remote — зелёный полный CI; нет незакоммиченных тестовых изменений вместе с DI. |
| Конец B | Повторный grep `@/infra` в `route.ts`; список только согласованных исключений; обновление `api.md` / `di.md` / архитектурных ссылок. |

## Маленькие reviewable commits

- Один логический шаг: например «удалён e2e X, замена route.test Y» **или** «cluster integrator GET: вынесен verify в сервис».
- После каждого шага: проверки по **test-execution-policy** (step по умолчанию; phase — когда закрыт кусок работы в приложении); **не** полный монорепо CI. Перед **пушем** в remote — **`pnpm install --frozen-lockfile && pnpm run ci`** (`.cursor/rules/pre-push-ci.mdc`).
- **Repo-scope (не путать с микрошагом):** затронуты shared-пакеты, lockfile, корневые конфиги тулчейна или несколько приложений — полный `pnpm run ci` уместен по `.cursor/rules/test-execution-policy.md` даже без немедленного пуша; это **не** отменяет запрет гонять полный CI после каждой точечной правки только в `apps/*`.
