# Промпты для агента (копипаста)

Используй по порядку сверху вниз. Внутри блока — целиком в чат агенту. Не нумеруй этапы вручную и не подставляй в промпт списки путей: контекст — инициатива в дереве репозитория и правила в каталоге правил редактора.

**Обязательные правила репозитория**

- Между коммитами: уровни проверок **step** → **phase** → полный CI — см. `test-execution-policy` в каталоге правил. Не гоняй полный набор тестов монорепы и не вызывай `pnpm run ci` после каждого микрошага «на всякий случай».
- Аудит: первый шаг — разбор диффа и scope, **не** немедленный прогон всех тестов — см. тот же файл, раздел Audit validation.
- **Пуш в remote:** только после успешного `pnpm install --frozen-lockfile && pnpm run ci` — см. `pre-push-ci` в каталоге правил. Повторно тот же `ci` без новых изменений кода не запускать.
- **GitHub CI и деплой:** workflow, job Deploy и связанные шаги **не менять** в рамках этой инициативы — пайплайн считается корректным. После merge в принятую ветку деплой идёт по текущему процессу команды; отдельной ручной задачи «задеплоить» в промптах нет.

**Где только коммит, где пуш**

- После каждого логически завершённого куска работы: **коммит** (локально).
- **Пуш** (и тем самым участие в обычном post-merge deploy, если это принято у команды): после закрытия всего объёма оптимизации тестов без смешивания с рефакторингом API; и снова после полного закрытия нормализации import-boundary и документации — каждый раз с полным CI перед пушем. Между этими точками — накопление коммитов без пуша допустимо, если так удобнее review.

Прочитай эти правила и кратко выпиши, какие уровни проверок применишь к будущим изменениям (step / phase / full CI). Пайплайн GitHub CI и деплой не меняй. Код и тесты пока не трогай; если найдёшь противоречие между документами инициативы и правилами — предложи минимальную правку только внутри папки инициативы. Затем сделай аудит без запуска тестов: сверь, что в execution rules явно зафиксирован запрет правок CI workflow.
