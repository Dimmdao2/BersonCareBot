# СХЕМА АВТО-ПРОХОДА АГЕНТАМИ · Agent Auto-Run Scheme  (v2, self-audited 2026-06-17)

> **Канонический метод оркестрации** разработки агентами BersonCare. Определён владельцем 2026-06-17.
> Заменяет ad-hoc описания (`/home/dev/orch/PIPELINE.md`, `ORCHESTRATION_SCHEME_V2.md`) — они становятся подчинёнными раннтайм-деталями.
> **Читать перед любой оркестрованной работой.** Все роль-промпты обязаны ссылаться на этот файл.

## 0. Принципы
- **Корректность + полнота > скорость > параллельность.** Но **строгость масштабируется по размеру задачи**.
- **Атомарный шаг** — дробим до объёма, который:
  - (фронтенд) доводит до **подтверждения скриншотом**;
  - (не-фронтенд / на будущее) агент **безопасно делает за один проход**; подтверждение = `curl`-воспроизведение или повторная детальная проверка в коде.
- ⛔ **Никогда git `main`.** Интеграционное (общее) дерево = **`feat`** — там dev-сервер `:5200`; туда вливаем после зелёного CI; там же финальное ревью. (Это НЕ `main`.)

## 1. Иерархия единиц работы (определяет, ЧТО и ГДЕ проверяется)
| Единица | Что это | Где проверка |
|---------|---------|--------------|
| **Атомарный шаг** | наименьшая единица исполнения (§0) | внутри этапа |
| **Этап (stage)** | набор атомарных шагов = один «цельный проход» | **проверка в КОДЕ, ДВАЖДЫ, ДО мержа** (аудит→фикс→аудит+рефикс) §4A |
| **Фаза (phase)** | _необязательная_ группировка этапов (крупная инициатива — с фазами; мелкая — только этапы) | — |
| **Фича (feature)** | единица **РЕВЬЮ и мержа**: один-несколько готовых этапов, доведённых «до первого ревью» | **фичевое ревью + мерж в feat + рендер/визуал + ДВА ревьюера** §4B |
| **Инициатива** | вся задача владельца; папка `docs/<NAME>_INITIATIVE/` | уровень планирования §3 |

**Два уровня строгости:** атомарный фикс (вне инициативы) → лёгкий цикл (исполнитель → 1 аудит → рендер/`curl` → мерж). Инициатива → полный пайплайн (§3 планирование + §4 цикл).

## 2. Роли и модели «по плечу»
| Роль | Модель | Что делает |
|------|--------|-----------|
| **Planner** | Opus | анализ задачи, дорожная карта, финальная сверка с интентом |
| **Executor** | Sonnet-high | пишет КОД + тесты по своему коду + heartbeat; своих серверов не поднимает |
| **Code-auditor** | Sonnet-high | первый независимый код-ревью всех пунктов |
| **Fixer** | Sonnet-high | фикс по аудиту, долги, покрытие тестами |
| **Opus-auditor** | Opus (**глубина мышления — по сложности задачи**) | второй независимый аудит после фиксов |
| **Visual / 2nd reviewer** | Sonnet-high / Opus | рендер-подтверждение на :5200 + контрольный ревьюер |
| **Orchestrator** | фоновый луп | спавн, параллельность, сериализация мержей, ведёт цикл, **пер-агентная liveness по тейлам логов** |
| **Governor** | скрипт (cron) | агрегат: нагрузка / число агентов / жив ли оркестратор |

Opus — только на гейтах высокого плеча (планирование, второй аудит, фичевые ревью). Остальное — Sonnet-high.

## 3. Фаза планирования (инициатива) — шаги 1–5
**1) Сильный агент (Opus):**
- 1.1 Первичный анализ и структурирование задачи владельца; **подтверждение актуальности задачи в коде**; сбор данных. **Фиксация требований владельца ДОСЛОВНО в `REQUIREMENTS.md`** (не только в памяти/переписке).
- 1.2 Главная дорожная карта `ROADMAP.md` + первичная декомпозиция на этапы (под исполнителя).

**2) Исполнитель (Sonnet-high):**
- 2.1 Инфраструктура инициативы (если не короткий план): `docs/<NAME>_INITIATIVE/` с **фазами и этапами** (или только этапами) + шаблоны `audit/` и `log.md`.
- 2.2 Детальная декомпозиция этапов — **отдельной задачей по каждому этапу**.
- 2.3 Анализ декомпозиции: пробелы/слабые места; **список нерешённых до старта вопросов** (закрыть до кода); оценка объёма по этапам **для своего уровня**.

**3) Новый сильный агент (Opus):** читает замечания+вопросы → идёт в код → отвечает на ВСЕ вопросы в планах → проверяет и **усиливает** планы, углубляет декомпозицию. Несколько фаз/этапов → **отдельной задачей по каждому**, не одним проходом.

**4) Финальная сверка (Opus):** читает главный план + все этапы → закрывает ли полный план **первичную инструкцию владельца** (все боли/задачи). План может **улучшить** задачу, но **не обрезать / не упростить «для удобства» без согласия владельца**.

**5) Фиксация планов + проверка качества и полноты инструкций** против критериев §6. План ОБЯЗАН содержать: детальные чек-листы, ведение лога, указание «прочитать правила».

## 4. Фаза исполнения

### 4A. Цикл на ЭТАП — проверка в КОДЕ, ДВАЖДЫ, ДО мержа (шаги 6–11)
> Этапы можно гнать **параллельно** при непересекающихся файловых скопах (§5). Цикл повторяется по каждому этапу фичи.
- **6. Executor** по плану этапа.
- **7. Auditor** — независимый код-ревью ВСЕХ пунктов: ошибки **и пропущенное** (правомерно ли отложено). Отчёт в `audit/`.
- **8. Fixer** — фикс ошибок + закрытие долгов + проверка пропущенных тестов и покрытия.
- **9. Opus-auditor** — независимый аудит после фиксов (**глубина мышления по сложности**) → `code-audit-log`.
- **10. Opus audit-fix** — сравнение + суммаризация в audit-файл; правки + переаудит **до чистого прохода**. Тесты — все затронутые файлы.
  > **«Проверка до мержа» = в КОДЕ, ДВАЖДЫ** (7→8, затем 9→10 + рефикс если есть остатки). Рендера/мержа здесь ещё нет.
- **11. Commit + push в СВОЁ дерево** (бэкап).

### 4B. Ревью + мерж на уровне ФИЧИ (шаги 12–14) — когда хвосты в коде закрыты
- **12. Opus-ревью всей фичи. ПОЛНЫЙ CI в своей ветке.** После зелёного — **аккуратный мерж в `feat`** (общее интеграционное дерево, где :5200; ⛔ НЕ `main`).
- **13. Финальное ревью на `feat` (:5200):**
  - **Нужно UI-подтверждение** → **visual-reviewer**: заходит на страницу, проверяет поведение, скрины **ВСЕХ шагов** → `.claude/screenshots/<понятное-имя>/`, сверяет с планом; расхождение → отчёт.
  - **Не нужно (код-фича)** → `curl`-воспроизведение ИЛИ детальная повторная проверка в коде.
  - **Второй ревьюер для контроля** — прошёл ли первый нужные пути / не пропустил ли.
- **14.** Замечания → назад в ветку → фикс → переаудит → commit/merge/перепроверка **до согласия ДВУХ глобал-ревьюеров (одинаковый вердикт)** → принять фичу. **Зафиксировать в доках.**

## 5. Параллельность и ОДИН сервер
- Параллелим **исполнителей** по **непересекающимся файловым скопам** (этапы/инициативы).
- **ОДИН Next-сервер `:5200`** (на `feat`) для визуал-ревью. **Воркеры своих серверов НЕ поднимают** (иначе перегруз — словили load 43 на 3 серверах). Визуал-ревью и тяжёлые Opus-аудиты — **сериализуем**.
- Лимиты: **≤3 параллельных воркера / ≤5 агентов всего**.

## 6. Always-критерии (в КАЖДОМ плане и роль-промпте; канон — `.cursor/rules/*` + `AGENTS.md`)
- ⛔ **Нет сырому SQL — только drizzle.**
- **Предельная чистота архитектуры + изоляция слоёв; импорты/экспорты модулей только через DI** (нет прямых связей).
- **Нет дублированию при возможности переиспользования:** запрет на кастомные UI/бэкенд-компоненты, если есть shared-аналог / кандидат в shared / удобнее опенсорс (React, shadcn и т.п.).
- **Нет серверному поиску, если можно сначала поиск в UI** (всё готовое есть).
- **Нет избыточной fetch-подгрузке, если можно быстро собрать SSR.**
- **Шаблоны — смысловыми блоками** (каждый таб = свой блок) с импортом в общий контейнер.
- **Общий верхний шаблон страниц** (эталон: «Сегодня», «Расписание») — заголовок + место под табы/инфо-бейджи.
- **Тесты — только по своему коду; полный CI — только контролёром перед пушем; метка теста в общем пространстве** (flock-мьютекс) для блока параллельности.
- **Опциональное — оценить пользу+выполнимость в ЭТОМ проходе и делать сразу.** Откладывать только то, что нельзя или противоречит пользе/более важным решениям. Если делать сразу ломает атомарность/безопасный проход — вынести **отдельным атомарным шагом** (но всё равно сделать).
- **Ведение лога + синхронизация документации.**

## 7. Оркестрация и governance (runtime)
- **Единый launcher** `dev-orchestra start|pause|status` — поднимает оркестратор + крон-скрипты + **единый PAUSE** (стоп всему; **авто-старт при возобновлении работы**; хук «пауза от агента выше репозитория»).
- **Разделение liveness (важно):**
  - **Governor (скрипт, cron)** — только АГРЕГАТ: снимок CPU/RAM, число агентов, перезапуск раздутого `:5200`, **жив ли оркестратор**. Дёргает оркестратор только если что-то не так. _Скрипт НЕ способен сказать, завис ли КОНКРЕТНЫЙ агент._
  - **Orchestrator** — ПЕР-АГЕНТНО: раз в **15 мин** читает **тейлы логов** агентов; **>20 мин без записи → агент подозрителен → проверка/реап** (это умеет только оркестратор — он знает, кто есть кто).
  - **Лог:** общий лог с пер-агентными метками (проще) — оркестратор тейлит по агенту. Агент **ОБЯЗАН писать после КАЖДОГО шага** (агенты забывают → ловит тейл).
- **Тесты** — всегда через flock-мьютекс (`run-tests.sh`), один прогон за раз.
- **Заготовки роль-промптов** в `orch/roles/` (чем больше, тем лучше) — агенту кидаем **ссылку на план + «фас, твой черёд»** (промпты не пишем заново).
- Раннтайм-детали (пути/портов) — `/home/dev/orch/` (`launch-round3.sh`, `watchdog-round3.sh`, `resource-governor.sh`, `heartbeat-monitor.sh`).

## 9. Безопасность (всегда)
- ⛔ Никогда `main`. ⛔ Миграции БД — **гейт** (флаг, не выполнять; dev test-seed — можно, идемпотентно). ⛔ Живые отправки — только авторизованные владельцем; dev-редирект держит D7 (ничего реальным клиентам).
- Single chokepoint, без дублирования. Честный статус — доказательство (скрин/тест/`curl`), а не «tsc зелёный».
- **Автономный луп требует РАЗОВОГО правила человека** в `~/.claude/settings.json`: `Bash(setsid bash /home/dev/orch/round3/launch-round3.sh:*)` — ИИ не вправе сам себя авторизовать на `--dangerously-skip-permissions`.
