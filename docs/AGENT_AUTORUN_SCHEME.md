## 0. Принципы
- **Корректность + полнота > скорость > параллельность.** Строгость масштабируется по размеру задачи.
- **Атомарный шаг** — дробим до объёма, который: (фронтенд) доводится **до подтверждения скриншотом**; (не-фронтенд) агент безопасно делает за один проход, подтверждение = `curl`-воспроизведение или повторная детальная проверка в коде.
- ⛔ **Никогда git `main`.** Интеграционное (общее) дерево = **`feat`** (там dev-сервер `:5200`). Это НЕ `main`.
- **САМООЦЕНКА ИСПОЛНИТЕЛЯ НЕ ЗАСЧИТЫВАЕТСЯ.** Ни Миньон-воркер, ни Миньон-оркестратор НЕ имеют права закрыть пункт. Закрытие — только по **ДВУМ ПЕЧАТЯМ** независимых ревизоров-доёбщиков (§4C). Корень прошлого провала: луп штамповал свои же ✅ (заявил ~21 готовых — независимо подтверждено ~8).

## 1. Иерархия единиц работы (определяет, ЧТО и ГДЕ проверяется)
| Единица | Что это | Где проверка |
|---------|---------|--------------|
| **Атомарный шаг** | наименьшая единица исполнения (§0) | внутри этапа |
| **Этап (stage)** | набор атомарных шагов = один «цельный проход» | **проверка в КОДЕ, ДВАЖДЫ, ДО мержа** (аудит→фикс→аудит+рефикс) §4A |
| **Фаза (phase)** | _необязательная_ группировка этапов | — |
| **Фича (feature)** | единица **РЕВЬЮ и мержа** | **фичевое ревью + мерж в feat + рендер/визуал + ДВА ревьюера-доёбщика + ДВЕ ПЕЧАТИ** §4B/§4C |
| **Инициатива** | вся задача владельца; папка `docs/<NAME>_INITIATIVE/` | уровень планирования §3 |

**Два уровня строгости:** атомарный фикс (вне инициативы) → лёгкий цикл (исполнитель → 1 аудит → рендер/`curl` → визуал-ревьюер-доёбщик ставит печать → мерж). Инициатива/фича → полный пайплайн (§3 планирование + §4 цикл + ДВЕ печати).

## 2. Роли и модели «по плечу»  (команда исполнения = **«МИНЬОНЫ»** — без надзора лепят брак, поэтому НЕ имеют права приёмки)
| Роль | Модель | Что делает | Право печати |
|------|--------|-----------|--------------|
| **Planner** | Opus | анализ задачи, дорожная карта, финальная сверка с интентом | — |
| **Plan-auditor** | Opus | независимый аудит плана: закрывает ли план дословные боли владельца, не обрезан ли | — |
| **Decomposer** | Opus/Sonnet-high | декомпозиция этапов (отдельной задачей по каждому) | — |
| **Orchestrator (Миньон-драйвер)** | фоновый луп | спавн, параллельность, сериализация мержей, ведёт цикл, пер-агентная liveness по тейлам; **закрывает пункт ТОЛЬКО при 2 печатях** | ⛔ нет |
| **Executor (Миньон-воркер)** | Sonnet-high | пишет КОД + тесты по своему коду + heartbeat; своих серверов НЕ поднимает | ⛔ нет |
| **Code-auditor** | Sonnet-high | 1-й независимый код-ревью всех пунктов (ошибки + пропущенное) | — |
| **Fixer** | Sonnet-high | фикс по аудиту; умеет стартовать **с точки аудит-фикса**, не с нуля | — |
| **Opus-auditor** | Opus | 2-й независимый код-аудит после фиксов | — |
| **Визуальный ревьюер #1 (доёбщик)** | Sonnet-high/Opus | рендер-подтверждение на :5200, режим ДОЕБАТЬСЯ (§4C) | ✅ **печать #1** |
| **Главный Ревизор #2 (доёбщик)** | **Opus** | контрольный независимый рендер-разбор, режим ДОЕБАТЬСЯ, держит **ГЛАВНУЮ печать** | ✅ **печать #2 (главная)** |
| **Governor** | скрипт (cron) | агрегат: нагрузка / число агентов / health :5200 / жив ли оркестратор / **ротация тиков** | — |

Opus — на гейтах высокого плеча (планирование, 2-й аудит, оба ревьюера). Качество ревизора держим **промптом + скиллом** (режим доебаться), а не вторым человеком.

## 3. Фаза планирования (инициатива) — шаги 1–5
**1) Сильный агент (Opus):** 1.1 первичный анализ + **подтверждение актуальности в коде** + сбор данных + **фиксация требований ДОСЛОВНО в `REQUIREMENTS.md`**. 1.2 `ROADMAP.md` + первичная декомпозиция на этапы.
**2) Исполнитель (Sonnet-high):** 2.1 инфра инициативы `docs/<NAME>_INITIATIVE/` (фазы/этапы + `audit/`, `log.md`). 2.2 детальная декомпозиция этапов — отдельной задачей по каждому. 2.3 анализ декомпозиции: пробелы, список нерешённых вопросов (закрыть до кода), оценка объёма.
**3) Новый сильный агент (Opus):** читает замечания+вопросы → в код → отвечает на ВСЕ вопросы → усиливает планы, углубляет декомпозицию (по этапу — отдельной задачей).
**4) ФИНАЛЬНАЯ СВЕРКА + АУДИТ ПЛАНА (Opus, независимый):** закрывает ли полный план **первичную инструкцию владельца** (все боли). План может улучшить, но **не обрезать/упростить «для удобства» без согласия владельца**.
**5) Фиксация планов + проверка качества/полноты** против §6. План ОБЯЗАН: детальные чек-листы, ведение лога, «прочитать правила».

## 4. Фаза исполнения

### 4A. Цикл на ЭТАП — проверка в КОДЕ, ДВАЖДЫ, ДО мержа (шаги 6–11)
> Этапы — параллельно при непересекающихся файловых скопах (§5).
- **6. Executor** по плану этапа.
- **7. Code-auditor** — независимый код-ревью ВСЕХ пунктов: ошибки **и пропущенное**. Отчёт в `audit/`.
- **8. Fixer** — фикс + закрытие долгов + проверка пропущенных тестов/покрытия.
- **9. Opus-auditor** — независимый аудит после фиксов → `code-audit-log`.
- **10. Opus audit-fix** — правки + переаудит **до чистого прохода**; тесты — все затронутые файлы.
- **11. Commit + push в СВОЁ дерево** (бэкап).

### 4B. Ревью + мерж на уровне ФИЧИ (шаги 12–15) — когда хвосты в коде закрыты
- **12. Opus-ревью всей фичи. ПОЛНЫЙ CI в своей ветке.** После зелёного — **аккуратный мерж в `feat`** (⛔ НЕ `main`; `git pull --rebase`, без force — `feat` общий с SaaS-чатом).
- **13. Свежий рендер на `feat` (:5200):** сервер прогрет/перезапущен на текущем `feat` (§7), `health`-чек ДО съёмки.
- **14. ДВА независимых ревьюера-доёбщика (§4C):** визуальный ревьюер #1 + Главный Ревизор #2. Каждый снимает скрины **ВСЕХ шагов/состояний** → `.claude/screenshots/<ITEM>/`, сверяет КАЖДый пункт требования, пытается СЛОМАТЬ. Каждый по итогу **ставит ПЕЧАТЬ или ОТКЛОНЯЕТ**.
- **15.** При ОТКЛОНЕНИИ хоть одним → оркестратор запускает **Fixer с точки аудит-фикса** (не с нуля, если остальное сделано и косяк локализован) → переаудит → новый рендер → снова к двум ревьюерам. **Закрытие фичи — ТОЛЬКО при ДВУХ печатях с одинаковым вердиктом ✅** (§4C). Зафиксировать в доках.

### 4C. 🔱 ПРОТОКОЛ ПЕЧАТИ (seal) — единственный сигнал приёмки
**Режим ревьюера = «ДОЕБАТЬСЯ»:**
- Вердикт по умолчанию = **ОТКЛОНИТЬ**. Презумпция «НЕ сделано».
- Открыть КАЖДЫЙ требуемый скрин; проверить КАЖДЫЙ пункт требования по отдельности; прогнать ВСЕ состояния (ховер/клик/модалка/пустое/ошибка/рефреш/диплинк/мобилка если применимо).
- **Авто-FAIL при любом:** «—»/«Загрузка»/плейсхолдер/пусто там где должны быть данные; ошибка консоли / «N Issue»; утёкший dev/spec-текст в UI; непокрытое состояние; скрин отсутствует/старый/не та страница; UI визуально не изменился против требования.
- **ПЕЧАТЬ ставится ТОЛЬКО когда доебаться НЕ ДО ЧЕГО.**

**Журнал печатей** `round3/SEAL_LEDGER.md` (append-only). Запись печати = `✅ ПЕЧАТЬ <ITEM> | reviewer=<role/agentId> | пункты:[…] | скрины:[…пути, должны существовать] | "пытался сломать: …" | UTC`.
- Печать #1 — визуальный ревьюер; печать #2 — Главный Ревизор (главная). **Две печати разными ревизорами = разрешение оркестратору закрыть.**
- **Миньоны (воркер/оркестратор) журнал НЕ пишут** — только читают. Любая печать от Миньона = нарушение, контролёр откатывает.
- **Guard `verify-seals.sh`** (cron + перед каждым закрытием): палит любой `✅`/`done`/`verified` в STATUS/QUEUE/ACCEPTANCE без ДВУХ валидных печатей в журнале на тот же ITEM, и печать без существующего скрина. Самоштамп ловится на корню.
- **Code-master** (§8) = верхний над печатями: на расхождение ревьюеров / повтор провала → нотификация владельцу (Telegram dev-redirect).

## 5. Параллельность и ОДИН сервер
- Параллелим **исполнителей** по непересекающимся файловым скопам. Лимиты: **≤3 воркера / ≤5 агентов всего**.
- **ОДИН Next-сервер `:5200`** (на `feat`) — воркеры своих серверов НЕ поднимают (ловили load 43 на 3 серверах). Визуал-ревью и тяжёлые Opus-аудиты — **сериализуем**.

## 6. Always-критерии (в КАЖДОМ плане и роль-промпте; канон — `.cursor/rules/*` + `AGENTS.md`)
- ⛔ Нет сырому SQL — только drizzle. ⛔ Предельная чистота архитектуры + изоляция слоёв; межмодульно только через DI.
- Нет дублированию при возможности переиспользования (shared-аналог / кандидат в shared / опенсорс — React, shadcn). Эталон UI — страница упражнений + `shared/ui/doctor/*`.
- Нет серверному поиску, если можно в UI. Нет избыточной fetch-подгрузке, если можно SSR. Шаблоны — смысловыми блоками (таб = блок) в общий контейнер; общий верхний шаблон («Сегодня»/«Расписание»).
- Тесты — только по своему коду; полный CI — контролёром перед пушем; всё через flock (`run-tests.sh`).
- Опциональное — оценить пользу+выполнимость В ЭТОМ проходе и делать сразу (или отдельным атомарным шагом). Ведение лога + синхронизация доков.

## 7. Оркестрация и governance (runtime) — ОПЕРАЦИОННАЯ ЗАКАЛКА (v3)
- **Единый launcher** `dev-orchestra start|pause|resume|status` — оркестратор + крон-скрипты + **единый PAUSE** (стоп всему; watchdog уважает PAUSE).
- **Liveness (разделение):**
  - **Governor (cron)** — АГРЕГАТ: снимок CPU/RAM, число агентов, **жив ли оркестратор**, **health :5200**, **ротация тиков**. Дёргает оркестратор если что-то не так.
  - **Orchestrator** — ПЕР-АГЕНТНО: раз в 15 мин тейлит логи агентов; **>20 мин без записи → подозрителен → проверка/реап**.
- **🩺 Health сервера (было: порт держит, а не отвечает):** Governor каждые ~5 мин `curl -m 5 :5200`; **таймаут/5xx → kill + чистый рестарт** dev-сервера. Не «жив ли порт», а «отвечает ли осмысленно».
- **🖥️ Жизненный цикл `:5200` (решение контролёра, v3):** держим **ОДИН ПОСТОЯННЫЙ** сервер (HMR подхватывает мержи) **+ чистый рестарт перед каждым батчем визуал-ревью И при разбухании `.next` кэша** (>~3–4 ГБ — ловили 22 ГБ-зависание). НЕ поднимаем эфемерно на каждый скрин (turbopack cold-start 30с–мин — дороже, чем рестарт-на-мерж). Свежесть рендера гарантируется рестартом+health-чеком на шаге 13.
- **🧟 Зависшие терминалы отработавших агентов:** на завершении воркера — **реап его процессов** (cwd под `.claude/worktrees/agent-*`, особенно осиротевший `next dev`) + `git worktree remove`. Governor добивает procs >30 мин stale.
- **♻️ Рестарт оркестратора (анти-перегруз контекста):** каждый проход = **свежий `claude -p`** (контекст НЕ копится между проходами). Доп.: **кап одного прохода ≤45–60 мин** (`timeout`) и/или **≤N пунктов за проход** → выход → launcher респавнит свежим. Так контекст одного прохода ограничен.
- **🌀 Ротация тиков/логов (чтобы не сдох сервак):** агенты пишут heartbeat после КАЖДОГО шага → файлы растут. Governor **ротирует**: `.r3-heartbeat` и общий лог — кап по размеру (напр. >5 МБ → срез последних N строк / `logrotate`-стиль), старые в `*.1`. Реап worktree-логов завершённых агентов.
- **Тесты** — всегда через flock-мьютекс (`run-tests.sh`).
- **Заготовки роль-промптов** в `orch/roles/ROLE_PROMPTS.md` — агенту кидаем **ссылку на план + «фас»** (промпты не пишем заново).
- Раннтайм-детали — `/home/dev/orch/round3/` (`launch-round3.sh`, `watchdog-round3.sh`, `resource-governor.sh`, `heartbeat-monitor.sh`, `verify-seals.sh`).

## 8. Code-master (верхний уровень)
- **Реализован через ПЕЧАТИ (§4C):** оркестратор не может закрыть пункт без двух печатей; над ревизорами — Code-master.
- На **BLOCKED / падение / расхождение ревьюеров / повтор провала** — **уведомление владельцу в Telegram** (стаб есть: dev-redirect → владелец). Хук-нотификация в оркестраторе.

## 9. Безопасность (всегда)
- ⛔ Никогда `main`. ⛔ Миграции БД — гейт (флаг; dev test-seed — можно, идемпотентно). ⛔ Живые отправки — только авторизованные владельцем; dev-редирект держит D7.
- Single chokepoint, без дублирования. Честный статус — доказательство (скрин/тест/`curl`), не «tsc зелёный».
- **Автономный луп требует РАЗОВОГО правила человека** в `~/.claude/settings.json` (`Bash(setsid bash …/launch-round3.sh:*)`) — ИИ не вправе авторизовать сам себя на `--dangerously-skip-permissions`. **Перезапуск/пауза/правка правил/файлов схемы — можно агенту; ЗАПУСК лупа — только человек.**

## 10. Контролёр (человеко-надзорный чат) поверх Миньонов
- Контролёр — независимый строгий гейт + разблок + правда владельцу. Владеет `QUEUE.md` пока луп на паузе. Может править схему/PIPELINE/промпты, останавливать/перенастраивать Миньонов. **Не плодит воркеров на `feat` параллельно живому лупу** (два мержера = конфликт) — либо луп, либо контролёр драйвит, не оба.
