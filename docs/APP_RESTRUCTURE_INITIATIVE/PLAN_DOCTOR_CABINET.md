## 0. Принцип порядка работ

Порядок выбран **CMS-first**: сначала навести порядок в CMS, потому что без этого редактор путается между статьями, разминками, ситуациями и уроками, а блоки главной пациента ссылаются на разделы по магическим slug. После CMS — параллельно меню/дашборд врача и каталоги назначений. Карточка пациента в текущем проходе **заморожена**, только пересобирается из имеющегося.

Последовательность шагов прохода (нумерация **как в сводной таблице §ниже**, не как «Этап N» в [`RECOMMENDATIONS_AND_ROADMAP.md`](RECOMMENDATIONS_AND_ROADMAP.md)):

1. **Этап 1 — CMS-разделение** — [`CMS_RESTRUCTURE_PLAN.md`](CMS_RESTRUCTURE_PLAN.md) (вариант C: `kind` + `system_parent_code`).
2. **Этап 2 — Меню врача** — группы по смыслу, аккордеон, «Библиотека файлов» в основном меню ([`DOCTOR_MENU_RESTRUCTURE_PLAN.md`](DOCTOR_MENU_RESTRUCTURE_PLAN.md)).
3. **Этап 3 — Бейджи** на «Онлайн-заявки» и «Сообщения».
4. **Этап 4 — Дашборд «Сегодня»** — inbox вместо метрик; метрики в `/stats`.
5. **Этап 5 — Сообщения** — единый чат ([`DOCTOR_MESSAGES_UNIFIED_CHAT_PLAN.md`](DOCTOR_MESSAGES_UNIFIED_CHAT_PLAN.md)): список тредов, `DoctorChatPanel`, модалка из карточки, автопрочтение.
6. **Этап 6 — Карточка пациента** — в текущем проходе **заморожена** (нет табов/hero): только дешёвые правки без смены IA.
7. **Этап 7 — Каталоги назначений** — «где используется» и предупреждение при архиве ([`ASSIGNMENT_CATALOG_USAGE_ARCHIVE_PLAN.md`](ASSIGNMENT_CATALOG_USAGE_ARCHIVE_PLAN.md)).
8. **Этап 8 — Плотность UI** — [`DOCTOR_UI_DENSITY_PLAN.md`](DOCTOR_UI_DENSITY_PLAN.md).

### Фокус текущего прохода

Главная цель — организовать работу врача с контентом, который видит пациент. Карточку пациента **не углубляем** новыми функциями. Курсы/программы лечения — базовое управление; глубокая переделка курсов отложена.

### Кто исполняет

Этот план — для агентов-исполнителей. Каждый шаг — отдельный проход с проверкой. Перед следующим шагом — отчёт в `LOG.md`.

## Этап 1. CMS-разделение (Вариант C)

**Статус этапа:** ✅ **Выполнено** (код, миграция, журнал, пост-аудит документов — [`LOG.md`](LOG.md)).

**Исполняется по отдельному документу:** [`CMS_RESTRUCTURE_PLAN.md`](CMS_RESTRUCTURE_PLAN.md).

**Цель:** в `content_sections` появляются `kind` (`article` | `system`) и `system_parent_code`. CMS-сайдбар получает визуальную иерархию: «Статьи» отдельно, системные папки («Ситуации», «SOS», «Разминки», «Уроки») — отдельно; **«Мотивации»** — отдельный маршрут и сущность (`motivational_quotes`), не значение `system_parent_code` у разделов. Содержимое блоков главной пациента ограничивается соответствующим `system_parent_code` (с исключениями, см. `CMS_RESTRUCTURE_PLAN.md`).

**Решения, уже принятые:**

- Тематические разделы (Антистресс, Молодая мама, Здоровье 50+, …) **не переносятся автоматически**. Все остаются `kind='article'`. Врач сам перекладывает в системные папки.
- «Полезный пост» — свободный выбор из любых опубликованных статей.
- Системные папки стартуют пустыми.
- «Уроки» остаются в CMS пока, переедут внутрь карточки курса при будущей переделке курсов.
- Бонус: создание раздела изнутри системной папки сразу проставляет `system_parent_code`.

**Шаги** — 7 шагов, расписаны в [`CMS_RESTRUCTURE_PLAN.md`](CMS_RESTRUCTURE_PLAN.md). Каждый — отдельный проход с проверкой.

**Definition of Done** — см. соответствующий блок в `CMS_RESTRUCTURE_PLAN.md`.

**Цель:** меню должно объяснять врачу, где ежедневная работа, где библиотеки назначений, где редакторские задачи, а где система. Во избежание длинного скролла — аккордеон с одним открытым блоком; состояние сохраняется в браузере.

### Целевое поведение

Меню делится на кластеры (точные названия — допиливаются по ходу):

```text
РАБОТА С ПАЦИЕНТАМИ
  Сегодня
  Пациенты
  Записи
  Онлайн-заявки
  Сообщения

НАЗНАЧЕНИЯ
  Упражнения
  Комплексы ЛФК
  Клинические тесты
  Наборы тестов

КОНТЕНТ ПРИЛОЖЕНИЯ
  Главная пациента
  CMS

БИБЛИОТЕКА ФАЙЛОВ        ← переезжает из CMS в основное меню (отдельный пункт, не в «Контент приложения»)

КОММУНИКАЦИИ
  Рассылки

СИСТЕМА
  Справочники
  Статистика
```

Аккордеон: открыт только один блок. Состояние раскрытия сохраняется в `localStorage` (per-устройство, без БД). Раскрытый по умолчанию блок при первом заходе — «Работа с пациентами».

## Этап 3. Бейджи и быстрые сигналы в меню

**Зависит от:** этап 2 (новая структура меню с явными пунктами «Онлайн-заявки» и «Сообщения»).

**Связь с этапом 4:** источники данных для бейджей и для inbox «Сегодня» общие. Если возникает выбор, что делать раньше — можно сначала сделать этап 4, тогда бейджи становятся тонким слоем поверх уже существующих query для «Сегодня».

**Цель:** врач должен видеть, что появились новые заявки или сообщения, не заходя в каждый раздел.

### Текущая проблема

`/online-intake` существует, `/messages` существует, но они не "светятся". Новая заявка или непрочитанное сообщение требуют ручной проверки.

### Целевое поведение

- У "Онлайн-заявки" есть счётчик новых заявок.
- У "Сообщения" есть счётчик непрочитанных.
- Счётчики отображаются одинаково в desktop/sidebar и mobile/sheet menu.

### Шаги

1. Проверить текущий источник unread messages (`useDoctorSupportUnreadCount`) и решить, можно ли переиспользовать его для меню.
2. Для online intake определить источник count: статус `new` или другой канонический статус из текущего модуля.
3. Выбрать один UI-pattern бейджа для всех doctor nav entries.
4. Реализовать счётчики без тяжёлых запросов на каждом render.
5. Если SSR menu не подходит для live count, сделать client-only badge layer без дублирования навигации.

### Проверки

- Tests для unread count API уже существуют: `apps/webapp/src/app/api/doctor/messages/unread-count/route.test.ts`.
- Tests для online-intake API: `apps/webapp/src/app/api/doctor/online-intake/route.test.ts`.
- Добавить/обновить тесты меню на rendering badge при mocked count.
- Manual smoke: новая online intake заявка увеличивает бейдж, после обработки бейдж меняется.

### Критерии закрытия

- Врач видит новые заявки и сообщения из меню.
- Бейджи не ломают SSR/CSR навигацию.
- Нет второго источника истины для unread messages.

## Этап 4. "Сегодня" врача вместо отчётного обзора

**Зависит от:** этап 2 (новое название «Сегодня» в меню). Независим от этапа 3, но если выполняется раньше, даёт готовые источники данных для бейджей.

Экран "Сегодня" показывает actionable inbox:

1. Записи на сегодня.
2. Новые онлайн-заявки.
3. Непрочитанные сообщения.
4. "К проверке": тесты или intake-анкеты, требующие действия врача.
5. Ближайшие приёмы на 2-3 дня.

Метрики переезжают в `/app/doctor/stats`.

### Шаги

1. Инвентаризировать текущие данные:
   - записи: `doctorAppointments`

   - tests/intake to review: определить текущий источник или явно оставить как empty-state slice, если данных пока нет.
2. Сформировать минимальную версию "Сегодня":
   - "Записи сегодня" с patient/time/format/CTA.
   - "Новые онлайн-заявки" с CTA.
   - "Непрочитанные сообщения" с CTA.
   - "Ближайшие приёмы" коротким списком.
3. Убрать с `/app/doctor` пассивные dashboard tiles или заменить ссылкой "Смотреть статистику".
4. Оставить `/stats` владельцем агрегатов.
5. Не добавлять patient event feed в этот этап.

### Проверки

## Этап 5. Сообщения: список чатов + универсальный layout + модалка

**Зависит от:** —. Можно вести параллельно с любыми этапами. Откладывается, если не уложимся.

**Исполняется по отдельному ТЗ:** [`DOCTOR_MESSAGES_UNIFIED_CHAT_PLAN.md`](DOCTOR_MESSAGES_UNIFIED_CHAT_PLAN.md).

**Статус:** закрыт 2026-05-02; post-audit и hardening fixes выполнены (phone/time в списке, unread badge в карточке пациента, `patient_not_found`/`conversation_ensure_failed` в ensure-flow, сетевые fallback в inbox/chat panel, refresh глобального doctor unread count). Детали: [`LOG.md`](LOG.md), [`DOCTOR_MESSAGES_UNIFIED_CHAT_EXECUTION_AUDIT.md`](DOCTOR_MESSAGES_UNIFIED_CHAT_EXECUTION_AUDIT.md).

**Цель:** у врача один сценарий переписки с пациентом. История одна. Из общего списка чатов и из карточки пациента открывается один и тот же layout чата.

### Целевое поведение

- Страница `/app/doctor/messages` — список всех чатов пациентов с фильтром «непрочитанные».
- Универсальный layout чата (UI-компонент, не отдельная страница) — конкретный диалог. Используется и как модалка, и потенциально как отдельная страница.
- В карточке пациента — пункт «Открыть сообщения» с бейджем «есть непрочитанные / сколько». При клике — модалка с тем же универсальным layout-ом для чата с этим пациентом.
- Из общего списка чатов клик по чату — та же модалка.
- При открытии диалога: автоскролл к первому непрочитанному. Все сообщения, попадающие в видимое поле во время прокрутки, помечаются прочитанными.
- Отдельная форма отправки в карточке пациента **удаляется** — остаётся один механизм переписки.

### Шаги

1. Проверить, что бэкенд уже умеет: список чатов, непрочитанные счётчики, mark-as-read, история сообщений. Зафиксировать в `LOG.md`, чего не хватает (вероятно немного).
2. Вытащить layout чата в отдельный компонент, переиспользуемый и страницей, и модалкой.
3. На `/app/doctor/messages` — список чатов с фильтром «непрочитанные» и открытием выбранного чата в модалке (или текущим способом).
4. В карточке пациента — кнопка «Открыть сообщения» с бейджем; клик открывает модалку с layout-ом для этого пациента.
5. Логика автопрочтения: IntersectionObserver на сообщения, mark-as-read батчем при попадании в видимое поле.
6. Удалить старую форму отправки из `ClientProfileCard`.
7. `/broadcasts` не трогать.

### Проверки

- Existing tests: `apps/webapp/src/app/api/doctor/messages/unread-count/route.test.ts`.
- `rg "SendMessageForm|Коммуникации|DoctorSupportInbox|doctorMessaging" apps/webapp/src/app/app/doctor/clients apps/webapp/src/app/app/doctor/messages`
- Тесты на: открытие модалки из списка чатов и из карточки пациента; автопрочтение при попадании в видимое поле.
- Smoke: непрочитанные пропадают из счётчика после прокрутки.

### Критерии закрытия

- У одного пациента один сценарий переписки.
- Один компонент чата работает в модалке и потенциально как страница.
- Старая форма из карточки пациента удалена.
- Бейдж непрочитанных в карточке пациента работает.
- `/broadcasts` не затронут.

## Этап 7. Каталоги назначений: "где используется" и безопасная архивация

**Зависит от:** ничего из предыдущих этапов. Можно вести параллельно с этапами 2–6. Не включать в этот этап объединение курсов и шаблонов программ — это stage 7 дорожной карты, отдельная инициатива.

**Исполняется по отдельному ТЗ:** [`ASSIGNMENT_CATALOG_USAGE_ARCHIVE_PLAN.md`](ASSIGNMENT_CATALOG_USAGE_ARCHIVE_PLAN.md).

**Цель:** врач должен понимать последствия изменения или архивации элемента каталога.

### Текущая проблема

Упражнение, комплекс, тест или рекомендация могут использоваться в шаблонах и активных назначениях, но в каталоге это не видно. Архивация выглядит безопасной даже там, где элемент нужен пациентам.

### Целевое поведение

- На карточке элемента видно "используется в N шаблонах", "назначено N пациентам" или похожая понятная формулировка.
- При архивировании показывается предупреждение, если элемент используется в активных назначениях.
- Врач понимает: это просто уборка каталога или действие, которое затронет лечение.

### Рекомендуемый порядок внутри этапа

1. Упражнения.
2. Комплексы ЛФК.
3. Клинические тесты.
4. Наборы тестов.
5. Рекомендации.
6. Шаблоны программ.
7. **Курсы** (счётчик «назначено пациентам», предупреждение при архивации опубликованного курса).

Не включать сюда courses/program templates merge — это отдельная инициатива.

### Шаги для каждого каталога

1. Найти существующий list/card/action flow.
2. Определить read-only usage counters.
3. Добавить usage summary на карточку/строку.
4. Добавить archive warning перед действием.
5. Все проверки делать в service layer / port, без изменения forbidden LFK schemas.

### Проверки

- Тесты archive action для выбранного каталога.
- Tests на usage summary для элемента с usage и без usage.
- Manual smoke: archive unused item, archive used item.
- `rg` на новые прямые infra imports в modules/routes.

### Критерии закрытия

- Врач видит последствия до изменения каталога.
- Архивация активного элемента не выглядит как безрисковая уборка.
- Нет schema changes в запрещённых LFK таблицах.
