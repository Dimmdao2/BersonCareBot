# Часть I — Рекомендации по пациентскому приложению

Цели приложения: «удобный помощник», «платформа для профилактики и реабилитации», «доступ к курсам», «выполнение назначений врача», «получение важных уведомлений».

## I.1. Сквозные проблемы IA

1. **Дублирующиеся сущности.**
   - Уведомления: аккордеон «Уведомления» в `/profile` (просто ссылка на `/notifications`) + сама страница `/notifications`. Две точки входа в одну функцию. ~~Темы рассылок на `/notifications` задавались хардкодом~~ — **исправлено (2026-05-01):** список тем и подписей хранится в `system_settings` (`notifications_topics`, scope admin), редактирование — `/app/settings` → вкладка «Параметры приложения» (блок «Темы рассылок (пациент)»).
   - Поддержка: `/messages` (чат) и `/support` (одно письмо админу) — для пациента это один сценарий «связаться с клиникой».
   - Контент: «Ситуации» на главной + страница `/sections` («Уроки и тренировки») — содержательно дублируются с разной формулировкой.

2. **Битые смысловые ссылки в `/cabinet`.** ~~Раньше~~ `CabinetInfoLinks` вела «Как подготовиться» / «Стоимость» на `/help` без контента — **исправлено (2026-05-01):** три плитки с честными подписями и каноническими маршрутами (см. [`LOG.md`](LOG.md)). Остаётся общая задача: наполнить `/help` и связать с IA отдельным этапом.

3. **Нет проактивности после редизайна Home.** Внутренние страницы регрессируют относительно нового home:
   - `/treatment-programs` — плоский список «Название + статус», без прогресса и ближайшего шага.
   - `/cabinet` — таблица «активных записей» без выделения ближайшего визита.
   - `/diary` — открывается дашбордом, а не «давай отметим занятие за сегодня».

4. **Скрытые/несвязанные сценарии.**
   - `/reminders` — мощная страница (правила, синхронизация с ботом, журналы), **в основной навигации её нет**. Заходят случайно с карточки «Следующее напоминание».
   - `/intake/nutrition` и `/intake/lfk` — **обёрнуты в `AppShell` (2026-05-01)**; видимой точки входа в продукте по-прежнему нет — попасть можно через скрытый deep-link.

5. **Лишние URL/редиректы.** `lessons`, `emergency`, `diary/symptoms`, `diary/lfk` — четыре route-файла существуют только ради `redirect()`.

## I.2. По продуктовым целям

### Выполнение назначений врача

- **`/treatment-programs` (список):** карточка показывает только заголовок и статус. Нет: прогресса этапа, ближайшего шага, даты окончания, бейджа «новые задачи». Архив (`completed`) лежит вперемешку с активными.
- **`/treatment-programs/[instanceId]` (детальная):** должна быть точкой входа в «сегодняшний шаг», а не длинной портянкой этапов.
- **`/diary` (Симптомы + ЛФК):** в пустом ЛФК-состоянии показано «Создайте комплекс упражнений» с формой `createLfkComplex`. Это пациентское создание комплекса противоречит модели «комплексы приходят из назначения врача» и смешивает источники.
- **`/reminders`:** содержательно полезна, но скрыта от навигации; не связана с `/notifications` (одну сущность можно настраивать в двух местах с разной семантикой).

### Профилактика и реабилитация (контент)

- **`/sections`:** заголовок «Уроки и тренировки» противоречит языку нового home («Ситуации»), функционально дубль ряда `situations` главной.
- **`/sections/[slug]`:** subscription callout + warmups reminder bar + grid `FeatureCard` — реализовано хорошо.
- **`/content/[slug]`:** одна из лучших страниц — hero, markdown, видео, отметка «выполнено», CTA курса. Не хватает «следующий материал», чтобы превращать чтение в привычку.

### Доступ к курсам

- **`/courses`:** каталог + highlight по `?highlight=`. Без биллинга «запись» = создание instance бесплатно, и тогда непонятно, чем `/courses` отличается от `/treatment-programs`.
- **`/purchases`:** до появления платежей — пустая страница без смысла; занимает строчку IA.

### Запись и кабинет

- **`/cabinet`:** ~~битые ссылки в `CabinetInfoLinks`~~ частично сняты честными плитками (2026-05-01); нет визуального приоритета ближайшего визита. История intake (`CabinetIntakeHistory`) и прошлых приёмов (`CabinetPastBookings`) — два разных списка с разной семантикой, стоит объединить.
- **Booking wizard (4–5 шагов):** избыточен на mobile при единственном варианте формата/города. Не интегрирован с `/intake/*` — онлайн-формат живёт отдельно от очного flow.
- **`/intake/*`:** без точки входа в основной IA; **shell добавлен (2026-05-01)**, интеграция с booking wizard — всё ещё открыта.
- **`/address`:** iframe плохо работает в TG WebApp, лучше блок «адрес + карта-ссылка + телефон» прямо в `/cabinet`.

### Получение важных уведомлений

- **`/notifications`:** ~~темы захардкожены в коде~~ — **снято по источнику данных (2026-05-01)** — см. раздел I.1 и ключ `notifications_topics`. По-прежнему нет связи с `/reminders` — одну и ту же «напоминание о ЛФК» можно настроить в двух местах с непересекающейся семантикой.
- **`/profile` → аккордеон «Уведомления»:** содержит только текст и кнопку перехода в `/notifications` — лишний хоп.
- **`/messages` + `/support`:** для пациента это один сценарий «связаться с клиникой» — стоит сделать одну точку входа.
- **`/help`:** очень бедная страница (3 ссылки + кнопка поддержки). ~~Из `/cabinet` раньше вели вводящие в заблуждение плитки~~ — **убрано (2026-05-01)**; наполнение `/help` по-прежнему отдельная задача.

### Системные

- **`/profile`:** 7 аккордеонов слишком дробно для роли «помощник». Реальных логических блоков три: «Аккаунт», «Уведомления» (если не убрать), «Данные».
- **`/bind-phone`:** норм. Может быть частью wizard-онбординга вместо самостоятельной обзорной страницы.
- **`/install`:** статичная справка, не привязана к триггеру (PWA-промт не предлагается в правильный момент).

## I.3. Целевая модель IA пациента (короткая сводка)

| Вкладка | Что туда уходит |
|---------|------------------|
| **Сегодня** | Без изменений (новый home) |
| **План** | Программы и шаги, прогресс, тесты к заполнению, материалы дня; абсорбирует `/treatment-programs/*` + ссылку «весь план» в архив |
| **Дневник** | Симптомы + ЛФК-журнал (без само-создания комплексов) |
| **Запись** | Единый wizard, включает онлайн-форматы (`/intake/*`); `/cabinet` как «Мои визиты» внутри той же ветки |
| **Профиль** (справа в шапке) | Аккаунт + Уведомления + Поддержка + Помощь + Установить приложение + Выход |

В этой модели исчезают как отдельные пункты IA: `/sections`, `/purchases`, `/address`, `/help`, `/support`, `/install`.

**Что не так относительно целевой модели** (см. также факт в `STRUCTURE_AUDIT.md` §II.3 и [`TARGET_STRUCTURE_DOCTOR.md`](TARGET_STRUCTURE_DOCTOR.md)):

- Экран по-прежнему смешивает **учётную запись** и **клинические данные** в одном скролле — целевая модель предполагает **табы** и hero «Что важно сейчас» (не в scope REPACK).
- **Заметки врача** — вверху страницы после REPACK, но без **таймлайна** и без выделенного таба.
- **Назначить ЛФК / программу** — доступны из плоских секций; не вынесены в таб «Назначения» (backlog этапа 6 глубокая часть).
- **Дневник симптомов** показан как «Симптомы: A, B, C. Последние записи: 3, 4, 2» — без графика, дат и связи с назначениями.
- **Коммуникации:** секция со старым журналом отправок рендерится **только при непустом** `messageHistory`; единый чат — CTA в шапке. Полное **сведение к табам** — после снятия заморозки этапа 6.
- При первом открытии **контакты** больше не доминируют по умолчанию — приоритет у клинической группы (после REPACK).

**Целевая структура — рабочие табы:**

```
Шапка (имя, телефон, аватар каналов, статус «активный/архив»)
├ Hero «Что важно сейчас»
│   • Назначения: активная программа + ближайший шаг
│   • Открытые тесты к проверке
│   • Последняя запись на приём (статус)
│   • Последнее сообщение в чате
├ Tab 1 — Заметки и история визитов  (основной рабочий tab)
│   • Хронологический фид: визиты ↔ заметки врача ↔ ключевые события
├ Tab 2 — Назначения
│   • Активная программа (с прогрессом + кнопка «изменить шаг»)
│   • Активные комплексы ЛФК
│   • Активные напоминания
│   • Кнопки «Назначить новое»
├ Tab 3 — Дневники
│   • Симптомы: график (как у пациента) + комментарий
│   • ЛФК: журнал занятий + комплаенс
├ Tab 4 — Сообщения (встроенный чат)
└ Tab 5 (свернуть) — Учётная запись (контакты, каналы, lifecycle, admin)
```

Заметки должны быть **первичны** в Tab 1: это основной артефакт врача и единственное, что нельзя восстановить из других данных.

## II.4. CMS — главная боль

**Примечание 2026-05-02:** в коде уже внедрён **вариант C** ([`CMS_RESTRUCTURE_PLAN.md`](CMS_RESTRUCTURE_PLAN.md)): у `content_sections` есть `kind` (`article` \| `system`) и `system_parent_code`. Подпункты ниже по-прежнему полезны как описание продуктовой проблемы и **дальнейшего** ужесточения типизации (отдельные `situation` / `subscription` / `course_lesson` и т.д.).

### II.4.1 Что сейчас

После **варианта C** ([`CMS_RESTRUCTURE_PLAN.md`](CMS_RESTRUCTURE_PLAN.md)) у `content_sections` уже есть **`kind`** (`article` \| `system`) и **`system_parent_code`** для кластеров CMS и фильтров patient-home; сайдбар CMS разделён на статьи и системные папки. Ниже — что **ещё** тянет техдолг и продуктовую путаницу относительно **целевой** модели отдельных `situation` / `subscription` / `course_lesson` и табов хаба (II.4.3).

`content_sections` по-прежнему **одна таблица** для разных смыслов контента. Часть различий всё ещё опирается на legacy-признаки и код:
- `requires_auth` — флаг «нужен tier patient»; используется как proxy для «по подписке»;
- `LESSON_CONTENT_SECTION = "lessons"` / `LESSON_CONTENT_SECTION_LEGACY = "course_lessons"` — **хардкоженый slug** в `modules/treatment-program/types.ts`, по которому фильтруются страницы для шаблона программы;
- `slug` секции «случайно» = коду блока главной (`situations`, `daily_warmup`, `subscription_carousel` ссылаются на `content_sections.slug` через `patient_home_block_items.target_type=content_section`).

`/app/doctor/content` — хаб с сайдбаром по таксономии варианта C; полное разнесение по отдельным табам/маршрутам под каждый продуктовый `kind` (как в черновике II.4.3) — **ещё впереди** (часть IV этап 3 roadmap).

### II.4.2 Главная нестыковка

В коде есть **два независимых уровня типизации**, которые должны были быть одним:
- `patient_home_blocks.code` — содержит «смысловой тип» блока (`situations`, `subscription_carousel`, `courses`, `useful_post`, `daily_warmup`...);
- `content_sections` — после варианта C есть **`kind`** / **`system_parent_code`**, но это **ещё не** целевой жёсткий enum из II.4.3 (отдельные `situation` / `subscription` / `course_lesson`).

Без дальнейшего ужесточения типизации и правил pickers один и тот же раздел теоретически может оказаться в нескольких смысловых ролях; вариант C снизил хаос (фильтры по кластеру, защита immutable slug), но **не заменяет** полностью целевую модель II.4.3.

### II.4.3 Предложение — типизация разделов CMS

**Факт репозитория:** в варианте C уже есть `content_sections.kind` ∈ {`article`, `system`} и `system_parent_code`. Таблица и шаги ниже — **целевое расширение** до отдельных продуктовых типов; не путать с уже вмерженной схемой.

Ввести **`content_sections.kind`** как **полный** enum (строгий):

| `kind` | Смысл | Где используется |
|--------|-------|------------------|
| `article` | Статьи / уроки / FAQ | `/app/patient/sections/[slug]`, прямые ссылки |
| `situation` | Ситуации для главной | Только блок `situations` |
| `subscription` | Материалы подписки | Только `subscription_carousel`, бейдж «по подписке» |
| `course_lesson` | Материалы шаблона программы | Конструктор `treatment_program_templates` (заменит хардкод slug `lessons`) |
| `system` | Технические разделы (например, `warmups`) | Прибит к коду через ID |

Дальше:
- На страницах `/content/sections`, `/content` — **разнести по табам/маршрутам** по `kind`.
- На `/patient-home` редактор **видит только разрешённые `kind`** для блока (для `situations` — выбор только из `kind=situation`; для `subscription_carousel` — `subscription`).
- На редакторе шаблона программы — выбор «уроков» только из `kind=course_lesson`, без хардкода slug.

### II.4.4 `requires_auth` vs `kind=subscription`

Сейчас «подписочный контент» = `requires_auth=true`. Это **семантически неверно**: «требует входа» ≠ «является частью подписки». После реформы:
- `requires_auth` → access-control (нужен tier patient).
- `kind=subscription` → визуальная маркировка и место (карусель «по подписке»).

## Этап 2. CMS-типизация: `content_sections.kind`

**Статус 2026-05-02 (факт репозитория):** первый шаг CMS-first выполнен как **вариант C** в [`CMS_RESTRUCTURE_PLAN.md`](CMS_RESTRUCTURE_PLAN.md): колонки `kind` ∈ {`article`, `system`} и `system_parent_code` для кластеров CMS и patient-home; детали и сверка с кодом — [`CMS_RESTRUCTURE_EXECUTION_AUDIT.md`](CMS_RESTRUCTURE_EXECUTION_AUDIT.md). Блок **«Конкретно»** ниже описывает **более широкую** целевую модель (отдельные `situation` / `subscription` / `course_lesson` и т.д.) и остаётся ориентиром для последующих этапов, а не дословным описанием уже вмерженного кода.

**Что меняется (для редактора):** редактор перестаёт «не понимать, куда уйдёт страница». В CMS-хабе появляются отдельные табы по типу контента: «Статьи», «Ситуации», «Подписочные», «Уроки курсов». При создании страницы редактор сразу выбирает тип и видит соответствующий набор полей.

**Что меняется (для пациента):** ничего пользовательски заметного, **если** на этапе миграции данных всё классифицировано корректно. Под капотом блоки главной начинают валидировать выбор items по `kind`.

**Зачем:** это **фундамент CMS-реформы**. Без него любой UI-апгрейд CMS остаётся «всё в одной куче». Также эта же миграция выносит хардкод slug `lessons` (`course_lesson` → `kind`).

**Конкретно:**
- Drizzle-схема: добавить `content_sections.kind` (enum: `article | situation | subscription | course_lesson | system`).
- Миграция данных: всем существующим разделам присвоить `kind=article` по умолчанию; вручную пометить:
  - `lessons` / `course_lessons` → `course_lesson`;
  - системные slug-и (warmups и пр.) → `system`;
  - использующиеся в `patient_home_block_items` блока `situations` → `situation`;
