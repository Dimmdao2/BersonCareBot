# Часть I — Рекомендации по пациентскому приложению

Цели приложения: «удобный помощник», «платформа для профилактики и реабилитации», «доступ к курсам», «выполнение назначений врача», «получение важных уведомлений».

## I.1. Сквозные проблемы IA

1. **Дублирующиеся сущности.**
   - Уведомления: аккордеон «Уведомления» в `/profile` (просто ссылка на `/notifications`) + сама страница `/notifications`. Две точки входа в одну функцию. Темы рассылок (`SUBSCRIPTIONS`) **захардкожены в коде**, а не в `system_settings` — нарушает `runtime-config-env-vs-db.mdc`.
   - Поддержка: `/messages` (чат) и `/support` (одно письмо админу) — для пациента это один сценарий «связаться с клиникой».
   - Контент: «Ситуации» на главной + страница `/sections` («Уроки и тренировки») — содержательно дублируются с разной формулировкой.

2. **Битые смысловые ссылки.** В `/cabinet` карточка `CabinetInfoLinks` показывает плитки «Адрес кабинета» (ок), **«Как подготовиться»** и **«Стоимость»** — обе ведут на `/help`, где этого контента нет.

3. **Нет проактивности после редизайна Home.** Внутренние страницы регрессируют относительно нового home:
   - `/treatment-programs` — плоский список «Название + статус», без прогресса и ближайшего шага.
   - `/cabinet` — таблица «активных записей» без выделения ближайшего визита.
   - `/diary` — открывается дашбордом, а не «давай отметим занятие за сегодня».

4. **Скрытые/несвязанные сценарии.**
   - `/reminders` — мощная страница (правила, синхронизация с ботом, журналы), **в основной навигации её нет**. Заходят случайно с карточки «Следующее напоминание».
   - `/intake/nutrition` и `/intake/lfk` — отдельные client-формы без `AppShell`, без видимой точки входа в продукте; попасть только через скрытый deep-link.

5. **Лишние URL/редиректы.** `lessons`, `emergency`, `diary/symptoms`, `diary/lfk` — четыре route-файла существуют только ради `redirect()`.

## I.2. По продуктовым целям

### Выполнение назначений врача

- **`/treatment-programs` (список):** карточка показывает только заголовок и статус. Нет: прогресса этапа, ближайшего шага, даты окончания, бейджа «новые задачи». Архив (`completed`) лежит вперемешку с активными.
- **`/treatment-programs/[instanceId]` (детальная):** должна быть точкой входа в «сегодняшний шаг», а не длинной портянкой этапов.
- **`/diary` (Симптомы + ЛФК):** в пустом ЛФК-состоянии показано «Создайте комплекс упражнений» с формой `createLfkComplex`. Это пациентское создание комплекса противоречит модели «комплексы приходят из назначения врача» и смешивает источники.
- **`/reminders`:** содержательно полезна, но скрыта от навигации; не связана с `/notifications` (одну сущность можно настраивать в двух местах с разной семантикой).

### Профилактика и реабилитация (контент)

- **`/sections`:** заголовок «Уроки и тренировки» противоречит языку нового home («Ситуации»), функционально дубль ряда `situations` главной.
- **`/sections/[slug]`:** subscription callout + warmups reminder bar + grid `FeatureCard` — реализовано хорошо.
- **`/content/[slug]`:** одна из лучших страниц — hero, markdown, видео, отметка «выполнено», CTA курса. Не хватает «следующий материал», чтобы превращать чтение в привычку.

### Доступ к курсам

- **`/courses`:** каталог + highlight по `?highlight=`. Без биллинга «запись» = создание instance бесплатно, и тогда непонятно, чем `/courses` отличается от `/treatment-programs`.
- **`/purchases`:** до появления платежей — пустая страница без смысла; занимает строчку IA.

### Запись и кабинет

- **`/cabinet`:** битые ссылки в `CabinetInfoLinks` (см. I.1). Нет визуального приоритета ближайшего визита. История intake (`CabinetIntakeHistory`) и прошлых приёмов (`CabinetPastBookings`) — два разных списка с разной семантикой, стоит объединить.
- **Booking wizard (4–5 шагов):** избыточен на mobile при единственном варианте формата/города. Не интегрирован с `/intake/*` — онлайн-формат живёт отдельно от очного flow.
- **`/intake/*`:** изолированы, без `AppShell`, без точки входа.
- **`/address`:** iframe плохо работает в TG WebApp, лучше блок «адрес + карта-ссылка + телефон» прямо в `/cabinet`.

### Получение важных уведомлений

- **`/notifications`:** темы захардкожены в коде. Нет связи с `/reminders` — одну и ту же «напоминание о ЛФК» можно настроить в двух местах с непересекающейся семантикой.
- **`/profile` → аккордеон «Уведомления»:** содержит только текст и кнопку перехода в `/notifications` — лишний хоп.
- **`/messages` + `/support`:** для пациента это один сценарий «связаться с клиникой» — стоит сделать одну точку входа.
- **`/help`:** очень бедная страница (3 ссылки + кнопка поддержки). При этом на неё ведёт `/cabinet` как на «Как подготовиться» и «Стоимость» — пустые обещания.

### Системные

- **`/profile`:** 7 аккордеонов слишком дробно для роли «помощник». Реальных логических блоков три: «Аккаунт», «Уведомления» (если не убрать), «Данные».
- **`/bind-phone`:** норм. Может быть частью wizard-онбординга вместо самостоятельной обзорной страницы.
- **`/install`:** статичная справка, не привязана к триггеру (PWA-промт не предлагается в правильный момент).

## I.3. Целевая модель IA пациента (короткая сводка)

| Вкладка | Что туда уходит |
|---------|------------------|
| **Сегодня** | Без изменений (новый home) |
| **План** | Программы и шаги, прогресс, тесты к заполнению, материалы дня; абсорбирует `/treatment-programs/*` + ссылку «весь план» в архив |
| **Дневник** | Симптомы + ЛФК-журнал (без само-создания комплексов) |
| **Запись** | Единый wizard, включает онлайн-форматы (`/intake/*`); `/cabinet` как «Мои визиты» внутри той же ветки |
| **Профиль** (справа в шапке) | Аккаунт + Уведомления + Поддержка + Помощь + Установить приложение + Выход |

В этой модели исчезают как отдельные пункты IA: `/sections`, `/purchases`, `/address`, `/help`, `/support`, `/install`.

Развёрнутая мысленная карта — [`TARGET_STRUCTURE_PATIENT.md`](TARGET_STRUCTURE_PATIENT.md).

# Часть II — Рекомендации по кабинету врача / админа

## II.1. Сквозные проблемы

1. **Меню плоское, без приоритетов;** два разделителя ничего не означают для пользователя.
2. **`/online-intake` не в меню вообще** (`DOCTOR_MENU_ENTRIES` его не содержит) — критическая дыра в SLA коммуникации с пациентом.
3. **Дублирующиеся точки входа:**
   - Рассылки в двух местах (`/messages` журнал + `/broadcasts`).
   - Сообщения в двух местах (страница `/messages` + аккордеон «Коммуникации» в карточке клиента).
   - Записи (`/appointments` + `/clients?scope=...`).
   - `/subscribers` — полностью legacy (redirect).
4. **Архитектурные нарушения:** `/content/news/page.tsx` и `/content/motivation/page.tsx` напрямую вызывают `pool.query(...)` из RSC (нарушение `clean-architecture-module-isolation.mdc` §1).
5. **Параллельные сущности контента:** `news_items` и `motivational_quotes` живут отдельно от `content_pages` без прозрачной причины (см. II.4).

## II.2. Дашборд `/app/doctor`

**Текущее состояние:** только метрики-плитки («всего пациентов», «отмен за месяц») и «ближайший приём». Этим врач не работает — это панель отчётности.

**Чего не хватает (actionable inbox):**
1. Записи на сегодня — карточки с пациентом, форматом, ссылкой на карту клиента.
2. Новые онлайн-заявки (`patient_online_intake`).
3. Непрочитанные сообщения в поддержке.
4. «К проверке» — пациенты, прошедшие тесты, заполнившие intake-вопросники, пропустившие N занятий.
5. Ближайшие приёмы 2–3 дня вперёд.

**Метрики** при этом стоит **переместить в `/stats`** (которая уже есть), убрав дубль.

## II.3. Карточка пациента `/clients/[userId]`

**Что не так** (см. факт в `STRUCTURE_AUDIT.md` §II.3):

- Это не «карточка пациента», а **раздел системных настроек учётной записи** + рудиментарные view-only данные дневников.
- **Заметки врача** (главный артефакт работы с пациентом) — один из последних аккордеонов, без приоритета и без таймлайна.
- **Назначить ЛФК / программу** спрятаны внутри аккордеонов — это главное действие, а не подопция.
- **Дневник симптомов** показан как «Симптомы: A, B, C. Последние записи: 3, 4, 2» — без графика, дат и связи с назначениями.
- **Коммуникации внутри карточки** дублируют `/messages`.
- Сразу при открытии раскрыт `contacts` (реквизиты), а не клиническая часть.

**Целевая структура — рабочие табы:**

```
Шапка (имя, телефон, аватар каналов, статус «активный/архив»)
├ Hero «Что важно сейчас»
│   • Назначения: активная программа + ближайший шаг
│   • Открытые тесты к проверке
│   • Последняя запись на приём (статус)
│   • Последнее сообщение в чате
├ Tab 1 — Заметки и история визитов  (основной рабочий tab)
│   • Хронологический фид: визиты ↔ заметки врача ↔ ключевые события
├ Tab 2 — Назначения
│   • Активная программа (с прогрессом + кнопка «изменить шаг»)
│   • Активные комплексы ЛФК
│   • Активные напоминания
│   • Кнопки «Назначить новое»
├ Tab 3 — Дневники
│   • Симптомы: график (как у пациента) + комментарий
│   • ЛФК: журнал занятий + комплаенс
├ Tab 4 — Сообщения (встроенный чат)
└ Tab 5 (свернуть) — Учётная запись (контакты, каналы, lifecycle, admin)
```

Заметки должны быть **первичны** в Tab 1: это основной артефакт врача и единственное, что нельзя восстановить из других данных.

## II.4. CMS — главная боль

### II.4.1 Что сейчас

`content_sections` — **одна таблица для всех типов** разделов: статьи, ситуации, подписочные материалы, уроки курсов. Различия между типами размазаны по неявным признакам:
- `requires_auth` — флаг «нужен tier patient»; используется как proxy для «по подписке»;
- `LESSON_CONTENT_SECTION = "lessons"` / `LESSON_CONTENT_SECTION_LEGACY = "course_lessons"` — **хардкоженый slug** в `modules/treatment-program/types.ts`, по которому фильтруются страницы для шаблона программы;
- `slug` секции «случайно» = коду блока главной (`situations`, `daily_warmup`, `subscription_carousel` ссылаются на `content_sections.slug` через `patient_home_block_items.target_type=content_section`).

`/app/doctor/content` — **один общий список страниц с боковой навигацией по разделам**, без типа. Редактор не понимает, **куда уйдёт** создаваемая страница: на главную как ситуация, в каталог как статья, как урок программы или в подписочную карусель.

### II.4.2 Главная нестыковка

В коде есть **два независимых уровня типизации**, которые должны были быть одним:
- `patient_home_blocks.code` — содержит «смысловой тип» блока (`situations`, `subscription_carousel`, `courses`, `useful_post`, `daily_warmup`...);
- `content_sections` — без типа.

В результате один и тот же раздел может попасть и в «ситуации», и в «подписки», и стать «уроком курса», и **редактор это никак не контролирует**.

### II.4.3 Предложение — типизация разделов CMS

Ввести **`content_sections.kind`** (enum, строгий):

| `kind` | Смысл | Где используется |
|--------|-------|------------------|
| `article` | Статьи / уроки / FAQ | `/app/patient/sections/[slug]`, прямые ссылки |
| `situation` | Ситуации для главной | Только блок `situations` |
| `subscription` | Материалы подписки | Только `subscription_carousel`, бейдж «по подписке» |
| `course_lesson` | Материалы шаблона программы | Конструктор `treatment_program_templates` (заменит хардкод slug `lessons`) |
| `system` | Технические разделы (например, `warmups`) | Прибит к коду через ID |

Дальше:
- На страницах `/content/sections`, `/content` — **разнести по табам/маршрутам** по `kind`.
- На `/patient-home` редактор **видит только разрешённые `kind`** для блока (для `situations` — выбор только из `kind=situation`; для `subscription_carousel` — `subscription`).
- На редакторе шаблона программы — выбор «уроков» только из `kind=course_lesson`, без хардкода slug.

### II.4.4 `requires_auth` vs `kind=subscription`

Сейчас «подписочный контент» = `requires_auth=true`. Это **семантически неверно**: «требует входа» ≠ «является частью подписки». После реформы:
- `requires_auth` → access-control (нужен tier patient).
- `kind=subscription` → визуальная маркировка и место (карусель «по подписке»).

### II.4.5 News, Motivation и Broadcasts

**Установленные факты по коду** (см. `STRUCTURE_AUDIT.md` §III, пункты 6, 8–10).

**Решение:**

1. **Мотивации** — оставить как сущность, **убрать как блок главной**, сделать **переиспользуемый виджет** `<MotivationCard />` для врезки на разные страницы (после завершения разминки в `PatientContentPracticeComplete`, после отметки занятия в `LfkSessionForm`, в empty-state дневника). Редактор остаётся простой плоский список с inline-редактированием.

2. **Новости** — **удалить целиком как сущность**: модель пуста (на главной не показывается), смысл совпадает с рассылками (короткое сообщение) или с `useful_post` (длинная карточка в CMS). Удаляем `news_items`, страницу `/app/doctor/content/news`, orphan-компоненты `PatientHomeNewsSection.tsx` / `PatientHomeMailingsSection.tsx` и legacy `HomeBlockId` `news` / `mailings`.

3. **Broadcasts → «Сообщение пациентам»** с полем `channels[]`. На старте — два канала (`bot_message`, `sms` — то, что сегодня по факту). Дальше расширяется без миграции модели:

   | Канал | Когда добавить | Что значит |
   |-------|----------------|------------|
   | `bot_message` | сейчас (есть) | TG/MAX через интегратор |
   | `sms` | сейчас (есть) | через интегратор |
   | `push` | позже | web push для PWA |
   | `home_banner` | позже | плашка на `/app/patient` (займёт нишу удалённых `news_items`) |
   | `notification_bell` | позже | колокольчик в шапке |

   В UI редактора — multiselect каналов, неактивные `disabled` с подписью «скоро».

### II.4.6 Целевая навигация CMS-хаба

```
КОНТЕНТ
  Статьи и уроки          (content_pages, kind=article + course_lesson)
  Ситуации                (content_pages, kind=situation)
  Подписочные             (content_pages, kind=subscription)
  ─
  Мотивации               (motivational_quotes — виджет, не блок)
  ─
  Главная пациента        (блоки + items)
  Библиотека файлов

КОММУНИКАЦИИ
  Сообщения               (чат)
  Рассылки                (broadcasts с channels[])
```

«Лента» как кластер исчезает — она была там только из-за `news` и `motivation`.

## II.5. Каталоги назначений врача

`Упражнения`, `Комплексы`, `Клинические тесты`, `Наборы тестов`, `Рекомендации`, `Шаблоны программ`, `Курсы` — **библиотека продукта**. Общий паттерн: список + фильтры + master-detail + создание/редактирование.

**Хорошо:** одинаковая структура, единый `doctorCatalogViewFromSearchParams`, общие фильтры (`q`, `region`, `load`).

**Что не хватает:**
1. **Обратная связь «сколько раз назначено»** на каждом элементе каталога («упражнение в 5 шаблонах», «комплекс назначен 12 пациентам»).
2. **Архивация без cascade-проверки** — нет предупреждения, если упражнение в активной программе.
3. **Курсы и Шаблоны программ** — по сути **одна сущность** (см. `/app/patient/courses` создаёт инстанс программы). Курс = «шаблон программы, опубликованный в каталог пациента». Стоит объединить административный экран либо как минимум привязать «Курс» к шаблону через UI.

## II.6. Inbox / событийная лента — чего не хватает

Чтобы кабинет стал рабочим, нужны **активные сценарии**, которых сейчас нет:

1. **Inbox онлайн-заявок** (`/online-intake`) — добавить в меню первым после «Сегодня». Бадж с количеством новых.
2. **Inbox непрочитанных чатов поддержки** — счётчик в меню/шапке.
3. **Лента событий по своим пациентам:**
   - «N отметила симптом 8/10 (выше порога врача)».
   - «M пропустил 5 занятий ЛФК подряд».
   - «K заполнил онлайн-тест A (нужна оценка)».
4. **Тесты к проверке** — отдельный inbox, потому что заполненный пациентом тест без оценки врача — незакрытый цикл.

## II.7. Целевая модель меню (5 кластеров)

```
РАБОТА С ПАЦИЕНТАМИ
  Сегодня (бывший «Обзор»)         — actionable inbox дня
  Пациенты                         — список + карточка
```

## Этап 2. CMS-типизация: `content_sections.kind`

**Что меняется (для редактора):** редактор перестаёт «не понимать, куда уйдёт страница». В CMS-хабе появляются отдельные табы по типу контента: «Статьи», «Ситуации», «Подписочные», «Уроки курсов». При создании страницы редактор сразу выбирает тип и видит соответствующий набор полей.

**Что меняется (для пациента):** ничего пользовательски заметного, **если** на этапе миграции данных всё классифицировано корректно. Под капотом блоки главной начинают валидировать выбор items по `kind`.

**Зачем:** это **фундамент CMS-реформы**. Без него любой UI-апгрейд CMS остаётся «всё в одной куче». Также эта же миграция выносит хардкод slug `lessons` (`course_lesson` → `kind`).

**Конкретно:**
- Drizzle-схема: добавить `content_sections.kind` (enum: `article | situation | subscription | course_lesson | system`).
- Миграция данных: всем существующим разделам присвоить `kind=article` по умолчанию; вручную пометить:
  - `lessons` / `course_lessons` → `course_lesson`;
  - системные slug-и (warmups и пр.) → `system`;
  - использующиеся в `patient_home_block_items` блока `situations` → `situation`;
  - использующиеся в блоке `subscription_carousel` → `subscription`.
- Сервис `contentSections` принимает фильтр по `kind`.
- Заменить `LESSON_CONTENT_SECTION = "lessons"` в `modules/treatment-program/types.ts` на запрос `contentSections.listByKind('course_lesson')`. Удалить хардкод.
- В `/app/doctor/content` ввести табы по `kind` (или отдельные маршруты `content/articles`, `content/situations`, `content/subscription`, `content/course-lessons`).
- В `patient_home` редакторе при выборе items — фильтровать по разрешённому для блока `kind` (для `situations` — только `kind=situation` и т.д.).
- В редакторе шаблона программы — выбор «уроков» из `kind=course_lesson`.

**Размер:** средний, 3–5 дней (миграция данных требует ручной классификации существующих разделов).
**Риск:** средний (миграция данных, но обратимая — кто угодно может перенастроить `kind`).
**Pre-conditions:** Этап 1 (мотивации/новости решены) — иначе путаница, какие kind нужны.
