## 1. Закреплённая доменная картина

### 1.1 Иерархия данных

```
treatment_program_template
  └── template_stage
       ├── template_stage_group       (NEW — A3)
       └── template_stage_item

           ↓ deep copy при назначении

treatment_program_instance
  └── instance_stage
       ├── instance_stage_group       (NEW — A3)
       └── instance_stage_item        (snapshot, status active/disabled)
```

Этап содержит **две оси организации**:
- **Группы** (NEW) — смысловой контейнер («Разминка», «Силовые для кора», «Шея»). У группы есть `title`, `description`, `schedule_text` (фристайл — «3 раза в неделю»), `sort_order`. Группа не управляет статусом, не имеет FSM. На **шаблоне** элементы иных типов, чем `recommendation` / **`clinical_test`** (плоская строка теста; разворот каталожного набора — отдельный API `from-test-set`), должны иметь `group_id` на группу шаблона; без группы допустимы только рекомендация и клинический тест. На **экземпляре** при назначении для каждого этапа создаются две **системные** группы (`system_kind`: `recommendations`, `tests`; `sort_order` 101/102; без `source_group_id`); ungrouped в шаблоне рекомендации/**клинические тесты** попадают туда (см. §1.1a).
- **Items** — упражнение / комплекс / клинический тест / рекомендация / урок. У instance_stage_item — `status` (`active` | `disabled`), `is_actionable` (для рекомендаций), `last_viewed_at`, `local_comment`, `snapshot`.

### 1.1a Системные группы на экземпляре этапа

- **Назначение с шаблона:** `assignTemplateToPatient` передаёт в `createInstanceTree` две системные группы на этап + пользовательские группы с шаблона; элементы с `group_id = null` в шаблоне типов `recommendation` / **`clinical_test`** получают `group_id` соответствующей системной группы.
- **Прямой ввод дерева** (тесты, скрипты): если в `groups` нет строк с `systemKind`, но есть ungrouped `recommendation` и/или **`clinical_test`**, репозиторий **дописывает** те же системные группы перед вставкой (`withDefaultSystemGroupsIfNeededForTreeStage`).
- **БД:** колонка `system_kind` на `treatment_program_instance_stage_groups` (миграция `0044_instance_stage_groups_system_kind.sql`, CHECK `NULL | recommendations | tests`); не более одной системной группы каждого вида на этап — частичные уникальные индексы (`0045_instance_stage_groups_system_kind_unique.sql`).

### 1.2 Особый этап «Общие рекомендации» (Этап 0)

Псевдо-stage с `sort_order = 0`. Поведение:
- всегда `available`, никаких `locked` / FSM;
- содержит **persistent**-рекомендации (режим/быт/долгосрочные инструкции);
- **остаётся видимым после завершения программы** (когда обычные пройденные этапы скрываются/архивируются).

Технически — обычный `instance_stage` с двумя признаками: `sort_order = 0` и нерабочий FSM (статусы не считаются). Отдельная таблица не нужна. См. §3 в части реализации.

### 1.3 Цели / задачи / срок этапа

На `template_stage` и `instance_stage` добавляются:
- `goals` TEXT (markdown) — клиническая цель этапа («снять отёк», «восстановить амплитуду»);
- `objectives` TEXT (markdown) — измеримые задачи («дойти до 90° сгибания», «3 занятия в неделю без болей > 3»);
- `expected_duration_days` INT (опц.) и/или `expected_duration_text` TEXT (опц.) — прогноз срока («2–3 недели», «до контрольного приёма»).

`goals/objectives/expected_duration_*` копируются в snapshot при назначении и видны пациенту в шапке этапа.

### 1.4 Item-types и поведение completion

| `item_type` | completion | Источник «выполнено» |
|---|---|---|
| `exercise` | да | галочка в чек-листе сессии (см. §1.5) |
| `lfk_complex` | да | галочка / окончание run-screen комплекса |
| `lesson` | да | пациент открыл и нажал «Изучил» / открытие |
| `clinical_test` | да | по снимку/попытке зафиксирован результат ожидаемого теста (legacy `test_set` в старых данных — см. миграция `0048`) |
| `recommendation` (actionable) | да | галочка в чек-листе |
| `recommendation` (persistent) | **нет** | элемент не имеет completion-логики, не считается в прогрессе этапа |

Для `recommendation` различие — поле `is_actionable: boolean` на **`instance_stage_item`** (а не на каталоге). Одна и та же рекомендация в каталоге может быть назначена и так и так в зависимости от контекста.

`completion` этапа считается только по items с `is_completable = true`. Для `recommendation` это эквивалентно `is_actionable = true`; для остальных типов — всегда `true`.

### 1.5 Лог действий (`program_action_log`)

Один общий лог действий пациента по программе. Поля (ключевые):

| Поле | Назначение |
|---|---|
| `id` | PK |
| `instance_id` FK | программа |
| `instance_stage_item_id` FK | конкретный item, к которому относится действие |
| `patient_user_id` FK | кто отметил |
| `session_id` UUID | группировка нескольких отметок «за один заход» (run-screen или ручной чек-лист) |
| `action_type` | `done` (выполнил), `viewed` (открыл), `note` (добавил заметку) |
| `quantity` JSONB? | опц. — для будущего «сколько раз/минут», сейчас не обязателен |
| `note` TEXT? | заметка из пост-сессионной формы ЛФК в A4 |
| `created_at` | timestamp |

`test_attempts` остаются отдельной таблицей (специфика данных теста); в `program_action_log` тест отмечается маркером «выполнил» с `instance_stage_item_id` соответствующего теста, а детали — в `test_attempts` / `test_results`.

**Несколько выполнений за календарный день (2026-05-06):** одна и та же позиция чек-листа / тот же ЛФК-комплекс / та же кнопка «выполнено» (`progress/complete`) может дать **несколько** строк `action_type = done` в пределах локальных суток пациента; каждая строка имеет свой `created_at`. Счётчики **`GET …/checklist-today`** (`doneTodayCountByItemId`, `doneTodayCountByActivityKey`) отражают **число таких строк** за день. Снятие галочки чек-листа за день по-прежнему очищает все «простые» отметки по элементу в окне, не затрагивая маркеры теста и строки ЛФК с `lfk_exercise_done`.

### 1.6 Бейдж «План обновлён»

Источник: уже существующая таблица `treatment_program_events`. После любой мутации программы (`item_added`, `item_disabled`, `stage_added`, `stage_skipped`, `comment_changed`, …) пациент при первом открытии Сегодня после правки видит бейдж «План обновлён N мая». Деталей diff не показываем; пациент открывает программу и видит изменения по списку.

Бейдж сбрасывается, когда пациент открыл программу.

### 1.7 Бейдж «Новое упражнение»

Колонка `last_viewed_at TIMESTAMPTZ NULL` на `instance_stage_item`. UI: если `last_viewed_at IS NULL` → бейдж «Новое». Endpoint «mark viewed» вызывается при открытии item-а в run-screen / на детали.

Бэкфилл при релизе: всем существующим items проставить `last_viewed_at = created_at`, чтобы старые элементы **не** показывались как «новые».

В первой итерации **cross-stage детекция** (тот же `item_ref_id` в другом этапе) **не делается** — бейдж считается per `instance_stage_item`. Cross-stage — backlog (см. §7).

### 1.8 «Отключение» вместо удаления

В **инстансе** программы основной путь для пациентского прогресса — переключение `status` на `disabled` (не показывается пациенту, строка и история сохраняются). Дополнительно в кабинете врача: **физическое удаление** строки элемента инстанса допустимо только при отсутствии `completed_at` и попыток теста по этой строке (`DELETE` API, событие `item_removed`); иначе сервис отклоняет удаление.

Disabled items:
- не появляются в чек-листе пациента и не считаются в completion;
- остаются в БД и видны во вкладке «История этапа» / «Все item-ы»;
- сохраняют связь с уже сделанными `program_action_log` записями.

В **шаблоне** программы удаление допустимо (там нет истории действий пациента).

В `treatment_program_events` мутация фиксируется как `item_disabled` или `status_changed payload.scope=item, to=disabled`. Обратимо — `item_enabled`.

«Замена шага» (item Y → item Z) реализуется как `disable Y` + `add Z` на нужный sort_order. Отдельной операции «replace» в UI не нужно.

### 1.9 Universal comment pattern (template + local override)

**Принцип:** у каждого item-в-контейнере (template-уровень) есть поле `comment TEXT NULL`. При создании инстанса контейнера это значение **копируется** в парное поле `local_comment TEXT NULL` на instance-уровне. Дальше:

- Если `instance.*.local_comment IS NOT NULL` — это override; пациенту показывается локальная заметка врача, специфичная для **этого** пациента.
- Если `local_comment IS NULL` (врач очистил поле или не правил его) — fallback на `template.*.comment`.
- В UI инстанса поле «Комментарий» показывает effective-значение; placeholder при пустом override — «Из шаблона: …», чтобы врач видел, что наследует.

**Применение** (полная карта — в [`ASSIGNMENT_CATALOGS_REWORK_PLAN.md`](ASSIGNMENT_CATALOGS_REWORK_PLAN.md) §3 B7):

| Уровень | template_comment | local_comment (instance override) |
|---|---|---|
| `template_stage_item` ↔ `instance_stage_item` | требует поля `comment` в template (если ещё нет) | `local_comment` уже зафиксирован выше в §1.4 |
| `lfk_complex_template_exercises` ↔ instance-LFK | поле в template уже есть | требует поле в instance (если ещё нет) |
| `test_set_items` ↔ instance test-set item | требует обоих полей | требует обоих полей |
| `course_lesson` ↔ enrollment-progress (когда появятся) | по решению `COURSES_INITIATIVE` | по решению там же |

**Итог:** «коммент везде» становится универсальным паттерном. Доктор может оставить нейтральный комментарий в шаблоне («Делать медленно, контролировать дыхание»), а в карточке Маши переопределить его на персональный («Маша — без скруток до 15 мая»). Маша видит персональный; Петя — шаблонный.

**Где живёт реализация:**
- Скелет (`instance_stage_item.local_comment`) — этап **A2** этого плана.
- Раскат на остальные item-контейнеры (`test_set_items.comment` + `local_comment`, copy-сервис, UI «Из шаблона: …») — этап **B7** в [`ASSIGNMENT_CATALOGS_REWORK_PLAN.md`](ASSIGNMENT_CATALOGS_REWORK_PLAN.md).

**Граница с описанием контента:** `recommendations.bodyMd` (содержание рекомендации в каталоге) и `template_comment`/`local_comment` (заметка в контексте конкретного назначения) — **разные сущности**. Описание принадлежит каталоговой записи; комментарий принадлежит item-у-в-контейнере. Не объединяются.

## 2. UX пациента — целевая модель

### 2.1 Точка входа на «Сегодня»

На главной патиента есть **карточка плана**:
- Hero «Сегодняшний шаг» — выбирается алгоритмически: первый незавершённый item первого `available`-этапа активной программы; персистентные рекомендации в Hero не попадают.
- Бейдж «План обновлён N мая» — по `treatment_program_events` (см. §1.6).
- CTA «Открыть план».

Если активной программы нет — карточка не показывается.

### 2.2 «План» (вкладка/раздел) — рабочее пространство

```
План
├ Шапка программы (название, % прогресса)
├ Этап 0 — «Общие рекомендации»  (всегда видим)
│    persistent-рекомендации списком, без галочек
├ Текущий этап (raскрыт)
│    Цель: …                   ← из stage.goals
│    Задачи: …                 ← из stage.objectives
│    Ожидаемый срок: …         ← expected_duration_*
│    ────
│    Группы (аккордеоны):
│      ▼ Разминка   (3 р/нед)
│         • дыхательная разминка   [▶] [✓]
│         • растяжка плеч          [▶] [✓]
│      ▶ Силовые для кора (2 р/нед)
│      ▶ Шея (через день)
│    ────
│    Actionable-рекомендации этапа:
│      • горячая ванна вечером     [✓]
│      • самомассаж 10 мин         [✓]
│    ────
│    Тесты этапа: «Тест X — пройти»     → run-screen теста
│    [Закончить занятие] → шкала «легко/средне/тяжело» + опц. заметка
├ Прошлые этапы (свернуты, доступны для просмотра)
└ Будущие этапы (locked-вид)
```

### 2.3 Run-screen ЛФК-комплекса

- Список «что и сколько» (без видео в линии).
- На каждом пункте — иконка ▶, по нажатию открывается видео упражнения.
- Чек-бокс «выполнил» на каждом пункте — пишет в `program_action_log` (action_type=done, по `instance_stage_item_id` упражнения внутри комплекса; если уровень детализации complex-only — пишется на сам комплекс).
- Кнопка «Закончить занятие» — закрывает сессию, открывает мини-форму оценки (см. §2.4).

В первой версии — детализация **на уровне комплекса**: одна запись «комплекс выполнен», галочки внутри — лог-сигнал. Расширение «галочки на каждом упражнении как отдельные записи» — backlog (§7).

### 2.4 Форма «Оценка занятия» (`LfkSessionForm` упрощённая)

После «Закончить занятие» — одна простая форма:
- Шкала «как было: легко / средне / тяжело» (3 значения).
- Опциональная заметка (TextField).
- Кнопка «Сохранить».

Шкала боли отдельно **не делается** — по принципу клиники работа через боль > 2–3 баллов не допускается, поэтому «тяжесть» включает в себя оценку боли. Если врач решит вернуть отдельное поле — backlog.

### 2.5 Run-screen теста

- Вопросы (по одному или сводно — зависит от типа scoring).
- Автосохранение черновика на каждом ответе.
- Финал: «Отправить врачу» → создаётся `test_attempt` + `test_results.normalized_decision` авто (по scoring), `decided_by = NULL`.
- Экран подтверждения «Ответ передан. Врач напишет в чат, когда оценит».

**Без лимитов попыток.** Врач — gate этапа: открывает следующий этап вручную или подтверждает решение. Пациент может проходить тест повторно, если не запретил врач (запрет — дополнительная функциональность, сейчас не делаем).

«История попыток» — отдельный экран на карточке шага (по запросу пациента «Все попытки»). На основной карточке — только текущее состояние (последняя попытка + решение врача, если есть).

### 2.6 Уведомления / напоминания

В первой версии MVP — **только бейдж «План обновлён» в Сегодня**. Push в бот о назначении/изменении/переходе на следующий этап — backlog (§7).

| Таблица | Назначение |
|---|---|
| `program_action_log` | Лог действий пациента (см. §1.5) |

### 3.3 Новые колонки (по этапам)

| Этап | Таблица | Колонка | Назначение |
|---|---|---|---|
| A1 | `treatment_program_template_stages`, `_instance_stages` | `goals TEXT`, `objectives TEXT` (markdown, O1), `expected_duration_days INT NULL`, `expected_duration_text TEXT NULL` | §1.3 |
| A2 | `treatment_program_instance_stage_items` | `is_actionable BOOLEAN NULL` (только для типа `recommendation`; иначе игнорируется) | §1.4 |
| A2 | `treatment_program_instance_stage_items` | `status TEXT NOT NULL DEFAULT 'active'` (`active` \| `disabled`) | §1.8 |
| A5 | `treatment_program_instance_stage_items` | `last_viewed_at TIMESTAMPTZ NULL` | §1.7 |

Все ALTER — нерушащие, на живой БД безопасны (nullable / DEFAULT).

### 3.4 Псевдо-«Этап 0» (часть A2)

Реализуется как обычный `instance_stage` с `sort_order = 0`. Никаких новых полей не нужно — поведение «не учитывается в FSM, всегда видим, переживает completion программы» определяется правилами в сервисе:
- `progress-service` исключает stage с `sort_order = 0` из расчётов FSM;
- UI пациента всегда показывает stage `sort_order = 0` (даже при `program.status = completed`).

Для шаблона — тот же приём (`template_stage.sort_order = 0`), при копировании в инстанс — копируется как обычный.

## 4. UX врача — что меняется в кабинете

### 4.1 Конструктор шаблона / правка инстанса

- Под title этапа — поля «Цель», «Задачи», «Ожидаемый срок» (см. §1.3).
- Поверх списка items — UI **группировки**: drag-and-drop items между группами, добавление/переименование/удаление группы; для группы — `title`, `description`, `schedule_text`, `sort_order`.
- При добавлении рекомендации в этап — на **экземпляре** toggle «Это actionable / persistent» (фактически в коде: новые рекомендации при копировании из шаблона и при добавлении в инстанс по умолчанию **persistent** (`is_actionable = false`); врач может перевести в actionable). Шаблон по-прежнему не хранит флаг — только ссылка на каталог.
- В инстансе у каждого item — «отключить» / «включить» (см. §1.8); при отсутствии прогресса по строке — **удалить** из инстанса. Жёсткое удаление из **каталожного шаблона** этапа — в конструкторе шаблона.

### 4.2 Каталог рекомендаций (`/app/doctor/recommendations`)

Расширяется до набора фильтров каталога с типизацией:
- `kind` (enum): `regimen`, `nutrition`, `device`, `self_procedure`, `external_therapy`, `lifestyle`, …
- `body_region` — справочник;
- `quantity` — короткая текстовая характеристика по умолчанию («10 минут», «10 сеансов», «3 подхода»);
- `frequency` — «3 р/нед», «ежедневно», «однократно» (фристайл-текст);
- `duration` — «4 недели», «бессрочно» (фристайл-текст);
- опционально `external_link` / `external_org` для `kind = external_therapy`.

Фильтры списка: `q`, `kind`, `body_region`. Master-detail как у других каталогов. «Где используется» / архивация — те же шаблоны, что в [`ASSIGNMENT_CATALOG_USAGE_ARCHIVE_PLAN.md`](done/ASSIGNMENT_CATALOG_USAGE_ARCHIVE_PLAN.md). Редакционные материалы блоков главной (ситуации, подписка, разминка) — [`CONTENT_PLAN.md`](CONTENT_PLAN.md).

Этот каталог по структуре близок к каталогу упражнений: список + фильтры + карточка с типизированными полями.

### 4.3 Inbox «К проверке» в карточке пациента

В Tab «Назначения» / «Дневники» карточки пациента — секция **«Тесты, ожидающие оценки»**:
- читает `test_results.decided_by IS NULL` для тестов в активных программах врача-curator;
- ссылка на конкретный тест → экран оценки (`overrideResultDecision`).

Кросс-пациентский inbox в «Сегодня» врача — backlog (мелкое расширение, на базе того же read-метода).

### 4.4 События обновления плана

В `treatment_program_events` уже пишется история. На стороне врача:
- никакого нового UX «уведомить пациента» — пациент сам видит бейдж «План обновлён» (см. §1.6);
- в карточке программы у врача — компактный блок «Последние изменения» как фид events (если уже не реализовано).

## 6. Этапы реализации (high-level)

> Каждый этап — потенциально отдельная инициатива со своим execution-документом и LOG-блоком. Зависимости — линейные: A1 → A2 → A3 → A4 → A5. Любой этап можно остановить — продукт остаётся работоспособным.

### A1 — Цели/задачи/срок этапа

- Drizzle schema: `goals`, `objectives`, `expected_duration_*` на `template_stage` + `instance_stage`.
- Сервис копирования template→instance переносит поля.
- UI пациента: header в этапе с этими полями.
- UI врача: поля в конструкторе и при правке инстанса.
- Тесты сервиса/UI.

**Размер:** малый, ~1–2 дня.

### A2 — Рекомендация actionable/persistent + Этап 0 + «отключение» item-а

- Drizzle: `instance_stage_item.is_actionable BOOLEAN NULL`, `instance_stage_item.status TEXT NOT NULL DEFAULT 'active'`.
- Сервис: `progress-service` исключает `disabled` items и persistent-рекомендации из completion.
- Псевдо-stage `sort_order = 0` (правила в сервисе и в UI; миграции таблиц нет).
- Каталог рекомендаций: расширение полей (см. §4.2).
- UI врача: toggle actionable, иконка «отключить» в инстансе.
- UI пациента: persistent-рекомендации в Этапе 0 без галочек; actionable-рекомендации с галочками в чек-листе.
- Тесты.

**Размер:** средний, ~3–4 дня.

### A3 — `tplStageGroups` / `instStageGroups`

- Drizzle: две новые таблицы (template + instance), `group_id` на items.
- Сервис копирования template→instance переносит группы и связи.
- UI врача: drag-n-drop items между группами, CRUD группы.
- UI пациента: аккордеоны по группам с `schedule_text` курсивом в шапке.
- Тесты.

**Размер:** средний, ~4–5 дней.

### A4 — `program_action_log` + чек-лист дня + Inbox «К проверке»

- Drizzle: `program_action_log`.
- Сервис: запись действий (`done`, `viewed`, `note`); генерация «чек-лист дня» как union actionable items текущего этапа + actionable рекомендаций.
- UI пациента: чек-лист в этапе с галочками; форма «Оценка занятия» в конце run-screen.
- Inbox «К проверке» в карточке пациента у врача — read-метод по `test_results.decided_by IS NULL` + секция в карточке.
- Тесты.

**Размер:** средний-крупный, ~5–7 дней.

### A5 — Бейдж «План обновлён» + бейдж «Новое»

- Бейдж «План обновлён» — read по `treatment_program_events` за период «после последнего открытия программы пациентом», UI в Сегодня. Без новых таблиц (опционально — лог открытий).
- Колонка `last_viewed_at` на `instance_stage_item` + endpoint mark-viewed + бэкфилл `created_at` существующих + UI-бейдж.
- `revalidatePath` / `revalidateTag` после мутаций программы.
- Тесты.

**Размер:** малый, ~1–2 дня.

## 8. Зафиксированные решения (журнал продуктовых выборов)

> Сюда добавляются решения по открытым вопросам §5 и любые корректировки модели. Каждое решение — короткий блок с датой и контекстом.

### 8.1 (2026-05-03) — Принципы инициативы

- «Любое назначение → `treatment_program_instance`». Параллельных «комплексов вне программы» больше не вводим (legacy `patient_lfk_assignments` не трогаем, но новые сценарии его не используют).
- Курсы — отдельная инициатива (геткурс-модель), к плану лечения не относится.
- Тесты — без лимитов попыток; врач — gate этапа.
- Рекомендации — каталог с типизированными полями; в назначении флаг `is_actionable`.
- Группы — отдельные таблицы (template + instance), не колонка.
- «Удалить» в инстансе → «отключить» (status active/disabled).
- Бейдж «План обновлён» делаем в первой волне; «Новое упражнение» — тоже (дёшево).
- Inbox «К проверке» делаем в карточке пациента вместе с A4.
- Все остальные расширения (расписание, push, метрики) — backlog с условиями.

### 8.2 (2026-05-03) — Universal comment pattern + scope раздела

- **Universal comment pattern** (template `comment` → instance `local_comment` override) фиксируется как общий принцип для **всех** item-контейнеров (см. §1.9). Базовое поле `instance_stage_item.local_comment` остаётся в этом плане (A2). Раскат на `test_set_items`, instance-LFK и др. — в [`ASSIGNMENT_CATALOGS_REWORK_PLAN.md`](ASSIGNMENT_CATALOGS_REWORK_PLAN.md) этап B7.
- **Описание (`bodyMd` рекомендации) и комментарий — разные сущности.** Не объединяем.
- **Sister-инициатива по каталогам** ([`ASSIGNMENT_CATALOGS_REWORK_PLAN.md`](ASSIGNMENT_CATALOGS_REWORK_PLAN.md), B1–B7) — отдельный документ. Делает: типизация клин. тестов (`assessmentKind`, `body_region`, структурированный `scoring`, `CreatableComboboxInput`), редактор наборов тестов как клон LFK-комплекса, регион/метрики у рекомендаций, фикс UX комплексов ЛФК, визуальный pass-1 конструктора шаблонов, два независимых фильтра «черновик/опубликован» × «активный/архив», universal comment pattern. **Не делает** доменную работу A1+A3 (цели/задачи/срок этапа, группы) — она остаётся в этом плане.
- Конструктор шаблона переписывается **визуально один раз** в B6, потом в A1+A3 расширяется доменно. Это исключает двойной refactor.

### 8.3 (2026-05-03) — Пред-реализационные решения для снятия stage-gates

- **O1:** `objectives` фиксирован как `TEXT` markdown в рамках `PROGRAM_PATIENT_SHAPE` (A1). JSONB-checklist не делаем в этой волне; при необходимости выносится отдельным улучшением.
- **O4:** `is_actionable` хранится только на `treatment_program_instance_stage_items`. Каталоговый `default_is_actionable` в A2 не вводится.
- **O2:** `program_action_log` для ЛФК в MVP фиксирован на уровне комплекса (не per-exercise).
- **O3:** заметка из формы `Как прошло занятие?` сохраняется в `program_action_log.note`; отдельное поле `lfk_session.note` не добавляется в A4.
- Для исполнения фиксирована карта кодовой базы в `../archive/2026-05-initiatives/PROGRAM_PATIENT_SHAPE_INITIATIVE/MASTER_PLAN.md` (§3.1): модули `modules/treatment-program`, API `doctor/patient treatment-program-instances|templates`, patient pages `app/patient/treatment-programs/*`, и Drizzle schema `treatmentProgram*.ts`.
