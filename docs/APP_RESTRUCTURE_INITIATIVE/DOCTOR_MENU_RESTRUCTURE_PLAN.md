## Цель

Сделать основное меню кабинета врача понятным рабочим навигатором:

- сгруппировать пункты по смыслу;
- заменить плоский список на аккордеон, где открыт один блок;
- сохранять открытый блок в `localStorage`;
- повторить одну и ту же структуру в desktop sidebar и mobile Sheet;
- перенести «Библиотеку файлов» из CMS-сайдбара в основное меню.

Это отдельный дешёвый проход. Он не должен превращаться в переделку дашборда, бейджей, CMS или карточки пациента.

Вне scope:

- не менять маршруты и URL;

- не делать бейджи новых заявок / непрочитанных сообщений — это этап 3;

## Поведение аккордеона

- Открыт один кластер.
- При первом заходе без сохранённого значения открыт кластер «Работа с пациентами».
- Состояние хранится в `localStorage` под стабильным ключом `doctorMenu.openCluster.v1`.
- Если сохранённый id больше не существует, использовать дефолтный кластер.
- Если пользователь открывает другой кластер, новое значение записывается в `localStorage`.
- В mobile Sheet клик по ссылке закрывает Sheet, но не сбрасывает выбранный кластер.
- Прямой пункт «Библиотека файлов» всегда виден рядом с кластерами и подсвечивается по активному маршруту.

Реализация должна быть SSR-safe: не читать `localStorage` во время server render. `DoctorHeader` и `DoctorAdminSidebar` уже client components, но чтение лучше делать после mount или через безопасный initializer с проверкой `typeof window`.

## Рекомендуемая техническая форма

В `doctorNavLinks.ts` заменить separator-based модель на смысловую:

- `DoctorMenuLinkItem` — обычная ссылка;
- `DoctorMenuClusterItem` — кластер с `id`, `label`, `items`;
- `DoctorMenuStandaloneLinkItem` или отдельный массив для «Библиотеки файлов»;
- `DOCTOR_MENU_CLUSTERS` / `DOCTOR_MENU_STANDALONE_LINKS`;
- `DOCTOR_MENU_LINKS` сохранить как плоский список ссылок для мест, где он может использоваться.

Допустимо выбрать другие имена, если структура остаётся понятной и тестируемой. Главное — больше не использовать `separator` как способ имитировать группы.

Чтобы не дублировать разметку, лучше вынести общий renderer в новый компонент, например:

- `DoctorMenuAccordion.tsx`;
- props: `variant: "sidebar" | "sheet"`, `pathname`, `onNavigate?`;
- внутри: кластеры, standalone-ссылки, active state, localStorage.

Если исполнитель решит не выносить отдельный компонент, он обязан сохранить одинаковое поведение и порядок пунктов в `DoctorHeader` и `DoctorAdminSidebar`.

## Шаги исполнения

### Шаг 0. Preflight

Проверить текущее состояние после CMS-прохода:

- перечитать `doctorNavLinks.ts`, `DoctorHeader.tsx`, `DoctorAdminSidebar.tsx`, `doctorScreenTitles.ts`;
- перечитать `ContentPagesSidebar.tsx` перед удалением ссылки библиотеки;
- выполнить `rg "Библиотека файлов|Обзор|Клиенты и подписчики" apps/webapp/src/shared/ui apps/webapp/src/app/app/doctor/content`.

Критерий закрытия: исполнитель понимает, какие строки реально менять, и не конфликтует с текущим CMS-проходом.

### Шаг 1. Структура меню

- Ввести кластерную структуру меню.
- Сохранить плоский `DOCTOR_MENU_LINKS`.
- Добавить ссылку «Библиотека файлов» на `/app/doctor/content/library` как отдельный пункт верхнего уровня.
- Переименовать пункты меню по решениям выше.

Проверки:

- unit test на состав кластеров и порядок ключевых пунктов;
- unit test, что `DOCTOR_MENU_LINKS` содержит все ссылки, включая библиотеку, но без служебных действий.

### Шаг 2. Общий UI аккордеона

- Реализовать общий renderer или синхронно обновить оба существующих меню.
- Добавить controlled state открытого кластера.
- Подключить `localStorage` с ключом `doctorMenu.openCluster.v1`.
- Добавить active state через `isDoctorNavItemActive`.
- Не ломать доступность: у кнопок кластеров должны быть `aria-expanded` и понятный label; ссылки остаются обычными `Link`.

Проверки:

- RTL-тест: по умолчанию открыт «Работа с пациентами»;
- RTL-тест: открытие другого кластера скрывает предыдущий;
- RTL-тест: сохранённый cluster id восстанавливается из `localStorage`;
- RTL-тест: невалидное значение в `localStorage` даёт дефолт.

### Шаг 3. Desktop sidebar

- Обновить `DoctorAdminSidebar`.
- Сохранить служебные действия внизу: «Профиль и настройки», «Выйти», имя пользователя.
- Проверить, что меню помещается в текущий `w-56`; если подписи не помещаются, уменьшить только локальные размеры/spacing меню, без общего UI density-прохода.

Проверки:

- render test на кластеры и standalone «Библиотека файлов»;
- active state для вложенного маршрута, например `/app/doctor/content/library`.

### Шаг 4. Mobile Sheet

- Обновить меню внутри `DoctorHeader`.
- Использовать тот же порядок и те же labels.
- Клик по любой ссылке закрывает Sheet.
- `hideMenuOnDesktop` оставить как есть.

Проверки:

- render test или smoke через существующий компонент: Sheet содержит те же кластеры;
- клик по ссылке вызывает закрытие Sheet, если это уже тестируемо текущими средствами.

### Шаг 5. Перенос «Библиотеки файлов» из CMS-сайдбара

- В `ContentPagesSidebar.tsx` убрать ссылку `Библиотека файлов`.

- Не менять CMS-системные папки, article sections, filters и query params.

Проверки:

- тест `ContentPagesSidebar` обновить: ссылки «Библиотека файлов» в CMS-сайдбаре нет;
- smoke: `/app/doctor/content/library` доступен из основного меню.

### Шаг 6. Заголовки экранов

- `getDoctorScreenTitle("/app/doctor")` должен вернуть «Сегодня».

- Добавить title для `/app/doctor/content/library`: «Библиотека файлов», если отсутствует.
- Обновить тесты `doctorScreenTitles.test.ts`.

Проверки:

## Stop conditions

Исполнитель должен остановиться и спросить, если:

- после CMS-прохода `ContentPagesSidebar.tsx` устроен иначе и непонятно, как безопасно убрать только библиотеку;
- выяснится, что «Библиотека файлов» технически жёстко зависит от CMS-сайдбара и без него теряет важную навигацию;
- текущие tests/setup не позволяют без большого объёма проверить `localStorage` поведение;
- для аккордеона потребуется новая зависимость;
- задача начинает тянуть бейджи, дашборд, маршруты или изменение CMS-логики.
