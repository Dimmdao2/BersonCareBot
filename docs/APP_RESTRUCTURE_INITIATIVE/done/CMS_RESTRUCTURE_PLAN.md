# CMS_RESTRUCTURE_PLAN — разделение CMS-разделов (Вариант C)

**Дата:** 2026-05-02. **Статус:** реализовано (вариант C: миграция `content_sections`, CMS-сайдбар и фильтры, patient-home кандидаты/валидация, защита slug, резолверы главной).

**Таксономия `system_parent_code` (канон для миграции и CHECK в БД):** допустимые не-null значения — `situations` | `sos` | `warmups` | `lessons`. Значение `null` — для статей (`kind=article`) и для системных слотов без «папки» в CMS (`emergency`, `materials`, `workouts`).

**«Мотивации» в CMS:** это отдельный маршрут `/app/doctor/content/motivation` и таблица `motivational_quotes`, **не** значение `system_parent_code` у `content_sections` в этом проходе. В сайдбаре остаётся ссылка «Мотивации»; системные папки в сайдбаре — Ситуации, SOS, Разминки, Уроки (без отдельной DB-папки «Мотивации»).

## Контекст и решение

В `content_sections` сейчас в одном плоском списке лежат тематические разделы для статей (Антистресс, Молодая мама, Здоровье 50+, Острая боль…) и системные слоты главной пациента (warmups, lessons, course_lessons, emergency, materials, …). Тип нигде не записан, всё держится на slug.

Принят **Вариант C** (см. обсуждение): визуальная иерархия через два новых поля у `content_sections`, без настоящей parent-иерархии в БД. Это не блокирует переход на Вариант A (настоящий `parent_id`) в будущем.

Системные верхние уровни в CMS (Ситуации, SOS, Разминки, Уроки) — это **логические папки**. В БД они не отдельная таблица, а только маркер `system_parent_code` у разделов `kind=system`, относящихся к кластеру. UI рисует иерархию по этому маркеру. «Мотивации» — отдельный экран, не кластер `content_sections` (см. шапку документа).

---

## Цель

- В CMS визуально отделить статьи от системного контента главной пациента.
- Раздел «Антистресс» (если он сейчас используется как Ситуация) лежит как подраздел системного кластера «Ситуации», а не вперемешку со статьями.
- На главной пациента редактор блоков выбирает кандидатов только из подразделов своего кластера (блок `situations` — из подразделов кластера `situations` и т.п.).
- Встроенные операционные slug защищены от переименования через UI и сервис.

---

## Что входит

- Миграция Drizzle: добавить у `content_sections` колонки `kind` (`article` | `system`) и `system_parent_code` (`situations` | `sos` | `warmups` | `lessons` | `null`). См. канон в шапке документа.
- Backfill известных slug — **ничего не переносится автоматически**:
  - `warmups` → `kind=system`, `system_parent_code=warmups`;
  - `lessons`, `course_lessons` → `kind=system`, `system_parent_code=lessons` (временно; концептуально уроки принадлежат курсам, переедут внутрь курса при переделке курсов);
  - `emergency`, `materials`, `workouts` → `kind=system`, `system_parent_code=null` (системные служебные слоты без своей папки в CMS);
  - **все тематические разделы** (антистресс / здоровье-50 / спортзал / острая-боль / дыхательная гимнастика / молодая мама / офисная работа / что делать при острой боли) → `kind=article`, **остаются как обычные статьи**.
- Системные папки «Ситуации» и «SOS» в CMS стартуют **пустыми**. Врач сам заходит в нужную папку и добавляет туда разделы (или убирает оттуда обратно в общие статьи). Никакой автоматической миграции.
- Перестройка sidebar CMS: «Статьи» с тематическими разделами + системные папки «Ситуации», «SOS», «Разминки», «Уроки»; ссылка «Мотивации» — как сейчас, отдельным пунктом (не `system_parent_code`).
- Список «Все страницы» в CMS — только страницы разделов с `kind=article`.
- На главной пациента (`/app/doctor/patient-home`) кандидаты для блоков `situations`, `sos`, `daily_warmup` ограничиваются разделами/страницами с соответствующим `system_parent_code`.
- Блок «Полезный пост» (`useful_post`) — **исключение**: кандидаты — **любая** опубликованная страница из разделов `kind='article'`, без привязки к системному кластеру.
- Создание раздела из контекста системного кластера CMS («Создать ситуацию» внутри папки «Ситуации»): новый раздел сразу получает соответствующий `system_parent_code` и не появляется в общем списке статей.
- Защита **встроенных** операционных slug (`warmups`, `lessons`, `course_lessons`, `emergency`, `materials`, `workouts`): в UI нет переименования для этого набора; `renameSectionSlug` отклоняет смену slug **только** для них. Пользовательские разделы с `kind=system` внутри папок кластера **могут** переименовываться (slug в форме редактирования по-прежнему не редактируется напрямую — смена через отдельный поток rename, где не immutable).

## Что НЕ входит

- Настоящая иерархия с `parent_id` (это Вариант A, отложен).
- Отдельная страница «Интерфейс клиента» (отказались в пользу единой CMS).
- Доработка курсов, тестов, шаблонов программ, рекомендаций (отдельные хвосты, см. [`CMS_AUDIT.md`](../CMS_AUDIT.md) §6).
- Перенос «Библиотеки файлов» из CMS в основное меню (отдельный шаг плана меню).
- Уменьшение плотности интерфейса, аккордеон меню (отдельные шаги).
- Изменение пациентских маршрутов `/app/patient/sections/[slug]` — остаются как есть.

## Шаги (каждый — отдельный проход с проверкой)

### Шаг 1. Решение по тематическим разделам

**Зафиксировано (backfill по умолчанию):**

| Slug / группа | `kind` | `system_parent_code` |
|---------------|--------|------------------------|
| `warmups` | `system` | `warmups` |
| `lessons`, `course_lessons` | `system` | `lessons` |
| `emergency`, `materials`, `workouts` | `system` | `null` |
| все остальные существующие строки | `article` | `null` |

Список slug для сверки с продом — `CMS_AUDIT.md` и `SELECT slug FROM content_sections` на момент миграции.

### Шаг 2. Миграция БД и backfill

- Drizzle schema: добавить `kind` и `system_parent_code` у `content_sections`.
- SQL миграция: `ALTER TABLE` + `CHECK` (`kind IN ('article','system')`).
- Backfill согласно решениям шага 1.
- Юнит-тест на backfill.
- UI не правится.

### Шаг 3. Сервис/порт

- `ContentSectionsPort` расширить методами с фильтром по `kind` и `system_parent_code`.
- Защита в сервисе: запрет смены slug у **встроенных immutable** slug (см. `IMMUTABLE_SYSTEM_SECTION_SLUGS` в модуле `content-sections`), а не у всех строк с `kind='system'`.
- Юнит-тесты.

### Шаг 4. CMS sidebar

- `ContentPagesSidebar`: дерево «Статьи» / системные папки. Системные папки определяются константой кодов на стороне UI/сервиса.
- Список «Все страницы» — только страницы из разделов с `kind='article'`.
- Smoke-тест на рендер.

### Шаг 5. Редактор содержимого системных разделов

- В CMS открытие системной папки показывает список её подразделов и страниц внутри них.
- Создание подраздела внутри системной папки автоматически проставляет `system_parent_code`.
- Юнит/RTL-тесты.

### Шаг 6. Главная пациента: фильтр кандидатов

- Picker блока `situations` — только разделы с `system_parent_code='situations'`.
- Picker блока `sos` — только разделы/страницы с соответствующим `system_parent_code`.
- Picker блока `daily_warmup` — страницы из разделов с `system_parent_code='warmups'`.
- Юнит-тесты на фильтр кандидатов.

### Шаг 7. Защита от переименования встроенных системных slug

- В UI формы раздела: поле slug не редактируется напрямую в режиме редактирования; для **immutable** встроенных slug скрыт/заблокирован поток переименования через диалог.
- В сервисе: при попытке `renameSectionSlug` для **immutable** slug — ошибка (пользовательские `kind=system` в папках кластера не попадают под этот запрет).
- Юнит-тест.

## Definition of Done

- В CMS sidebar видны «Статьи», системные папки (Ситуации, SOS, Разминки, Уроки) и отдельно ссылка «Мотивации» (не папка `content_sections`).
- Список «Все страницы» CMS содержит только страницы статей.
- Редактор главной пациента не показывает системные разделы как кандидатов «не своего» кластера.
- Встроенные операционные slug (immutable-набор) нельзя переименовать через UI и через сервис.
- Существующие данные не потеряны: backfill пометил их корректно; **шаблон** контрольного `SELECT` и инструкция для ops — в [`LOG.md`](../LOG.md) (запись «CMS Composer — реализация варианта C»); факт результата на конкретной БД команда при необходимости добавляет туда же строкой.
- Затронутые тесты webapp зелёные; перед push — `pnpm run ci`.

## Решения, принятые до старта (2026-05-02)

- Тематические разделы (Антистресс, Молодая мама, Здоровье 50+, Острая боль в спине, Дыхательная гимнастика, Домашний спортзал, Офисная работа) — **никакой автоперенос в системные папки**. Все остаются `kind='article'`. Врач сам решит, что отнести в «Ситуации» или «SOS», заходя в нужную системную папку и добавляя раздел туда вручную.
- «Полезный пост» (`useful_post`) — кандидаты остаются свободным выбором: любая опубликованная страница из article-разделов CMS. Не вводим отдельный системный кластер для этого блока.
- «Уроки» (`lessons` / `course_lessons`): концептуально принадлежат курсам, при переделке курсов переедут под управление карточкой курса. **Сейчас остаются как системная папка «Уроки» в CMS-сайдбаре** — `kind='system'`, `system_parent_code='lessons'`. Это ничего не ломает и убирается одной правкой при переделке курсов.
- Бонус-фича (включается, если не дорого): при создании раздела из системной папки («Создать ситуацию» внутри папки «Ситуации») новый раздел сразу получает соответствующий `system_parent_code` и автоматически выпадает из общего списка CMS-статей. См. шаг 5.
- Блок главной пациента `subscription_carousel` — **оставляем свободный выбор** в этом проходе. Кандидаты — любые опубликованные article-разделы / страницы / курсы (как сейчас). Отдельный системный кластер `subscription` не вводим. Если позже окажется, что подписочный контент путается со статьями, добавим кластер отдельным шагом.
