# Единый перехватываемый ствол доступа к БД (DB access chokepoint)

**Приоритет:** высокий
**От:** владелец (2026-06-17) — «всё это — отдельная задача ДО старта SAAS»
**Статус:** заготовка в `_TODO/` (не активна; стартует по команде владельца)

## Контекст
Мультитенантная SaaS-миграция (см. инициативу `SAAS_FOUNDATION/`) ставит Postgres RLS,
которому на каждый запрос нужно выставить tenant-принципала на соединении. Это возможно
только если **весь** доступ к БД проходит через **одну перехватываемую точку** на процесс.
Сейчас это не так — есть сырой SQL и пути в обход общего пула.

Доказательная база — аудит `../SAAS_FOUNDATION/RAW_SQL_AUDIT.md` (+ `scope-derivation/raw-sql-sites.tsv`).

## Ситуация (из аудита, факты)
- **Хорошо:** в webapp ~весь сырой SQL идёт через 2 моста — `runWebappPgText` (легаси `$1..$n`)
  и `runWebappSql`/`sql\`\``/`db.execute` — и садится на singleton `getDrizzle()` ⇒ **естественный
  ствол**. Интегратор аналогично через `runIntegratorSql` на свой пул.
- **Поверхность в обход одного ствола:**
  - **~30 выделенных `.connect()`-клиентов** (своя TCP-сессия — `SET LOCAL` на пуле туда не дойдёт),
    в т.ч. 2 в `app-layer/` и 1 в `modules/` (ещё и нарушение слоёв).
  - **3 отдельных пула в рантайме** (integrator `client.ts`, media-worker `main.ts`, webapp
    `platformUserFullPurge` → integratorPurgePools) + **boot-`migrate.ts`** пул + **ops-скрипты**.
  - **~22 port-байпасса**: сырой SQL в `modules/**` (10), `app-layer/**` (~8), routes/pages (4).
  - **3 байпассера `system_settings`** мимо аксессора (`pgBookingEngine`, `pgBookingRubitimeBridge`,
    `configAdapter`).

## Задача
Привести доступ к БД к **ЕДИНОМУ, защищённому CI-guard'ами, перехватываемому стволу** на каждый
процесс, при **НУЛЕВОМ изменении поведения**. Сырой SQL — только в `infra/repos`. Сделать так,
чтобы выставление принципала позже стало «однострочником» в стволе.

## Последовательность
**ПЕРВАЯ инициатива, строго ДО `SAAS_FOUNDATION`.** Она вынимает из SAAS-скоупа подготовительные
этапы (P0.5 часть / P0.6 / P0.7 / P0.11 + T0 connection-audit). После неё SAAS = `organization_id`
+ enrollment + RLS-политики + default-deny tiers + fixtures, а SAAS T0 = «принципал в готовом стволе + флип GUC».

## Границы (ВНЕ scope — это SAAS)
НЕ добавляем `organization_id`; НЕ пишем RLS-политики; НЕ трогаем мультитенантную семантику,
enrollment, tiers. Только рефактор доступа к данным с сохранением поведения.

## Боли / критерии приёмки
- Любой новый **сырой SQL** / **`.connect()`** / **`new Pool`** мимо санкционированных хелперов
  и провайдеров → **падает CI**.
- **100% доступа к БД перечислимо** идёт через ствол/провайдер/`withClient`; «беглецов» нет
  (кроме явно перечисленных отдельных процессов, у которых появился dormant hook для принципала).
- Полный CI зелёный; ключевые страницы рендерятся идентично (поведение не изменилось).
