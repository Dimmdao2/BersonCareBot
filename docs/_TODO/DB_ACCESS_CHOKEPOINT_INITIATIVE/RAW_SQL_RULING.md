# Raw-SQL ruling — два независимых Opus-ревьюера (2026-06-17)

Два свежих независимых Opus-агента (A и B; ни один не видел другого) разобрали вопрос
«оставить сырой SQL или мигрировать на Drizzle?» + нюанс владельца + планку владельца.
**Оба сошлись (HIGH confidence).** Полные вердикты: `/home/dev/orch/raw-sql-verdict-A.md`, `…-B.md`.

## Консенсус (вердикт)
1. **НЕ делать сплошной переписи сырой SQL → Drizzle-builder.** Отрицательная ценность: уцелевшие
   сырые фрагменты — замороженные/покрытые тестами очереди и CTE, уже идущие через типизированный транспорт.
2. **Планка «общего порта» уже выполнена на ~90%.** Webapp: всё → `getDrizzle()` → один singleton `getPool()`.
   Интегратор: всё → `runIntegratorSql` → один `DbPort`/`Pool`. **Это и есть общий порт.** Значит планка
   владельца («обёрнуто + через общий порт, без разбросанного прямочтения») в основном уже соблюдена;
   реальная работа = **достроить chokepoint, закрыв небольшой перечислимый набор байпассов** — ровно эта
   инициатива — а НЕ миграция SQL.
3. **Планка владельца — правильная рамка и есть сам предмет работы.**

## Нюанс владельца (legacy-таблицы интегратора / миграция в public-схему) — сведено
Ревьюеры назвали его по-разному (A: «неверно как сформулировано»; B: «верно, это суть»), но **сходятся в действии**:
- Drizzle-моделить таблицы интегратора можно **на месте** — жёсткого «сначала перенести схему» нет
  (`pgSchema(` не используется нигде; обычный `pgTable` работает). Т.е. ORM-работа технически **не заблокирована**
  расположением схемы (A).
- НО **бессмысленно** Drizzle-моделить legacy-таблицы `integrator`-схемы (`identities`, `conversations`,
  `message_drafts`, `telegram_state`, `users`, `contacts`, `rubitime_*`): у владеющего процесса интегратора
  **нет Drizzle-моделей** для них, доступ — через неквалифицированные имена + `search_path`, и они помечены
  move-to-public/deprecate (LOG Drizzle-миграции, фаза 08). → любую ORM-миграцию там надо **СВЯЗАТЬ с решением
  о консолидации схемы / жизненном цикле** (legacy умирают после sunset, не мигрировать ради ORM) (B).
- Сырой SQL по `public.*` такой зависимости НЕ имеет → конвертировать оппортунистически (или просто держать
  обёрнутым — он уже через порт).
- **Латентный баг (отмечен):** одна Drizzle-модель таблицы интегратора ошибочно размещена как `public` — починить при консолидации.

→ **Итог:** инстинкт владельца («нет смысла мигрировать на Drizzle без переноса таблиц интегратора в public»)
**директивно верен для legacy-зоны интегратора**; решать это в рамках консолидации схемы, не как ORM-рутину.

## Настоящие байпассы — ЧИНИТЬ (независимо от raw-vs-Drizzle) — объединение A+B
Это и есть нарушения планки владельца = scope инициативы chokepoint:
- **Отдельный пул в процессе:** `apps/webapp/src/infra/platformUserFullPurge.ts:77` (`new PgPool`, search_path интегратора).
- **Процессные / миграционный пулы:** integrator `client.ts`, media-worker `main.ts`, `migrate.ts:307` (boot-миграции), пулы ops-скриптов → per-process identity + решение по роли.
- **~30 выделенных клиентов `.connect()`** — пул центральный, но RLS-контекст надо ставить на КЛИЕНТ → единый `withRlsClient()`. (Крупнейший SaaS-пункт.)
- **SQL не в том слое (port-bypass):** `modules/auth/channelLink.ts` (импортит infra `getPool`), выделенные клиенты в `app-layer` (`createDoctorClient`, `messengerPhoneHttpBindExecute`), сырой SQL в ~4 routes/pages.
- **Байпассеры чтения `system_settings` (×3)** — это DRY/консолидация, не chokepoint (уже через пул).
- **НОВОЕ (нашёл ревьюер A, мой аудит пропустил):** сырой `getPool().query("SELECT role FROM platform_users …")` в `app/api/doctor/clients/[userId]/permanent-delete/route.ts:37` + `modules/doctor-clients/clientArchiveChange.ts:24`.

## Одно честное расхождение (не блокирующее)
Разница была только интерпретационная (true/false-ярлык нюанса выше); рекомендуемое ДЕЙСТВИЕ — единогласно.
По правилу владельца («если сошлись → ок») это **зелёный вердикт**: строить эту инициативу (chokepoint),
не переписывать SQL; legacy-интегратор связать с консолидацией схемы.
