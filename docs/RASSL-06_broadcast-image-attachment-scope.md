# RASSL-06 — Прикрепление картинки к рассылке (scope + решения)

**Статус:** заблокировано на решении владельца (taskdb bcb #21). Автономный проход 2026-06-24 провёл полную разведку пайплайна; код НЕ менял — фича большая и кросс-аппная, нужен продуктовый выбор объёма, чтобы не строить лишнее.

## Что уже есть
- **UI пикер медиа** — `MediaPickerPanel` / `MediaPickerShell` (desktop Dialog / mobile bottom-sheet), отдаёт `{ id, url, kind, mimeType, … }`. Загрузка картинок: `POST /api/media/upload` → S3, возвращает `{ mediaId, url }`. Переиспользуется в упражнениях/контенте.
- **In-app чат** — таблица `support_conversation_messages` УЖЕ имеет колонки `media_url` / `media_type` (на чтение сериализуется), но порт записи `appendWebappMessage` их не принимает.
- **Markdown-форматирование рассылки** (жирный/списки/строки) — сделано отдельно (task #20).

## Чего НЕТ (net-new работа)
- **Отправка фото в мессенджер отсутствует целиком**: в `apps/integrator/src` нет `sendPhoto/sendMedia`; контракт `unifiedMessage.ts` не имеет поля image/photo/media/attachment. Telegram/MAX-адаптеры умеют только `text`.
- У `BroadcastCommand.message` нет поля медиа; нет колонки в аудите рассылок; в draft-порте нет медиа; в `BroadcastForm` нет поля «картинка».

## Объём по каналам
| Канал | Сейчас | Работа | Размер |
|---|---|---|---|
| Telegram | только текст | новый путь `sendPhoto` (фото+caption), поле в контракте, проброс из deliveryJobs | средне |
| MAX | только текст | вложение-картинка; **поддержка MAX API не подтверждена** | средне + риск |
| In-app чат | колонки media есть | расширить `appendWebappMessage(mediaUrl, mediaType)` + проброс | мелко |
| Email | text/html | inline `<img>` в HTML-теле | мелко |
| Web-push | без image | поле `imageUrl` в payload + service worker | мелко |
| SMS | — | невозможно (текст-протокол) | — |

## Кросс-каттинг (общее)
схема `BroadcastCommand` + zod → `BroadcastForm` (пикер) → draft-порт → колонка `broadcast_audit.media_id` (миграция) → контракт `unifiedMessage.imageUrl` (2 приложения) → `deliveryJobs` проброс → адаптеры.

## РЕШЕНИЯ владельцу (без них не начинаем)
1. **Какие каналы обязаны показать картинку?** Рекоменд.-минимум: **Telegram + in-app чат + email** (там, где пациенты реально читают; быстрый высокий эффект). MAX и web-push — отложить (MAX API под вопросом). SMS — невозможно.
2. **Картинка опциональна, и если её отправка упала — текст всё равно уходит?** Рекоменд.: да (fallback на текст).
3. **Одна картинка на рассылку?** Рекоменд.: да (одна, ≤ ~5 МБ, jpg/png).
4. **MAX обязателен в первой версии?** Рекоменд.: нет (вынести в отдельную задачу после подтверждения MAX API).

## Рекомендованный план (после решения)
- **Фаза 1 — фундамент:** поле `mediaId` в команде/схеме/draft, пикер в `BroadcastForm`, колонка аудита, проброс в сервисе.
- **Фаза 2 — in-app чат:** расширить `appendWebappMessage` (media), привязать к рассылке (мелко, БД готова).
- **Фаза 3 — Telegram:** `imageUrl` в контракт, `sendPhoto` в адаптер, fallback на текст.
- **Фаза 4 — email:** inline `<img>` в HTML.
- (**Фаза 5 — MAX/web-push:** отдельно, по подтверждению API.)

Минимальный ценный срез = Фазы 1–4. После «да» владельца по объёму — разбить на под-задачи bcb и вести по фазам.
