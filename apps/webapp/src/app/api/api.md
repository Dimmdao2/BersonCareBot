# api

Обработчики HTTP API вебаппа (Next.js route handlers).

- **auth/** — авторизация: обмен токена интегратора на сессию (exchange), вход по Telegram initData (`telegram-init`), вход по MAX Mini App initData (`max-init`), выход (logout).
- **integrator/** — эндпоинты для бота/интегратора: приём событий (events), завершение channel-link (`channel-link/complete`), **опционально** привязка телефона по подписи для внешнего вызывающего (`messenger-phone/bind` — не hot path при unified DB), дневники (symptom-trackings, lfk-complexes), напоминания (reminders/dispatch). Доступ по подписи запроса (общий секрет). Для `POST /api/integrator/events`: типизированные `MergeConflictError` / `MergeDependentConflictError` из слоя проекций **не** приводят к бесконечному retry (`503`): при наличии `conflictAudit` запись в `admin_audit_log` (`auto_merge_conflict` или `auto_merge_conflict_anomaly`) и ответ **`202` accepted**; прочие ошибки ingestion остаются retryable (`503`) или `422` по прежним правилам. См. `docs/ARCHITECTURE/PLATFORM_USER_MERGE.md` (projection conflicts). **Идемпотентность:** ключ в `idempotency_keys` считается по **семантическому** телу события (стабильный JSON без шумовых полей вроде верхнеуровневого `occurredAt`); подробнее — `apps/webapp/src/modules/auth/auth.md` → «Integrator → webapp».
- **media/** — `GET /api/media/[id]`: требуется **сессия** (любая роль); без сессии — **401**. При записи в БД с `s3_key` — **302** на presigned GET в **private** MinIO-бакет (`S3_PRIVATE_BUCKET`); без прямых публичных URL MinIO в ответе. Модель угроз и internal purge — `docs/REPORTS/S3_PRIVATE_MEDIA_EXECUTION_LOG.md`, nginx для `/api/internal/` — `deploy/HOST_DEPLOY_README.md`. **Легаси / малые файлы (single PUT):** `POST /api/media/presign` (JSON `filename`, `mimeType`, `size`, опционально `folderId` UUID или `null` для корня) → `{ ok, mediaId, uploadUrl, readUrl }`, затем клиент PUT на `uploadUrl`, затем `POST /api/media/confirm` с `{ mediaId }` → `{ ok, url }`. **Крупные файлы (multipart direct-to-S3, без отдельного `confirm`):** `GET /api/media/s3-status` → `{ s3Multipart: boolean, ... }`; при `s3Multipart: true` — `POST /api/media/multipart/init` → `POST /api/media/multipart/part-url` (presigned **UploadPart**) → PUT частей с чтением **ETag** из ответа → `POST /api/media/multipart/complete` с `parts: [{ PartNumber, ETag }]` → `{ ok, url: /api/media/{id} }` и строка `media_files` в `ready`. Операции **`complete` / `abort` / internal cleanup** по одному `sessionId` сериализуются **advisory lock** на `sessionId` (см. `infra/multipartSessionLock.ts`), чтобы не было гонок с финализацией и отменой. **`complete`:** при повторном запросе, если сессия уже в `completing` (например обрыв после успешного S3 `CompleteMultipartUpload`), повторно **не** вызывается S3 complete — выполняется проверка объекта (Head) и идемпотентная финализация в БД; ошибки: `404 session_not_found`, `409 session_expired`, `409 session_state_conflict` (если `complete` недоступен: нет строки, истёк TTL, или терминальный/неконсистентный статус), `409 finalize_inconsistent_state`, `409 integrity_mismatch`, `400 invalid_parts`, `500 finalize_failed` (в теле может быть `retryable: true` — клиент может повторить `complete` с тем же `parts`). **`POST /api/media/multipart/abort`:** в транзакции с тем же lock — `SELECT … FOR UPDATE`, при необходимости удаление `pending` `media_files` (CASCADE убирает строку сессии), затем best-effort `AbortMultipartUpload` в S3 по сохранённым ключам; ответы `alreadyGone` / `alreadyCompleted` / `alreadyFinal` — идемпотентность. **`part-url`:** `404 session_not_found`; `409 session_expired` (истёк TTL); `409 session_state_conflict` (статус не `initiated`/`uploading`); `400 part_out_of_range`. Лимит размера и MIME — [`uploadAllowedMime.ts`](../../modules/media/uploadAllowedMime.ts) (до **3 GiB**). Fallback без полного S3: `POST /api/media/upload`, multipart `file`, опционально поле **`folderId`** (UUID папки из `media_folders` или строка `root` для корня; несуществующая папка — `400 folder_not_found`), только роль doctor/admin (до ~50 МБ). Прокси-загрузка: MIME из белого списка (изображения incl. HEIC/AVIF/TIFF/SVG, видео incl. WebM, аудио incl. Ogg/AAC/M4A, документы Office Open XML и legacy OLE, `text/plain`, `text/csv`; для SVG — не инлайнить сырой XML в `<img>`). Ошибки: `413`, `415`, `403`, `409`, при сбое S3 — `502`/`500` по сценарию.
- **admin/media/** — библиотека файлов CMS (`GET /api/admin/media`) с фильтрацией/сортировкой, опционально **`folderId`** (UUID) и **`includeDescendants=true`** (рекурсивный листинг поддерева). `PATCH /api/admin/media/[id]`: `displayName` (строка или `null`), опционально **`folderId`** (перенос в папку). Удаление (`DELETE /api/admin/media/[id]`) с precheck использования в `content_pages` и подтверждением `confirmUsed=true`. После подтверждения запись переводится в очередь на фоновое удаление в S3; ответ содержит `scheduled: true`. **`/api/admin/media/folders`:** `GET` (дерево или `?flat=true`), `POST` создание; **`/api/admin/media/folders/[id]`:** `PATCH` (имя, `parentId`), `DELETE` (только пустая папка — иначе `409`). Экран сбойных удалений: `GET /api/admin/media/delete-errors` (`items`, `total`); доступ doctor/admin.
- **internal/media-pending-delete/purge** — `POST` с `Authorization: Bearer <INTERNAL_JOB_SECRET>` обрабатывает батч строк `media_files` в статусе **`pending_delete`** или застрявшем **`deleting`** (удаление объекта в S3 и строки в БД; при ошибке S3 — **retry** с `delete_attempts` и `next_attempt_at`). Ответ: `{ ok, removed, errors }`. Параметр `?limit=` (по умолчанию 25). Без `INTERNAL_JOB_SECRET` в env — `503`.
- **internal/media-multipart/cleanup** — `POST` с тем же Bearer: истёкшие активные строки `media_upload_sessions` обрабатываются **по одному `sessionId`** под тем же advisory lock, что и user-facing `abort`: удаление `pending` `media_files` при возможности (CASCADE убирает сессию), иначе пометка сессии `expired`; затем best-effort **AbortMultipartUpload** в S3. Ответ `{ ok, cleaned, errors }`, `?limit=` (по умолчанию 25, cap в репозитории). Рекомендуется cron на loopback рядом с purge — см. `deploy/HOST_DEPLOY_README.md`.
- **admin/audit-log** — `GET /api/admin/audit-log` — пагинированный журнал `admin_audit_log` (операции админки и деструктивные действия по мере подключения сценариев). Доступ: `role === admin`, включённый admin mode (`requireAdminModeSession`). Параметры: `page`, `limit`, `action`, `target`, **`involvesPlatformUserId`** (UUID — строки, где `target_id` совпадает, или для `action=auto_merge_conflict` пользователь в `details.candidateIds`, или для `action=user_merge` совпадает `details.targetId` / `details.duplicateId`), `status`, `from`, `to` (даты `YYYY-MM-DD`). В ответе дополнительно **`openAutoMergeConflictCount`**: число **открытых** строк `auto_merge_conflict` (`resolved_at IS NULL`) для бейджа в UI. См. `docs/REPORTS/STRICT_PURGE_MANUAL_MERGE_EXECUTION_LOG.md`.
- **admin/system-health** — `GET /api/admin/system-health`, только **admin + admin mode** (`requireAdminModeSession`). Агрегирует косвенные health-сигналы в стабильный контракт: `webappDb` (локальная DB проверка), `integratorApi` (проба `{INTEGRATOR_API_URL}/health`), `projection` (проба `/health/projection` через proxy helper), `fetchedAt`. Пробы запускаются параллельно (`Promise.allSettled`), ошибки нормализуются в `status: unreachable | error` без раскрытия внутренних ответов внешних сервисов; логируются только технические поля `probe`, `status`, `durationMs`, `errorCode`.
- **doctor/clients/[userId]/permanent-delete** — `POST`, тело `{ confirmUserId: uuid }` (должен совпадать с `:userId`). Только **admin + admin mode**, клиент в архиве, `role === client`. Вызывает strict purge (`runStrictPurgePlatformUser`): advisory lock, preflight артефактов, S3/integrator post-commit, запись `user_purge` в `admin_audit_log`. Для повторной внешней очистки audit хранит `artifact`, `phoneNormalized`, `webappIntegratorUserId` и `resolvedIntegratorUserIds`; повтор через `retryStrictPurgeExternalCleanup` пишет отдельно `user_purge_external_retry`. Успех: `{ ok: true, outcome, integratorSkipped, details }` (`outcome`: `completed` | `partial_failed` | `needs_retry`). Ошибки: `400/404/409/500` (`purge_transaction_failed` при сбое транзакции webapp).
- **doctor/clients/[userId]/merge-candidates** — `GET`, только **admin + admin mode**. Список других канонических клиентов, потенциально дубликатов якоря (общий телефон, email, `integrator_user_id` или общий messenger binding). Query: опционально `q` (подстроковый фильтр). Ошибки: `400 not_client`, `404`, `409 anchor_is_alias`. См. `docs/ARCHITECTURE/PLATFORM_USER_MERGE.md` (Manual merge — preview).
- **doctor/clients/merge-preview** — `GET`, query `targetId`, `duplicateId` (оба UUID). Только **admin + admin mode**. Read-only preview ручного merge: профили (camelCase), биндинги, OAuth, dependent counts, hard blockers, конфликты, auto-merge поля, `recommendation`, `mergeAllowed`, **`v1MergeEngineCallable`**, **`platformUserMergeV2Enabled`** (значение admin-настройки `platform_user_merge_v2_enabled` на момент запроса). **Флаг выкл. (v1):** при двух разных non-null `integratorUserId` — hard blocker `different_non_null_integrator_user_id` (как раньше). **Флаг вкл. (v2):** для той же пары preview вызывает integrator M2M `canonical-pair`; если canonical id совпали после integrator merge — блокер снимается; иначе `integrator_canonical_merge_required`; при отсутствии конфигурации integrator M2M или timeout/недоступности ответа — `integrator_merge_status_unavailable`. Ошибки: `400` (`invalid_query`, `same_id`, `not_client`), `404 missing_user`. Контракт: `docs/ARCHITECTURE/PLATFORM_USER_MERGE.md`, v2 пакет: `docs/PLATFORM_USER_MERGE_V2/STAGE_5_FEATURE_FLAG_AND_FLOW_SWITCH.md`.
- **doctor/clients/integrator-merge** — `POST`, только **admin + admin mode**. Тело `{ targetId, duplicateId, dryRun?: boolean }`. Требует **`platform_user_merge_v2_enabled`** в `system_settings` (admin); иначе `400 feature_disabled`. Обе строки — канонические клиенты с **разными** non-null `integrator_user_id`; иначе `400 integrator_ids_not_divergent`. Перед M2M вызовом route перечитывает обе строки `platform_users` под короткой транзакцией `FOR UPDATE`, чтобы winner/loser integrator ids не drift'нули посреди запроса. Проксирует HMAC `POST` на integrator `/api/integrator/users/merge` (winner = integrator id **целевого** platform user, loser = **дубликата**). Ошибки integrator пробрасываются в JSON (`integrator_merge_failed` + `details`); без URL/секрета — `503 integrator_unconfigured`; при timeout — `503 integrator_timeout`.
- **doctor/clients/merge** — `POST`, только **admin + admin mode**. Тело JSON: `{ resolution: ManualMergeResolution }` (см. `apps/webapp/src/infra/repos/manualMergeResolution.ts` и архитектурный doc): явный выбор победителей по скалярам, `telegram`/`max`/`vk`, OAuth по провайдеру, стратегия `channelPreferences`. `resolution.bindings.* = "both"` допустим только для каналов **без** конфликта; при конфликтных `external_id` backend требует `target` или `duplicate`. Gate по `integrator_user_id`: **v1** (флаг выкл.) при двух разных id **не** отвечает отдельным `409` — как до Stage 5, `mergePlatformUsersInTransaction` бросает `MergeConflictError`, `runManualPlatformUserMerge` возвращает ошибку и пишет audit на фазе транзакции, HTTP **`409`** `merge_failed` с `message` вида `merge: two different non-null integrator_user_id`. **v2:** до транзакции M2M `canonical-pair`; relaxed merge допустим только если snapshot проверенной пары совпал с `integrator_user_id`, перечитанными под lock в merge tx. При drift сервер отвечает `409` с `code` `integrator_ids_changed_since_gate`; при несогласованной canonical паре — `409`/`502`/`503` с `code` `integrator_canonical_merge_required`, `integrator_canonical_status_failed`, `integrator_merge_status_unavailable`. Выполняет `runManualPlatformUserMerge` (dual advisory lock, merge tx, перенос `media_files.uploaded_by`, `media_upload_sessions.owner_user_id`, `email_verified_at`, **`user_merge`** в `admin_audit_log` после commit). Сервер дополнительно отклоняет пары, где хотя бы одна строка не `role === client`. Успех: `{ ok: true, targetId, duplicateId }`. Неверное тело: `400` (`invalid_body`).
- **health/** — проверка работоспособности сервиса.
- **me/** — данные текущего пользователя (если нужны для клиента).
- **menu/** — меню по роли (если используется отдельным запросом).

Роуты остаются тонкими: проверка прав и параметров, вызов сервисов из `buildAppDeps()` или модулей.
