# api

Обработчики HTTP API вебаппа (Next.js route handlers).

**Граница импортов:** файлы `**/route.ts` здесь не тянут `@/infra/*` напрямую — доступ к инфраструктуре через `@/app-layer/**` (фасады) и `buildAppDeps()`; см. `docs/archive/2026-04-docs-cleanup/test-api-di-optimization/api-di-boundary-normalization/ALLOWLIST_REMAINING_INFRA_ROUTE_IMPORTS.md` и § «API route import-policy» в `docs/ARCHITECTURE/LOW_LEVEL_ARCHITECTURE_AUDIT_AND_REORG.md`.

- **auth/** — авторизация: обмен токена интегратора на сессию (exchange), вход по Telegram initData (`telegram-init`), вход по MAX Mini App initData (`max-init`), выход (logout).
- **admin/settings** — `GET` / `PATCH` настроек `scope=admin` (роль `admin`). Для **`video_default_delivery`** допустимы только **`mp4`**, **`hls`**, **`auto`** (сохраняется в нижнем регистре); иначе **400** `invalid_value`.
- **integrator/** — эндпоинты для бота/интегратора: приём событий (events), завершение channel-link (`channel-link/complete`), **опционально** привязка телефона по подписи для внешнего вызывающего (`messenger-phone/bind` — не hot path при unified DB), дневники (symptom-trackings, lfk-complexes), напоминания (reminders/dispatch). **`GET /api/integrator/diary/lfk-complexes`:** при **`includeTreatmentPrograms=true`** в query добавляется массив **`treatmentProgramLfkBlocks`** — элементы **`lfk_complex`** из **активных** экземпляров программ пациента (`SYSTEM_LOGIC_SCHEMA.md` §11; заголовок из `snapshot`). Доступ по подписи запроса (общий секрет). Подписанные **GET** integrator (кластер G трека B) проходят guard через `assertIntegratorGetRequest` в `@/app-layer/integrator/assertIntegratorGetRequest` — заголовки `x-bersoncare-timestamp` и `x-bersoncare-signature`, без прямого импорта `@/infra/webhooks/*` из `route.ts`; крипто-проверка остаётся в infra внутри этого модуля. Для `POST /api/integrator/events`: типизированные `MergeConflictError` / `MergeDependentConflictError` из слоя проекций **не** приводят к бесконечному retry (`503`): при наличии `conflictAudit` запись в `admin_audit_log` (`auto_merge_conflict` или `auto_merge_conflict_anomaly`) и ответ **`202` accepted**; прочие ошибки ingestion остаются retryable (`503`) или `422` по прежним правилам. См. `docs/ARCHITECTURE/PLATFORM_USER_MERGE.md` (projection conflicts). **Идемпотентность:** ключ в `idempotency_keys` считается по **семантическому** телу события (стабильный JSON без шумовых полей вроде верхнеуровневого `occurredAt`); подробнее — `apps/webapp/src/modules/auth/auth.md` → «Integrator → webapp».
- **media/** — `GET /api/media/[id]`: требуется **сессия** (любая роль); без сессии — **401**. При записи в БД с `s3_key` — **307** на presigned GET в **private** MinIO-бакет (`S3_PRIVATE_BUCKET`); срок подписи — admin **`video_presign_ttl_seconds`** (миграция **`0023`**, по умолчанию 3600 с, допустимый диапазон 60…604800); без прямых публичных URL MinIO в ответе. Модель угроз и internal purge — `docs/REPORTS/S3_PRIVATE_MEDIA_EXECUTION_LOG.md`, nginx для `/api/internal/` — `deploy/HOST_DEPLOY_README.md`. **Легаси / малые файлы (single PUT):** `POST /api/media/presign` (JSON `filename`, `mimeType`, `size`, опционально `folderId` UUID или `null` для корня) → `{ ok, mediaId, uploadUrl, readUrl }`, затем клиент PUT на `uploadUrl`, затем `POST /api/media/confirm` с `{ mediaId }` → `{ ok, url }`. При **`video_hls_pipeline_enabled`** и **`video_hls_new_uploads_auto_transcode`** (admin) для `video/*` после **первичного** успешного confirm (переход в `ready`) ставится job в **`media_transcode_jobs`** идемпотентно; при выключенных флагах или не-видео — нет; повтор confirm при уже **`ready`** не вызывает enqueue. **Крупные файлы (multipart direct-to-S3, без отдельного `confirm`):** `GET /api/media/s3-status` → `{ s3Multipart: boolean, ... }`; при `s3Multipart: true` — `POST /api/media/multipart/init` → `POST /api/media/multipart/part-url` (presigned **UploadPart**) → PUT частей с чтением **ETag** из ответа → `POST /api/media/multipart/complete` с `parts: [{ PartNumber, ETag }]` → `{ ok, url: /api/media/{id} }` и строка `media_files` в `ready`. Тот же **авто-enqueue** транскода для `video/*`, что и после `confirm`, при тех же двух admin-флагах. Операции **`complete` / `abort` / internal cleanup** по одному `sessionId` сериализуются **advisory lock** на `sessionId` (см. `infra/multipartSessionLock.ts`), чтобы не было гонок с финализацией и отменой. **`complete`:** при повторном запросе, если сессия уже в `completing` (например обрыв после успешного S3 `CompleteMultipartUpload`), повторно **не** вызывается S3 complete — выполняется проверка объекта (Head) и идемпотентная финализация в БД; ошибки: `404 session_not_found`, `409 session_expired`, `409 session_state_conflict` (если `complete` недоступен: нет строки, истёк TTL, или терминальный/неконсистентный статус), `409 finalize_inconsistent_state`, `409 integrity_mismatch`, `400 invalid_parts`, `500 finalize_failed` (в теле может быть `retryable: true` — клиент может повторить `complete` с тем же `parts`). **`POST /api/media/multipart/abort`:** в транзакции с тем же lock — `SELECT … FOR UPDATE`, при необходимости удаление `pending` `media_files` (CASCADE убирает строку сессии), затем best-effort `AbortMultipartUpload` в S3 по сохранённым ключам; ответы `alreadyGone` / `alreadyCompleted` / `alreadyFinal` — идемпотентность. **`part-url`:** `404 session_not_found`; `409 session_expired` (истёк TTL); `409 session_state_conflict` (статус не `initiated`/`uploading`); `400 part_out_of_range`. Лимит размера и MIME — [`uploadAllowedMime.ts`](../../modules/media/uploadAllowedMime.ts) (до **3 GiB**). Fallback без полного S3: `POST /api/media/upload`, multipart `file`, опционально поле **`folderId`** (UUID папки из `media_folders` или строка `root` для корня; несуществующая папка — `400 folder_not_found`), только роль doctor/admin (до ~50 МБ). Прокси-загрузка: MIME из белого списка (изображения incl. HEIC/AVIF/TIFF/SVG, видео incl. WebM, аудио incl. Ogg/AAC/M4A, документы Office Open XML и legacy OLE, `text/plain`, `text/csv`; для SVG — не инлайнить сырой XML в `<img>`). Ошибки: `413`, `415`, `403`, `409`, при сбое S3 — `502`/`500` по сценарию.
- **`GET /api/media/[id]/preview/[size]`** — превью изображения (`sm`|`md`); при редиректе на presigned GET для объекта в private-бакете срок подписи — **`video_presign_ttl_seconds`** (как у **`GET /api/media/[id]`** и playback JSON).
- **`GET /api/media/[id]/playback`** — VIDEO_HLS_DELIVERY phase-04: JSON-дескриптор воспроизведения при **`video_playback_api_enabled=true`** в `system_settings` (иначе **503** `feature_disabled`). Требуется **сессия** (любая роль), как у `GET /api/media/[id]`. Опционально **`?prefer=mp4|hls|auto`** — только для **`role=admin`**. Резолв: `video_default_delivery` (после миграции **`0022`** значение по умолчанию в БД — **`auto`**) → `media_files.video_delivery_override` → admin `prefer`. Для видео с HLS: `delivery: "hls"`, `hls.masterUrl` (presigned), опционально `posterUrl` и `hls.qualities`; всегда `mp4.url` = **`/api/media/{id}`** (прогрессивный источник / fallback). `fallbackUsed` если запрошен HLS, а выдан MP4. Не-видео: `delivery: "file"`. Если presign **постера** падает при уже успешном master, **`delivery`** остаётся **`hls`**, **`posterUrl`** в JSON будет **`null`** (клиент не должен считать постер обязательным при HLS). **`expiresInSeconds`** в теле — TTL presigned URL (**`video_presign_ttl_seconds`**, как у redirect `GET /api/media/[id]`). Клиент обновляет JSON при длинном просмотре / истечении подписи (phase-09). Логи `playback_resolved` / `playback_presign_failed` без полных presigned URL.
- **admin/media/** — библиотека файлов CMS (`GET /api/admin/media`) с фильтрацией/сортировкой, опционально **`folderId`** (UUID) и **`includeDescendants=true`** (рекурсивный листинг поддерева). Элементы `items[]` могут включать опциональные поля метаданных HLS/транскода (`videoProcessingStatus`, `hlsMasterPlaylistS3Key`, …) после миграции БД — см. `modules/media/types.ts`; до pipeline значения **null**. `PATCH /api/admin/media/[id]`: `displayName` (строка или `null`), опционально **`folderId`** (перенос в папку). Удаление (`DELETE /api/admin/media/[id]`) с precheck использования в `content_pages` и подтверждением `confirmUsed=true`. После подтверждения запись переводится в очередь на фоновое удаление в S3; ответ содержит `scheduled: true`. **`/api/admin/media/folders`:** `GET` (дерево или `?flat=true`), `POST` создание; **`/api/admin/media/folders/[id]`:** `PATCH` (имя, `parentId`), `DELETE` (только пустая папка — иначе `409`). Экран сбойных удалений: `GET /api/admin/media/delete-errors` (`items`, `total`); доступ doctor/admin.
- **internal/media-pending-delete/purge** — `POST` с `Authorization: Bearer <INTERNAL_JOB_SECRET>` обрабатывает батч строк `media_files` в статусе **`pending_delete`** или застрявшем **`deleting`** (удаление объекта в S3 и строки в БД; при ошибке S3 — **retry** с `delete_attempts` и `next_attempt_at`). Ответ: `{ ok, removed, errors }`. Параметр `?limit=` (по умолчанию 25). Без `INTERNAL_JOB_SECRET` в env — `503`.
- **internal/media-multipart/cleanup** — `POST` с тем же Bearer: истёкшие активные строки `media_upload_sessions` обрабатываются **по одному `sessionId`** под тем же advisory lock, что и user-facing `abort`: удаление `pending` `media_files` при возможности (CASCADE убирает сессию), иначе пометка сессии `expired`; затем best-effort **AbortMultipartUpload** в S3. Ответ `{ ok, cleaned, errors }`, `?limit=` (по умолчанию 25, cap в репозитории). Рекомендуется cron на loopback рядом с purge — см. `deploy/HOST_DEPLOY_README.md`.
- **internal/media-transcode/enqueue** — `POST` JSON `{ mediaId: uuid }` с `Authorization: Bearer <INTERNAL_JOB_SECRET>`. Ставит строку в `media_transcode_jobs`, если включён admin-флаг **`video_hls_pipeline_enabled`** (иначе **`503`** `pipeline_disabled`), файл — `video/*`, в библиотеке (не `pending` / delete pipeline), с `s3_key`. Идемпотентно при уже активной job (`alreadyQueued`). Уже готово HLS (`video_processing_status=ready` и master в БД) → **`200`** `{ ok: true, skipped: "already_ready" }`. Тяжёлый FFmpeg **не** выполняется в Next.js — обработка в **`apps/media-worker`**.
- **internal/media-playback-stats/retention** — `POST` с `Authorization: Bearer <INTERNAL_JOB_SECRET>`. Обрезка **`media_playback_stats_hourly`**: удаляются строки с `bucket_hour` старше `days` суток (**`?days=`**, по умолчанию **`90`**, минимум 1). **`?dryRun=1`** или **`?dry_run=1` / `true`** — только подсчёт (**`deleted`** = число попавших под порог строк без DELETE). Таблица **`media_playback_user_video_first_resolve`** **не** затрагивается. Ответ **`{ ok, deleted, dryRun, retentionDays }`**, лог **`media_playback_stats_retention_job`**. Cron на loopback — `deploy/HOST_DEPLOY_README.md`.
- **admin/audit-log** — `GET /api/admin/audit-log` — пагинированный журнал `admin_audit_log` (операции админки и деструктивные действия по мере подключения сценариев). Доступ: `role === admin`, включённый admin mode (`requireAdminModeSession`). Параметры: `page`, `limit`, `action`, `target`, **`involvesPlatformUserId`** (UUID — строки, где `target_id` совпадает, или для `action=auto_merge_conflict` пользователь в `details.candidateIds`, или для `action=user_merge` совпадает `details.targetId` / `details.duplicateId`), `status`, `from`, `to` (даты `YYYY-MM-DD`). В ответе дополнительно **`openAutoMergeConflictCount`**: число **открытых** строк `auto_merge_conflict` (`resolved_at IS NULL`) для бейджа в UI. См. `docs/REPORTS/STRICT_PURGE_MANUAL_MERGE_EXECUTION_LOG.md`.
- **admin/system-health** — `GET /api/admin/system-health`, только **admin + admin mode** (`requireAdminModeSession`). Агрегирует косвенные health-сигналы в стабильный контракт: `webappDb` (локальная DB проверка), `integratorApi` (проба `{INTEGRATOR_API_URL}/health`), `projection` (проба `/health/projection` через proxy helper), `mediaPreview` (счётчики preview-пайплайна по `media_files`), **`videoPlayback`** (VIDEO_HLS_DELIVERY через Drizzle: при **`video_playback_api_enabled`** в БД **`false`** проба **не** читает `media_playback_*`; в **`SystemHealthSection`** числовой блок воспроизведения скрыт, чтобы не выдавать выключенный API за «нулевую статистику»; при **`true`** — суммы за последние 24 ч UTC в **`media_playback_stats_hourly`** (`totalResolutions`, `byDelivery` hls/mp4/file, `fallbackTotal`); **`uniquePlaybackPairsFirstSeenInWindow`** — `COUNT(*)` из **`media_playback_user_video_first_resolve`** где **`first_resolved_at`** попадает в окно (новая когда-либо пара платформенный пользователь + `media_id`; повторные успешные резолвы только наращивают `totalResolutions`); длинный HLS‑сеанс с повторными `GET …/playback` даёт несколько инкрементов только в почасовике), `fetchedAt`. Ретенция почасового агрегата: **`internal/media-playback-stats/retention`**. Пробы запускаются параллельно (`Promise.allSettled`), ошибки нормализуются в `status: unreachable | error` без раскрытия внутренних ответов внешних сервисов; логируются только технические поля `probe`, `status`, `durationMs`, `errorCode`.
- **doctor/clients/[userId]/permanent-delete** — `POST`, тело `{ confirmUserId: uuid }` (должен совпадать с `:userId`). Только **admin + admin mode**, клиент в архиве, `role === client`. Вызывает strict purge (`runStrictPurgePlatformUser`): advisory lock, preflight артефактов, S3/integrator post-commit, запись `user_purge` в `admin_audit_log`. Для повторной внешней очистки audit хранит `artifact`, `phoneNormalized`, `webappIntegratorUserId` и `resolvedIntegratorUserIds`; повтор через `retryStrictPurgeExternalCleanup` пишет отдельно `user_purge_external_retry`. Успех: `{ ok: true, outcome, integratorSkipped, details }` (`outcome`: `completed` | `partial_failed` | `needs_retry`). Ошибки: `400/404/409/500` (`purge_transaction_failed` при сбое транзакции webapp).
- **doctor/clients/[userId]/merge-candidates** — `GET`, только **admin + admin mode**. Список других канонических клиентов, потенциально дубликатов якоря (общий телефон, email, `integrator_user_id` или общий messenger binding). Query: опционально `q` (подстроковый фильтр). Ошибки: `400 not_client`, `404`, `409 anchor_is_alias`. См. `docs/ARCHITECTURE/PLATFORM_USER_MERGE.md` (Manual merge — preview).
- **doctor/clients/merge-preview** — `GET`, query `targetId`, `duplicateId` (оба UUID). Только **admin + admin mode**. Read-only preview ручного merge: профили (camelCase), биндинги, OAuth, dependent counts, hard blockers, конфликты, auto-merge поля, `recommendation`, `mergeAllowed`, **`v1MergeEngineCallable`**, **`platformUserMergeV2Enabled`** (значение admin-настройки `platform_user_merge_v2_enabled` на момент запроса). **Флаг выкл. (v1):** при двух разных non-null `integratorUserId` — hard blocker `different_non_null_integrator_user_id` (как раньше). **Флаг вкл. (v2):** для той же пары preview вызывает integrator M2M `canonical-pair`; если canonical id совпали после integrator merge — блокер снимается; иначе `integrator_canonical_merge_required`; при отсутствии конфигурации integrator M2M или timeout/недоступности ответа — `integrator_merge_status_unavailable`. Ошибки: `400` (`invalid_query`, `same_id`, `not_client`), `404 missing_user`. Контракт: `docs/ARCHITECTURE/PLATFORM_USER_MERGE.md`, v2 пакет (архив): `docs/archive/2026-04-initiatives/PLATFORM_USER_MERGE_V2/STAGE_5_FEATURE_FLAG_AND_FLOW_SWITCH.md`.
- **doctor/clients/integrator-merge** — `POST`, только **admin + admin mode**. Тело `{ targetId, duplicateId, dryRun?: boolean }`. Требует **`platform_user_merge_v2_enabled`** в `system_settings` (admin); иначе `400 feature_disabled`. Обе строки — канонические клиенты с **разными** non-null `integrator_user_id`; иначе `400 integrator_ids_not_divergent`. Перед M2M вызовом route перечитывает обе строки `platform_users` под короткой транзакцией `FOR UPDATE`, чтобы winner/loser integrator ids не drift'нули посреди запроса. Проксирует HMAC `POST` на integrator `/api/integrator/users/merge` (winner = integrator id **целевого** platform user, loser = **дубликата**). Ошибки integrator пробрасываются в JSON (`integrator_merge_failed` + `details`); без URL/секрета — `503 integrator_unconfigured`; при timeout — `503 integrator_timeout`.
- **doctor/clients/merge** — `POST`, только **admin + admin mode**. Тело JSON: `{ resolution: ManualMergeResolution }` (см. `apps/webapp/src/infra/repos/manualMergeResolution.ts` и архитектурный doc): явный выбор победителей по скалярам, `telegram`/`max`/`vk`, OAuth по провайдеру, стратегия `channelPreferences`. `resolution.bindings.* = "both"` допустим только для каналов **без** конфликта; при конфликтных `external_id` backend требует `target` или `duplicate`. Gate по `integrator_user_id`: **v1** (флаг выкл.) при двух разных id **не** отвечает отдельным `409` — как до Stage 5, `mergePlatformUsersInTransaction` бросает `MergeConflictError`, `runManualPlatformUserMerge` возвращает ошибку и пишет audit на фазе транзакции, HTTP **`409`** `merge_failed` с `message` вида `merge: two different non-null integrator_user_id`. **v2:** до транзакции M2M `canonical-pair`; relaxed merge допустим только если snapshot проверенной пары совпал с `integrator_user_id`, перечитанными под lock в merge tx. При drift сервер отвечает `409` с `code` `integrator_ids_changed_since_gate`; при несогласованной canonical паре — `409`/`502`/`503` с `code` `integrator_canonical_merge_required`, `integrator_canonical_status_failed`, `integrator_merge_status_unavailable`. Выполняет `runManualPlatformUserMerge` (dual advisory lock, merge tx, перенос `media_files.uploaded_by`, `media_upload_sessions.owner_user_id`, `email_verified_at`, **`user_merge`** в `admin_audit_log` после commit). Сервер дополнительно отклоняет пары, где хотя бы одна строка не `role === client`. Успех: `{ ok: true, targetId, duplicateId }`. Неверное тело: `400` (`invalid_body`).
- **doctor/clinical-tests** (TREATMENT_PROGRAM_INITIATIVE, библиотека) — только **doctor** или **admin** (`getCurrentSession` + `canAccessDoctor`). **`GET /api/doctor/clinical-tests`** — query: `q`, `testType`, `includeArchived`, опционально **`region`** (UUID `reference_items.id`), **`assessment`** (код вида оценки: активные строки справочника `clinical_assessment_kind` / `reference_items.code`; при пустом справочнике в БД допускаются коды сида v1 из миграции D2); неверный UUID в `region` или код вне allowlist для `assessment` — **`400`** `{ ok:false, error:"invalid_query", field:"region"|"assessment" }`; ответ `{ ok: true, items }` (каждый элемент с B2: **`scoring`** — структурированный JSON с корневым **`schema_type`** (`numeric`|`likert`|`binary`|`qualitative`) и **`measure_items`**, **`rawText`**, **`assessmentKind`**, **`bodyRegionId`**). **`POST /api/doctor/clinical-tests`** — JSON: `title`, опционально `description`, `testType`, **`assessmentKind`** (код `reference_items.code` категории `clinical_assessment_kind`; валидация allowlist как у формы врача), `media[]`, `tags`; `{ ok: true, item }`. **`GET/PATCH/DELETE /api/doctor/clinical-tests/[id]`** — один тест; `PATCH` частичное обновление; **`DELETE`** — архивация (`is_archived`). Устаревший вариант с одной строкой и `source: "lfk_session"` в истории может встречаться у старых данных.
- **patient/treatment-program-instances/[instanceId]/plan-opened** (A5) — **`POST`** `{ ok: true, recorded: boolean }` — если экземпляр **не найден** у текущего пациента → **404**; если программа **`completed`** (не активна) → **`{ ok: true, recorded: false }`** без изменения `patient_plan_last_opened_at`; для **`active`** — запись метки открытия и **`recorded: true`**; **`revalidatePath`** (Today / список программ) вызывается **только** при **`recorded: true`**.
- **patient/.../items/[itemId]/mark-viewed** (A5) — **`POST`** `{ ok: true, updated: boolean }` — если **`last_viewed_at IS NULL`**, проставляет метку просмотра (идемпотентно); иначе **`updated: false`**. Бейдж **«Новое»** на элементе; при **`updated: true`** — **revalidatePath** для Today и списка программ.
- **doctor/comments** (фаза 5, `SYSTEM_LOGIC_SCHEMA.md` §7) — только **doctor** или **admin**. **`GET`** — query `targetType` (enum как в §7: `exercise` \| `lfk_complex` \| `test` \| `test_set` \| `recommendation` \| `lesson` \| `stage_item_instance` \| `stage_instance` \| `program_instance`), `targetId` (UUID); ответ `{ ok: true, items }` (по `created_at` по возрастанию). **`POST`** — JSON `targetType`, `targetId`, `commentType` (`template` \| `individual_override` \| `clinical_note`), `body` (1…32000); `author_id` берётся из сессии; `{ ok: true, item }`. Ошибки: `401`, `403`, `400` (`invalid_query` \| `invalid_body` \| текст сервиса).
- **doctor/comments/[id]** — **`GET`** одна запись `{ ok, item }` или `404`. **`PATCH`** — хотя бы одно из `body`, `commentType`; разрешено автору комментария или **admin**. **`DELETE`** — то же правило мутации. Ошибки: `401`, `403`, `400` (`invalid_id` \| `invalid_body`), `404`.

- **health/** — проверка работоспособности сервиса.
- **me/** — данные текущего пользователя (если нужны для клиента).
- **menu/** — меню по роли (если используется отдельным запросом).
