# api

Обработчики HTTP API вебаппа (Next.js route handlers).

- **auth/** — авторизация: обмен токена интегратора на сессию (exchange), вход по Telegram initData (telegram-init), выход (logout).
- **integrator/** — эндпоинты для бота/интегратора: приём событий (events), дневники (symptom-trackings, lfk-complexes), напоминания (reminders/dispatch). Доступ по подписи запроса (общий секрет). Для `POST /api/integrator/events`: типизированные `MergeConflictError` / `MergeDependentConflictError` из слоя проекций **не** приводят к бесконечному retry (`503`): при наличии `conflictAudit` запись в `admin_audit_log` (`auto_merge_conflict` или `auto_merge_conflict_anomaly`) и ответ **`202` accepted**; прочие ошибки ingestion остаются retryable (`503`) или `422` по прежним правилам. См. `docs/ARCHITECTURE/PLATFORM_USER_MERGE.md` (projection conflicts).
- **media/** — `GET /api/media/[id]`: требуется **сессия** (любая роль); без сессии — **401**. При записи в БД с `s3_key` — **302** на presigned GET в **private** MinIO-бакет (`S3_PRIVATE_BUCKET`); без прямых публичных URL MinIO в ответе. Модель угроз и internal purge — `docs/REPORTS/S3_PRIVATE_MEDIA_EXECUTION_LOG.md`, nginx для `/api/internal/` — `deploy/HOST_DEPLOY_README.md`. Загрузка для кабинета врача: при настроенном S3 — `POST /api/media/presign` (JSON `filename`, `mimeType`, `size`) → `{ ok, mediaId, uploadUrl, readUrl }` (`readUrl` = `/api/media/{id}`), затем клиент PUT на `uploadUrl`, затем `POST /api/media/confirm` с `{ mediaId }` → `{ ok, url: /api/media/{id} }`. Fallback без полного S3: `POST /api/media/upload`, multipart `file`, только роль doctor/admin. Прокси-загрузка: до 50 МБ, MIME из белого списка в [`uploadAllowedMime.ts`](../../modules/media/uploadAllowedMime.ts) (изображения incl. HEIC/AVIF/TIFF/SVG, видео incl. WebM, аудио incl. Ogg/AAC/M4A, документы Office Open XML и legacy OLE, `text/plain`, `text/csv`; для SVG — не инлайнить сырой XML в `<img>`). Успех upload: `{ ok: true, mediaId, url }` с `url` вида `/api/media/{uuid}`. Ошибки: `413`, `415`, `403`, при сбое presign на чтение — `503`.
- **admin/media/** — библиотека файлов CMS (`GET /api/admin/media`) с фильтрацией/сортировкой, editable названием (`PATCH /api/admin/media/[id]`, поле `displayName`: строка или `null`; пустая строка/пробелы очищают custom title и включают показ исходного имени), удаление (`DELETE /api/admin/media/[id]`) с precheck использования в `content_pages` и подтверждением `confirmUsed=true`. После подтверждения запись переводится в очередь на фоновое удаление в S3; ответ содержит `scheduled: true`. Экран сбойных удалений: `GET /api/admin/media/delete-errors` (`items`, `total`) — строки с `delete_attempts > 0` в очереди удаления; доступ doctor/admin.
- **internal/media-pending-delete/purge** — `POST` с `Authorization: Bearer <INTERNAL_JOB_SECRET>` обрабатывает батч строк `media_files` в статусе **`pending_delete`** или застрявшем **`deleting`** (удаление объекта в S3 и строки в БД; при ошибке S3 — **retry** с `delete_attempts` и `next_attempt_at`). Ответ: `{ ok, removed, errors }`. Параметр `?limit=` (по умолчанию 25). Без `INTERNAL_JOB_SECRET` в env — `503`.
- **admin/audit-log** — `GET /api/admin/audit-log` — пагинированный журнал `admin_audit_log` (операции админки и деструктивные действия по мере подключения сценариев). Доступ: `role === admin`, включённый admin mode (`requireAdminModeSession`). Параметры: `page`, `limit`, `action`, `target`, **`involvesPlatformUserId`** (UUID — строки, где `target_id` совпадает **или** для `action=auto_merge_conflict` пользователь входит в `details.candidateIds`), `status`, `from`, `to` (даты `YYYY-MM-DD`). В ответе дополнительно **`openAutoMergeConflictCount`**: число **открытых** строк `auto_merge_conflict` (`resolved_at IS NULL`) для бейджа в UI. См. `docs/REPORTS/STRICT_PURGE_MANUAL_MERGE_EXECUTION_LOG.md`.
- **doctor/clients/[userId]/permanent-delete** — `POST`, тело `{ confirmUserId: uuid }` (должен совпадать с `:userId`). Только **admin + admin mode**, клиент в архиве, `role === client`. Вызывает strict purge (`runStrictPurgePlatformUser`): advisory lock, preflight артефактов, S3/integrator post-commit, запись `user_purge` в `admin_audit_log`. Для повторной внешней очистки audit хранит `artifact`, `phoneNormalized`, `webappIntegratorUserId` и `resolvedIntegratorUserIds`; повтор через `retryStrictPurgeExternalCleanup` пишет отдельно `user_purge_external_retry`. Успех: `{ ok: true, outcome, integratorSkipped, details }` (`outcome`: `completed` | `partial_failed` | `needs_retry`). Ошибки: `400/404/409/500` (`purge_transaction_failed` при сбое транзакции webapp).
- **doctor/clients/[userId]/merge-candidates** — `GET`, только **admin + admin mode**. Список других канонических клиентов, потенциально дубликатов якоря (общий телефон, email, `integrator_user_id` или общий messenger binding). Query: опционально `q` (подстроковый фильтр). Ошибки: `400 not_client`, `404`, `409 anchor_is_alias`. См. `docs/ARCHITECTURE/PLATFORM_USER_MERGE.md` (Manual merge — preview).
- **doctor/clients/merge-preview** — `GET`, query `targetId`, `duplicateId` (оба UUID). Только **admin + admin mode**. Read-only preview ручного merge: профили (camelCase), биндинги, OAuth, dependent counts, hard blockers, конфликты, auto-merge поля, `recommendation`, `mergeAllowed`, **`v1MergeEngineCallable`** (совместимость с **авто** `mergePlatformUsersInTransaction` без `ManualMergeResolution`: `false` при разных non-null телефонах или при hard blockers). Ошибки: `400` (`invalid_query`, `same_id`, `not_client`), `404 missing_user`. Контракт: `docs/ARCHITECTURE/PLATFORM_USER_MERGE.md`.
- **doctor/clients/merge** — `POST`, только **admin + admin mode**. Тело JSON: `{ resolution: ManualMergeResolution }` (см. `apps/webapp/src/infra/repos/manualMergeResolution.ts` и архитектурный doc): явный выбор победителей по скалярам, `telegram`/`max`/`vk`, OAuth по провайдеру, стратегия `channelPreferences`. `resolution.bindings.* = "both"` допустим только для каналов **без** конфликта; при конфликтных `external_id` backend требует `target` или `duplicate`. Выполняет `runManualPlatformUserMerge` (dual advisory lock на оба id, транзакция merge, перенос `media_files.uploaded_by`, сохранение `email_verified_at` у surviving email, запись **`user_merge`** в `admin_audit_log` после commit). Сервер дополнительно отклоняет пары, где хотя бы одна строка не `role === client`; жёсткий запрет на два разных non-null `integrator_user_id` остаётся как в preview. Успех: `{ ok: true, targetId, duplicateId }`. Ошибка merge: `409` (`merge_failed`, `message`). Неверное тело: `400` (`invalid_body`).
- **health/** — проверка работоспособности сервиса.
- **me/** — данные текущего пользователя (если нужны для клиента).
- **menu/** — меню по роли (если используется отдельным запросом).

Роуты остаются тонкими: проверка прав и параметров, вызов сервисов из `buildAppDeps()` или модулей.
