# diary

Подраздел «Дневники» в разделе пациента.

## Текущий MVP (страница `page.tsx`)

- Одна секция **«Самочувствие за неделю»**: Recharts `ComposedChart` — среднее за день (`general_wellbeing`, только **instant**), линия отметок в течение дня (**только instant**; записи `daily` на линию не попадают), точки «после разминки» (`warmup_feeling`, значения 1/3/5).
- Данные: `loadPatientDiaryWeekWellbeing` → `buildWellbeingWeekChartData`; границы недели в ms передаются из лоадера в хелпер (**одинаковое окно** с SQL `fromRecordedAt` / `toRecordedAtExclusive`).
- TZ календарного дня — `resolveCalendarDayIanaForPatient` + `patientCalendarTimezone` (**тот же выбор IANA**, что `GET /api/patient/mood/week` и полоса «Ваша неделя» на главной).
- Ось Y графика самочувствия в коде — **1–5** (шкала чек-ина; ноль не используется); при необходимости расширения до 0–10 — отдельная задача.
- Вкладки **«Симптомы»** и **«ЛФК»**, форма «Отметить занятие», `QuickAddPopup` — **временно не монтируются** (код компонентов сохранён в репозитории). На странице есть TODO-блок с путями для восстановления.

### Чек-лист реализации (после графика)

Детальная постановка — в разделе **«Продуктовая цель»** ниже.

- Блок **разминок** (столбики по дням, слоты, «огонёк»).
- Блок **упражнений плана** (полоски по дням в порядке «План → Программа»).
- **Снимки в БД** для прошлых дней + правила закрытия дня.
- По продукту: вкладки симптомов/ЛФК, судьба ручной отметки ЛФК в вебе.

## Продуктовая цель (после графика самочувствия)

График «Самочувствие за неделю» (`ComposedChart`) считается **реализованным**; этот раздел описывает **остальной** экран дневника — задачи, механику и дизайн из продуктового обсуждения.

### Роль дневника

- Дневник — **история и сводка** того, что пациент уже делает на **главной «Сегодня»**, в **плане**, по **разминкам** и отметкам самочувствия, а не второй полный дубль всех форм.
- Заполнение в идеале **событийное**: данные подтягиваются из действий, а не только из ручного ввода на странице дневника.
- **Отслеживание симптомов** (врач/бот, отдельный модуль) на MVP **не показывать** — доработка после выхода MVP; не путать с базовыми шкалами самочувствия на уже сделанном графике.
- **Механика «ведения занятия ЛФК» в вебе** — скрыта; нужен отдельный **TODO** в проекте: замена на автоматику из плана/разминок, синхронизация с ботом, что видит пациент.

### Композиция экрана (остальные блоки)

На **одной общей горизонтали недели** (синхрон с графиком по дням и **таймзоне календаря пациента**):

1. **График самочувствия** — готов, не описывается здесь.
2. **Разминки по дням** — столбики выполнения относительно слотов напоминаний.
3. **Упражнения из плана** — полоски по дням, порядок как в «План → Программа».

**Месячный режим** — отложен; в фокусе **неделя**. **Текстовые заметки дня / полноценный текстовый дневник** — позже.

### Блок «Разминки»

**Смысл:** дозировка активности по разминкам (сколько слотов «план» и сколько закрыто) + лёгкая игровая мотивация.

**Разметка:**

- Общая с графиком ось **дней недели** (те же границы календарных дней в TZ пациента).
- **Ось Y** — число делений = **число запланированных напоминаний о разминке на этот день** (источник — механизм слотов/напоминаний, когда будет готов).

**Визуал:**

- В каждом дне — **вертикальные столбики** по числу выполнений (от **столбика из кружков** ушли к **меньшему шуму**: в приоритете **столбики**, **один заметный акцент на день**).
- Не делал / минимум: **серая точка** или **серая полоска внизу** при отсутствии столбика.
- Делал: **выше и зеленее** по мере закрытия доли слотов.
- Все слоты дня закрыты — **иконка огонька** над столбиком.

**Пока нет стабильных слотов:** временно **минимум три деления** по вертикали; если пользователь настроил больше — показывать больше (деталь — привязка к напоминаниям).

**Пример правила без перерисовки истории:** вчера лимит **3**, все закрыты — **огонёк**; сегодня лимит стал **5** — **сегодня** считаем от **5**, **вчера** остаётся **3/3** и огонёк **не снимается** (см. снимки ниже).

### Блок «Упражнения из плана»

**Смысл:** визуальная связь нагрузки по плану с неделей самочувствия.

**Разметка:**

- **Один календарный день** — вертикальный столбец **полосок**.
- **Каждая полоска** — одно **упражнение** (элемент плана), **сверху вниз**.
- **Порядок** — **глобальный** порядок элементов **без разбиения по этапам**, как на **«План → Программа»** для текущего дня (правило сортировки зафиксировать в спеке/API).

**Состояния:**

- **Серая** — нет выполнения за этот календарный день.
- **Зелёная** — выполнено **хотя бы раз за этот день**.

**Изменение плана:** при добавлении/удалении, перестановке, смене этапа — **прошлые дни не перерисовываются**; только **снимок** на тот день (иначе история ложная).

### Снимки в БД для прошлых дней

**Цель:** не пересчитывать историю при каждом открытии и не переписывать прошлое при смене **сегодняшних** напоминаний или плана.

- **Текущий день** — по актуальным напоминаниям и текущему плану.
- **Прошлые дни** — из **снимка**, не из «текущий план задним числом».

**Минимальный состав смыслов:**

- **Разминки:** на день — лимит слотов «как для того дня», число выполнений, признак полного дня (огонёк).
- **План:** на день — зафиксированный порядок id элементов + факт выполнения за день (или эквивалент для воспроизведения UI).

**Закрытие дня для снимка:** полночь по TZ пациента + **ленивая фиксация** при **первом заходе** на следующий календарный день (продуктовая договорённость; альтернативы — отдельное решение).

**Краевой случай:** поздние правки «задним числом» — нужно явное правило (перезапись снимка / запрет / отдельное событие).

**Объём данных:** для MVP небольшой; сложность — в **дисциплине правил**, а не в размере таблиц.

### Таймзона и граница «сегодня»

- Все блоки страницы используют **одинаковые** правила календарного дня в **IANA пациента**.
- Порог вида **«сегодня с 03:00»** — спорный; варианты: полночь по TZ или будущая **настройка** «день начинается с HH:MM» — вынести в TODO продукта/настроек.

### Мотивация и карточка сверху (отдельный слой)

- Карточка в духе: «на прошлой неделе — **N%** активности, среднее самочувствие **±N**…» с **ротацией** текстов и привязкой к **реальным** метрикам.
- Мягкие CTA при плохом самочувствии (запись, онлайн-вопрос) — **без** тона медицинского диагноза.
- Допустимый минимум — **одна сухая строка** средних; «юмор» и яркие формулировки — контент поверх тех же цифр.

### Явно вне текущего объёма

- Месячный вид.
- Полноценные вкладки симптомов/ЛФК и QuickAdd на странице — до продуктового решения.

## Прежняя структура (архив описания)

Две ветки: дневник симптомов и дневник ЛФК. Данные хранятся в БД вебаппа; добавлять записи и отслеживания можно и из бота (интегратор шлёт события в вебапп), и из вебаппа (форма «Отметить занятие» на странице ЛФК).

- **symptoms/** — страница дневника симптомов: список отслеживаемых симптомов и список записей (балл, тип, дата). Добавление симптомов и записей — через бота.
- **lfk/** — страница дневника ЛФК: список комплексов, форма «Отметить занятие» (выбор комплекса и кнопка), список отмеченных занятий. Добавление комплексов — через бота; отметку занятия можно сделать и здесь.

Логика чтения/записи — в модулях `diaries` (symptom-service, lfk-service) и репозиториях в `infra/repos`.

**RSC:** единая страница дневника и страницы журналов (`symptoms/journal`, `lfk/journal`) перед чтением из БД вызывают **`patientRscPersonalDataGate`** (`requireRole.ts`). Мутации — server actions с **`requirePatientAccessWithPhone`**.
