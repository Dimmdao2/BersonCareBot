# Операции с идентичностью и телефоном (обход UI)

Нормативная модель tier и доверия к номеру: [`docs/PLATFORM_IDENTITY_ACCESS/SPECIFICATION.md`](../../docs/PLATFORM_IDENTITY_ACCESS/SPECIFICATION.md), таблица trusted-путей в коде — [`docs/PLATFORM_IDENTITY_ACCESS/SCENARIOS_AND_CODE_MAP.md`](../../docs/PLATFORM_IDENTITY_ACCESS/SCENARIOS_AND_CODE_MAP.md) §8.

Эта инструкция — для **ручных** действий: SQL, скрипты в этой папке, восстановление целостности. Обычные сценарии в приложении (OTP, OAuth, кабинет врача, интегратор) уже проходят через продуктовый код и **не требуют** этого чек-листа.

---

## 1. Одно правило чтения tier для клиента

Приложение считает пользователя **patient** (полный доступ к patient-бизнесу) только если у **канонической** строки `platform_users` одновременно:

- заполнен нормализованный телефон;
- заполнена отметка доверия **`patient_phone_trust_at`** (не NULL).

Только телефон без отметки доверия → **onboarding**, не patient. Это не «баг», а политика безопасности.

Любое ручное изменение, которое трогает телефон или пользователей, должно **явно** решить: нужен ли после операции статус **patient** по продукту, и отразить это в **обоих** полях согласованно.

## 2. Порядок действий (строгая логика)

**Шаг A — По возможности не писать «голый» SQL по `platform_users`.**

1. **Удаление / полная зачистка по номеру или UUID** — сначала смотреть [`user-phone-admin.ts`](user-phone-admin.ts): там уже используется инфраструктура вроде `runStrictPurgePlatformUser` и описаны каскады webapp/integrator. Это снижает риск забыть связанные таблицы.
2. **Слияние двух клиентских учёток** — через **продуктовые** пути: UI врача (merge) или код репозитория `pgPlatformUserMerge` / процедуры, согласованные с [`docs/ARCHITECTURE/PLATFORM_USER_MERGE.md`](../../docs/ARCHITECTURE/PLATFORM_USER_MERGE.md). Там переносится и логика вокруг `patient_phone_trust_at`, а не только `phone_normalized`.
3. **Назначить номер «как после доверенного сценария»** без UI — по смыслу это дублирует OTP/интегратор; предпочтительно довести пользователя через **штатный** поток. Если это невозможно, переходите к шагу B с явным решением в тикете/runbook.

**Шаг B — Если без прямого SQL/скрипта не обойтись.**

Перед выполнением зафиксируйте в задаче (тикете) ответы:

| Вопрос | Если «да» | Если «нет» |
|--------|-----------|------------|
| После правки пользователь должен сразу иметь **patient**-доступ (как после подтверждённого номера)? | Выставить **`patient_phone_trust_at`** (например `now()`), согласованно с телефоном на **каноне** после `merged_into_id`. | Обновить только то, что нужно; **не** выставлять `patient_phone_trust_at`, если не хотите автоматического patient-tier. |
| Меняется ли канон (merge, смена победившего UUID)? | Все дальнейшие проверки — по **каноническому** id после цепочки merge. | — |

**Шаг C — После операции.**

- Проверить в БД одну строку канона: телефон и `patient_phone_trust_at` соответствуют решению из шага B.
- При сомнениях зайти под пользователем или вызвать `GET /api/me` и посмотреть `platformAccess.tier` (не полагаться только на «телефон в профиле»).

**Шаг D — Повторяющийся операционный сценарий.**

Если одна и та же ручная процедура понадобится **регулярно**, оформите изменение в **продуктовом коде**: новый путь в `TrustedPatientPhoneSource` + writer в репозитории + ревью (как в SCENARIOS §8). Иначе каждый раз риск расхождения «как договорились» vs «как в коде».

## 3. Runtime worker (`apps/integrator`) и webapp

**Worker не выполняет** `apps/webapp/scripts/*` и **не пишет** напрямую в БД webapp.

Процесс `worker:start` (см. `apps/integrator/package.json`) крутит два контура: очередь job’ов (integrator) и **projection outbox** → HTTP-вызов webapp (`emit` событий). Запись в webapp-таблицы `platform_users` / телефон / `patient_phone_trust_at` для проекций идёт **только** в обработчиках webapp (`handleIntegratorEvent` → `pgUserProjection` и т.д.), т.е. через те же доверенные пути, что и при живом webhook. С точки зрения Platform Identity & Access **воркер согласован** с архитектурой: отдельного «тихого» обхода tier через воркер нет.

## 4. Инвентарь скриптов webapp и `platform_users`

| Файл | Пишет `platform_users` | `patient_phone_trust_at` |
|------|------------------------|----------------------------|
| [`backfill-person-domain.mjs`](backfill-person-domain.mjs) | INSERT/UPDATE из integrator | **Да** — при непустом телефоне из источника integrator (как проекция). |
| [`backfill-appointments-domain.mjs`](backfill-appointments-domain.mjs) | INSERT/UPDATE по записям приёмов | **Да** — `now()` при INSERT; при ON CONFLICT `COALESCE(..., now())` для догонки legacy-строк. |
| [`backfill-rubitime-history-to-patient-bookings.ts`](backfill-rubitime-history-to-patient-bookings.ts) | INSERT при создании пользователя по телефону из истории | **Да** — `now()` (тот же смысл, что `ensureClientFromAppointmentProjection`). |
| [`user-phone-admin.ts`](user-phone-admin.ts) | DELETE / служебные UPDATE | Удаление identity; не создаёт patient без прохождения продуктовых потоков. |
| `reconcile-*.mjs` | как правило только чтение / сверка id | по сценарию — без массовой записи телефона; перепроверять при изменении скрипта. |
| `*.sql` (repair и пр.) | вручную | проходить шаги §2 B–C. |

Прочие `backfill-*` / `reconcile-*`, если начнут писать телефон в `platform_users`, нужно либо выровнять с этой таблицей и §1, либо явно задокументировать исключение в шапке скрипта.

## 5. Прочее в этой папке
