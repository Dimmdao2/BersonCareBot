Домен - бизнес логика

принимает подготовленное (очищенное) событие от eventGayeway

подгружает связанный контекст (админ ли юзер, в каких каналах он идентифицирован - знаем его телеграм/макс/вк/инстаграм/почту/телефон)

отправляет подготовленные данные в оркестратор

получает от оркестратора при необходимости запрос на дополнительный контекст (например настройки рассылок юзера в конкретных каналах, или список записей на прием или еще что-то)

отправляет собранный доп контекст оркестратору

получает итоговыу инструкции на выполнение

в виде doAction(action, context, channel, now/shedule, retry_at, attempts, fallback, fallback_message) (это как пример)

например:
doAction (sendMessage, {phone:phone, message:"Вы записаны на прием в 12:00"}, telegramm, now, every1min, 2, smsc, message + telegram_link) 

здесь домен пробует  найти юзера по номеру в этом мессенджере (смотрит привязку телефона в таблице юзеров мессенджера), если ок то отправляет сообщение, а если не находит или ошибка в процессе отпарвки (не прошел вебхук исходящий) то ставит воркеру задачу вызывать это действие еще два раза с паузой в 1 минуту, и если не получилось то отправляет смс через smscб но добавляет к сообщению еще ссылку на телеграм/другие мессенджеры.

эту ссылку мы потом сделаем.

или

doAction (sendMessage, {chat_id:id, message:"Выберите пункт меню"}, telegramm, now,,adminMessage, "ошибка отправки меню в телеграмм юзеру" + контекст)

здесь пытаемся отправить сообщение с кнопками меню в телеграм, если не получилось - отправляем админу дебаг

или

doAction (createRecord, {branch_id, cooperator_id, service_id, status, record, rk}, rubitime, now,,adminMessage, "ошибка создания записи в рубитайм через телеграм-бота" + контекст)

ПРАВИЛА:

домен олучает снаружи только три вида событий:

1) подготовленный универсальный incommingEvent от eventGatevay 
2) запрос на контекст от Orchestrator (загрузить из БД такие то данные по такому то юзеру / событию)
3) универсльную инструкцию на выполнение от 

4) Никто больше не пишет логи и метрики кроме домена.

5) Домен не хранит и не парсит скрипты и шаблоны сообщений, не делает дедуп, очистку данных


-------

Назначение

Домен — центр бизнес-логики системы.

Здесь реализуется поведение системы.

Основные задачи

Домен:
	1.	собирает контекст
	2.	работает с базой данных
	3.	выполняет действия
	4.	формирует задачи
	5.	вызывает интеграции

Что делает

После получения события:

event
 ↓
домен

домен:
	1.	загружает данные из базы
	2.	формирует контекст
	3.	вызывает оркестратор
	4.	получает список действий
	5.	выполняет действия

Типы действий

Домен может:

прочитать данные из БД
записать данные в БД
отправить сообщение
создать задачу
запланировать повтор

Результат работы

Домен может вернуть:

intent
db mutation
scheduled job

Что НЕ делает

Домен:
	•	не знает HTTP
	•	не знает webhook
	•	не знает Telegram API
	•	не знает очередь задач

Он работает через порты.
