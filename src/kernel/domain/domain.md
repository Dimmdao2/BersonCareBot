Домен — бизнес-логика

Принимает подготовленное (очищенное) событие от EventGateway.

Подгружает связанный контекст (админ ли пользователь, какие идентификаторы/привязки есть по каналам и контактам).

Вызывает Orchestrator: передаёт incomingEvent + baseContext.

Получает от Orchestrator:
	•	либо план шагов (steps) на выполнение,
	•	либо запрос на дополнительный контекст (что именно догрузить из БД/портов).

Догружает требуемый контекст и повторно вызывает Orchestrator (если требуется) до получения итогового плана.

Исполняет план:
	•	делает DB reads/writes через порты,
	•	формирует и отправляет outgoing intents (доставка через dispatcher/интеграции),
	•	создаёт jobs (отложенные/повторные попытки) через queue/job-порт,
	•	применяет retry/fallback согласно политике (политика может быть частью плана, но решение и исполнение — в домене).

Пример логики исполнения (иллюстративно)

Вместо doAction(sendMessage, …, now/every1min/attempts/fallback…) домен исполняет шаги плана, например:
	•	message.send (channel=channel-a, target=contact, template=…, params=…)
	•	домен резолвит target (по привязкам в БД)
	•	пытается доставить intent
	•	при ошибке/нет получателя — создаёт job на retry
	•	при исчерпании попыток — выполняет fallback (например message.send через channel-b с добавленным текстом/линком)
	•	admin.notify (channel=channel-a, target=admin, message=…)
	•	при критической ошибке доставки/исполнения шага отправляет админу debug-уведомление (если включено настройками)
	•	booking.createRecord / booking.upsert / и т.п.
	•	домен выполняет действие через соответствующий порт (не через HTTP-SDK напрямую)

Логи и метрики

Домен пишет бизнес-логи и метрики.
Параметры — из appSettings.json:
	•	какие события логировать (успехи/отказы),
	•	логировать ли дедуп/rate-limit результаты (если эти статусы переданы из EventGateway),
	•	отправлять ли debug админу и в какой канал.

ПРАВИЛА
	1.	Домен получает снаружи только:

	•	IncomingEvent от EventGateway (accepted или blocked/rejected как результат обработки),
	•	Job (из runtime worker) на исполнение отложенного действия,
	•	(опционально) SchedulerTick/внутренний системный event, если домену нужно реагировать на расписание — но это не должно задевать Orchestrator как “парсер сценариев”.

	2.	Orchestrator не управляет доменом. Orchestrator — “планировщик/подбор сценария”: возвращает план и (опционально) запросы на недостающий контекст.
	3.	Никто, кроме домена, не принимает решений retry/fallback и не пишет бизнес-метрики. Runtime ведёт только технические логи выполнения job (по необходимости).
	4.	Домен не хранит и не парсит скрипты/шаблоны сообщений.
	5.	Домен не делает dedup/rate-limit/очистку входящих данных (это EventGateway/integrations).

Назначение

Домен — центр поведения системы.

Основные задачи

Домен:
	1.	собирает контекст (через read-порты),
	2.	вызывает Orchestrator за планом,
	3.	исполняет шаги плана,
	4.	планирует отложенные/повторные действия (jobs),
	5.	инициирует доставку сообщений/уведомлений через порты (dispatcher/интеграции).

Что НЕ делает

Домен:
	•	не знает HTTP/webhook,
	•	не знает SDK внешних каналов напрямую,
	•	не знает реализацию очереди/воркера/планировщика,
	•	не знает файловую структуру content.

Он работает через порты.
