# Этап 6 — Абонементы

> ТЗ: `STAGE_CHECKLISTS.md` §Этап 6 (ТЗ §11,13.5). Зависит от этапов 1,2,5 (запись, услуги, платёжный слой). Учитывает этап 4 (отмены/переносы).

## Контекст существующего кода

- Прямого аналога абонементов нет. Платёжный слой — этап 5 (`modules/payments/*`). Услуги/доступность — этап 1. Запись/статусы — этап 1/2 (статус «списана по абонементу» уже в enum §17). `user_subscriptions_webapp` — это mailing-topic, НЕ абонементы (не путать).
- Курсы (`modules/courses/*`) как потребитель продуктового слоя — смежно с этапом 7.

## Scope boundaries

- **Можно трогать:** новый модуль `modules/memberships/*` (service+ports), Drizzle абонементных таблиц, интеграция со слот-движком/записью (этап 2) и платёжным слоем (этап 5), admin/doctor UI создания/назначения, пациентский блок абонементов, карточка (срез), docs.
- **Вне scope:** общая модель продуктов/акций/подписок (этап 7 — здесь только абонемент-продукт как частный случай). Полная карточка клиента — этап 9 (здесь срез абонементов).

## Декомпозиция

### Шаг 6.1 — Модель (todo s6-model) — ТЗ §11.1,21.4
- Drizzle: `package`(subscription_package) + `package_item` (произвольный состав: разные услуги×количества, напр. 2×90+4×60); срок действия; правила списания. `patient_package` (экземпляр) + `patient_package_balance` (остатки по позициям) + `package_usage` (списания, append-only).
- Инвариант: НЕ «N одинаковых занятий»; произвольный состав.
- Чек: модель «2×90 + 4×60» создаётся; баланс производный от `package_usage`.

### Шаг 6.2 — Ручное создание (todo s6-manual) — ТЗ §11.2
- Специалист/админ: выбор пациента, состав (услуги+количества), цена, срок, правила списания; генерация ссылки на оплату (этап 5); отправка предложения; показ в кабинете пациента.
- Чек: индивидуальный абонемент создаётся и появляется у пациента со ссылкой на оплату.

### Шаг 6.3 — Готовые продукты-абонементы (todo s6-ready) — ТЗ §11.3,13.5
- Готовый абонемент с предзаданным составом (пациент не компонует); активация после оплаты (модель продукта — заготовка под этап 7).
- Чек: покупка готового абонемента активирует его.

### Шаг 6.4 — Списание (todo s6-usage) — ТЗ §11.4,17
- Списание: авто по подтверждению визита ИЛИ вручную (настраиваемо). При отмене — по правилам этапа 4 (не списывать/штраф/ручное/перенос занятия). Статус «списана по абонементу» увязан с балансом (C2,C6).
- Чек: списание/возврат занятия при отмене проходят тестами; баланс детерминирован.

### Шаг 6.5 — Абонемент при записи (todo s6-booking) — ТЗ §11.5
- При записи пациент видит доступные по абонементу услуги; система знает что входит/осталось/использовано/забронировано/списано/отменено/перенесено.
- Чек: запись с использованием абонемента списывает корректную позицию.

### Шаг 6.6 — UI (todo s6-ui) — [`UI_SURFACES_CHECKLIST.md`](../../docs/OWN_BOOKING_ENGINE_INITIATIVE/UI_SURFACES_CHECKLIST.md) §A11,§B-package,§C-package
- Admin: конструктор готовых абонементов. Врач/админ: создание/назначение индивидуального, состав/остаток/списания/истории/комментарии в карточке пациента.
- Пациент (`app/app/patient/*`, shared-примитивы): активные абонементы, состав, остаток/использовано, срок, история посещений по абонементу, будущие записи, отменённые/перенесённые, статус оплаты.
- Чек: e2e создания/использования абонемента.

### Шаг 6.7 — Верификация (todo s6-verify)
- Тесты баланса/списания/отмены; `typecheck`/`lint`; обновить `api.md`, `LOG.md`, `ROADMAP.md`.

## Definition of Done (этап 6)
- [ ] Составной абонемент (ручной и готовый) создаётся, оплачивается (этап 5), активируется (§11.1–11.3).
- [ ] Абонементные сущности tenant-aware (`organization_id`) и не пересекаются между клиниками (C1).
- [ ] Списание авто/вручную; отмена/перенос корректно влияют на баланс (§11.4,C6).
- [ ] При записи видны доступные по абонементу услуги и остатки (§11.5).
- [ ] Остатки/история видны пациенту и специалисту/админу (§11.6–11.7).
- [ ] UI §A11/§B-package/§C-package; тесты/typecheck/lint зелёные; docs/статусы обновлены.
