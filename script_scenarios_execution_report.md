# Script Scenarios Execution Report

Date: 2026-03-07  
Scope: `src/content/telegram/scripts.json`, `src/content/rubitime/scripts.json`

## Grouped problems

1. **Нужна realtime-проверка уже внесенных исправлений**
  - В код уже внесены изменения по generic template rendering, menu rendering, execution-state и generic facts.
  - Но до подтверждения на реальном runtime эти части нельзя считать окончательно закрытыми.

2. **Нужно обновить фактическую классификацию сценариев после realtime-проверки**
  - Часть сценариев внизу отчета все еще отмечена как частично работающая по старому состоянию.
  - После реальной проверки их нужно перевести в блок полностью выполняемых, если поведение подтвердится.

3. **Нужно подтвердить стабильность generic facts contract на реальных интеграционных данных**
  - Требуется убедиться, что нужные данные действительно стабильно приходят через `facts` и не теряются в реальном потоке.

## Исправлено в коде — проверить в realtime

Ниже список частей, которые уже исправлены в коде, но их нужно отдельно проверить на реальном runtime-потоке интеграций.

1. **Рендер шаблонных переменных в runtime**
  - Подключен реальный `TemplatePort` в production pipeline.
  - Generic template rendering перенесен в executor/runtime path.
  - Это должно исправить:
    - шаблоны сообщений с `{{...}}`,
    - `params.vars`,
    - template-based текст для `message.send`, `message.edit`, `admin.forward`.

2. **Отправка Telegram-меню и кнопок**
  - Убран временный workaround из orchestrator.
  - Полный generic rendering меню теперь должен происходить в executor.
  - Это должно исправить:
    - `textTemplateKey`,
    - `prefixTemplateKey`,
    - reply keyboard,
    - inline keyboard,
    - notification menu after toggle.

3. **Явная загрузка состояния notification-экрана**
  - В сценарий первого открытия notification-экрана добавлен явный шаг `notifications.get`.
  - Экран больше не должен зависеть от скрытого состояния.

4. **Передача runtime execution-state между шагами**
  - `result.values` теперь переносится между шагами одного сценария.
  - Это должно исправить перерисовку notification-экрана после `notifications.toggle.*`.

5. **Уникальные menu-данные переведены в generic facts**
  - Данные для меню теперь передаются как универсальный контейнер `facts`.
  - В сценариях заменены прямые ссылки на:
    - `context.adminChatId`
    - `context.bookingWidgetUrl`
    - `actor.displayName`
  - Вместо этого используются generic facts paths.

## Classification rule

- **Полностью выполняется** — сценарий не только матчится и парсится, но и проходит текущий runtime-путь без выявленных потерь смысла.
- **Частично / не полностью** — сценарий выбирается и запускается, но в текущем runtime есть выявленные проблемы: пустые кнопки, неотрендеренные `{{...}}`, либо ссылки на поля контекста, которые не заполняются.

## Telegram

### Полностью выполняется

- `telegram.ask`  
  Пришел текст / действие `question.ask` -> перевести пользователя в состояние `waiting_for_question` и отправить сообщение с просьбой описать вопрос.

- `telegram.bookings.show`  
  Пришел callback / действие `bookings.show` при уже привязанном телефоне -> отредактировать сообщение и показать экран «Мои записи» + кнопку назад.

### Частично / не полностью

- `telegram.start`  
  Пришел текст `/start` -> сбросить состояние в `idle` и показать главное меню.  
  **Проблема:** само меню открывается, но подписи кнопок берутся через `textTemplateKey` и в реальном pipeline становятся пустыми.

- `telegram.ask.question`  
  Пришел обычный текст, пока пользователь находится в состоянии `waiting_for_question` -> переслать вопрос админу, вернуть пользователя в `idle`, отправить подтверждение «вопрос принят».  
  **Проблема:** шаблон `admin.forward` не дорендеривается по `vars`, а `context.adminChatId` не найден как гарантированно заполняемое поле.

- `telegram.booking.open`  
  Пришло действие `booking.open` при `linkedPhone = true` -> сбросить состояние в `idle` и показать inline-кнопку открытия записи.  
  **Проблема:** текст кнопки берется через `textTemplateKey` и становится пустым; `context.bookingWidgetUrl` не найден как гарантированно заполняемое поле.

- `telegram.contact.link.request.booking`  
  Пришло действие `booking.open` при `linkedPhone = false` -> перевести пользователя в состояние ожидания контакта и показать клавиатуру с запросом номера телефона.  
  **Проблема:** подпись кнопки запроса контакта берется через `textTemplateKey` и становится пустой.

- `telegram.contact.link.request.bookings`  
  Пришел callback `bookings.show` при `linkedPhone = false` -> перевести пользователя в состояние ожидания контакта, показать клавиатуру запроса номера, ответить на callback.  
  **Проблема:** подпись кнопки запроса контакта становится пустой.

- `telegram.contact.link.confirm`  
  В состоянии `await_contact:subscription` пришел контакт / телефон -> привязать телефон, вернуть состояние `idle`, показать главное меню.  
  **Проблема:** финальное меню строится с `textTemplateKey`, поэтому подписи кнопок становятся пустыми.

- `telegram.contact.link.remind`  
  В состоянии `await_contact:subscription` пришло сообщение без телефона -> повторно показать клавиатуру запроса контакта.  
  **Проблема:** подпись кнопки запроса контакта становится пустой.

- `telegram.more.menu`  
  Пришло действие `menu.more` -> показать inline-меню с переходом к уведомлениям и моим записям.  
  **Проблема:** подписи inline-кнопок берутся через `textTemplateKey` и становятся пустыми.

- `telegram.notifications.show`  
  Пришел callback `notifications.show` -> отредактировать сообщение и показать экран настроек уведомлений + ответить на callback.  
  **Проблема:** часть кнопок рисуется через `textTemplateKey` / `prefixTemplateKey`; в реальном pipeline их текст неполный или пустой.

- `telegram.notifications.toggle.spb`  
  Пришел callback `notifications.toggle.spb` -> переключить настройку уведомлений СПб, затем перерисовать экран уведомлений и ответить на callback.  
  **Проблема:** экран после перерисовки использует template-based подписи кнопок, которые сейчас рендерятся не полностью.

- `telegram.notifications.toggle.msk`  
  Пришел callback `notifications.toggle.msk` -> переключить настройку уведомлений Москва, затем перерисовать экран уведомлений и ответить на callback.  
  **Проблема:** экран после перерисовки использует template-based подписи кнопок, которые сейчас рендерятся не полностью.

- `telegram.notifications.toggle.online`  
  Пришел callback `notifications.toggle.online` -> переключить настройку онлайн-уведомлений, затем перерисовать экран уведомлений и ответить на callback.  
  **Проблема:** экран после перерисовки использует template-based подписи кнопок, которые сейчас рендерятся не полностью.

- `telegram.notifications.toggle.all`  
  Пришел callback `notifications.toggle.all` -> переключить все категории уведомлений, затем перерисовать экран уведомлений и ответить на callback.  
  **Проблема:** экран после перерисовки использует template-based подписи кнопок, которые сейчас рендерятся не полностью.

- `telegram.menu.back`  
  Пришел callback `menu.back` -> отредактировать сообщение и вернуть пользователя в предыдущее меню.  
  **Проблема:** подписи inline-кнопок берутся через `textTemplateKey` и становятся пустыми.

- `telegram.menu.default`  
  В состоянии `idle` пришел обычный текст, который не является `/start`, `booking.open`, `question.ask`, `menu.more` -> снова показать главное меню.  
  **Проблема:** подписи кнопок главного меню берутся через `textTemplateKey` и становятся пустыми.
